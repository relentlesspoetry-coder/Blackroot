# V0.20.45 — The Camp Folk Actually Walk

## What I got wrong in V0.20.44

V0.20.44 made camp NPCs move and set every field a walk animation needs — `moveBlend`, `walkCycle`,
`facingX`/`facingY`. I verified **those fields** (day-walker frames 0/180 → 120/120) and declared the
walk animation fixed.

It wasn't. The report came back: *"they really do not walk, they kinda teleport around slowly, there's
no walking animation at all."* That was correct. **I verified a proxy** — the node state — and the
render discarded that state before anything could use it. Exactly the trap this project keeps catching.

## Root cause: two hardcoded kills in the same chain

**(1) The render actor was built with the answer hardcoded** (`systems/npc-system.js`):

```js
facingX: 0,
facingY: 1,
isMoving: false,                          // always
action: npcSwimming ? 'swim' : 'idle',    // always
```

So the model *and* the sprite cache were always told "standing still, facing south," regardless of what
the NPC was actually doing.

**(2) `drawNpcClassModelDirect` clobbered it again**, one line before the draw:

```js
actor.isMoving = false;   // unconditional
```

**Why that froze the sprite:** the runtime sprite cache keys its animation bucket off `action`/`isMoving`
(`actionFrameMs` + `animationBucket`). `walk` gets 6 buckets at 100ms; `idle` gets 2 at 250ms. Told
"idle" forever, every camp NPC baked to essentially **one frozen frame** — and then slid around wearing
it. That is precisely the "teleporting" look.

## The fix

Derive all four from the live node instead:

- `moveBlend > 0.08` → `action: 'walk'` + `isMoving: true`
- `facingX`/`facingY` passed through instead of hardcoded
- `facingName` derived from the movement vector via an 8-way mapping — necessary because
  `faceNodeToward` only writes the vector, so without it an NPC would walk north while drawn facing south
- `drawNpcClassModelDirect` now derives `isMoving` rather than forcing `false`

A genuinely idle NPC still reports `idle`/`false`, so nothing else changes.

## Verified at the bridge — not on the node

The lesson from the miss: verify the thing that was broken. I ticked until NPCs were mid-stride, then
forced a real render frame and captured what the **model actually received**:

| | before (hardcoded) | after |
|---|---|---|
| `action` | always `'idle'` | **21 of 22 draws `'walk'`** |
| `isMoving` | always `false` | **`true`** for walkers |
| `facingName` | always `'south'` | **northwest / north / south / southeast** |
| idle NPC (`moveBlend` 0.01) | — | still `'idle'` / `false` ✅ |

Sprite-cache frame math confirms the payoff: `walk` yields **5 distinct frames per 600ms** vs `idle`'s
**2**.

## Two diagnosis traps worth recording

- **Wrapping several hot render-path functions at once hung the renderer** — producing zero draw counts
  that looked like "NPCs never render."
- **An idle background tab throttles `requestAnimationFrame`**, so the live loop appeared not to run
  (0 NPCs moved in 26s) and I nearly blamed my own working update code. Manual ticking isolated it:
  **27 NPCs moved, no error** — the update was fine all along.

Both produced misleading zeros mid-diagnosis. Neither was a real defect.
