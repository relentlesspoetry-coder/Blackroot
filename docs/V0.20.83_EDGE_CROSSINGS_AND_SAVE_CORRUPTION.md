# V0.20.83 — Edge Crossings, Correct Zone Naming, and a Save-Corrupting Bug

Reported: *"the warp needs to be at the edge of the map in both zones… Also the zone map should not say
dark woods it should say the correct zone map."*

Both were real. Chasing the second one uncovered a third, much worse problem.

## 1. Edge crossings

| | was | now | tiles from edge |
|---|---|---|---|
| Dark Woods | x = 352 | **x = 357** of 360 | **2** |
| Ashen Valley | x = 34 | **x = 3** of 450 | **3** |

Arrivals still land ~11–12 tiles from the opposite crossing, preserving the V0.20.82 offset that stops
walk-through travel bouncing the player between zones.

## 2. The zone map named the wrong zone

`render/minimap-renderer.js` → `mapZoneLabel()` handled dungeon and cave, then:

```js
return 'Dark Woods';
```

A hardcoded string for *any* overworld. It now reads the active zone's authored name.

## 3. Fog — the same bug, one level deeper

`fogZoneKey()` had the identical shape:

```js
return 'dark_woods';
```

So Ashen Valley wrote into **Dark Woods' exploration memory**, and did it at Dark Woods' **360** size
against a **450** map. That is exactly the reported symptom: the new zone opened as *"Dark Woods —
Explored 0.2%"* with a single small revealed circle. It was reading another zone's buffer, sized wrong.

Exploration is now keyed per overworld zone.

**Worth noting these two were found by following the reported symptom rather than stopping at it.** The
label was the visible complaint; the fog was the same mistake in the same file, and would have looked
like a separate bug later.

## 4. The serious one — and it is a direct consequence of the V0.20.80 design

`systems/world-serializer.js`:

```js
function getOverworldMap(game) {
  return game.overworldMap || ...;      // "the overworld" == Dark Woods, pre-V0.20.80
}
```

The world save schema stores **exactly one** overworld, under the key `dark_woods`. But since V0.20.80,
`game.overworldMap` is an accessor onto **whichever zone is active**.

**So an autosave that fired while the player stood in Ashen Valley serialised its 450×450 terrain into
the save's `dark_woods` slot — permanently replacing the starting zone.**

Confirmed directly:

```
cleanBootDarkWoods: 450          // a "Dark Woods" 450 tiles wide
sameArrayObject:    true         // both zone slots holding the SAME array
```

This is precisely the risk the accessor approach carries, and it deserves to be recorded plainly: the
refactor was chosen because it avoided touching ~70 call sites, and the cost is that **every one of those
sites silently changed meaning**. Reads that meant "Dark Woods" now mean "the active zone". Most are
correct under the new meaning. The serializer was not, because it is the one place that writes a
*named* slot to disk.

Both readers now take the `dark_woods` slot **by name**.

## 5. The damage lives in the save, not in memory

Which means it survives every reboot — the corruption reloads itself. `apply()` now rejects a saved
overworld whose dimensions are not Dark Woods':

```
Saved overworld is 450x450, expected 360x360 for Dark Woods.
```

and falls back to procedural generation. That loses editor decoration, and that is the right trade: **a
wrong Dark Woods is unrecoverable; a regenerated one is not.**

## Verified

| check | result |
|---|---|
| guard against the actually-corrupted 12MB local save | **rejected**, with a specific reason |
| Dark Woods after rejection | regenerated **360×360**, crossing present at x=357 |
| serialise **while standing in Ashen Valley** | writes Dark Woods' **360×360** ✅ |
| saved grid identity | matches the Dark Woods slot, not merely the size |
| zone map label | **"Ashen Valley"** there, **"Dark Woods"** at home |
| fog keys | `ashen_valley` / `dark_woods`, separate |
| Ashen Valley fog buffer size | **450** |
| crossings from map edge | **2** and **3** |

Suite ok, 0 console errors.

## Lesson recorded

The accessor refactor was the right call for avoiding a 70-site cascade, and I still think so — but the
report for V0.20.80 should have listed *"every existing reader of these three names now means something
different"* as a risk to audit, not just as an implementation detail. The serializer was findable by
asking "which readers mean **Dark Woods specifically** rather than **the current zone**?" That audit was
never done, and a save-destroying bug shipped for three versions.
