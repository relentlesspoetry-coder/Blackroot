# V0.20.63 — Prop Distance LOD, and a WebGL Backend Scoping Report

Two tasks: ship prop LOD (1), and scope the WebGL backend without touching it (3).

---

## Part 1 — Prop distance LOD (shipped)

### The finding

Per-prop cost at camp:

| prop | count | ops each | ops/frame |
|---|---|---|---|
| tree | 15 | 191 | 2,870 |
| **webbedBush** | 8 | **274** | **2,192** |
| grassTuft | 15 | 55 | 825 |
| flower | 9 | 62 | 558 |

**A webbed bush cost more than a whole tree.** ~200 of its 274 ops is a single call —
`drawAnchoredOrbWeb` with 9 spokes, 3 rings and dew — drawn identically at any distance.

### The change

New `DR.CONFIG.PERFORMANCE.propDetailRadius` (11 tiles). `drawObject` sets `this._propFarDetail` once
per prop (set on the renderer rather than threaded through ~180 `drawX(s, obj)` signatures). Beyond the
radius:

- **webbedBush** skips the orb web entirely
- **tree** skips branch strokes, 2 of 4 crown blobs, and the sub-20%-alpha moss/leaf accents

**Silhouette and palette are preserved** — the two largest crown masses still draw, so the tree line
looks unchanged.

**This is not caching.** No bitmaps, no textures, nothing retained between frames. That is precisely what
separates it from V0.20.49 / V0.20.53 / V0.20.59, all of which regressed by trading draw calls for
texture churn.

### Measured (drift-checked)

| arm | prop ops | total ops |
|---|---|---|
| LOD off | 9,214 | 29,495 |
| **LOD on** | **6,554** | **27,090** |
| LOD off again (drift) | 9,214 | 29,775 |

**−29% prop cost, −8% of the frame.** The drift arm reproduces arm 1 exactly.

### Fifth measurement trap

**`DR.CONFIG` and `.PERFORMANCE` are `Object.frozen`.** A/B-ing by assigning to them **fails silently** —
two arms measured byte-identical because the write never landed. Confirmed: `Object.isFrozen(PERF) === true`,
`writeTook: false`.

Also: **`propDetailRadius: 0` does not disable the LOD** — it makes *everything* "far". Raise the radius
to disable.

The technique that worked: `Object.defineProperty` on the game instance to intercept `_propFarDetail`.

Running list of traps in this codebase, all of which produced confident wrong answers:

1. `performanceSettings()` is memoized — config mutations don't reach readers.
2. `runtimeSpriteCache.beginFrame()` only runs from `loop(now)` — synthetic loops disable the cache.
3. Instance-level canvas patches shadow the prototype — reload between instrumentation runs.
4. Synthetic render loops drift (127 → 148ms on identical settings) — always repeat arm 1 last.
5. **`DR.CONFIG` is frozen — writes are silently discarded.**

## Part 2 — WebGL backend scope (investigation only)

`systems/render-backend-system.js` — **3,401 lines. A real prototype, not a stub.**

### What already works

| signal | value |
|---|---|
| WebGL2 available | ✅ |
| `webglPrototypeReady` | **true** |
| GPU | ANGLE (AMD Radeon RX 7900 XT), max texture 16384 |
| **`backendBatchDrawablePct`** | **99.3%** |
| `backendBatchOverflow` | **0** |
| batched renderables | 200 sprite / 120 object / 153 world / 48 terrain |
| overlay cache | 684 hits / 28 misses |

**99.3% batchable with zero overflow** is the headline. Batching is exactly what a draw-call-bound
renderer needs, and the design already fits this scene.

### What blocks it — its own audits, not missing code

| gate | state |
|---|---|
| `rendererMode` | `canvas2d`, `renderBackendSafeMode: true` |
| `hybridPromotionBlockedByAudit` | **true** |
| `hybridCameraDepthAuditState` | **blocked** — round-trip **1.313 tiles** |
| `webglVisibleSpriteLastError` | `final-qa-block:final QA projection/depth audit` |
| `webglVisibleTerrainLastError` | `final-qa-block:final QA cave/fringe terrain safety` |
| all visible layers | `enabled: false`, `active: false` |
| sprite/terrain GL programs | not compiled — prototype is off |
| `hybridDepthOrderViolations` | 0 (nothing promoted yet) |

In plain terms: **the WebGL path can draw, but its camera projection does not yet agree with the canvas2d
one** — a 1.313-tile round-trip error — and the depth audit correctly refuses promotion, because a
projection mismatch means wrong occlusion (actors drawing in front of things they stand behind). That is
the same class of correctness constraint that ruled out baking props into terrain chunks.

The system's own recommendation, verbatim: *"Enable Hybrid WebGL Prototype and scene preview overlay
before promoting WebGL to visible rendering."*

### Verdict

**60 FPS is reachable**, and the remaining work is well-defined rather than speculative:

1. Make the WebGL projection round-trip match canvas2d within tolerance (currently 1.313 tiles).
2. Clear the final QA projection/depth audit.
3. Resolve the cave/fringe terrain safety case.
4. Promote layers one at a time behind the **scene-preview overlay that already exists** — it renders
   WebGL offscreen and scores alignment against canvas2d, so each step is verifiable before it becomes
   visible.

That last point matters given this project's history: the infrastructure supports incremental,
measurable promotion rather than a flag-flip that either works or corrupts the scene.

### Standing frame budget after this version

| phase | ops |
|---|---|
| `drawObject` | ~6,600 |
| `drawEntity` | ~5,400 |
| `drawTile` | ~3,500 |
| `npcRuntime` | ~320 |
| **total at camp** | **~27,000** vs a ~5–10k budget for 60 FPS |

Canvas2d tuning has roughly 3,000–5,000 more ops available (thinning decorative props, tile detail).
**The remaining gap to 60 is WebGL's to close.**
