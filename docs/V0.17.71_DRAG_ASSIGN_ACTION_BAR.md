# V0.17.71 — Drag-to-Assign Paged Action Bar (Talent spec BUG 1)

**Type:** Pre-existing bug fix / feature, its own pass (Talent System v1 spec, PART 8 BUG 1 / PART 9)
**Previous version:** V0.17.70
**New version:** V0.17.71

The spec called this "arguably a bigger win than talents. Half your combat content is currently
invisible."

---

## The bug

The compiled class spellbook has always held **20** spells per class (`MAX_HOTBAR_SLOTS = 20`), but the
action bar exposed only **9**: the DOM had 9 spell slots and only keys `2`–`0` were bound. Every
ability from roughly level 10 up (book indices 9–19) was authored, iconed, tooltipped, and balanced —
and completely **uncastable** (no slot, no key).

## The fix — decouple bar *slot* from spell *index*

Chosen approach (user pick): **a second action bar + drag-to-assign with a spellbook window.**

### Assignment model (`systems/spell-system.js`)
- `player.hotbar` — an 18-slot array (two bars × 9) of **stable spell ids** (`spell.id`, fallback
  `spell.name`), persisted per character. Default = identity fill of the first 18 class spells.
- `ensureHotbarAssignments()` / `normalizeHotbar()` — create defaults; on load drop ids that no longer
  resolve to a class spell and pad/truncate to 18.
- `hotbarSlotSpellIndex(slot)` / `hotbarSlotSpell(slot)` — resolve a slot's stored id to the **current**
  book index, so talent spell-patches (which rebuild the compiled book) never desync bindings.
- `useHotbarSlot(slot)` → resolves → `useClassSpell(bookIndex)`.
- `assignHotbarSlot` / `swapHotbarSlots` / `clearHotbarSlot`.

### Bars (`Blackroot ….html`, `renderHotbarCombatState`, `updateSpellHotbar`)
- Bar 1: auto-attack (key `1`) + slots 0–8 (keys `2`–`0`).
- Bar 2: slots 9–17 (keys `Shift+2`–`Shift+0`) + a **Spellbook** button.
- Each spell slot carries `data-hotbar-slot` and is `draggable`. `updateSpellHotbar` and
  `renderHotbarCombatState` now display each slot's **assigned** spell; cooldown is keyed by the
  resolved book index (so two slots holding the same spell share its cooldown).

### Spellbook window
All 20 class abilities as cards (icon / name / level / cost / cd, "on bar" marker). **Drag** a card
onto any slot to bind it; **click** a card to cast it directly (so nothing is ever uncastable, even
unassigned); **right-click** a bar slot to clear it; **drag** slot→slot to swap. Toggle via the bar
button or the new **`v`** keybind.

### Input (`game.js`)
- `1` = auto-attack (now guarded from `Shift`).
- `2`–`0` → `useHotbarSlot(0..8)`; `Shift+2`–`Shift+0` → `useHotbarSlot(9..17)`.
- `v` → toggle Spellbook. `spellbook` added to `defaultKeyBindings` + the rebind labels.

### Persistence (`systems/save-system.js`)
Serializes `player.hotbar`; on load restores it, runs `normalizeHotbar()`, and refreshes the bar.

## Result

All 20 abilities are reachable: 18 keybound across two bars, the remaining 2 (and any unassigned) via
click-to-cast in the Spellbook — all freely drag-reassignable and persisted.

## Verification (in-browser, port 8899)

Level-20 Paladin: 18 DOM slots across two non-overlapping bars + Spellbook button; default fill
resolves (slot 17 = Sanctuary Vow, previously uncastable). `useHotbarSlot(17)` casts and sets its
cooldown; a DOM click on bar-2 slot 12 cast Hammer of Reckoning; right-click cleared it; casting an
empty slot returns false (no crash). Spellbook lists all 20 including the 2 not on bars; drag-assigning
Avatar of the Oath onto a slot rebinds it; swap works. `serializeCharacterState` → wipe →
`applyCharacterState` round-trips assignments; `normalizeHotbar` drops a bogus id and pads a
wrong-length array to 18. No console errors.

## Not in this pass

The other 13 talent classes (273 nodes) and the caster balance tuning motivated by V0.17.70.
Auto-attack keeps key `1`, so each bar exposes 9 keybound slots (18 total); the 2 overflow abilities
live in the click-to-cast Spellbook by design.
