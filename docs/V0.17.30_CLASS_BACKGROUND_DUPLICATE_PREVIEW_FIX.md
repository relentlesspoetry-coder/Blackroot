## Root Cause
V0.17.29 successfully made class background art visible, but multiple background ownership paths were active at the same time:

1. `drawClassBackgroundPanelBackdrop()` drew the class background directly into the paper-doll model canvas.
2. `.paperDollPreview` / `.classBackgroundSurface` CSS and inline style still painted the same class background art on the DOM surface behind the canvas.
3. `drawClassBackgroundPanelBackdrop()` also drew a second contained/hero copy of the class background near the top of the tall preview panel.

That produced the visible smaller duplicate class-art image at the top of the Character Model preview.

## Fix Details
- Removed the contained/hero duplicate draw from `drawClassBackgroundPanelBackdrop()`.
- Kept one cover-fit canvas draw as the only class-art render pass behind the character model.
- Changed `applyClassHudVisualStyle()` so `.paperDollPreview` receives only neutral dark readability gradients, not a second class-art image layer.
- Updated the final Character Model preview CSS fallback to remove duplicate `var(--pd-class-bg-image)` layers.
- Reduced the Character Model preview call site to pass only `coverAlpha`, since hero-copy options are no longer used.

## Player HUD Behavior
Player HUD class-background styling was left intact. This phase only removes the Character Model preview duplicate layer.

## Fallback Behavior
If class background art is not loaded yet, the preview uses a clean dark/vignette fallback until the canvas draw path can render the resolved asset.
