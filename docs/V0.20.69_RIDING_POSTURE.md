Reported: *"the character model is just standing on top of the mount."*

Two root causes, plus a third found only by checking all fourteen classes instead of the one in the
screenshot.

## Cause 1 â€” the riding pose never reached the character model

V0.20.65 put the riding pose in `getStandingPuppetPose` / `drawStandingLegs` (`entity-renderer.js`).

Instrumenting both proved the player calls **neither**:

```
onFoot:  { riding: 0, standing: 0 }
mounted: { riding: 0, standing: 0 }
```

Zero calls in *both* states. The Paladin renders through the bespoke class-identity model, which derives
its posture from `HumanoidAnimationSystem.buildPose` â€” a completely different pose builder. The V0.20.65
work was correct code in a function the player never executes.

### The fix, in the right place

`ride` is now a **first-class action**, resolved in `resolveAction` beside `swim` / `sit` / `jump`, with
`rideGait` and `rideMoving` on the pose. `humanoid-base-renderer` gains **`buildRidingRig`** alongside
`buildSittingEmoteRig` and `buildSwimmingRig`.

The rig differs from sitting in the way that matters: a seated character folds its legs **forward**; a
rider's thighs go **out across the animal's back** and the shins hang **straight down** its flanks. That
silhouette is the whole reason a rider reads as riding.

Because it is a rig, **all 14 classes inherit it from one place**. The bespoke Paladin and Fighter models
now let `ride` fall through to the shared renderer exactly as they already do for `dance` and `sit`.

## Cause 2 â€” one seat height for animals of wildly different heights

`drawRiddenMount` lifted the rider a flat `26 Ã— scale` (~29px) for **every** mount. Measured back lines
from the beasts' own artwork:

| mount | back above anchor | old lift | rider sat |
|---|---|---|---|
| Dusk Bat | 38px | 23 | 15px too low |
| Crystal Crawler | 47px | 29 | 18px too low |
| Gloom Wolf / Briar Boar | 63px | 29 | **34px too low** |
| Hollow Stag | 95px | 29 | **67px too low** |

A wolf rider sat **34px below the animal's back**. That is precisely why the character read as standing
*in front of* the beast rather than on it.

**`mountSaddleHeight()`** renders the beast once to an offscreen canvas and samples the topmost pixels in
a band across the **saddle area** â€” not the whole silhouette, because the highest point of a stag is its
antlers, which is not where you sit. It sinks 5px into the back so the rider is seated rather than
perched, and caches per `beastKey + scale`.

Verified **0 re-measurements across 30 frames**. Measuring beats authoring a number per mount: it cannot
go stale when art changes.

## Cause 3 â€” two classes still stood

Found by testing all 14 rather than the Paladin in the screenshot: **Bard and Druid** kept standing.
Their models pass an **explicit** action via `renderActionForHumanoid`, which overrode the derived
`ride`. Added the mounted check there too.

Had I stopped at the reported class, this would have come back as a separate bug report.

## Method note

The V0.20.65 fix looked right, was tested, and reported passing numbers â€” because I verified the *pose
object* rather than whether the model consumed it. The instrumentation that actually settled it was
counting calls to the drawing functions themselves, which returned **zero in both states** and made the
mistake obvious immediately.

**Verifying that a value is correct is not the same as verifying that anything reads it.**
