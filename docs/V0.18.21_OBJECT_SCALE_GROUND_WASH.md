# V0.18.21 — World-object scale + Dark Woods ground-wash fix

Two screenshot-driven requests from the user:

1. "remove all these weird overlapping circles on the ground"
2. "the tents are way too small the character models are way bigger, tents should be bigger than the character models, so should the well. any kind of actual object should be sized correctly."

---

## 1. Objects too small (tents, wells, buildings)

### Root cause
The generic prop draws in `render/object-renderer.js` (`drawTent`, `drawWell`,
`drawTownHouse`, `drawShop`, …) are authored at a small **fixed** footprint:

- `drawTent`: apex at `s.y - 55`, base at `s.y + 14` → ~70px tall
- `drawWell`: roof at `s.y - 42`, base at `s.y + 10` → ~52px tall

versus a ~90px-tall character model. So these props rendered *smaller than the
player*. The existing `startingCampObjectScale(obj)` returned sensible per-type
multipliers (tent 2.35/3.1, etc.) but was **only** applied to the two special
early-return paths (`largecamptent`/`camptentlarge` → `drawLargeCampTent`, and
`vendorstall` → `drawVendorStall`). Every other object type fell through to the
`switch` in `drawObject` and drew at its raw base size, unscaled.

### Fix (root cause, no cascade patch)
Added `overworldObjectScale(obj)` — a per-type "real world" size multiplier:

| Category | Types (matched by type/name substring) | Multiplier |
|---|---|---|
| Large built structures | tent, leanto, hut, house, cabin, building, shop, vendor, stall, forge, smith, well, shrine, gate, cave, dungeon, tower, barn, windmill | **2.2** |
| Mid furnishings / stations | loom, cookingspit, weaponrack, studytable, ritualcircle, sparringdummy, herbalisttable, supplystack, stashchest, altar, anvil, cart, wagon, table, rack, bench, statue, obelisk, totem, banner, fence, rootarch, signpost | **1.5** |
| Small ground clutter | crate, barrel, bedroll, logseat, stool, pot, pouch, coin, candle, lantern, mushroom, rock, stone, bush, brush, bone, skull, trap, cloak, sack, basket, bucket, debris | **1.12** |
| Natural foliage (already tuned) | tree, log, stump, root, flower, grass, reed, vine, fern | **1** |
| Default | everything else | 1.25 |

`drawObject` now applies this as a transform **about the object's ground-contact
point** so props grow *upward from the ground* rather than drifting off their tile:

```js
const objScale = this.overworldObjectScale?.(obj) ?? 1;
if (objScale === 1) return this.drawObjectShape(obj, s);
ctx.save();
ctx.translate(s.x, s.y);
ctx.scale(objScale, objScale);
ctx.translate(-s.x, -s.y);
this.drawObjectShape(obj, s);
ctx.restore();
```

Because every `case` in the dispatch `switch` ends in `return this.drawX(...)`,
a naive `ctx.save()` before the switch would never reach a trailing
`ctx.restore()` and the transform would **leak into the next object** (cumulative
scaling). To keep `save`/`restore` balanced cleanly (no `try/finally` wrapping a
130-case switch), the contact-shadow lookup + dispatch switch were extracted
**verbatim** into a new `drawObjectShape(obj, s)` helper. `drawObject` scales,
calls it, then restores — the switch's per-case `return`s simply return from the
helper, and control always comes back to the `ctx.restore()`.

Resulting heights: tent 70 → ~154px, well 52 → ~114px — both clearly larger than
the ~90px character, as requested. Small clutter (crates/barrels/rocks) stays
near its authored size; foliage is untouched.

Note: the two early-return paths (`drawLargeCampTent`, `drawVendorStall`) already
receive `startingCampObjectScale` and are unchanged — no double-scaling. Placed
objects don't author `obj.scale` anywhere in `data/`/`systems/`/`world/`, so the
handful of draws that read `obj.scale` internally (e.g. `drawStashChest`) compose
cleanly with the outer transform.

## 2. Overlapping ground circles

### Root cause
`render/terrain-renderer.js` (~L574) painted the Dark Woods depth-darkening wash
as a per-tile **filled circle** of radius `scale * 0.62`. Since 0.62 tile > 0.5
tile, each tile's circle overlapped its neighbours, tiling out into a
ring / fish-scale pattern across the forest floor.

### Fix
Replaced the per-tile ellipse with a per-tile `fillRect` covering the tile
(same `depthFactor` gating, `0.22` alpha, `#050807` colour), so depth areas
darken **uniformly** with no circles:

```js
const depthFactor = this.darkWoodsDepthFactor?.(x, y) || 0;
if (depthFactor > 0.02) {
  c.globalAlpha = depthFactor * 0.22;
  c.fillStyle = '#050807';
  c.fillRect((x - worldMinX) * scale - 0.5, (y - worldMinY) * scale - 0.5, scale + 1, scale + 1);
}
```

The `-0.5 / +1` matches the base tile fill so tiles abut with no seams.

## Verification
- `node --check` on both edited files — OK.
- Game boots with **0 console errors**.
- `overworldObjectScale` returns the intended multipliers for all sampled types
  (tent/well/townHouse/shop/campLeanTo 2.2, stashChest/rootArch 1.5,
  crate/barrel/rock 1.12, tree 1, fire 1.25).
- Exercised the new render path live: `drawObject` for all 14 sampled types →
  **0 throws**, and the canvas transform returns to identity (`1,0,0,1,0,0`)
  after each — proving `ctx.save`/`ctx.restore` pair correctly through the
  switch's `return`s with **no transform leak** between props.
- Visual sign-off is the user's (this WebGL/canvas page cannot be screenshotted
  by the preview tooling).
