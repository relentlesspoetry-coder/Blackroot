# V0.20.6 — Thirty-Three Faces Nobody Had Seen (Roadmap Item 5)

Item 5's core requirement: **"display a face portrait of the NPC currently speaking."**

## The find

Every one of the game's **33 NPCs** already authors a portrait in `data/npcs.js`:

```js
portrait: { family: 'villager', color: '#d8bc78', glyph: '?' }
```

**33/33. Zero missing.** And:

```
$ grep -rn "\.portrait\b" --include=*.js .
(nothing)
```

**Somebody designed a portrait for every character in this game and not one of them was ever put on
screen.** Same dead-data pattern as `coneDegrees` (V0.19.6), the east/west poses (V0.19.9) and the
`slam` style (V0.20.0) — except this one was an entire authored **feature**, not a field.

The dialogue window already existed — name, role/faction/level, body text, quest/vendor/trainer actions.
The portrait is the piece it never had.

## The build

`drawNpcPortrait(canvas, npc)` renders **from the authored descriptor**:

- **family → silhouette**: hood, helm, brim, toque, cap, bare, wisp — plus a collar (cloth / leather /
  robe / apron / mail)
- **colour → tint** on every layer (frame, shoulders, collar, face, headwear, badge)
- **glyph → corner role badge**
- `PORTRAIT_FALLBACK` covers an unknown family **and** an NPC with no portrait at all — the spec's
  required *"fallback portrait for minor NPCs without custom artwork"*

**Procedural rather than artwork, deliberately.** Every NPC, mob and prop in this game is already drawn
from code. A bust drawn the same way matches the established visual style *by construction*, and
satisfies *"no stretched, cropped, or low-resolution portraits"* at any UI scale for free — with no
assets to ship or lose.

The **dream spirits** are drawn as a wisp with **no bust**. Giving them shoulders would be a lie about
what they are.

## Layout — part of the window, not an overlay

The canvas is a **flex child of the existing `.windowHeader`**, beside the name/role block. The spec
requires the portrait be *"part of the dialogue window layout, not a detached overlay"* — and as
siblings sharing a row, the text and the portrait **cannot** overlap.

Verified: `position` computes to `static`, and it resolves inside `.windowHeader`.

Speaker changes repaint it in `openNpcWindow` — the single entry point for showing an NPC — so *"update
the portrait immediately when the active speaker changes"* holds **without a watcher**.

## Verification — every NPC, by pixels

Drove the **real `openNpcWindow`** for all 33 NPCs and hashed the canvas:

| check | result |
|---|---|
| all 33 draw | **0 blank** |
| distinct portraits | **32 of 33** |
| unknown family | **falls back and draws** |
| no portrait data at all | **falls back and draws** |
| in header, not overlay | **`static`, inside `.windowHeader`** |

**The one identical pair was investigated, not accepted.** *Cavern Spirit Healer* and *Spirit Healer*
have **byte-identical authored portrait data** (`spirit` / `#d8f6ff` / `✦`), and the renderer was proven
to **vary with colour** (recolouring one changes its hash). So the duplicate is **authored**, not a
renderer collapsing data.

### The eye was wrong; the pixel was right

A first pass drew headwear at `base-25` and the hoods and helms **read as hair**. I raised it to
`base-70` against a `base+42` face — then compared two screenshots and concluded **nothing had
changed**. It had. Sampling the pixel showed the hood at `89,129,63` = **exactly `base-70`**, and the
face at `201,241,175` = exactly `base+42`.

That's the fourth time this session a check has lied, and the first time the liar was my own eyes.
