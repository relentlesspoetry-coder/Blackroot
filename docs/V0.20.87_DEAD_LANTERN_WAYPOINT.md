# V0.20.87 — Dead Lantern Camp Waypoint

Requested: *"add a waypoint to dead lantern camp"*.

## Placement

**106, 105** — just off the south-east road out of camp, **7.1 tiles from the spawn point**.

Beside the hub rather than inside it, deliberately. The camp carries **60 props in a 26×26 box**; a
waypoint dropped into that reads as more clutter, and §5.3 wants it to be *"easy to find"* and *"a natural
gathering point"*, which needs standing room. A 5×5 apron is cleared around it for the same reason.

Named **Dead Lantern Waypoint**, matching the spec's own example list in §5.11 (*"Dark Woods — Dead
Lantern Waypoint"*).

## Dark Woods is now the first zone with two waypoints — which broke an assumption I had baked in

Travel was refused whenever the destination's zone matched the player's current zone:

```js
if (def.zoneId === game.activeOverworldZoneId) return 'You are already in that region.';
```

With one waypoint per zone that was indistinguishable from correct. With two, **Dead Lantern and Stone
Hedge could never reach each other** — which is most of the point of putting a waypoint at the camp.

Fixed at both levels:

- The waypoint refusal is now **per waypoint**: you are refused only if you are standing at the one you
  picked.
- `travelToOverworldZone` takes an `allowSameZone` option, so intra-zone arrival runs through the **same
  safety path** as any other travel — walkable landing, transition cooldown, cancelled fishing and
  gathering — rather than getting a second, shortcut movement path that would drift.
- The arrival log is suppressed for same-zone hops; *"You arrive in Dark Woods"* reads wrong when you
  never left it.

## Two placement faults, both silent

### 1. Written into a grid that was about to be replaced

The ensure ran inside `generateMap` **before `storeOverworldZone`**. Because it targets the `dark_woods`
slot *by name* — the correct pattern, learned in V0.20.81 — it wrote into the slot's **previous** grids,
which `storeOverworldZone` then swapped out a few lines later.

It ran. It returned `1`. The object was not in the world.

### 2. Cleared by the camp rebuild

Moving it after the store still failed. `ensureStartingCampRevamp` runs later, from `game.js:610`, and
clears **every object inside the camp ellipse**:

```js
if (compactTile(x, y)) clearAndSet(x, y, ...);   // clearAndSet sets objects[y][x] = null
```

`compactTile(106,105)` = `hypot(6/11.4, 5/9.2)` = **0.756** — well inside. The waypoint was placed and
then wiped, on every boot.

It is now called from `game.js` immediately **after** the camp revamp. The redundant earlier calls were
**removed**, not left in "just in case" — a call that provably cannot survive is dead code that reads as
working code.

## The pattern is worth naming

Three versions running, the same shape of bug:

| version | what happened |
|---|---|
| V0.20.81 | crossing placed before the object-pruning pass → deleted |
| V0.20.81 | crossing placed into `this.map` during a load, before the live map existed → no-op |
| **V0.20.87** | waypoint placed before the grid swap, then again inside a region that gets cleared |

Every one of them **reported success.** `ensureX()` returning 1 means "I wrote to an array", not "the
player will see this."

**"Placed" is not "present."** The only proof is reading the world back at a clean boot — which is what
caught all three, and what the earlier versions' verification skipped by checking the function's return
value instead of the world.
