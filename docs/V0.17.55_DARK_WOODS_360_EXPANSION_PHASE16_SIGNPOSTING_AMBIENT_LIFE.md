# Phase 16 — Dark Woods 360x360 Expansion: Creature Territory Signposting + Ambient Life

**Phase:** 16 of 28 (Dark Woods Zone Expansion + Resource Identity master plan)
**Previous version:** V0.17.54
**New version:** V0.17.55

---

## 1. Goal

Per the master plan: add environmental signs before each mob family (webs/egg sacs before spiders, broken
arrows before bandits, floating motes before wisps, claw marks/bones before wolves, root clusters before
treants, dead trees/crows before danger areas) and lightweight, camera-culled, cosmetic-only ambient life
(crows, fireflies, butterflies, frogs, squirrels, deer, beetles, moths, bats), with no new
collection/combat system. Validation: territory transitions are readable; ambient life appears in correct
regions; no combat targeting bugs; no performance spike.

## 2. Audit First — Two Major Findings

Grepped the whole codebase before writing anything, per this project's established discipline, and found
this phase needed far less new infrastructure than expected:

- **`DR.DEFAULT_WORLD.zones.dark_woods.regions[].identity.signpostProps`** has existed since Phase 3
  (V0.17.42) — `['brokenArrow']` for Bandit's Fall, `['floatingMote']` for Stone Hedge, `['webbedBush',
  'eggSac']` for Silk Web Approach, `['lanternPost']` for Dead Lantern Outskirts, `['clawMarks']` for
  Bramblefen Thicket, `['heavyRoots']` for Gloamroot Depths — almost a literal 1:1 match for this phase's
  requirement list. Confirmed by grep that it was **never read anywhere** in the codebase until this phase;
  Phase 3's own report explicitly called it "metadata only."
- **`systems/atmosphere-system.js`** is a pre-existing, already-installed (`DR.AtmosphereSystem.install`),
  already-called-every-frame (`game.js:3997-3998`) lightweight ambient system covering fog pockets,
  fireflies, and 5 ambient creature kinds (firefly, butterfly, moth, beetle, lizard) — camera-culled by
  player-relative range, capped counts (`MAX_AMBIENT_CREATURES = 26`), fully stateless/cosmetic (no
  collision, no combat, no persistent entities). This is exactly the "ambient life" system the plan asks
  for, already built and already Dark-Woods-scoped (`isOverworld()` checks `currentZone === 'dark_woods'`).

Given both, this phase's job was almost entirely **extend existing systems**, not build new ones.

## 3. A Real Bug Found During Audit

While reading `atmosphere-system.js` to understand its render-bounds logic, found that both
`renderDarkWoodsAtmosphere` and `renderAmbientCreatureLayer` computed their loop bounds using
`DR.CONFIG?.MAP_SIZE || 200` — the shared cave/dungeon map size — instead of `this.activeMapSize?.()`, the
pattern every other post-Phase-2 system uses (`game.js`, `collision-system.js`, `world-system.js`,
`save-system.js`, etc.) to get the real per-zone size (360 for the resized Dark Woods overworld, 200 for
caves). This system was written before/independent of the Phase 2 resize and was never updated.

**Confirmed the severity by a standalone Node simulation** before fixing (not just by reading the code): for
any player x-position ≥ 200 — more than half of the 360-wide map — the computed `maxX` bound was actually
*below* the player's own position, meaning the render loop's `for (x = minX; x <= maxX; ...)` never even
reached the player's own column. Fog, fireflies, and all ambient creatures have been silently failing to
render across roughly the outer half of the Dark Woods ever since the V0.17.41 resize. Fixed by switching
both functions to `this.activeMapSize?.() || DR.CONFIG?.MAP_SIZE || 200`, matching the established pattern.

## 6. Implementation Notes

- **Signposting density**: deliberately much sparser than both the Phase 14 general forest-dressing pass and
  the Phase 6/9 region-specific dressing passes (own dedicated noise seed, `>0.988` threshold) — these are
  meant to read as an occasional warning sign, not general clutter, and the existing region dressing already
  provides the bulk of each territory's visual identity.
- **`dead_lantern_outskirts`'s `lanternPost` entry is explicitly skipped** in the signpost function — that
  region's signage is already fully saturated by the Phase 4 Dead Lantern Trail system; adding more lantern
  posts here would just duplicate that system's job.
- **Deer "flee" behavior is a deliberate scope decision**: `atmosphere-system.js` is entirely stateless (no
  persistent per-creature entities, just per-frame procedural jitter reseeded from tile coordinates). A true
  multi-frame flee/pathing behavior would require a new persistent-entity system, which risks exactly the
  "new duplicate collection/combat-adjacent system" the golden rules forbid for what's supposed to stay a
  cosmetic layer. Instead, the deer's rendered offset each frame points directly away from the player's
  *current* position when within 7 tiles — a stateless, per-frame approximation of "startled and stepping
  back," not persistent fleeing. Documented here rather than silently under-delivering on the plan's wording.
- **Region/depth biasing for new ambient kinds**: reuses `darkWoodsDepthFactor` (Phase 13) and the region
  system (Phase 3) rather than a new geography calculation — crows bias toward `depth > 0.4`, deer bias
  toward `depth < 0.5`, bats require proximity to one of the two real cave entrances (Phase 5/8's coordinates,
  looked up via `DR.CAVE_BY_ID`, same pattern as Phase 6/8's own cave-approach dressing).
- **No collision, no combat, no new persistent entities** for any of the 5 new ambient kinds — same guarantee
  the pre-existing 5 kinds already had, preserved throughout.

## 7. Validation

- `find . -name "*.js" -print0 | xargs -0 -n1 node --check` — clean, all files.
- `find . -name "*.json" -print0 | xargs -0 -n1 python3 -m json.tool > /dev/null` — clean, all files.
- **Functional simulation of the signpost placement logic**: re-implemented the exact region-lookup and
  selection logic from `ensureDarkWoodsTerritorySignposts` standalone in Node, using the real `seededNoise`
  function and the real region definitions, swept the full 360x360 coordinate space. Confirmed by execution:
  each region only ever produces its own named signpost prop(s) (e.g. `bramblefen_thicket` → only
  `clawMarks`, never `brokenArrow`); `dead_lantern_outskirts` produces zero placements (correctly skipped);
  overall density across all signpost-eligible tiles is 1.19% — low, as designed.
- **Functional simulation of the MAP_SIZE bug and fix** (Section 3): reproduced the exact bounds computation
  for a range of player x-positions, proving the pre-fix bound fell below the player's own position for
  every x ≥ 200, and that the post-fix bound is correct across the full 0-360 range.

**Not validated (no browser available, same limitation as Phases 2–15):** actual visual appearance in-game,
console-error-free boot, and no frame-rate impact. The new ambient kinds reuse the exact same per-frame cap
(`MAX_AMBIENT_CREATURES = 26`, unchanged) and the same cheap draw-call style as the 5 pre-existing kinds, so
no measurable performance change is expected, but this was not benchmarked.

## 9. Known Limitations

- Deer "flee" is a stateless per-frame visual offset, not persistent flee AI/pathing — see Section 6.
- Territory signposting is scattered across each whole region rather than concentrated specifically at the
  region's entrance/leading edge — consistent with how Phase 6/9's own region dressing already works, but
  means a sign could technically appear deep inside a region rather than strictly "before" it.
- "Dead trees before danger areas" is satisfied by Phase 14 (dead tree density already scales with region
  depth) and "crows before danger areas" is satisfied by this phase's depth-biased crow spawning — neither
  required new work, but they're not literally new Phase 16 deliverables, just already-satisfied plan items
  noted for completeness.
- No frame-rate measurement was performed (Section 7) — the "no performance spike" claim is a design
  argument (same cap, same draw-call cost class as existing kinds), not a benchmark.
