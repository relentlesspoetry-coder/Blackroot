- Runtime UI ownership remains in the existing UI/menu systems; no duplicate Patch Notes panel or duplicate gameplay UI was added.

## Root Cause
The Patch Notes UI did not use a live parser/loader for `PATCH_NOTES.md`. Both Patch Notes windows were hardcoded HTML lists. The splash/main-menu list had been manually updated only partially, while the in-game Menu list was stale and started at older V0.16.05/V0.15.x entries. Because there was no shared current embedded source, recent versions after V0.16.96 could be missing from one or both panels even though `PATCH_NOTES.md` contained later entries.

The issue was content embedding/UI ownership drift, not gameplay state, cache, sorting, or a combat-system issue.

## Changes Made
- Added a `V0.17.03` entry to `PATCH_NOTES.md`.
- Added a transparent `V0.16.98 - Current Baseline Handoff` marker because no standalone V0.16.98 phase report or feature summary was present in the project files. No undocumented gameplay changes were invented for that entry.
- Rebuilt both existing Patch Notes panels from the canonical `PATCH_NOTES.md` content.
- Replaced the stale one-line list treatment with structured cards that show version headings, section headings, bullet lists, and readable spacing.
- Added `refreshPatchNotesPanels()` to the existing UI owner so the in-game panel mirrors the embedded current Patch Notes content before either Patch Notes window is opened.
- Updated the existing in-game Patch Notes click binding to refresh the panel before toggling visibility.
