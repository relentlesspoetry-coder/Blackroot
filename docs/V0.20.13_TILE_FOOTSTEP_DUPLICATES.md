# V0.20.13 — The Same Words, Twice

Continuing the dead-field retirement, with **V0.20.12's lesson applied first**: ask what a field would
*mean* if it were read, before touching it.

## The duplication

Every tile in `data/tiles.js`:

```js
terrainType: 'grass', footstepEffect: 'grass_blade_fx', footstepSound: 'grass_soft', footstepRateModifier: 1.0,
footstepProfile: { terrainId: 'grass', terrainName: 'Grass', particleEffect: 'grass_blade_fx',
                   decalEffectOptional: null, soundSet: 'grass_soft', stepIntervalModifier: 1.0, … }
```

The **same three values**, loose on one line and again inside the profile on the next, under different
names.

**Only the profile is read:**

| consumer | reads |
|---|---|
| `terrain-footstep-system.js` | `def.footstepProfile` → particles |
| `audio-system.js:535` | `profile.soundSet` → the sound |

So the loose three are exact duplicates. **33 declarations removed**, behaviour-neutral. Footsteps are
unchanged — grass still sounds like grass, because `soundSet: 'grass_soft'` *inside the profile* was
always the one being played.

## What was NOT cut — the lesson paying for itself

The scanner also flagged **`terrainName`** (10×) and **`decalEffectOptional`** (10×), and **an earlier
pass of mine called them dead**.

They have **zero loose declarations**. They exist **only inside `footstepProfile`** — which is read
wholesale, and whose shape `terrainFallbackProfile` deliberately mirrors. **The scanner was reporting
the profile's own contents as if they were tile fields.**

Cutting them would have been the heavy-weapon mistake again, in the opposite direction: acting on *"not
read"* without asking what it would mean.

### Two tools, two wrong answers, same field

- **The scanner** said `terrainName` was dead — it was reading inside the profile.
- **My triage grep** said `terrainName` had 10 reads — it was counting `terrainName: 'Grass'` in
  `terrainFallbackProfile`, the game **defining** a default, not reading tile data.

Both confident. Both wrong. Neither was resolved until I opened the file and looked at the actual
structure.

## Verified

| check | result |
|---|---|
| `footstepEffect` / `footstepSound` / `footstepRateModifier` | **11 → 0** each |
| `terrainType` (which **is** read) | **untouched, 11** |
| `footstepProfile` declarations | **all 11 intact** |
| runtime: tile defs with a truthy profile | **10 of 11** |
| the 11th | **Cave Wall — deliberate `null`** |
| grass profile still carries | `soundSet`, `particleEffect`, `stepIntervalModifier` |
| suite | **PASS** — 15/15, 396/396, 0 issues, 0 faults |

**The Cave Wall null looked like a hole in the data and isn't.** You can't walk on a wall, and
`terrain-footstep-system.js:84` checks `def.footstepProfile === null` for exactly that case. One more
thing that would have been a false "fix".
