# V0.18.89 — Stat Pipeline: Set Bonuses as a First-Class Stat Source (Roadmap Item 14 begins)

Roadmap Item 14 asks for *one authoritative and inspectable calculation order for every character
stat*, with every listed source flowing through it. This version audits the current pipeline,
documents the authoritative order, and closes the one missing source — **set bonuses**.

## Audit: the current pipeline is already centralized

The player's stats are computed by one function, `Game.prototype.recalculatePlayerStats`
(`systems/inventory-system.js`), which runs on every equip / unequip / talent or trainer change:

1. Start an empty `totals` (via `DR.StatSystem.cleanStats`).
2. `add()` trainer bonuses (`npcTrainingBonuses`).
3. `add()` talent stat ranks (`talentStatBonuses()`).
4. `add()` each equipped item's `stats` (and `armor`).
5. **(new, V0.18.89)** `add()` each fully-equipped set's `bonus.stats`.
6. Call `DR.StatSystem.computePlayerDerived(className, level, baseStats, totals, raceStatMods)` — this
   applies base class values, level growth, race mods, and turns the aggregated `totals` into every
   final/derived value: `maxHp`, `maxMana`, `attack`, `defense`, `speed`, `armorReduction`,
   `magicResistReduction`, crit/cdr/damage multipliers, regen, attributes.
7. Assign the derived values onto the player; clamp `hp`/`mana` to the new maxima.

The UI (character sheet) and combat both read the output of this one function — they never compute
stats separately.

## Authoritative calculation order (documented)

```
base class values
  -> level growth
  -> permanent progression / race mods
  -> equipment base values + affixes (item.stats, item.armor)
  -> skill proficiency (trainer + talent bonuses)
  -> SET BONUSES            (V0.18.89 — new stage)
  -> derived + final clamps (computePlayerDerived: multipliers, reductions, regen, HP/mana maxima)
```

Buffs / debuffs / food / potions are handled by the status-effect system on top of this base; folding
those explicitly into the same documented order (additive vs multiplicative stages 8–11 in the
roadmap) is the next Item 14 slice.

## The fix — set bonuses as a stat source

Before this version, of the roadmap's listed stat sources only **set bonuses** had no stage. The
V0.18.88 Wanderer set applied XP and meditation bonuses via live-checked hooks, but a set could not
grant *stats*.

`recalculatePlayerStats` now, after the equipment loop:

```js
const armorSets = DR.ARMOR_SETS || {};
for (const setId of Object.keys(armorSets)) {
  const set = armorSets[setId];
  if (set?.bonus?.stats && this.hasFullArmorSet?.(setId)) add(set.bonus.stats);
}
```

- Set stats go into the **same `totals`** as gear, so `computePlayerDerived` applies them once.
- Because the whole function re-runs on every equip/unequip, the bonus is **added once and removed
  immediately** when a piece comes off — it is **never written into base stats**. This satisfies the
  roadmap's requirements ("added once", "removed immediately when conditions fail", "do not
  permanently write temporary bonuses into base stats").

The **Dark Woods Wanderer** set gains `bonus.stats { speed: 0.03, hp: 10 }` to exercise and showcase
the stage. Its `xpPercent` / `meditationSpeedPercent` remain at their own hooks (`awardPlayerXp` /
`meditationTickRate`) because they are not stats.

## Verification

- `node --check` clean; JSON valid; boots with **0 console errors**.
- **Isolated aura test** (same full-set equipment, toggling `bonus.stats` on/off): final `maxHp`
  changes by **exactly +10** and `speed` by **exactly +0.03** — the aura flows through the real
  derived pipeline, not a shortcut.
- **Full-set gating**: removing any one set piece drops the entire set contribution (`hasFullArmorSet`
  → false), confirming add-once / remove-on-fail behaviour through `recalculatePlayerStats`.
