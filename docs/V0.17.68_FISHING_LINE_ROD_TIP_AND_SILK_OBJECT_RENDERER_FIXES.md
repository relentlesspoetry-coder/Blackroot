# V0.17.68 — Bug Fixes: Fishing Line Rod-Tip Anchor + Silk Web Cavern Object Renderers

**Type:** Post-expansion bug fixes (not a plan phase)
**Previous version:** V0.17.67
**New version:** V0.17.68

Two unrelated, pre-existing bugs found while acting on a user report ("the fishing line isn't
attached to the actual fishing pole"). Both fixed at the root cause in the owning render module; no
new systems, wrappers, or duplicate logic.

---

## Bug 1 — Fishing line not attached to the rod

### Symptom
While fishing, the line started up near the top of the screen instead of at the tip of the rod
(reported with a screenshot of Aelwyn fishing at a Silk Web Cavern pool).

### Root cause
The fishing system draws the line from a **live rod-tip screen anchor** published by the character
renderer:

- `systems/fishing-system.js` → `getFishingLineStartScreen()` prefers
  `getFreshFishingRodTipScreen()` (which returns `player.fishingRodTipScreen` when it is fresh and the
  camera matches), and only falls back to `worldFishingPointToScreen(getFishingPoleTipWorld(...))`
  when no live anchor is available.
- The fallback `getFishingPoleTipWorld()` returns a world point at `z = elev + 3.25`, which projects
  **near the top of the screen** once the camera zoom is applied — this is the detached line.

`player.fishingRodTipScreen` was **only ever published by the humanoid-base path**
(`render/entity-renderer.js` `drawFishingActionOverlay`, used by `drawStandingBody`). But a real
player is a **class-identity actor** (Fighter/Cleric/Rogue/Enchanter/Summoner/Necromancer), rendered
via `drawClassIdentityModel` → the shared `drawClassFishingRig` in
`render/class-identity-procedural-model.js`, which draws the rod but **never published the anchor**.
So every fishing player always hit the too-high world-space fallback. (The Phase 23
`s`→`screen` fix to `drawFishingActionOverlay` only ever affected the humanoid-base path, which
players do not use — which is why the line stayed detached.)

### Fix (owning modules)
1. **`render/class-identity-procedural-model.js` `drawClassFishingRig`** — after computing the rod
   `tip` (the exact point the pole is drawn to), record it as a model-space offset on the source
   entity: `source.fishingRodTipLocal = { x: tip.x, y: tip.y }`. This is the same offset space the
   model's anchors already use (offsets from the model foot at `screenX/screenY`), and it is
   independent of the sprite-cache bake origin, so it is correct whether the frame is live-rendered or
   blitted from cache (the cache's `bakeActor.sourceEntity` is the real entity).
2. **`render/entity-renderer.js` `drawEntity`** — after the model dispatch, when the entity is fishing
   and has published a `fishingRodTipLocal`, convert it to the live **unzoomed `worldToScreen`
   anchor** the fishing system consumes:
   `fishingRodTipScreen = { x: visualFoot.x + local.x*scale, y: visualFoot.y + local.y*scale }`,
   plus the freshness timestamp and camera state (`yaw/zoom/x/y`) exactly like the humanoid-base
   publisher. Players render at `scale = 1` and the character is drawn at `visualFoot` with anchors as
   offsets from it (the nameplate uses the identical `foot + anchor` mapping), so `visualFoot +
   local*scale` is the real on-screen rod tip.

This publishes the anchor generically for **every** cached model path that records
`fishingRodTipLocal` (all class-identity classes and mercs share `drawClassFishingRig`). The
humanoid-base path is unaffected — it still publishes its own anchor inside
`drawFishingActionOverlay` and never sets `fishingRodTipLocal`.

### Verification (in-browser, no-cache server)
- `drawClassFishingRig` now sets `fishingRodTipLocal` for all facings (east/west/south/north):
  ~`(±127, -108)` — a sane "up and out from the hands" rod tip.
- Rendered a fishing Cleric on a probe canvas at a known foot and pixel-sampled the actual rod
  (brown `#4b2e19` / gold `#c79b56`) around the computed anchor: **the published anchor sits 1px from
  the nearest rendered rod pixel, exactly at the rod's tip** (rod bbox top-left corner). So
  `foot + fishingRodTipLocal` lands on the drawn pole tip; the line now starts at the pole.

## Bug 2 — Silk Web Cavern object render faults

### Symptom
`TypeError: this.drawEggCluster is not a function` (caught per-object as "render fault in
world-object") spamming every frame in the Silk Web Cavern, with the affected objects invisible.

### Root cause
`render/object-renderer.js` `drawObject` dispatched six placed cavern object types to `draw*`
functions that were never defined anywhere:
`eggCluster`, `chitinColumn`, `broodGrowth`, `webBones`, `royalPylon`, `silkReliquary`. (Same bug
class as the Phase 21 `drawSilkStrands` fix; these were missed because they only fault when their
specific objects are on-screen.) All six are actually placed in the dungeon data/generation.

### Fix
Routed the six cases to the existing shared `drawSilkCavernHighFidelityDetail(s, obj)` renderer (the
same renderer already used by ~16 other silk-cavern detail types), and added a concise dedicated
visual branch for each type (clustered eggs; ribbed chitin column; brood-sac mass; bone pile in web;
crested royal pylon; silk reliquary shrine). No crash, and each renders distinctly.

### Verification
- `render/object-renderer.js` no longer references any of the six missing functions; a sweep of every
  `this.draw*(s, obj)` dispatch confirms **zero** undefined renderers remain.
- Confirmed served-file content: line 177 now routes `eggCluster` to
  `drawSilkCavernHighFidelityDetail`; `this.drawEggCluster` is gone (the console errors were stale
  pre-reload buffer entries).

## Not changed
- The world-space fallback (`getFishingPoleTipWorld`, `z = elev + 3.25`) is left as-is: it is now only
  reachable for render paths that publish no `fishingRodTipLocal` and no humanoid-base anchor (e.g.
  Bard/Druid, which use their own separate procedural fishing rigs). Those classes could be given the
  same `fishingRodTipLocal` treatment in a follow-up; the reported bug (and all `drawClassFishingRig`
  classes) is fixed.
