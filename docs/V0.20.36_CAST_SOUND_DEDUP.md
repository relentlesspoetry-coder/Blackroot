# V0.20.36 — One Cast Sound, The Right One (fixes V0.20.35)

## A same-day fix of my own bug, caught by continuing the audit

Continuing the audio work onto the **enemy** side is what exposed it. Checking whether enemy abilities
were silent revealed that **`spawnCastCue`** — the shared cast-cue funnel, called in the *player* cast
pipeline (`spell-system.js` 2479 cast-time, 2648 instant) **and** by enemies/bots/mercs — **already
plays `magic_cast`.**

My V0.20.35 test called `resolveClassSpell` **directly**, bypassing that pipeline. So I saw "silent"
and added a **second** cast sound in `resolveClassSpell`. In the real instant-cast flow that meant:

```
spawnCastCue → magic_cast     (already there)
resolveClassSpell → my sound  (V0.20.35)
```

- magic spell: `magic_cast` + `magic_cast` → same id, **deduped** by the cooldown → fine.
- **Lightning: `magic_cast` + `thunder_roll` → different ids → NOT deduped → double.**
- **Melee: `magic_cast` + `attack_slash` → double.**

`playSfx` only dedupes **identical** ids within the cooldown window. Verified the double empirically:
the real flow played `magic_cast + thunder_roll + hit_thud`.

## The fix — unify on the one real funnel

- **`spawnCastCue`** gains an `options.sfx` override (default `magic_cast`, so **enemy/bot/merc callers
  are unchanged**).
- The **two player cast paths** pass `{ sfx: this.spellCastSfxId(spell) }`.
- The **V0.20.35 `resolveClassSpell` sound is removed.**

`resolveClassSpell` stays silent, which is **correct** — it's also called by item procs and bots that
never showed a cast cue and were never meant to announce a player cast.

## Better than V0.20.35, not just de-duped

The kind-aware cast sound now fires in **both** cast paths — cast-time *and* instant — where the
removed `resolveClassSpell` line only covered the instant path. Slower cast-time spells now get the
right cast sound too.

## The lesson

The session's most-repeated one, this time on my own work from the same day: **verify through the real
path, not a direct call.** My V0.20.35 test drove `resolveClassSpell` directly and never saw the cast
cue that the real pipeline plays first — so "spells are silent" was measured on a path players never
take. The fix, like the good ones this session, was to find the one true funnel (`spawnCastCue`) and
route everything through it once.
