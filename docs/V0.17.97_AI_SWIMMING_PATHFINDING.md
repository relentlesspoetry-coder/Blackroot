# V0.17.97 — AI overhaul I: swimming + pathfinding (movement)

First slice of the AI work: fix the two concrete movement complaints — creatures get
stuck on trees, and nothing but the player will cross water. Combat/skill smarts and the
AI-performance pass come next.

## Root causes

**Water was player-only.** In `systems/collision-system.js`:
- `tryMove`: `if (enteringWater && !movingPlayer) return false;` — any non-player was
  hard-blocked from stepping into a water tile.
- `isWalkable` water branch: returned a swim result only for `entity.kind === 'player'`,
  else `return false`.

Since **all** movement and pathfinding route through these (`findPath` →
`pathTileBlocked` → `isWalkable`), enemies/mercs/pets/bots could never enter water and A*
never routed a path through it.

**Enemies never pathfound.** `entities/enemy.js` moved chasers with raw `moveToward`
(local steering + a few side-steps) only. The A* pathfinder (`findPath`) and the async
path-worker exist and companions use them, but enemies never called them — so a tree,
rock, or water edge between a mob and its target left it stuck.

## Fixes

### 1. Universal swimming
- **`Game.actorCanSwim(entity)`** (new): `true` for player/enemy/merc/pet/bot/remote by
  default. Opt a specific actor out with `cannotSwim = true` (or `canSwim = false`). The
  player still honours an external `playerCanSwim === false` disable.
- **`tryMove`**: admits any swimmer to water (`if (enteringWater && actorCanSwim !== true)
  return false;`) and now sets `entity.swimming = enteringWater` for **all** actors (the
  player's flag is still refined per-frame by `swimmingSystem`, so no player change).
- **`isWalkable`** water branch: `return this.actorCanSwim(entity) === true;` — swimmers
  may enter shallow or deep water; non-swimmers are blocked.

Because everything funnels through `isWalkable`, this single change lets mobs, mercs,
pets, and bots cross water **and** makes A* route through water for swimmers (and refuse
it for `cannotSwim` actors) — companions now follow the player across water too.

### 2. Enemy pathfinding
- **`Game.moveEnemyToward(enemy, tx, ty, dt, stopRange)`** (new): if
  `hasLineWalkPath` is clear, steer directly with the existing `moveToward` (cheap, no
  behaviour change). If blocked, A* around the obstacle with a **per-enemy cached,
  throttled** route (`findPath`, repath ~0.5 s, `maxNodes: 520`, `allowPartialPath`),
  advancing waypoints as it goes; falls back to a direct steer if fully walled off.
- `entities/enemy.js`: the aggro **chase** and the **home-return** now call
  `moveEnemyToward` instead of raw `moveToward`. `handlePathingFailure` stays as the
  last-resort unstick.

Cost is bounded: A* runs only for enemies whose straight line is blocked, and only on the
throttle — so many chasers don't blow up the frame.

## Verification (in-browser)
- `actorCanSwim`: player/enemy/merc/pet = true; `{cannotSwim:true}` = false.
- A found water tile is `isWalkable` for enemy/merc/pet/player, **not** for a `cannotSwim`
  actor.
- An enemy physically `tryMove`s onto a water tile (its `swimming` flag sets true).
- `findPath` returns a 2-step route **across** water for a swimmer (`crossesWater` true);
  a `cannotSwim` actor gets **no** path across.
- Live play: 5 enemies swimming + 1 actively A*-routing, **0 console errors**; camp render
  unchanged (screenshot).
