# V0.20.97 — Art Direction Phase 10: The Authored Asset Pipeline

The contract an artist exports against, plus the loader, validator and diagnostics that consume it.

**Shipped before the art, on purpose.** Fixing the format now means work commissioned against it drops
in; discovering the format after the art arrives means a second migration.

## What this is not

`systems/sprite-atlas-system.js` already exists and is a **different thing**. It indexes sprites *baked
from our own procedural renderers* (`dream-realms-entity-atlas`) — no anchors, no emissive masks, no
occlusion height, no animation grouping, because a bake of our own output needs none of those. Authored
art needs all of them, so this is a separate system rather than an extension.

Worth recording, found while doing this: `assets/atlases/entities.json` is an **empty placeholder**
(`"pages": [], "frames": {}`), with the real V0.20.53 bake preserved beside it as
`entities.baked-v0.20.53.manifest.js`.

## The format

Schema `blackroot-authored-atlas-v1`. Full documentation lives in the file header and, in artist-facing
form, in `docs/ASSET_EXPORT_SPEC.md`.

```json
{
  "schema": "blackroot-authored-atlas-v1",
  "name": "props_waypoints", "kind": "prop",
  "pages": [{ "file": "props_waypoints_0.png", "w": 1024, "h": 1024 }],
  "frames": {
    "shrine": {
      "page": 0,
      "rect":   { "x": 0, "y": 0, "w": 192, "h": 192 },
      "anchor": { "x": 96, "y": 168 },
      "occlusionHeight": 72,
      "contactShadow": { "rx": 44, "ry": 16, "alpha": 0.24 },
      "light": { "radius": 140, "intensity": 0.85, "color": "#78aaff" }
    }
  },
  "animations": { "walk": { "directions": 8, "frames": 8, "fps": 12, "pattern": "walk_{dir}_{i}" } }
}
```

API: `registerAuthoredAtlas`, `loadAuthoredAtlas`, `authoredFrame`, `hasAuthoredArt(kind?)`,
`authoredAtlasReport()`, and the pure/synchronous `DR.AuthoredAtlas.validateManifest(man, label)`.

## The validator is the point

Every rule exists because getting it wrong is **silent**.

| rule | severity | what happens without it |
|---|---|---|
| anchor is the foot | **warn** | prop floats — the exact failure V0.20.94 chased in terrain |
| anchor inside its rect | error | draws offset, no error at all |
| rect inside its page | error | samples garbage |
| every anim frame resolves | error | invisible until a player faces that way |
| page ≤ 2048 | **warn** | ~6.6× slower blits — the V0.20.53/54 revert |
| schema / kind / pages present | error | — |

Two are warnings rather than errors because both are *usually* but not *always* mistakes — a centre
anchor is legitimate for a floating VFX frame, and a large page may be a deliberate call. They warn
loudly and load.

`registerAuthoredAtlas` **refuses** an invalid atlas rather than half-loading it: a partially valid
atlas produces missing sprites at runtime with no explanation.

Wired into `runValidationSuite` as an `authoredAtlas` section, so a bad atlas fails **at boot, where
the reason is still visible**, rather than at render time. With none registered it reports
`no authored atlases registered - procedural rendering in use` — not a failure, just the truth.

## Verified by negative control

A validator that passes everything is worthless.

| case | result |
|---|---|
| valid manifest | **accepted**, 0 errors, 0 warnings |
| centre anchor | ok, **1 warning** — *"anchor sits in the upper half; anchors should be the FOOT"* |
| page 4096² | ok, **1 warning** — *"pages above 2048 blit far slower (see V0.20.53/54 revert)"* |
| anchor (400,168) in a 192×192 frame | **rejected** — *"lies outside its 192x192 frame"* |
| rect x=960 w=192 on a 1024 page | **rejected** — *"rect falls outside page 0"* |
| 8-dir walk, no frames authored | **rejected** — *"64 animation frame(s) missing, e.g. walk:walk_0_0…"* |
| wrong schema | **rejected** — names both the found and expected schema |
| no pages | **rejected** |
| `registerAuthoredAtlas` good / bad | **true / false** |
| `hasAuthoredArt('actor')` with only a prop atlas | **false** |

One correction worth recording: my first assertion run reported `PASS_warningsNotFatal: false`. That was
**my test's bug, not the code's** — I compared `.warnings` (an array) to `0` instead of
`.warnings.length`. The underlying results were correct. Worth noting because an assertion that fails
for its own reasons is indistinguishable from a real failure until you read it.

## Nothing here invents art

No atlas exists. `hasAuthoredArt()` returns false, every path degrades to "not available", and the game
renders exactly as it did. Stated in the file header so nobody reads this as more than it is.
