# V0.18.4 — Animations only played fully in Debug mode (fixed)

## Symptom (user report)
> "i've noticed the animations only play fully in Debug mode"

In normal play the player and nearby creatures looked nearly frozen (idle) or
choppy (walk/attack). Opening the Debug overlay made everything animate smoothly
at full frame rate.

## Root cause
`render/entity-renderer.js` funnels **every** procedural model draw (player,
merc, race, class, bard, druid, spider, dark-woods mob, wolf, pet) through one
choke point: `drawCachedModelOrFallback(actor, model, scale, options)`.

That method decides whether to use the **runtime sprite cache**
(`systems/runtime-sprite-cache-system.js`):

```js
const debugFacing = !!(actor.debugFacing || this.debugMode || this.showFacingDebug);
const cacheAllowed = !debugFacing && this.runtimeSpriteCache?.drawModel && ...;
```

The Debug overlay sets `actor.debugFacing = true` for every actor, so
`cacheAllowed` becomes `false` and **every model is drawn live at 60fps** — the
smooth look the user saw only in Debug.

In normal play (`cacheAllowed = true`), the sprite cache bakes each model into a
**bounded flip-book of quantized animation frames** (`animationBucket` /
`buildKey` → `t${timeBucket}`):

| State  | Buckets (frames)              | Frame interval                  |
|--------|-------------------------------|---------------------------------|
| idle   | `spriteCacheIdleBuckets: 2`   | `spriteCacheIdleFrameMs: 250`   |
| moving | `spriteCacheMovingBuckets: 6` | `spriteCacheMovingFrameMs: 100` |
| action | `spriteCacheActionBuckets: 8` | `spriteCacheActionFrameMs: 125` |

So idle was **2 poses sampled 250 ms apart** — with the breathing/sway cycle,
those two samples are often nearly identical, so idle characters looked
**frozen**. Walk/attack were low-frame flip-books. This is fine (imperceptible)
on the distant crowd, but it was being applied to the player and the monsters
right next to them too — exactly what the eye tracks.

## Fix (root cause, keeps the perf net)
The sprite cache is a legitimate optimization for the **distant crowd** (a
360×360 map carries ~400 enemies). The fix keeps it there but renders the things
the player actually watches **live**:

`render/entity-renderer.js` — new method:

```js
shouldRenderModelLive(actor) {
  const src = actor?.sourceEntity || actor;
  if (src === this.player || actor?.kind === 'player' || actor?.isPlayer || src?.kind === 'player') return true;
  const player = this.player;
  if (!player) return false;
  const perf = this.performanceSettings?.() || DR.CONFIG?.PERFORMANCE || {};
  const liveRadius = Number(perf.spriteCacheLiveRadius);
  if (!(liveRadius > 0)) return false;
  const ex = Number(src?.x), ey = Number(src?.y);
  if (!Number.isFinite(ex) || !Number.isFinite(ey)) return false;
  const dx = ex - player.x, dy = ey - player.y;
  return (dx * dx + dy * dy) <= liveRadius * liveRadius;
}
```

and the cache gate becomes:

```js
const cacheAllowed = !debugFacing && !this.shouldRenderModelLive(actor)
  && this.runtimeSpriteCache?.drawModel && DR.CONFIG?.PERFORMANCE?.enableSpriteCache !== false;
```

`core/config.js` — new tunable in `CONFIG.PERFORMANCE`:

```js
spriteCacheLiveRadius: 18,   // tiles; live-render radius around the player
```

**18** sits just inside `nearAiRadius: 20`, so every live-rendered entity is
also in the near-AI band — full AI *and* full animation, consistent. Only a
handful of things are ever within 18 tiles at once, so the frame rate is
unaffected; the cache still does its job on everything farther out.

## Verification (in-browser, port 8899)
- `shouldRenderModelLive` present; returns **true** for the player, **true** for
  an entity 8 tiles away, **false** for one 40 tiles away; `spriteCacheLiveRadius = 18`.
- After real frames: `player._runtimeSpriteCached` is **false** (drawn live);
  nearby enemies within 18 tiles are **not** cache-flagged; the sprite cache is
  still active and working on the mid/far on-screen crowd (**4184 hits / 62
  entries**).
- **0 console errors.**

Animation is temporal, so a screenshot can't prove smoothness — the motion
itself is the user's visual sign-off. Structural behavior is confirmed above.
