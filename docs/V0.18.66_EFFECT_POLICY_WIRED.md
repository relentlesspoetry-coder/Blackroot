# V0.18.66 — Effect Policy Wired Into the Live Paths (Item 13, slice 2)

Slice 1 ([V0.18.65](V0.18.65_EFFECT_POLICY_FOUNDATION.md)) created `core/effect-policy.js` but only the Bard song cap read from it. This slice makes the policy the **single automatic enforcement point** for effects and consumables. No player-visible change — it's foundation for the content systems (food/potions, Roadmap Item 3).

## Changes
1. **Central concurrency enforcement** — `DR.StatusEffects.applyToEntity` (`systems/status-effect-system.js`), the one place a *new* effect is added to an actor's `buffs`, now resolves `DR.effectPolicy.categoryOf(normalized)` and calls `enforceConcurrency`: drop the oldest of that category to make room, or deny the incoming when `overflow: 'rejectNew'`. Uncategorized effects (`categoryOf → null`) are left completely untouched.
2. **Carry `effectCategory` through `normalizeEffect`** — it previously stripped it (only the `majorSong` flag survived), so a categorized food/potion effect would lose its category the moment it was stored. Now the category survives normalization, so `categoryOf` keeps recognizing it.
3. **Consumable cooldown via the policy** — the consumable-use cooldown key (`systems/item-compiler-system.js`) is now `item.cooldownGroup || effectPolicy.cooldownGroupFor(item.effectCategory) || item.itemId`, so a categorized consumable shares its category cooldown group (copies/stacks can't bypass it). Inert today (no shipped item sets `effectCategory`); an explicit `item.cooldownGroup` still wins.

## Why this matters
The effect and consumable systems now enforce the central policy **automatically**. The upcoming food, potions and other consumables (Item 3) become policy-governed simply by declaring an `effectCategory` — no new hard-coded caps anywhere.
