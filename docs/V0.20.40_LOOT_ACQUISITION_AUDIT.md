# V0.20.40 — The Loot Audit That Kept Catching Itself (Roadmap Item 12)

## Result

The loot & acquisition system is **healthy**:

- **0 dangling loot references** — all 193 runtime loot `itemId`s resolve to the 396-item catalog.
- **0 genuinely unobtainable items** — every catalog item is obtainable via loot, shop, crafting,
  gathering, fishing, quest, or starter gear.

No code fix needed. But the *journey* is the story, and it's the session's recurring lesson turned on my
own audit.

## 133 → 71 → 1 → 0, and every step was my methodology, not the game

The first automated sweep reported **133 "unobtainable" items.** Almost every one was a flaw in how I
was *looking*:

| # | the artifact | the fix |
|---|---|---|
| 1 | my object-walk didn't collect **bare strings inside arrays**, so it missed **every shop item** (shops are `id → ['item_x', …]`) | walk array elements as values |
| 2 | gathering/fishing reference items by **display name** (`'Gloomleaf'`), not id | build a name→id map |
| 3 | **starter gear** (`item_starter_v01689_*`, 70 items) is granted at **runtime onto `game.equipment`** — not the player, not in any data blob (V0.20.2: belongings live on the GAME) | start each class, read `game.equipment` |
| 4 | the last candidate, `item_torn_lookout_ledger`, is granted by a **quest POI** (`quests.js:74 grantItem`) | — |

Correcting each layer: **133 → 71 → 1 → 0.**

The near-miss: I almost reported "133 unobtainable items" as a finding. The tell was **shops
contributing 0 refs** (`afterLoot 193 = afterShops 193`) — a source that obviously *should* add items
adding none is a methodology smell, not a game bug.

## The one lasting change — a regression guard

The loot-reference invariant (every loot `itemId` must exist) is worth locking in, so this ships
`tools/check-loot-references.js`, wired into `validate.sh`:

- Cross-references every `itemId: 'item_…'` in `data/loot-tables.js` / `data/loot.js` against every
  `id: 'item_…'` in `data/items.js`.
- Fails the build on a dangling reference.
- **Verified not vacuous** by a negative control: an injected fake `item_this_does_not_exist` is flagged
  — and *only* it, while all 178 real refs resolve.

Scope note in the checker: it covers **ID-based loot drops** only. Gathering/fishing (name-based) and
runtime-granted starter gear aren't loot-table drops and were verified separately in this audit.

## The lesson

The one this session keeps teaching, now applied to auditing itself: **when a sweep reports a big
number, distrust the sweep before the game.** Every layer of "unobtainable" here was an artifact —
arrays of strings, names vs ids, runtime grants, game-vs-player storage, quest POIs. The only honest
way to land on "0" was to verify each source against reality (start the class, read the equipment, grep
the ledger) rather than trust the walk. A checker that ships is only worth shipping once you've proven
it can fail.
