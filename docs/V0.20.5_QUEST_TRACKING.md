# V0.20.5 — What Am I Actually Doing (Roadmap Item 15, tracking)

The HUD quest panel showed a **single aggregate progress bar summed across every active quest**, plus a
nearby-source prompt. It never named one objective. The spec's *"show tracked objectives clearly"* had
**nothing behind it at all**.

## The audit — spec vs data

| tracking requirement | reality |
|---|---|
| define a tracked-quest limit | **already existed** — `selectedQuestId`, single-valued, driven by the journal's Select button. Nothing used it. |
| show tracked objectives clearly | **absent** — aggregate bar only |
| update progress immediately | `refreshPanel` already called from **16 sites** — no new plumbing needed |
| show timed conditions | **already built** — `quest.deadlineAt` drives a countdown (V0.17.85) |
| show night-only conditions | **`nightOnly` on 3/38 tasks — enforced since V0.17.87, never displayed** |
| show escort state | `this.escort.actor` — runtime, available, unshown |
| distinguish optional / required | **NO DATA** — not built, see below |

**I got one wrong on the first pass.** I called timed conditions "NO DATA" by checking *task* fields
only. Reading `refreshPanel` showed `quest.deadlineAt` already driving a `⏱ 42s` countdown. Corrected
by reading the code, not the data — and left alone, because it already works.

## The real find — a small cruelty, since V0.17.87

`task.nightOnly` gates three objectives: `advanceObjective` **skips a nightOnly task by day**. The
wispbloom won't be gathered in daylight, the Lumen-Wisp won't come, the ring won't open.

The engine has enforced that for many versions and **nothing displayed it**. A player could stand in
exactly the right place, doing exactly the right thing, at two in the afternoon, and watch the counter
refuse to move with no way to work out why.

Those objectives now render `☾ <label> 0/5 — waits for dark`, and the marker clears at night.

(Two of the three carry "(night only)" inside their authored *label* text, which helped — but the third
didn't, and a label is a string an author remembered to write, not a fact the UI knows.)

## What was built

`renderTrackedObjectives()`, called from `refreshPanel`:

- names the **tracked quest** and each objective with live `n/required`
- **✔** a completed objective, **☾** one waiting for dark, **·** one in progress
- **⚑ Escort underway — keep them alive (43/90 HP)** when `this.escort.actor` is live
- prompts *"No quest tracked · press L to select one"* when nothing is selected

## Deliberately NOT built

**"Distinguish optional and required objectives."** No task carries an optional flag anywhere in the
data — `required` is a **count** of how many, not a required/optional marker. The label would read
"Required" on **all 38 objectives, forever**.

A word that never varies is not information; it's furniture. When an optional objective exists, it can
have a marker.

### A false alarm, checked rather than assumed

The escort line **looked** concatenated with the objective above it (`…trail 1/5⚑ Escort underway…`).
That was **`innerText` collapsing line breaks on an element that isn't rendered** — the HUD is hidden at
the title screen — not a layout bug. Inspecting the DOM directly showed **three separate
`display: block` divs**. The test was unreliable; the code was fine.
