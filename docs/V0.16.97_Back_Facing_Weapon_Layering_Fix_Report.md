# V0.16.97 - Back-Facing Weapon Layering Fix Report

## Phase source

- Phase brief: "Back-Facing Weapon Layering / Body Occlusion Fix Pass" (user-supplied, targeted "V0.16.95" against an assumed baseline of V0.16.94; the project's actual shipped version at the start of this phase was V0.16.96 - Party EXP Bond Bonus (V0.16.95) and Class Buff Spell Icon Linking Pass (V0.16.96) had already landed in this session - so this phase was produced as the next real version, V0.16.97, per the auto-increment convention already established earlier in this session).
- Baseline used: `Blackroot V0.16.96`
- Target produced: `Blackroot V0.16.97`

## Where weapon/body layer ownership lives

`render/humanoid-base-renderer.js`'s `Base.draw(ctx, actor, options, nowMs)` is the single shared draw-order owner for every humanoid model (`render/class-identity-procedural-model.js`, `render/bard-procedural-model.js`, `render/druid-procedural-model.js`, `render/race-identity-procedural-model.js` all funnel into it). It calls a fixed sequence of body-part draws and, for equipment, calls out to `drawPaperdollLayer(ctx, rig, layer, arg)`, a thin wrapper around `render/paperdoll-equipment-renderer.js`'s `DR.render.PaperdollEquipmentRenderer.drawBodyLayer(ctx, rig, layer, arg)`.

`render/paperdoll-equipment-renderer.js` remains the sole owner of weapon *visual resolution* (item -> shape, established by the V0.16.89/V0.16.90 weapon-visual passes: `slotItem`, `resolveVisualSlotItem`, `weaponKind`, `drawWeaponShape`). This phase did not touch any of that resolution logic - only *when* the already-resolved shapes are drawn relative to the torso.

## Root cause

Confirmed by tracing the full draw sequence in `Base.draw()` (`render/humanoid-base-renderer.js:1112-1135`, before this phase): `drawPaperdollLayer(ctx, rig, 'weapon')` and `drawPaperdollLayer(ctx, rig, 'offhand')` were called at one fixed position each - always *after* `drawTorso`/`drawPaperdollLayer('torso')` - regardless of `rig.dir.backVisible` (the existing per-direction flag, true for north/northeast/northwest, computed in `render/humanoid-animation-system.js`'s `DIRECTIONS` table). `render/paperdoll-equipment-renderer.js`'s `drawWeapon`/`drawOffhand`/`drawBodyLayer` had zero facing-aware branches (confirmed by reading the full file) - the `side`/`dx` values they *do* read from `rig.dir` are used only to mirror the sprite left/right, never to change draw order.

The codebase already had the correct pattern for this exact problem, just not applied to the paperdoll weapon layer: `render/bard-procedural-model.js`'s `instrumentLayer(rig)` and `render/druid-procedural-model.js`'s `staffLayer(rig)` both return `'back'` when `rig.dir.backVisible` (unless actively casting/attacking, which always shows `'front'`), and are drawn from two different hook call sites (`hooks.back`, pre-torso; `hooks.front`, post-torso) so only the matching one actually draws. This same pair-of-call-sites pattern is also how `class-identity-procedural-model.js`'s `drawBackProp`/`drawFrontProps` (the Ranger/Warden class-locked weapon art) already worked - but for every other class, `shouldHideDefaultMainWeapon()` suppresses that class-owned back/front pair as soon as any weapon is equipped (which is true for virtually every character, including starter gear), so rendering fell through to the single, non-facing-aware paperdoll `'weapon'` layer call. That is the confirmed root cause.

## What changed

**`render/paperdoll-equipment-renderer.js`:**
- Added `weaponDepthLayer(rig)`, directly mirroring `instrumentLayer`/`staffLayer`: returns `'front'` while `pose.action` is `'cast'`/`'attack'` (a weapon mid-swing/mid-cast always shows in front, regardless of base facing), otherwise returns `'back'` when `rig.dir.backVisible` else `'front'`.
- `drawBodyLayer(ctx, rig, layer, arg)`: for `layer === 'weapon'` and `layer === 'offhand'` only, it now computes `requestedPass = arg === 'back' ? 'back' : 'front'` and skips drawing unless `weaponDepthLayer(rig) === requestedPass`. Every other layer (`back`, `legs`, `torso`, `arm`, `head`) is untouched. `arg` was previously used only by the `'arm'` layer (to pick `'near'`/`'far'`), so reusing it for `'weapon'`/`'offhand'` introduces no collision.

**`render/humanoid-base-renderer.js`:**
- `Base.draw()` now calls `drawPaperdollLayer(ctx, rig, 'weapon', 'back')` and `drawPaperdollLayer(ctx, rig, 'offhand', 'back')` once, right after the existing pre-torso `drawPaperdollLayer(ctx, rig, 'back')` call (in the same conceptual group as `hooks.back`/cloak/cape). The original post-torso calls remain in their original positions, now explicitly passed `'front'`. Because `drawBodyLayer`'s new gate ensures only one of the two calls for a given layer actually draws per frame (they're mutually exclusive by construction - `weaponDepthLayer(rig)` returns exactly one value), there is no double-draw and no new render pass; only the position of the existing draw call changes per facing.

No other file was changed. No new function/module was added - `weaponDepthLayer` lives beside `activityHidesWeapons` in the same file that already owns weapon drawing, and the two-call-site pattern in `Base.draw()` is the exact idiom the codebase already used for Bard/Druid.

## How back-facing weapon occlusion now works

For north/northeast/northwest (`rig.dir.backVisible === true`) and not actively casting/attacking: `drawPaperdollLayer(ctx, rig, 'weapon'/'offhand', 'back')` fires before `drawTorso`, so the torso/robe/cloak shape (drawn immediately after) paints over the parts of the weapon/shield that fall behind the body silhouette; only the parts extending past the silhouette (e.g. a blade tip over the shoulder) remain visible. The post-torso `'front'`-pass call for the same layer is skipped for these facings, so the weapon is never drawn a second time on top of the body. During `cast`/`attack` states, `weaponDepthLayer` always returns `'front'`, so the weapon still draws in its normal after-torso position mid-swing/mid-cast, exactly like every other facing - matching the existing Bard/Druid exception for the same states.

## How side-facing near/far weapon layering works

Side-facing (east/west, southeast/southwest, south) all have `rig.dir.backVisible === false`, so `weaponDepthLayer` returns `'front'` for them exactly as before this phase - their draw position is unchanged. The pre-existing arm-level near/far depth split (`layerProfile.farArmAfterTorso`, `drawArms(ctx, rig, palette, 'far'|'near')`) was not touched. Extending a *new* near/far split to the weapon/offhand layers themselves (as opposed to preserving the existing behavior) was deliberately out of scope for this pass: the confirmed, observed bug reports were exclusively back-facing (`backVisible`) cases, side-facing was not reported as broken, and there is no existing precedent anywhere in the renderer for a weapon-level near/far split to extend safely - inventing one would have risked a new, untested regression on facings that work correctly today. This is a scope decision, documented here per the phase's root-cause-only instruction ("Fix the root cause, not the symptom" - the diagnosed and confirmed root cause is the back-facing occlusion gap, not side-facing).

## Classes/weapon categories tested

Live-tested in the actual gameplay canvas renderer (not just code review) via a scripted Playwright session: created and drove real characters in the actual Dark Woods start zone.

- **Cleric** (routes through the generic `class-identity-procedural-model.js` -> paperdoll path): idle and walking, back-facing (north) - confirmed the weapon (mace) no longer cuts across the robe/torso; only a small edge is visible past the shoulder silhouette. Front-facing idle confirmed unchanged (weapon held normally in front).
- **Fighter** (same generic path): idle and walking, back-facing (north) - confirmed the same correct occlusion behavior (blade tip visible past the shoulder, not across the chest/back).
- **Druid** (dedicated `druid-procedural-model.js`, own `staffLayer` hook for its class-default staff): confirmed its pre-existing back-facing slung-staff visual is unchanged/still correct (this class-owned path was already facing-aware before this phase and was not touched), and confirmed east-facing (side) still shows the staff held normally in front - no regression.
- Screenshots were captured for idle, walking/moving, front, side (east), and back (north) states across these classes; all matched expected behavior with no console errors during the session.

Not independently live-tested this pass (code-reviewed only, via the shared `drawBodyLayer`/`weaponDepthLayer` chokepoint every class routes through identically): Assassin, Bard (instrument path already facing-aware and unchanged, same as Druid), Paladin, Warden, Ranger (both class-locked via `classLocksWeaponArt`, so their own already-facing-aware `drawBackProp`/`drawFrontProps` continues to own their weapon art untouched by this phase), Wizard, Enchanter, Shaman, Summoner, Necromancer - all of these share the exact same `drawBodyLayer(ctx, rig, 'weapon'/'offhand', arg)` chokepoint as Cleric/Fighter, so the fix applies uniformly; see Manual Tests Needed for the remaining classes.

## Weapon identity/stats/combat confirmation

No changes were made to `slotItem`, `resolveVisualSlotItem`, `weaponKind`, `drawWeaponShape`, item data, equipment data, combat/damage code, or auto-attack timing. `render/humanoid-base-renderer.js`'s `drawSharedAutoAttackOverlay` (the V0.16.91-established trail/impact-only overlay) was not touched and still fires immediately after the front-pass weapon draw, unchanged.

## White-stick / global-crossbow / arm-stretch regression check

None of the functions those V0.16.90/V0.16.91 fixes touched (`drawWeaponShape`, `weaponKind`, the auto-attack overlay's trail-only rendering) were modified in this pass - only the position of the pre-existing, unmodified `drawWeapon`/`drawOffhand` calls changed. Live-tested Cleric and Fighter auto-attack-adjacent states (idle/moving) showed normal weapon art with no placeholder geometry or stretched limbs.

## Known limitations / manual tests needed

- Side-facing near/far weapon depth (beyond the existing arm-level split) was explicitly not implemented this pass - see the scope decision above.
- The remaining 10 classes not live-tested this pass (Assassin, Paladin, Warden, Ranger, Wizard, Enchanter, Shaman, Summoner, Necromancer, plus Bard's paperdoll-equipped-weapon case) should be spot-checked in-browser per the phase brief's manual test checklist, though they all route through the identical, now-fixed chokepoint.
- Live auto-attack mid-swing while back-facing (an actual hostile-target combat exchange) was not captured this pass - `weaponDepthLayer`'s `cast`/`attack` front-pass exception was verified by code, not by a live combat screenshot.
- Bot/mercenary-rendered entities (as opposed to the player character) were not independently live-tested, though they render through the exact same `Base.draw()`/`drawBodyLayer` path.
