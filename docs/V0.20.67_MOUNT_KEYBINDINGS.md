# V0.20.67 — Mount/Taming Keys: All Three Collided

## What was reported

> "M is for Map, we need a different button for mounts!"

## What auditing the whole table found

Swapping one letter would have left two more collisions in place. Checking every binding:

| my key | already bound to |
|---|---|
| `m` | **map** |
| `h` | **hireMerc** |
| `t` | **portal** |

**All three of my choices were taken.**

## The real bug — why they could collide at all

`taming-system.js` and `mount-system.js` checked raw `event.key` letters:

```js
if (key === 'h' && !event.ctrlKey) { ... }      // mount
if (key === 'm' && !event.ctrlKey) { ... }      // panel
if (key.toLowerCase() === 't' && ...) { ... }   // tame
```

This bypassed `Game.defaultKeyBindings` / `bindingForAction` — the single table that defines every other
control. Two consequences:

1. **The collisions were invisible.** Nothing compared these keys against anything, because they were
   never in the table being compared.
2. **The actions never appeared in the remapping UI.** Roadmap Item 20 requires remappable hotkeys, and
   this game already has full remapping — three actions were simply outside it.

Picking a different hardcoded letter would have fixed the symptom and left the mechanism intact.

## The fix

- Registered `tameBeast` / `mountToggle` / `mountPanel` in `defaultKeyBindings`.
- Defaults **`z` / `x` / `y`** — the only free letters. The table already uses every other letter except
  `q` (a/b/c/d/e/f/g/h/i/j/k/l/m/n/o/p/r/s/t/u/v/w all bound).
- Added to `keyBindingLabels` under *Interactions / Professions* (tame, mount) and *Windows / UI* (stable),
  so they show in settings and can be rebound.
- Both systems switched to `game.isActionKey(event, ...)`.
- The panel hint renders `bindingForAction()` instead of hardcoded letters — a baked-in "Press H" would
  go stale the moment anyone rebinds.

## Verified

| check | result |
|---|---|
| duplicate bindings across the whole table | **only `space`** (jump + ascend) — intentional, pre-existing |
| new actions listed in settings | ✅ all three |
| **Y** opens the stable | ✅ |
| **M** no longer touches mounts | ✅ |
| hint reflects live bindings | ✅ "Press Z … Press X" |
| suite | ok, 396/396, 0 bait issues, 0 console errors |

## Controls

| key | action |
|---|---|
| **Y** | open the stable |
| **X** | mount / dismount |
| **Z** | tame the targeted beast |
| **1 / 2 / 3 / 4** | soothe / hold / feed / give ground (during a taming attempt only) |

The four answer keys stay fixed: they are modal, captured only while an attempt is running, and shown
on screen throughout it.

## Lesson

I added three controls to a game with a keybinding system without using it. The collision the user hit
was one of three, and the other two would have surfaced later as separate bug reports. **When a codebase
already owns a concern, going around it does not just risk a conflict — it removes the mechanism that
would have reported one.**
