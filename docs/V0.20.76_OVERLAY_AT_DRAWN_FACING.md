# V0.20.76 — The Overlay Is Measured at the Facing It Is Actually Drawn At

Three reports: front view still has a slight clipping issue; the back view still shows the head; side-on
still shows the whole chest and both arms.

## 1. Front — the clip rect was aimed 44px off the head

`mountOverlayBox` probed the beast at a **canned** facing — straight south for front, straight north for
rear — while the game draws **diagonals**.

| black wolf | head centre | head width | head span |
|---|---|---|---|
| probed `south` | dx **−11** | 64px | −84..−54 |
| drawn `southeast` | dx **+33** | 43px | −83..−61 |

A **44px** horizontal error. The rect clipped one side of the head off and repainted blank canvas across
the rider's chest.

**This is the fourth time this exact mismatch has caused a mount bug:**

1. **V0.20.65** — which way the *animal* faced
2. **V0.20.69** — whether the riding posture reached the model
3. **V0.20.73** — which seat the *rider* adopts
4. **V0.20.76** — where the overlay is cut

Fixed at the measurement this time: `mountOverlayBox(def, scale, mode, facing)` takes the entity's real
facing via `mountProbeFacing(e)` and caches per quantised heading, so the box always describes the
silhouette that reaches the screen.

## 2. The ×1.55 stretch is removed

Added in V0.20.74 to reach a muzzle the box appeared to be cutting off. But the muzzle was never the
problem — the box was simply aimed at the wrong place. With measurement corrected the multiplier was pure
over-reach:

| | box | head |
|---|---|---|
| with ×1.55 | −86..**−39** | −83..−61 |
| now (+9px jaw pad) | −86..**−53** | −83..−61 |

It ran 22px below the head, repainting the animal's neck across the rider's thighs — the reported "slight
clipping issue" that survived V0.20.75.

**A workaround layered on a bad measurement hid the bad measurement.** Removing it was only safe once the
measurement was right.

## 3. Side view — half the body

`buildRidingRig` kept `shoulderW`/`hipW` at full breadth side-on. V0.20.74 moved both **hands** to the near
side, but the chest still read face-on with an arm hanging off each edge, which is what "still looks like
they're sitting sideways" was pointing at.

Spans now compress to **0.46** side-on — **7.6–11.5 against 22** in front (35–52%). This is what actually
turns the body into a profile: it pulls the far shoulder behind the near one so the far arm tucks against
the torso instead of framing it. The standing rig has always done this via its own `sideCompress`; the
riding rig never did.

## 4. The back was already fixed — and the measurement proves it

The true back heading (`northwest`, the only one resolving to camera view `back`) measures **5.6% black,
90.9% gold** in the rider's torso band. An ASCII colour map of the live frame shows the tabard clean from
dy −100 to −56 with no black intrusion.

The reported screenshot matches the **V0.20.74** rear box (−90..−49, which was the head almost exactly),
fixed in V0.20.75. Worth confirming the running file is the current one.

## The measurement itself — why four versions of "I can't verify this"

**This is the first version whose render claims come from real pixels.** Sampling had failed silently
since V0.20.71, and the cause was never a render fault.

The world camera transform is applied to the **context**, so there are three distinct spaces:
`worldToScreen()` output, the `foot` argument inside `drawEntity`, and canvas pixels. Measured live:

```
transform: a = d = 0.9216,  e = -135.07,  f = -127.61
foot (670, 633)  ->  canvas (482, 456)
```

Every window aimed with `worldToScreen()` or the raw `foot` missed the character by ~190px in x and ~177px
in y, returning near-zero **with the change both ON and OFF**. Zero in both arms reads like "no fault"
but means "not sampling the character" — which is why the failure survived so long.

Fix: wrap the draw call, capture **both** the `foot` argument and `ctx.getTransform()`, then map
`canvasX = m.a*(foot.x+dx) + m.e`. Note `ctx` is `window.ctx`; `this.ctx` is undefined on the game object.

Two compounding traps, both previously recorded: the canvas is `{alpha:false}` so alpha metrics are
meaningless (use RGB darkness), and the browser pane is `document.hidden` so pane screenshots return
**stale** frames — only a forced `g.render()` followed by `getImageData` reflects current code.
