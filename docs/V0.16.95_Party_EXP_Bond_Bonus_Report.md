# V0.16.95 - Party EXP Bond Bonus Report

## Phase source

- Phase brief: user-supplied "Feature Rule" message specifying the Party EXP Bond Bonus formula, validation cases, and patch note text.
- Baseline used: `Blackroot V0.16.94`
- Target produced: `Blackroot V0.16.95`

## Audit before implementation

Player EXP was **not** centralized before this pass — `player.xp` was incremented directly at four independent sites, each re-implementing the same "human race +2%" bonus:

1. `systems/combat-system.js` `killEntity` (solo-kill fallback path).
2. `systems/party-system.js` `awardPartyEnemyXp` (party kill-split path).
3. `systems/party-system.js` `receivePartyReward` (multiplayer party-broadcast XP share).
4. `systems/quest-system.js` `completeQuest` (quest reward XP).

This phase introduces one canonical award function and routes all four sites through it, per the "Applied the bonus through the central EXP award path" requirement.

## Ownership

- **Central XP award**: `Game.prototype.awardPlayerXp(baseExp, options)` in `systems/combat-system.js` (placed immediately before `killEntity`, which already owns "shared XP/bot XP routing" per that file's own header comment). It applies **only** the new party-size bonus and increments `player.xp` — it does not re-derive the pre-existing human-race +2% bonus, which each call site still computes exactly as before and passes in as `baseExp`. This was a deliberate choice: folding the race bonus into the same multiplication as the party bonus would have compounded them (e.g. 100 base -> 107 for a human with 5 party members) and broken the phase's own validation case (100 base, 5 members -> exactly 105). Keeping the two bonuses on separate, additive-to-input stages guarantees the literal formula and both validation cases hold regardless of player race.
- **Active party member count**: `Game.prototype.getActiveExpBonusPartyMemberCount()` in `systems/party-system.js`, added directly after the pre-existing `Game.prototype.getPartyCombatMembers` (reused, not duplicated — it already resolves alive, same-zone, currently-active party members across bots/mercs/pets/remote players). The new helper filters that list to `actor !== this.player && type !== 'merc'` and clamps to 5.

## Design decision: mercenaries do not count

The codebase has an explicit, repeated, well-documented convention that mercenaries (and pets) are auxiliary followers that do **not** consume party slots (`Game.prototype.partyMemberConsumesSlot` in `systems/party-system.js`: *"party capacity is reserved for real player characters and bot party members only... Mercenaries and pets... never consume an invite/member slot."*). Since the phase brief's "party max size 6, player counts as one, max 5 non-player members" language describes exactly this slot-based party (bots + remote human players), the bonus counts only slot-consuming members and excludes mercenaries/pets. This matches the patch note's "Added safe handling for... mercenary party members" line — mercenaries are safely and intentionally excluded rather than silently mis-counted.

## Formula implementation

```js
Game.prototype.awardPlayerXp = function(baseExp, options = {}) {
  const base = Math.max(0, Math.floor(Number(baseExp) || 0));
  const activeBonusMemberCount = Math.min(5, Math.max(0, this.getActiveExpBonusPartyMemberCount?.() || 0));
  const partyBonusPercent = activeBonusMemberCount * 0.01;
  const finalExp = Math.floor(base * (1 + partyBonusPercent));
  this.player.xp += finalExp;
  ...
};
```

This is the exact formula from the phase brief. `getActiveExpBonusPartyMemberCount` itself also clamps to 5 (defense in depth), so stale/bad party data that reports 6+ members cannot exceed the +5% cap.

## Call sites updated

All four sites now compute the pre-existing race bonus exactly as before, then call `awardPlayerXp` (or `game.awardPlayerXp` in quest-system.js) with the race-adjusted amount, and use the function's returned `logText`/`amount`/`partyBonusPercent` for their existing log lines instead of re-deriving the awarded amount:

- `systems/combat-system.js` `killEntity` — solo-kill branch.
- `systems/party-system.js` `awardPartyEnemyXp` — player's share; the returned `logText` now includes both the party-kill-split suffix and, when active, a "Party Bond Bonus: +N%" suffix.
- `systems/party-system.js` `receivePartyReward`.
- `systems/quest-system.js` `completeQuest` — also fixed a pre-existing display inconsistency where the final log line showed the raw unmultiplied `rewards.xp` instead of the amount actually added to `player.xp`.

`checkLevelUp()` (the existing Combat Level-Up Popup owner, `systems/combat-system.js`) was not modified — it only reads `player.xp`/`player.nextXp` and needed no changes; it is still called from the same places, after XP is awarded, exactly as before.

## Validation

```
find . -name "*.js" -print0 | xargs -0 -n1 node --check
find . -name "*.json" -print0 | xargs -0 -n1 python3 -m json.tool > /dev/null
python3 -m json.tool _script_order.json > /dev/null
```

All passed with zero errors. The formula was additionally verified in isolation with a standalone Node script against both required cases:

- Base EXP 100, 5 active non-player party members -> **105** (matches spec).
- Base EXP 100, 7 members (stale/bad data) -> clamps to 5 -> **105** (matches spec, confirms overflow safety).
- Base EXP 100, 0 members (solo) -> **100** (+0%, no bonus for playing alone, per spec).

## Known limitations

- No new file was created for this phase (`systems/combat-system.js` and `systems/party-system.js` already owned the relevant logic); this keeps the change surgical per the phase's "preserve project structure" rule.
- The human-race +2% bonus was left exactly as it was (same formula, same call-site placement) — only its *result* is now passed into the new central function rather than being applied to `player.xp` directly. No combat/quest/party XP amounts change for players who were not previously eligible for a party bonus.
