Dream Realms V0.15.84 — Phase 6: Map Editor Parity + Patch Notes Correction
==============================================================================

Scope: this phase started as the originally-planned Phase 6 (map/layer/
attribute/warp/resource/crafting parity per docs/PHASE_5_EVENT_QUEST_
CONDITION_VARIABLE.txt's "next recommended phase"), then was redirected
mid-phase by an explicit user interrupt requiring two corrections before
continuing: (A) stale/missing patch notes, and (B) an Intersect-style map
editor layout/tool pass. Both are covered below. Phase 7 was NOT started -
see section 8.


1. PART A — PATCH NOTES CORRECTION
--------------------------------------
Audit: searched the whole project for "patch"/"release"/"changelog"/
"VERSION"/"README"/launcher or in-game panel owners. Found the canonical,
already-wired-up owner: two static HTML `<ul><li>` lists in
`Dream Realms V0.15.52.html` - `#splashPatchNotesPanel` (302 entries, title
screen) and `#menuPatchNotesPanel` (208 entries, in-game Settings/menu),
both toggled by existing buttons/JS in game.js/systems/ui-system.js. Both
lists were stale: their newest entry was V0.15.78, meaning Phases 0-5
(V0.15.79 through V0.15.83) were invisible to a player using the in-game
Patch Notes panel, even though the underlying work was real and already
shipped.

The two panels are independently-authored, not templated from one source
(confirmed: different entry counts, 302 vs 208) - a pre-existing, lower-
risk duplication not resolved this phase (see Known Limitations). Fixed by:
- Adding one prose entry per shipped version (V0.15.79-V0.15.83) to the top
  of BOTH panels, in the same descriptive-sentence style as every existing
  entry, summarizing what actually shipped (verified against
  docs/PHASE_1-5_*.txt and core/config.js's changelog comments - nothing
  invented).
- Creating `PATCH_NOTES.md` at the project root: the structured, canonical
  Added/Changed/Fixed/Validation/Known-Limitations reference the brief
  requested, covering every version to date. This is the detailed
  companion to the in-game prose panels, not a competing/duplicate system -
  the in-game HTML remains the actual UI-connected owner (this file is
  static, hand-authored to match it, not fetched at runtime - see Known
  Limitations for why).
- `VERSION.txt`/`README.txt`/`core/config.js` were already in sync with
  each other (all reported V0.15.83) before this phase began; they are
  bumped together with the rest of this phase's version increment.


2. PART B — MAP/EDITOR AUDIT AND SCOPE DECISION
----------------------------------------------------
Before writing any editor code, read systems/editor-system.js (~5317
lines, F10, the original/legacy editor) and systems/layered-map-editor-
system.js (~456 lines, F9, the 16-layer sub-editor) end to end against the
full Intersect-style target checklist the brief specified (toolbar, map
explorer, canvas overlays, tabbed inspector, status bar, tile/layer/
attribute/warp/NPC/resource/event tools, map properties, validation).

Key findings:
- F9 and F10 are confirmed to be the SAME editor shell, not two competing
  editors: F9 forces the shell to the "Layered Map" tab and opens it; F10
  toggles the shell on whatever tab was last active. This was re-verified,
  not assumed.
- A surprising amount of the target checklist already exists in some form,
  just split inconsistently across the two panels: undo/redo, rectangle/
  fill/eyedropper/selection tools, a real fixed status bar (layered editor
  only), zoom, grid, and a substantial existing validation system
  (editor-system.js's `collectValidationDetails`/`validateEditedWorld`,
  which already checked missing NPC/resource/event/quest/item/loot-table/
  recipe/station references before this phase).
- Several checklist items are genuinely absent and would require a large,
  risky rewrite to add for real: a map-explorer tree (there is no multi-
  map/zone-list concept at all today - only 2 hardcoded zones exist), a
  tabbed form-based inspector (entity editing today is ~198 chained
  `window.prompt()` calls, not panel forms), per-layer opacity/drag-
  reorder, and roughly ten requested attribute types with NO backing
  gameplay system at all (No Mount, No Combat, Slide/forced-movement,
  Sound Trigger, Player Light, Fog, Item Spawn) - adding paint tools for
  these would be the exact "vocabulary exists, nothing consumes it"
  anti-pattern this whole effort has been fixing in every prior phase, not
  introducing.

Given this, the user was asked to confirm scope before any code was
written (a genuine judgment call, not a unilateral decision) and chose the
"safe subset" option: extend real, working validation/overlay/tool gaps
in the two existing editor owners, and explicitly defer the large
structural items rather than attempt them destabilizing-risk-first.


3. WHAT WAS IMPLEMENTED
-------------------------

a) Dungeon zone-key bug fix (systems/layered-map-editor-system.js)
   `zoneKey()` had no branch for `game.currentZone === 'dungeon'` -
   confirmed via audit that this silently fell through to `'dark_woods'`,
   meaning every layered-map attribute lookup while inside a dungeon
   (including `Game.prototype.isWalkable`'s `layeredMapCanEnter` check,
   systems/collision-system.js:70) was checking Dark Woods' painted
   Blocked/NPC Avoid data at the numerically-same tile coordinates - a
   wrong-zone data lookup, not merely a missing one. Fixed by returning
   the dungeon's own id (via the already-established `DR.Ids.
   currentZoneId(game)` helper, reused rather than re-derived) when in a
   dungeon. Since no layered map exists for any dungeon id, this makes the
   check correctly find no data (safe no-op) instead of consulting an
   unrelated zone's paint. Dungeon walkability itself is unaffected either
   way - it was already independently enforced by
   systems/dungeon-system.js's own tile.blocked/reachable-tile checks,
   which never consulted the layered map to begin with (a related, lower-
   priority gap - see Known Limitations).

b) Dungeon/puzzle descriptor registry validation (core/registry.js)
   Added `dungeon`/`puzzle`/`dungeonRoom` lookup kinds (over the existing
   `DR.DUNGEON_BY_ID`/`DR.PUZZLE_BY_ID`/`DR.DUNGEON_ROOM_BY_ID` exports -
   data/dungeons.js remains the sole data owner) and cross-reference checks
   in the boot validation report: `dungeon.bossIds[]` -> boss,
   `dungeon.puzzleIds[]` -> puzzle, `dungeon.lootTableId` -> lootTable,
   `dungeon.keyItemId` -> item, `dungeon.caveZoneId` -> zone, and
   `puzzle.dungeonId` -> dungeon. `zoneId`/`entranceZoneId` are
   deliberately NOT checked against the zone table - they are floor-
   composite ids (e.g. "blackroot_catacombs_f4") that never exactly match
   a plain zone id, so validating them would be a guaranteed false
   positive. A dry run against the real project data found zero new
   warnings (all dungeon/puzzle references resolve correctly) - this
   addition is confirmed clean, not just theoretically safe.

c) Editor validation additions (systems/editor-system.js
   `collectValidationDetails`)
   Added checks that did not exist before: warp targets (both the legacy
   `editorAttributes` warp attribute and the layered map's own
   `attributes.warps`) for unknown target zone/map, out-of-bounds target
   coordinates, and self-loop (a warp targeting its own exact tile); mob
   spawn markers placed on a blocked tile or on a Dark-Woods cave entrance
   tile; and resource nodes placed on a blocked tile. All additions feed
   the same `report`/`add()` accumulator the existing validation already
   used - purely additive, no existing check was changed, and the report
   remains non-fatal (warnings only, never blocks saving).

d) Overlay visibility toggles (systems/editor-system.js)
   The attribute/resource/event/NPC/mob-spawn/dungeon marker-drawing
   functions in `render()` previously always ran whenever the editor was
   open, with no way to hide any one category. Added a `this.overlays`
   state object (all six keys default `true`, preserving the exact prior
   always-on behavior) and six new toggle buttons/actions
   (`toggleOverlayAttributes` etc.) that flip one key each; `render()` now
   checks the corresponding flag before each draw call. Button labels
   ("Attributes: On"/"Off") are kept in sync every status refresh.

e) "Validate Only" bug fix (systems/editor-system.js `runAction`)
   The "Validate Only" button has existed in the panel's HTML with
   `data-editor-action="validateOnly"`, but `runAction` had no matching
   branch - clicking it did nothing at all. Fixed by adding
   `else if (action === 'validateOnly') this.validateEditedWorld({repair:
   false});`, directly relevant to strengthening the same validation panel
   this phase already extends.

f) Warp self-loop prevention + Clear Layer (systems/layered-map-editor-
   system.js)
   `saveWarpDialog()` had bounds/target-exists validation but no self-loop
   guard - a warp could target its own exact tile on its own map, trapping
   a player in a zero-distance teleport loop. Added a check rejecting
   that case with a clear log message; existing bounds/target-exists
   validation is unchanged. Also added a new `action_clearLayer()` tool
   (button: "Clear Layer") that empties every tile on the active layer in
   one action - previously the only way to remove tiles was per-cell erase
   or overwriting a selection. Respects the same per-layer lock flag
   `action_cut` already honors, and goes through the existing transaction/
   undo system correctly (each cleared cell is recorded via `setLayer`,
   not a direct dict wipe - a bug caught and fixed during this phase's own
   testing, see Validation Results).

6. WHAT WAS AUDITED BUT DELIBERATELY NOT CHANGED
------------------------------------------------------
- Resource node concurrency ("claimed by" locking so two actors can't
  double-harvest the same node): a real gap, but fixing it in isolation
  would be inconsistent with the rest of the codebase's already-
  documented, accepted non-authoritative local-multiplayer model (Phase 0
  audit's "Prioritized risk list for a future authoritative-server
  phase") - the same gap exists for combat kills/loot rolls and was
  explicitly deferred there too. Not fixed here to avoid a partial,
  inconsistent fix.
- Dungeon actor/mob placement wiring into the layered map's attribute
  check (the same category of fix as (a) above, but for
  `dungeon-system.js`'s `isValidDungeonActorTile`/`dungeonTraversalTile`):
  deferred because no layered map exists for any dungeon today, so the
  fix would be 100% inert in practice, and threading a new parameter
  through several tightly-coupled function signatures in a 3255-line file
  for zero present-day behavior change was judged higher risk than value
  for this pass.
- `noSpawnNearTown` (a vestigial spawn-group flag, found in Phase 4):
  still not wired up - out of this phase's file scope.
- The two independent patch-notes lists (splash vs menu panel, 302 vs 208
  entries): kept as two hand-synced lists rather than unified into one
  templated source, since doing so safely would require introducing a
  small runtime templating mechanism to a currently 100%-static HTML file
  - judged out of proportion to the actual problem (both are now
  correctly caught up through V0.15.83; keeping them in sync going
  forward is a process discipline, not a code gap).


7. VALIDATION RESULTS
-----------------------
- `node --check` on every changed `.js` file: 0 syntax errors.
- Full-project sweep (`find . -name "*.js" ... node --check`, excluding
  vendor/three.min.js and the bundled blender/ tooling): 0 syntax errors.
- HTML sanity check: `<li>`/`</li>` tag counts remain balanced (520/520)
  after the patch-notes edits.
- Built 47 automated test cases across four Node harnesses (including a
  purpose-built lightweight DOM/canvas stub with a small regex-based
  innerHTML "parser," since these two editor files build their UI via
  template-string innerHTML rather than `document.createElement`+
  `appendChild`, so real `querySelector`/`querySelectorAll` calls needed
  something to find):
  - 6 cases for the `zoneKey()` dungeon fix: confirms overworld/cave
    resolution is byte-identical to before; confirms a dungeon zone no
    longer resolves to `'dark_woods'` and instead resolves to the real
    dungeon id; confirms `layeredMapAttributeAt`/`layeredMapCanEnter`
    correctly treat the dungeon zone as "no map data" (safe pass) rather
    than checking Dark Woods' paint.
  - 7 cases for the new dungeon/puzzle registry cross-references: valid
    references produce no warning, each of the 5 broken-reference types
    (boss/puzzle/lootTable/item/zone) is individually flagged with the
    correct id in the message, and a valid `puzzle.dungeonId` is not
    falsely flagged.
  - 18 cases against the real, installed editor-system.js runtime
    (through its actual `registerDreamRealmsSystem`/`install(game)`
    bootstrap, not a reimplementation): every new warp-validation branch
    (unknown zone, unknown map, out-of-bounds x2, self-loop x2, valid x2,
    disabled-warp-skipped); mob-spawn-on-blocked-tile and mob-spawn-on-
    cave-entrance (plus confirming the check clears once the tile is
    fixed); resource-on-blocked-tile; the `validateOnly` bug fix (confirms
    `lastValidation` actually populates now); and the overlay toggle
    system end-to-end (default all-on draws all 6 categories, toggling one
    off skips exactly that draw call and no others, toggling back on
    restores it).
  - 10 cases against the real, installed layered-map-editor-system.js
    runtime: Clear Layer empties the active layer, creates exactly one
    undo entry, marks the map dirty, and undo correctly restores every
    cleared tile; a locked layer is not cleared; clearing an already-empty
    layer does not push a spurious undo entry; a self-loop warp (same map,
    same tile) is rejected with a clear log message and does not mutate
    the warp table; a valid warp to a different tile on the same map is
    still accepted normally.
  All 47 cases passed. One real bug was caught BY this testing process
  itself before being shipped: the first draft of `action_clearLayer`
  mutated the layer's tile dictionary directly, which silently bypassed
  the undo-recording system (an empty transaction is discarded, not
  pushed to the undo stack) - rewritten to clear tiles one at a time
  through the existing `setLayer()` function, which correctly records
  each change, exactly matching the established pattern `action_cut`
  already uses.

9. KNOWN LIMITATIONS / FOLLOW-UPS FOR A FUTURE PHASE
--------------------------------------------------------
- This was verified via Node-based test harnesses against the real edited
  files (including a custom lightweight DOM stub), not a live browser
  session (this environment cannot run a browser). Actually opening F9/
  F10, clicking the new overlay/validate/clear-layer buttons, and painting
  a real warp in the live editor are things only a live session can truly
  confirm - see the test checklist in README.txt.
- Full Intersect-style editor layout parity was NOT attempted this phase
  (by explicit, confirmed user choice): no map-explorer tree (there is no
  multi-map concept to explore - today there are exactly 2 zones with
  layered map data), no tabbed form-based inspector (entity editing
  remains ~198 `window.prompt()` calls), no per-layer opacity/drag-
  reorder, and none of the ~10 requested attribute types with no backing
  gameplay system (No Mount, No Combat, Slide/forced-movement, Sound
  Trigger, Player Light, Fog, Item Spawn, Zone/Region marker,
  Respawn Point as a distinct concept from spawnPoint). Building these for
  real is a substantially larger, separate undertaking than this phase's
  "smallest safe" additions.
- Dungeon actor/mob placement still does not consult the layered map
  (section 6) - currently inert either way since no dungeon has layered
  map data, but would need real wiring if dungeon layered maps are ever
  authored.
- Resource-node multi-actor concurrency locking was not implemented
  (section 6).
- The two patch-notes lists remain two independently-maintained static
  HTML blocks, not unified into one templated source (section 6).
