## Correctness method

The spec's prose field names don't always match the real spell fields (as found with Paladin:
`tickDamage` vs `power`, `absorbFlat` vs `absorb`). So before authoring, the exact spell **names** and
**patchable fields** for every class were dumped from `data/spells.js` (which mixes quoted and
unquoted object keys — a parser quirk, not a gameplay one). Every `k:'spell'` node was then mapped to a
field that actually exists on its target spell; where the spec referenced a non-existent field or an
un-expressible mechanic (armor-break %, "2 traps active", stun duration), the node was adapted to the
nearest real field or a stat/mod, preserving intent.

An **automated in-browser validator** then checked all 294 nodes against the compiled books:
- every `empowers` entry and every `k:'spell'` match name resolves to a real spell,
- every patched field exists on its matched spell (or is a universal compiler field),
- every `k:'stat'` stat is in `DR.STAT_KEYS`, every `k:'mod'` is a known mod, every capstone trigger is
  wired. **Result: 0 mismatches.**

## Small engine additions

- **`fireTalentProc` reentrancy guard** — a proc's own effect can no longer trigger another proc
  (balance rail: no `onKill→onKill` / `onHealCast→onHealCast` chains).
- **`onHealCast` trigger** — fired from `healEntity` for the player healer (guarded against
  proc-triggered heals) so the Cleric/Druid healer capstones have a trigger.
- **`talentPetMultiplier` + `damageEntity` pet hook** — Summoner/Necromancer `petDamage` mods now scale
  the player pet's outgoing damage (owned by the player's talents, per spec).
- `damageTarget` proc effect now falls back to `ctx.source`, so `onDamageTaken` capstones retaliate
  against the attacker.

## Capstone fidelity (deferred to the balance/polish pass)

Capstones are implemented with the existing proc vocabulary — status buff (`damageTakenMultiplier` /
`holyDamageMultiplier` / `lethalSave` / `moveSpeedMultiplier`), timed `dynamicMod`, `healPct`/`manaPct`,
and `damageTarget`. The spec's more elaborate mechanics (detonating all DoTs, chaining lightning,
summoning one of each element, re-stealth) are approximated as **bounded procs with the correct trigger
and a visible tell** (ring + log). Making them fully faithful is part of the later balance pass.

Per-rank node values are provisional and sit inside the engine's clamps; the balance sim pass re-tunes
them.
