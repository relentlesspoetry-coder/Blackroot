# V0.20.2 ‚Äî Your Second Character Is Their Own Person

**Reported:** creating a new character on an account takes the first character's gear and items ‚Äî a
level-1 Fighter at **Gear Score 50** wearing a Rare Silkfang Dagger, a Set Piece Webbed Queen Earring
and a Warden Lantern Amulet.

## Root cause ‚Äî a character's belongings never lived on the character

`entities/player.js` **has no inventory at all**. Every one of these hangs off the **Game**:

```
this.inventory   this.equipment   this.bags   this.bank
this.classStartingGearGranted
this.questSystem.state   this.craftingSystem.state   this.resourceGatheringSystem.state
this.fog
```

`game.js start()` ‚Äî which its own comment calls *"the single owner of world entry"* ‚Äî does:

```js
this.player = new Player(className, appearance);
```

‚Ä¶and leaves every one of them exactly where it was. **A fresh Player is not a fresh character.**

### Why loading worked and creating didn't

The **load** path got away with it purely by an accident of ordering (`save-system.js:1789-1790`):

```js
this.start(‚Ä¶);                 // fresh Player, stale belongings
this.applyCharacterState(‚Ä¶);   // ‚Üê overwrites them with the SAVED character's
```

The **create** path (`ui-system.js:2747`, the Enter Realm button) calls `start()` and has **nothing to
overwrite them with**. So character 2 simply kept character 1's things.

`resetRuntimeSessionForCharacterSwap()` is **not** this and never was ‚Äî it clears actors, party, targets
and zone (the *session*) and deliberately says nothing about what a character *owns*. The create path
doesn't call it either, but calling it would not have fixed this.

### Second order ‚Äî the bug gave *and* took away

`classStartingGearGranted` also survived, so `grantClassStartingGear()` early-returned and the new
character was **denied its own starter kit** while standing there in someone else's rares.

## The leak was wider than reported

Found by **reproducing** it, not by assuming. A new character also inherited:

| leaked | |
|---|---|
| inventory | ‚úî reported |
| equipped items | ‚úî reported |
| bags | ‚úî reported |
| **bank** | not reported |
| **completed quests** | not reported |
| **profession levels** | not reported |
| **explored fog map** | not reported |

## The fix ‚Äî one owner, mirroring the restore

`Game.prototype.resetCharacterOwnedState()` in `systems/save-system.js`, placed **directly above
`applyCharacterState()` as its deliberate mirror**: whatever that restores per character, this clears.
A field added there and not here becomes the next leak ‚Äî that's why they live together.

Called from `start()`, before the new Player is built. Each system is reset via **its own
`importState(null)`** ‚Äî the same entry point `applyCharacterState` uses with saved data ‚Äî so a reset
cannot drift from a load. Fog is cleared and `_fogLoaded` re-armed so the new character reads its **own**
`dream-realms.fog.v1:<characterId>` key.

The load path is unaffected: `applyCharacterState()` still runs immediately after `start()` and restores
the saved character over the top.

### Test hygiene

My reproduction wrote a fake quest completion into the **global** `dream-realms.quest-runtime.v1` key
(via `questSystem.saveState()`). It was **cleaned up** by resetting that key through the system's own
defaults, and `dream-realms.accounts.v1` was checked and contains **no test characters**.

## Found while verifying ‚Äî NOT fixed here (see V0.20.3)

`render/object-renderer.js` defines **`drawOrbWeb` twice**: line 608 (options form) and line 2269
(positional form, taking `rnd` as an argument). **The later definition silently wins**, so the options
form is dead code and its three callers ‚Äî including `drawWebbedBush` ‚Äî pass an object where a number is
expected, leaving `rnd` undefined and throwing **`rnd is not a function` every frame**.

Confirmed **pre-existing**: `object-renderer.js` is byte-identical to the shipped `Blackroot V0.20.1.zip`
backup and untouched by this session. Webbed bushes have not been drawing.

## Related known issue (not in scope)

`dream-realms.quest-runtime.v1`, and the crafting/gathering equivalents, are **global** localStorage keys
rather than per-character. The reset above clears them on character creation and `applyCharacterState`
overwrites them on load, so both paths are now correct ‚Äî but the storage key itself is not
character-scoped the way `fog` is, which is a latent version of the same ownership confusion.
