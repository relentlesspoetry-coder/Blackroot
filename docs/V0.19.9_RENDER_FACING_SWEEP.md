# V0.19.9 — The Pose Nobody Could Reach (render-layer facing sweep)

Sweeping V0.19.7's facing bug through the render layer found **one more instance — and it was hiding a
whole authored feature.**

## The find

`render/entity-renderer.js` → `standingDirectionProfile(e)`:

```js
const fy = Math.sign(e.facingY || 1);
```

This makes `fy === 0` **impossible**. `facingY` is legitimately `0` when an entity faces due **east** or
**west**, and `0 || 1` is `1`.

The function's own key selector has an explicit `fy === 0` branch:

```js
const key = fy < 0 ? (…'north'…)
  : fy > 0 ? (…'south'…)
  : (fx < 0 ? 'west' : 'east');   // <- UNREACHABLE
```

So the authored `'east'` and `'west'` profiles were **dead code that could never render**:

```js
east: { sideProfile: true, torsoTurn: 1, headTurnX: 7, visibleEyeCount: 1, … }
west: { sideProfile: true, torsoTurn: -1, headTurnX: -7, visibleEyeCount: 1, … }
```

Every entity facing straight east or west was drawn as a **southeast/southwest three-quarter diagonal**
(`sideProfile: false`, 2 eyes).

**Same `||` defect as V0.19.7, bigger blast radius.** It didn't merely mis-aim a value — it made an
entire authored feature unreachable, and left the code for it sitting in plain sight looking finished.

## Why the fix is right — checked, not assumed

I nearly talked myself out of this one. The override at `entity-renderer.js:1904` converts **north/south**
into side profiles when the camera says `'side'`, which suggested world facing and screen appearance are
decoupled — and that forcing `'east'` might paint a side profile where the camera shows a front view.

Reading `entityFacingView()` settled it:

- It does the camera maths properly — rotate by `camera.yaw`, project to iso, classify. **Notably it uses
  `??`, not `||`**, so it handles `facingY === 0` correctly.
- At the default yaw, the iso projection resolves **all four cardinals** to `mode: 'side'`.
- That is *exactly why* north/south need the override — and the override's condition
  (`profile.sideProfile === false`) **deliberately excludes** east/west, because those **already had side
  profiles authored**.

All four cardinals were meant to end up as side profiles. East/west simply never got there.

**A missing facing must still default to south**, so the fix defaults only a **non-finite** value —
`Number.isFinite(rawFy) ? rawFy : 1`. Zero is a real value.

## Also fixed — the fishing cast that turned you

`systems/fishing-system.js` → `setPlayerFishingPose`:

```js
p.fishingTargetX = safeNumber(target?.x, p.x + (p.facingX || 1) * 2);
p.fishingTargetY = safeNumber(target?.y, p.y + (p.facingY || 0) * 2);
…
p.setFacingFromDelta?.(p.fishingTargetX - p.x, p.fishingTargetY - p.y);   // <- fed the mistake back
```

The untargeted cast aimed 45° off when facing north/south — and the next line fed that wrong heading into
`setFacingFromDelta`, **physically turning the player**. Now uses `actorFacingVector`.

## Deliberately NOT changed — four false positives

The idiom is a **clue, not a verdict**. Four sites match the pattern and are **not bugs**:

| site | why it's fine |
|---|---|
| `humanoid-animation-system.js:385` | `direction.side \|\| 0 \|\| (source.facingX \|\| 0)` — defaults to **0**, the value being defaulted |
| `fishing-system.js:289-290` | `facingX \|\| 0`, `facingY \|\| 0` — same |
| `silk-web-spider-procedural-model.js:76` | `facingX \|\| 1`, but `fx` is only tested as `fx < 0 ? -1 : 1` — **0 and 1 give the same result** |
| `dark-woods-mob-procedural-model.js:127` | identical |

Pattern-matching the idiom would have churned **three files for zero behavioural gain**. The memory note
listing five "unfixed" render sites was itself over-broad, and has been corrected.

## Where the facing thread stands

**Closed.** Every real instance of the `||` facing defect found across V0.19.7 and V0.19.9 is fixed; the
remaining matches are verified false positives. What it cost, once counted: six mis-aimed abilities
(including shipping Cleave and Backstab), a fishing cast that turned the player, and two authored poses
that could never render.
