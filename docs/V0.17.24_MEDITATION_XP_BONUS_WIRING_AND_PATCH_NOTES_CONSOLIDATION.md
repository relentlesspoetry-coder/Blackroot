## Part 1: Meditation XP Bonus Wiring

### What was dormant
V0.17.23 restored Meditation skill EXP as a flat, conservative per-tick rate (`MEDITATION_TUNING.skillXpPerTick = 5`) and explicitly deferred wiring in three pre-existing config fields that had never been consumed by anything:
- `MEDITATION_BALANCE.safeZoneXpMultiplier` (1.25)
- `MEDITATION_BALANCE.greatStillnessXpMultiplier` (1.5)
- Each `MEDITATION_ENVIRONMENTS[x].xpMultiplier` (all `1` except `inn: 1.25`)

### What changed
`systems/meditation-system.js: calculateMeditationRegen` now computes an `xpMultiplier` from these three sources and applies it to the flat rate:
```js
let xpMultiplier = 1;
if (environment && Number(environment.xpMultiplier) > 0) xpMultiplier *= environment.xpMultiplier;
if (DR.MeditationHelpers.isSafeZoneActive(this, entity)) xpMultiplier *= MEDITATION_BALANCE.safeZoneXpMultiplier;
if (isGreatStillnessActive(this)) xpMultiplier *= MEDITATION_BALANCE.greatStillnessXpMultiplier;
result.xpPerTick = MEDITATION_TUNING.skillXpPerTick * xpMultiplier;
```
These are direct multiplicative factors (e.g. `1.25` = "1.25x"), so they stack multiplicatively with each other. They are intentionally kept separate from the additive HP/resource regen-percentage multiplier computed earlier in the same function - `MEDITATION_BALANCE` already splits these two concerns itself (e.g. `greatStillnessRegenMultiplier: 2.0` for HP/resource vs `greatStillnessXpMultiplier: 1.5` for XP), so reusing the regen multiplier for XP would have mixed two deliberately-different config values.

Worst case stacking (Inn environment + safe zone + Great Stillness, all simultaneously) is `1.25 x 1.25 x 1.5 = ~2.34x`, i.e. `5 x 2.34 ≈ 11` XP per tick after flooring - not dangerous, so no additional cap was added on top of the existing per-tick constant.

### Latent zero-XP bug fixed
`systems/skills-system.js: gainMeditationXp(amount)` previously computed `Math.max(1, Math.floor(Number(amount) || 1))`, which silently floored a `0` (or falsy) argument up to `1`. This had no observable effect in V0.17.23 because `xpPerTick` was always the flat constant `5` (never `0` or falsy), but it became a real correctness risk in this phase since `xpPerTick` can now be any positive multiple of `5` depending on multipliers - and, more importantly, a future environment/tuning change that intentionally sets a multiplier to `0` would have silently still granted `1` XP instead of `0`. Changed to `Math.max(0, Math.floor(Number(amount) || 0))`; `grantMeditationXp` already no-ops on non-positive amounts, so this is a pure correctness fix with no behavior change under current tuning (every currently-defined multiplier is `> 0`).

### What was deliberately not changed
- No new multiplier fields were invented - only the three that already existed in the codebase were wired in.
- No cap was added on the XP multiplier stack (the realistic worst case is small; adding one would be speculative).
- HP/resource recovery math, tick timing, and interruption rules are untouched.

## Part 2: Patch Notes Consolidation

### What was found (documented, not yet fixed, in V0.17.23's review)
Two independent, hand-duplicated copies of the entire patch-notes version history were baked as static HTML: `#splashPatchNotesPanel` (main menu) and `#menuPatchNotesPanel` (in-game escape menu). Both were stale (last entry present was `V0.17.18`, missing `V0.17.19`-`V0.17.23`), and being two separate hand-authored blocks meant every future version bump had to remember to update both or they would drift from each other in addition to drifting from `PATCH_NOTES.md`.

### Existing (unmodified) safety net discovered
`systems/ui-system.js: Game.prototype.refreshPatchNotesPanels` already existed and already runs before either panel is shown (both the splash "PATCH NOTES" button handler and the in-game menu "Patch Notes" button handler call it first). It picks whichever panel's `.patchNotesEntries` element is non-null as `canonical` (preferring the splash panel), and clones that canonical content into the other panel if their `innerHTML` differs. This function was not touched - it already implements exactly the "render both panels from one source" pattern the consolidation needed; the panels' static HTML just wasn't structured to use it as a single source.

### What changed
1. The splash panel's `.patchNotesEntries` (now the sole authored source) had five missing entries backfilled - `V0.17.23` through `V0.17.19` - transcribed directly from `PATCH_NOTES.md`, plus this phase's own new `V0.17.24` entry, all inserted above the existing `V0.17.18` entry in the same markup style already used there.
2. The in-game menu panel's entire ~2,380-line duplicated `.patchNotesEntries` content was removed and replaced with an empty shell: `<div class="patchNotesEntries" data-patch-notes-source="PATCH_NOTES.md"></div>`. No JavaScript changed - `refreshPatchNotesPanels()` already replaces an existing-but-differing `.patchNotesEntries` element with a clone of the canonical one, so the very next time either Patch Notes button is opened (by either path), the menu panel is populated from the splash panel automatically, exactly as it already did for genuine cross-panel drift before this phase.

### Why this approach
This was chosen over building a new fetch/render pipeline from `PATCH_NOTES.md` because:
- The project ships as local `file://`-opened HTML in some setups, where `fetch()` of a sibling `.md` file can be blocked by the browser's local-file CORS policy - reusing the existing DOM-clone sync avoids that risk entirely.
- It required zero JavaScript changes, which keeps this a documentation/content consolidation rather than a new code path to validate.
- It directly matches the phase-control rule against introducing duplicate systems or new patch-cascade code - the sync mechanism already existed and was already correct; the panels' content just hadn't been restructured to rely on it as the single source.

### Verification
- Confirmed via `grep`/structural read that no other code in the project reads `.patchNotesEntries` or `data-patch-notes-source` besides `refreshPatchNotesPanels` (so nothing else assumed the menu panel had static baked content).
- Confirmed both of `refreshPatchNotesPanels`'s two call sites (`game.js`'s in-game menu button handler, `systems/ui-system.js`'s splash button handler) run before the corresponding panel becomes visible, so the menu panel can never be shown empty on first open.
- Verified HTML tag balance (`<div>`/`</div>`: 209/209, `<section>`/`</section>`: 199/199) is self-consistent after the edit.
- Manually traced `refreshPatchNotesPanels`'s logic against the new empty-shell menu panel to confirm the "target exists but content differs" branch (`target.replaceWith(canonical.cloneNode(true))`) fires correctly rather than the "target missing" branch, since the shell `<div class="patchNotesEntries">` element itself is still present (just empty).

## Known Limitations
- The XP multiplier stacking model (multiplicative, uncapped) is a reasonable, conservative choice given the current small multiplier set, but has not been explicitly play-tested against a hypothetical future addition of many more environments/multipliers; if that set grows substantially, consider revisiting whether a cap (mirroring `MEDITATION_BALANCE.maxTotalRegenMultiplier` for HP/resource) is warranted.
- The patch-notes consolidation relies on JavaScript running (`refreshPatchNotesPanels`) to populate the in-game menu panel; a user who somehow views the raw HTML with JavaScript disabled would see an empty in-game panel. This matches how every other dynamic panel in this project already behaves (the whole game requires JavaScript), so it is not a new risk class, just noted for completeness.
