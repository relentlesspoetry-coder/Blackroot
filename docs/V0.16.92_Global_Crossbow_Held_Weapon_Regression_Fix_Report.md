## Root cause

The visual resolver had no hard class-validation layer between equipped item visuals and class default visuals. If a stale/mismatched equipped visual item, such as the Assassin starter crossbow, reached the paperdoll renderer for another class, that item could still be accepted as the live weapon visual. The previous white-stick fix made the renderer trust hydrated equipment metadata more broadly, which fixed missing visuals but allowed a wrong class weapon visual to bleed through when equipment/class state was stale or mismatched.

The sprite cache also used an equipment signature that ignored common runtime item identifiers such as `itemId` and `sourceItemId`. That meant different starter weapon instances with the same broad type/rarity could share cache signatures too aggressively. In a worst-case cache sequence, an Assassin/crossbow frame could be reused for another class if the class/equipment identity was incomplete.

## Explicit root-cause answers

1. **Why did all classes resolve to crossbow?**
   A stale/mismatched Assassin crossbow visual could be accepted by the generic paperdoll weapon path instead of being rejected for non-Assassin classes and replaced with the actor's own class default visual.

2. **Was it caused by a global fallback?**
   Not by a single intended `fallbackWeapon = crossbow` assignment. The effective fallback behavior was still too permissive: a mismatched equipped crossbow could act like a universal visual because it was not class-validated.

3. **Was it caused by a shared cache?**
   The cache key was a contributing risk. It did not include `itemId` / `sourceItemId`, item slot/icon family, class restrictions, or a stable entity identity. V0.16.92 strengthens the key so weapon visuals cannot bleed across classes or entities through an under-specified cache signature.

4. **Was it caused by auto-attack bypassing the paperdoll resolver?**
   No new auto-attack bypass was added in this pass. V0.16.91 already removed the attack-only white-stick overlay weapon. This pass preserves that fix and corrects the shared resolver/cache below it.

5. **Was it caused by class default weapon mapping?**
   The missing part was not the existence of class defaults, but the lack of a canonical class-default fallback inside the paperdoll resolver when equipment was missing, invalid, or mismatched.

6. **Was it caused by a mutable object reference?**
   No single mutable crossbow template was found as the direct owner. V0.16.92 still returns fresh default visual objects for class fallback items so class default visuals do not share mutable item state.

7. **What code now prevents this regression?**
   `render/paperdoll-equipment-renderer.js` now includes canonical class/default helpers, validates `classRestrictions` against the actor class, and falls back to the actor's own class default starter visual item when equipped data is missing or invalid. `systems/runtime-sprite-cache-system.js` now includes stable identity and stronger equipment signatures.

## Changes made

### `render/paperdoll-equipment-renderer.js`

- Added `CLASS_NAME_ALIASES` for canonical 14-class class-name normalization.
- Added `CLASS_DEFAULT_VISUAL_ITEM_IDS` for per-class main-hand/off-hand default visual fallbacks.
- Added `canonicalActorClassName()`.
- Added `classDefaultVisualItem()`.
- Added `itemAllowsActorClass()`.
- Added `resolveVisualSlotItem()`.
- Renamed the raw equipment lookup to `slotItemRaw()`.
- Made public `slotItem()` resolve through class validation and class default fallback.
- Updated `hasAny()` so class default visuals can render even when an actor has no equipment object.
- Updated `weaponKind()` so crossbow classification only applies when the item is valid for the actor class.

### `systems/runtime-sprite-cache-system.js`

- Expanded `equipmentSignature()` to include more equipment slots and runtime item identity fields:
  - `itemId`
  - `sourceItemId`
  - `sourceId`
  - `id`
  - item type/slot
  - icon family
  - class restrictions
  - rarity key
- Added stable `identityKey` into the sprite cache key.
- Included `classId` in the class key.

The fix applies anywhere the shared paperdoll equipment renderer and runtime sprite cache are used:

- player characters
- bots
- mercenaries
- companion/party actors that use shared humanoid/class renderers
- class previews or NPCs that pass equipment through the same renderer

## Protected behavior

- Starter gear definitions were not changed.
