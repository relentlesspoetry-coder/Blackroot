# V0.20.27 — The Resist That Never Resisted (Roadmap Item 19)

## First step into enemy/boss combat, and it starts with a broken promise

`controlResistancePct` was authored on **three player defensive buffs**:

| buff | class | promised resist |
|---|---|---|
| Unstoppable Footwork | Fighter | 0.5 |
| Spirit Walk | Shaman | 0.3 |
| Blink | Wizard | 0.2 |

Each tells the player it shrugs off roots and stuns. **It was triply dead:**

1. **Authored** on the buffs.
2. **Stripped by `normalizeEffect`** — not in its field whitelist, so the *stored* buff never even
   carried the value.
3. **Read in zero places.**

Verified exhaustively: 6 occurrences codebase-wide, all authoring, **0 reads**. A player popped
Unstoppable Footwork before a pull and it did **literally nothing**.

## The fix — the missing central CC-resistance gate

Item 19 requires "crowd-control resistance." There was no gate at all. This adds one at
`StatusEffects.applyToEntity` — the single point **every** status effect passes through (the same
funnel Item 13's effect policy uses).

1. **`normalizeEffect` now preserves `controlResistancePct`**, so the buff carries it.
2. **`applyToEntity`**, when the incoming effect is a control (typed `control` **or** carrying a
   `stun`/`root`/`silence`/`slow` tag), scales its duration by `(1 − resist)`, where `resist` is the
   target's aggregate `controlResistancePct` — summed from active buffs plus an optional base field on
   the entity/`baseType` — **capped at 0.9**, so resistance *shortens* control but never fully
   immunizes through this path (full immunity would be a separate explicit flag). **Non-control
   effects are untouched.**

## Verified end to end

| case | result |
|---|---|
| field survives normalization | ✅ `0.3` preserved |
| no resist → 3.2s root | **3.2s** (unchanged) |
| **Unstoppable Footwork (0.5)** | 3.2 → **1.6s** |
| **Spirit Walk (0.3)** | 3.2 → **2.24s** |
| **Blink (0.2)** | 3.2 → **2.56s** |
| stacked resist 0.8 | 3.2 → **0.64s** |
| **DoT applied under a 0.5 resist buff** | **stays 10s** — resistance touches only control ✅ |

The three buffs work for the first time, tested by casting the real spells and applying the real enemy
`root_snare` shape from `enemy.js`.

## Scope — honest

This fixes the dead field and makes **player** control-resistance real. It also **builds the gate**
Item 19's enemy-side CC resistance needs: `applyToEntity` already reads `target.controlResistancePct`
and `target.baseType.controlResistancePct`, so **a boss becomes CC-resistant the moment its data
carries one field.**

**I did not invent boss resistance values.** That's a balance decision — how much a Silk Web boss or a
zone boss should resist stun-locking is a tuning call, and I left it with the user rather than guess,
exactly as I did with the channel tooltips. The lever is now installed and proven; pulling it is one
data field per enemy, whenever you want to decide the numbers.

## Item 19 — first finding of many

This is one requirement (crowd-control resistance) of a long review list — ability variety, telegraph
readability, enrage conditions, boss phases, one-shot avoidance, and more remain unexamined. What this
version establishes is that the CC-resistance *mechanism* now exists and works; the enemy-side
*tuning* is the next deliberate pass.

## The lesson

The session's dead-field theme, one layer deeper. Usually a dead field is *authored but unread*. This
one was also **stripped in transit** — even code that *wanted* to read it off the buff would have found
nothing, because normalization dropped it before storage. When a field seems dead, check all three:
is it authored, does it survive to storage, and does anything read it? This one failed at every stage.
