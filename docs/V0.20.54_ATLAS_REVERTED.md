# V0.20.54 — The Atlas Made Rendering Slower. Reverted.

V0.20.52 and V0.20.53 rested on my claim that draw calls were the bottleneck and that a sprite atlas
would fix the frame rate. **The claim has now been tested and it failed.**

## The test I should have run first

Every number in V0.20.52/53 came from the **splash screen**, because this preview has no save file — so
`render()` was painting a menu, not a world. I reported "1 drawImage, 0 path ops" as a limitation and
moved on, when it should have stopped me: *I was making performance claims about a world I had never
rendered.*

The fix took one call. `ui-system.js` shows the Enter Realm button does `this.start(className, data)`, so:

```js
g.activeAccountId = 1;
g.start('Paladin', { className: 'Paladin', name: 'PerfTest' });
```

boots a real character into a real world. With that in place, A/B'd with `atlas.enabled`, alternating
arms to cancel drift, `update()` between frames so nothing is measured on a frozen scene:

| arm | ms/frame | path ops | drawImage |
|---|---|---|---|
| atlas **ON** | **311.66** / **383.74** | ~26,900 | ~83 |
| atlas **OFF** | **63.65** / **54.62** | ~37,500 | 65 |

**The atlas made drawing 5–6× slower.** It matches the report from the machine that counts: 12 FPS → 9.

## Root cause: page size, not the atlas concept

Isolated benchmark — same destination size, same op count, only the source texture differs:

| source | warm | cold |
|---|---|---|
| 4096×4096 atlas page | **4.17 µs** | 45.8 µs |
| small per-frame canvas (176×192) | **0.63 µs** | 1.17 µs |

**6.6× slower purely because the source image is enormous.** Nineteen 4096² pages decode to **1.28 GB**
of texture memory, and only **0.03%** of those pixels are opaque — nearly all of that bulk is empty space
being dragged through the cache.

The idea is sound. **The packing is what killed it.** In small pages the blit costs ~0.6 µs, and trading
~11k path ops for ~22 blits would be a genuine win.

## Second bug from the same job: clipped Paladin feet

The baked frame is **176×192 with the origin at y=150**, leaving **42px** below it. Paladin feet sit
**44px** below origin — the exact class-specific depth V0.20.51 was about. Pixel inspection:

```
contentBottomY: 191   frameH: 192   touchesBottomEdge: true
last rows opaque: h-3: 20, h-2: 20, h-1: 21
```

Content runs off the bottom edge — the feet are clipped by the frame box.

**Also worth recording for any re-bake:** `footAnchorY === anchorY` in *every* baked frame. The baker
never captured a real foot line; it copied the origin. I cited that field in V0.20.52 as proof the
V0.20.51 root fix would survive the atlas. It would not have.

## What the frame time is actually spent on

Selective stubbing of a real world frame (atlas off):

| configuration | ms/frame | path ops | attributable |
|---|---|---|---|
| baseline | 41.4 | 37,942 | — |
| entities removed | 26.4 | 26,951 | **entities ≈ 36%** |
| world objects removed | 35.2 | 28,672 | objects ≈ 15% |
| effects removed | 43.3 | 37,323 | noise |

**~26ms and ~27,000 path ops remain with zero entities on screen**, and only ~22 actors survive culling
in the first place.

**Terrain and ground are ~63% of the frame. Actors were never the main cost** — so no actor atlas,
however well packed, could have reached 60 FPS alone. That is where the next attempt belongs.

## What this version does

Restores the placeholder manifest so the atlas no longer loads. **Nothing is destroyed** — the bake is
preserved as `entities.baked-v0.20.53.json` and `entities.baked-v0.20.53.manifest.js` beside the 19 PNGs.

### To re-enable it later (after a re-pack)

1. Re-pack into **small pages** (≤512², ideally per-model) and **trim transparent margins** — the current
   pages are 88% covered by frame boxes that are 99.97% empty.
2. Give the frame box **real bottom clearance**, and compute `footAnchorY` from the leg/foot anchors
   rather than copying `anchorY`.
3. Rename the baked manifest back to `entities.json` (and `.manifest.js`).
4. **A/B it in-world with `atlas.enabled` before believing it helps.**

## The lesson

This is the second time this session that a verification was *true and irrelevant*. In V0.20.51 it was
testing the roots fix on a Fighter when only Paladins were broken. Here it was measuring a menu and
reporting it as render performance.

Both times the tell was the same: **the measurement could not have failed.** A splash screen will always
look fast. When a check cannot come back negative, it is not evidence.
