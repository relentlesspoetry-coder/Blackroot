# Phase 11 — Dark Woods 360x360 Expansion: Wisps + Waypoint Integration

**Phase:** 11 of 28 (Dark Woods Zone Expansion + Resource Identity master plan)
**Previous version:** V0.17.49
**New version:** V0.17.50

---

## 1. Goal

Make Stone Hedge Ruins the wisp and waypoint landmark: place the Dark Woods waypoint in the exact (already
clear) center of Phase 10's stone ring, add outer-ring weaker and inner-ring stronger wisps, and persist
waypoint discovery using whatever the existing save system already supports — without building a duplicate
waypoint framework or an unrelated travel system. No waypoint animation yet (Phase 12's scope).

## 2. Audit First (Same Discipline as Phases 7–9)

Before writing code, I confirmed via audit that `player.unlockedWaypoints` — which I recalled seeing during
earlier save-system work — is genuine but **entirely dormant** scaffolding: `systems/save-system.js`
persists it on save/load and `core/character-model.js` mirrors it into a read-only projection, but nothing
anywhere reads it to gate behavior, and nothing anywhere writes an entry into it. A code comment in
`core/character-model.js` explicitly says "no owning system exists yet for any of these... so a future
system has a stable place to read from and write to without another save-format migration." This phase is
that system — built from scratch, but landing on save/load plumbing that already safely exists, so no save
schema migration was needed.

The audit also corrected a misremembered detail: "Grave Wisp" (which I'd recalled as an existing Dark Woods
wisp) is actually only in the older, separate `data/enemies.js` legacy table used by caves/dungeons, not in
the active `data/npcs.js` MOB_DRAFTS system this landmark's spawns live in. The two wisps that actually exist
in the active system are `mob_duskwisp` (level 5-8, non-elite) and `mob_lumen_wisp` (level 6-8,
`elite: true`, `rareNameplate: true`, already a "named rare"). Both already use `animationProfile:
'floatingCaster'` and a dedicated `wisp` procedural renderer — floating/hovering visuals and behavior are
already fully modeled, needing only new placements, not new mob data or new AI/animation code.

The audit also confirmed there is no shared "nearby interactable object" system — every existing interaction
type (chests, NPCs, crafting stations, dungeon entrances, quest sources, resource nodes, puzzle nodes, fishing
water) independently registers its own `window.addEventListener('keydown', ...)` handler that checks
`game.isActionKey(event, 'interact')`, scans a small window of `game.objects` near the player for a matching
type, and calls `event.stopImmediatePropagation()` on a hit. `systems/chest-system.js` was used as the direct
template for the new waypoint interaction, following this same established convention rather than inventing
a different pattern.

## 4. Root Cause / Implementation Notes

**Why a new file, not an addition to an existing system.** No existing file owns anything resembling
waypoints/fast-travel, and this is a genuinely new, distinct player-facing feature likely to grow (more
waypoints in future zones) — consistent with this codebase's established one-file-per-concern architecture
(`chest-system.js`, `npc-system.js`, `quest-system.js`, etc. are all separate files). `systems/waypoint-system.js`
follows the exact `registerDreamRealmsSystem({id, install(game){ const runtime = {...}; runtime.init(); return runtime; }})`
pattern chest-system.js uses, which self-registers into `window.DreamRealmsSystems` and is picked up
generically by `game.js`'s `installExternalSystems()` — no edits to that big explicit install-call list were
needed.

**Scope boundary: discovery/persistence only, no travel UI.** The plan explicitly warns against "unrelated
travel systems." With exactly one waypoint in the entire game, a fast-travel menu would have nothing
meaningful to travel between yet, so `tryAttuneWaypoint()` only marks the waypoint attuned in
`player.unlockedWaypoints` and logs a message — it does not teleport the player anywhere. This is deliberately
conservative; a travel menu is natural follow-up work once a second waypoint exists in a future zone, not
part of this phase.

**Wisp tiers: reused, not invented.** The plan calls for "outer-ring weaker wisps, inner-ring stronger wisps,
optional rare/named wisp." Only two wisp tiers exist in the active Dark Woods mob data — Duskwisp (regular)
and Lumen-Wisp (already `elite`/`rareNameplate`). Rather than inventing a third, non-named "strong" tier
purely to fill out the plan's bullet list, Lumen-Wisp was used to satisfy both the "inner-ring stronger" and
"optional named wisp" requirements at once — it is both, by virtue of already being the strongest and only
named wisp available. This is documented explicitly here as a deliberate reuse decision, not an oversight.
The outer-ring Duskwisp camp (`stone_hedge_wisps_outer`, center `210,266`, radius 9/roam 12) sits outside the
Phase 10 stone ring's radius-9 extent; the inner Lumen-Wisp anchor (`210,256`) sits between the waypoint's
radius-5 clear zone and the stone ring, i.e. genuinely "inner" relative to the outer patrol.

**Waypoint placement and "keep center uncluttered."** `ensureStoneHedgeWaypointObject` places the object at
exactly `(210,250)` — the same center `ensureStoneHedgeRuinsLandmark` (Phase 10) deliberately kept clear
(verified then: nothing closer than radius 5). It's registered non-blocking (`blocked: false`, and `waypoint`
was added to `decorOnly`), satisfying "ensure player can stand on/near it" and "collision does not block
access" directly — the player can walk onto the exact same tile as the waypoint object.

**The `attuned` visual flag has a known, accepted staleness edge case.** The object's initial `attuned` value
is computed once at world-generation time from `this.player?.unlockedWaypoints`. For a fresh character this
is always `false` (correct). For a loaded save where the world's object grid is itself restored from the
save's own serialized data (per Phase 2's world-serializer work) rather than freshly regenerated, this
generation-time computation may not even run. In both cases, the moment the player actually interacts with
the waypoint, `tryAttuneWaypoint()` sets `obj.attuned = true` directly on the live in-memory object, so the
visual self-corrects immediately in the one case where it could matter — not treated as a deeper problem.

## 5. Validation

Commands run:

```bash
find . -name "*.js" -print0 | xargs -0 -n1 node --check
find . -name "*.json" -print0 | xargs -0 -n1 python3 -m json.tool > /dev/null
```

**Results:** both passed with zero errors, including the new `systems/waypoint-system.js` file and the
updated `_script_order.json`.

**Functional validation (executed in Node, not just traced):**

1. Confirmed both reused spawn ids (`spawn_dark_woods_duskwisps`, `spawn_dark_woods_named_lumen_wisp`)
   genuinely exist in `data/npcs.js` — no dangling references introduced.
2. Loaded the real `DARK_WOODS_MOB_CAMPS` array and computed distance from the new
   `stone_hedge_wisps_outer` camp to every other camp: **nearest is 116.1 tiles away**, far clear of the
   20-tile `minCampSpacing` requirement.
3. Re-ran the Phase 3/9/10 region-lookup logic against the waypoint, outer-wisp-camp, and named-wisp-anchor
   coordinates — all three resolve to `stone_hedge_clearing`, confirming correct region containment.
4. Re-implemented `findNearbyWaypoint`'s exact scan logic in a standalone script against a synthetic
   360x360 object grid with a waypoint placed at `(210,250)`: a simulated player standing on the tile is
   found; a player 2 tiles away (within the default 2.2 range) is found; a player 10 tiles away correctly
   returns no match. This is a genuine functional test of the interaction detection logic, not just a syntax
   check.

**Not validated (no browser available, same limitation as Phases 2–10):** actually pressing the interact key
in a live browser session and confirming the on-screen prompt/log message appears, the save round-trips
`unlockedWaypoints` correctly end-to-end through a real save/load cycle, and the wisps visually render and
patrol as expected.

## 7. Known Limitations

- No fast-travel/teleport menu exists yet — deliberately deferred until a second waypoint exists somewhere
  in the game (a different zone), per this phase's own scope boundary.
- Only 2 wisp tiers are used (Duskwisp outer, Lumen-Wisp inner/named) rather than 3 distinct tiers, since
  only 2 exist in the active Dark Woods mob data — documented as a deliberate reuse decision in Section 4.
- The waypoint's `attuned` visual flag can be momentarily stale immediately after loading an old save before
  the player revisits it (Section 4) — self-corrects on next interaction, not treated as a functional bug.
- No in-browser interaction/save-round-trip/visual test was possible in this environment (see Section 5).
