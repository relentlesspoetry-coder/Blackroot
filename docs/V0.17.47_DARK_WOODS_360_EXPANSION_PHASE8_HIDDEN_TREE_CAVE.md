# Phase 8 — Dark Woods 360x360 Expansion: Hidden Tree Cave Exterior + Interior

**Phase:** 8 of 28 (Dark Woods Zone Expansion + Resource Identity master plan)
**Previous version:** V0.17.46
**New version:** V0.17.47

---

## 1. Goal

Make Hidden Tree Cave a small, concealed secret exploration cave: dense exterior concealment around its
entrance, and exactly one small piece of interior content — explicitly not a second dungeon, no multi-floor
logic, no boss progression.

## 2. Audit First (Same Discipline as Phase 7)

Following the same practice as Phase 7, I audited Hidden Tree Cave's current implementation before writing
any code. Unlike Silk Web Cavern, this cave has no `directDungeon` flag and goes through the fully generic,
organic cave-floor generator in `systems/cave-system.js` (`buildRooms` → `carveOrganicBlob`/
`carveOrganicTunnel`/`smoothCaveOpenCells`) — 5 room anchors for a `size:'small'` cave (entrance, patrol,
resource, nest, treasure), not a hand-authored room graph. This audit found one genuine, actionable problem
that directly conflicts with this phase's own rule, plus two smaller content-consistency issues left over
from Phase 5's rename — all addressed below.

## 4. The Real Problem Found: A Live Second Dungeon Inside Hidden Tree Cave

`systems/swimming-system.js`'s `UNDERWATER_ENTRANCES` list included an entry
(`id: 'underwater_cave_blind_pool', zoneId: 'mossfang_cave', x:106, y:107, dungeonId: 'dungeon_blind_pool_depths'`)
that, every session, force-carves a deep water pocket at those fixed coordinates inside this exact cave
(`ensureWaterPocket`, called from `ensureEntrances()` on every frame update while in a cave zone) —
overwriting whatever the cave's own procedural generator placed there. Diving into that water and
interacting routes into `dungeon_blind_pool_depths`, a real, playable 2-floor dungeon with boss
`boss_silk_mother` (`data/dungeons.js:159-177`, `data/enemies.js:100-118`), via the exact same generic
dungeon generator (`generateDungeonFloor`) used by any other non-Silk-Web dungeon draft. This predates the
Mossfang→Hidden Tree Cave rename (its own notes say "Pass 48 cave-pool underwater dungeon") and directly
contradicts this phase's explicit rule — it is invisible to and completely independent of
`cave-system.js`'s own generation, so nothing in Phase 5's cave-count work would have caught it (Phase 5 only
touched `CAVE_DEFINITIONS` and cave-entrance connector trails, not the swimming system).

**Correction to Phase 7's report:** Phase 7 stated `boss_silk_mother` was "never spawned by any code path,"
based on `dungeon_silk_web_depths` (a different, separately-superseded draft that also references this same
boss id) being unreachable. That was incomplete — `boss_silk_mother` *was* reachable, via
`dungeon_blind_pool_depths`'s live underwater entrance inside Hidden Tree Cave, until this phase's fix.
Noting the correction here for the record.

**Resolution:** removed the `underwater_cave_blind_pool` entry only (the other two `UNDERWATER_ENTRANCES`
entries target `dark_woods` overworld water, not this cave, and are unaffected). This satisfies "no second
dungeon in Hidden Tree Cave" directly. Following the same precedent set in Phase 5 for Gloom's Crypt,
`dungeon_blind_pool_depths` and `boss_silk_mother`'s data were **not deleted** — they're fully intact, just
now genuinely unreachable, pending a future decision (re-attach elsewhere, or accept as cut content).

## 5. Exterior Concealment

`ensureHiddenTreeCaveConcealment` places two concentric rings around the entrance at (76,74) — read from
`DR.CAVE_BY_ID.mossfang_cave.entrance`, not a duplicated literal coordinate, so it can't drift out of sync:

- **Inner ring (radius 8–11, just outside the ~7.8-tile entrance-clearing disk placed by the existing
  generic `placeEntrance`)**: denser (~46% of eligible cells get a prop), includes large blocking props —
  mossy tall trees (`tree` with `scale` up to ~1.34 and the new `mossy` overlay), root arches (`rootArch`,
  already a blocking prop type), plus non-blocking brush, hanging roots (reusing the existing
  `hangingRoots`/`drawHangingRoots` — already a vine-like hanging-strand prop, no new render code needed),
  and thick grass. This ring is what makes the mouth read as tucked behind foliage rather than sitting in an
  open clearing.
- **Outer ring (radius 11–20)**: sparser (~18%), the same prop palette minus root arches, for a general
  transition back into ordinary forest.

Both rings skip water, already-blocked tiles, and any already-occupied tile (never overwrites the existing
entrance dressing or the maintained road), and are deterministically seeded by tile coordinate — not
randomized per frame, matching every other dressing pass added in this project.

**The maintained road to the entrance was deliberately left alone.** `carvePath([{x:100,y:100},{x:90,y:89},{x:76,y:74}], ...)`
(Phase 2, `world-system.js:238`) still leads directly to the cave mouth. Re-routing or removing an
established road is materially bigger surgery than this phase's scope (it could affect other trail
connections, per Phase 4's own notes about shared road endpoints), so instead the concealment ring makes the
immediate mouth itself read as hidden even though a path leads *near* it — consistent with "discoverable but
not obvious" rather than "literally unreachable by road."

## 6. Interior Content: One Hidden Lore NPC

Of the plan's options (rare gathering node, hidden NPC, lore object, small enemy pocket, chest, secret named
mob, small quest objective), a stationary lore NPC was chosen as the lowest-risk, most self-contained choice:

- A rare gathering node was already effectively covered by the existing generic resource seeding
  (`decorateCave` already places 2-3 nodes from `resourceTypes` in the resource/treasure rooms) — adding a
  *new* rare resource type is explicitly Phase 20-23 scope (the plan's dedicated resource-identity phases),
  so it was left alone rather than improvised early.

- A chest already exists generically (the forced `treasure`-kind room already places a `dungeonTreasure`
  prop + a matching `chest`-category resource-grid node, resolved through the shared, reusable
  `systems/chest-system.js` — no new code needed for a chest to exist here).
- A "secret named mob" would require extending `createCaveEnemy`/`spawnCaveEnemies` in `cave-system.js`
  itself (core generation logic shared with any future plain cave), a bigger and riskier change than adding
  one inert NPC draft + one placement entry + one dialogue string — and the existing named-rare pattern
  (`mob_mossfang_alpha`, `data/npcs.js`) is built for the overworld mob-spawn system, not compatible with
  the cave's separate `createCaveEnemy` path, so reusing it would have required new plumbing anyway.

`npc_hidden_tree_hermit` ("Hollow Root Hermit") was placed at `(77,100)` — the cave's deterministically
computed **'nest' room center** for this cave's fixed `seed:4103`, floor 1. Room positions are seeded (not
`Math.random()`), so this coordinate is reproducible: verified by re-implementing `buildRooms`' exact
algorithm in Node (Section 7) rather than assuming. The room center is additionally guaranteed to be open
floor, not a wall — `carveOrganicBlob`'s carve condition is satisfied at `dist2 = 0` for any room, so the
exact center of every room is always carved. NPC placement reuses the existing, already cave-zone-aware
`DEFAULT_NPC_PLACEMENTS`/`ensureDefaultNpcPlacements()` pipeline (confirmed already pre-initializes a
`mossfang_cave` bucket and iterates placements generically by zone) — no new NPC plumbing was needed.

**Results:** both passed with zero errors.

**Functional validation (executed in Node, not just traced):**

1. Confirmed the underwater entrance removal: `grep -c zoneId: systems/swimming-system.js` → 2 (both
   `dark_woods`), and the only remaining occurrences of the string `mossfang_cave` in that file are an
   explanatory comment and an unrelated pre-existing fallback default — no entrance entry remains.
2. Re-implemented `buildRooms`' exact seeded-RNG algorithm (`createRng`, the same mulberry32-style PRNG) in
   a standalone script and ran it for `{seed:4103, size:'small', floors:1}`, floor 1:
   ```
   entrance (123,172), patrol (111,154), resource (86,129), nest (77,100), treasure (116,86)
   ```
   confirming `(77,100)` is genuinely the deterministic 'nest' room center before committing to it as a
   fixed NPC coordinate.
3. Re-implemented `ensureHiddenTreeCaveConcealment`'s exact threshold logic using the project's real
   `seededNoise` function and ran it across the full ring area:
   ```
   Violations inside the 8-tile entrance clear radius: 0
   Inner ring (184 cells): hangingRoots 19, brush 17, rootArch 15, grassTuft 15, tree(mossy) 19 (~46%)
   Outer ring (880 cells): brush 42, tree 48, grassTuft 31, hangingRoots 36 (~18%)
   ```
   confirming the entrance stays clear and the density gradient (denser inner ring, sparser outer ring)
   works as designed.

**Not validated (no browser available, same limitation as Phases 2–7):** the actual rendered appearance of
the concealment ring and moss overlay, whether the entrance genuinely "reads" as hidden when approached
along the existing road, and whether the hermit NPC is discoverable/interactable in a live session.

## 9. Known Limitations

- The maintained road still leads directly to the entrance (Section 5) — concealment is achieved by dressing
  around the mouth, not by removing the road. If a fully hidden, off-road discovery experience is wanted
  later, that would mean rerouting or removing this road, which is out of scope here.
- `dungeon_blind_pool_depths` and `boss_silk_mother` are now genuinely orphaned (Section 4) — same
  unresolved-pending-decision status as Phase 5's Gloom's Crypt.
- The hermit NPC has no quest or unique loot — pure lore/flavor only, per this phase's "small" mandate.
- Hidden Tree Cave's `enemyNames`/`resourceTypes`/`lootTableId` still reflect the old Mossfang-era content
  (wolves/rotling/thorn widow, moss/mooncap/tinstone/cavepool) — the plan's Hidden Tree Cave description
  doesn't strictly require changing these, and doing so was judged out of this phase's "small" scope; flagged
  in case a future phase wants tighter thematic alignment.
- No in-browser visual/interaction test was possible in this environment (see Section 7).
