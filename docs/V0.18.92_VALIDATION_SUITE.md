# V0.18.92 — Consolidated Validation Suite (Roadmap Item 22)

Item 22 asks for automated validation beyond syntax checks, run as part of every release. This version
gathers the game's scattered launch-time checks into **one release-time report**, adds a save/load
regression check, and fixes a stale obtainability count.

## The suite — `runValidationSuite()`

New `Game.prototype.runValidationSuite()` (`systems/inventory-system.js`, after
`runStatPipelineSelfTest`) aggregates:

- **registry** — descriptor references, compiler errors, item obtainability
- **statPipeline** — the regression self-test (now 8 checks)

into one report `{ ok, summary, sections, checkedAt }`, with one overall pass/fail:

- **Hard failures**: compiler errors, or a failed stat-pipeline self-test.
- **Advisory**: descriptor *warnings* (the game continues; the 7 pre-existing quest-giver-name
  warnings belong to the Item 1 quest-data pass, not this suite).

### Boot log

Invoked once during init (`game.js`, after `bindEvents`), replacing the standalone stat-self-test log
with one consolidated line:

```
[Dream Realms] validation suite: PASS - stat-pipeline 8/8, items sourced 396/396,
  0 compiler error(s), 7 descriptor warning(s). Call DarkWoodsGame.runValidationSuite() for details.
```

`console.info` on pass, `console.warn` + `sections` on fail. Advisory-only; a bare `try/catch`
guarantees it never blocks the loop. Single manual entry point: `DarkWoodsGame.runValidationSuite()`.

## New regression check — save/load safety (self-test #8)

Round-tripping a scratch player's equipment through `JSON.parse(JSON.stringify(...))` (as a save does)
must reproduce **identical** stats — no loss, **no duplicated modifiers** (roadmap: "save/load cannot
duplicate modifiers"). Passes `295 == 295`.

## Validation-quality fix — stale obtainability count

The boot-time `runBootValidation` (`data/default-zones.js`) fires **before** all source data (loot
tables, shop lists, crafting outputs) is registered, so its cached obtainability was **premature** and
under-reported sources: **340/396 (56 false-unsourced)**. A fresh audit run later shows the truth:
**396/396, 0 unsourced**.

Fix: the suite **re-runs `runBootValidation` fresh** (it runs late, after full init), so its number is
accurate **and** the corrected report is re-cached for anything else reading
`DR.Registry.getReport()`. Confirmed: after the suite runs, `getReport().obtainability` reads
`396/396`.
