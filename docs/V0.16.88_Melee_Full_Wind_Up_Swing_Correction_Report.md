# V0.16.88 - Melee Full Wind-Up Swing Correction Report

## Scope
This phase corrects the V0.16.87 melee auto-attack swing animation only. It does not change auto-attack damage, speed, range, hit chance, crit chance, threat, resources, spell animation, class balance, save data, UI, inventory, party, mercenary, pet, quest, map, dialogue, or renderer settings.

## Where the V0.16.87 melee swing lived
- Visual event timing owner: `systems/combat-system.js`
  - `MELEE_AUTO_ATTACK_IMPACT_PHASE`
  - `meleeSwingPulse()`
  - `startActorMeleeAutoAttackSwing()`
  - `refreshPlayerAutoAttackVisualState()`
- Pose/arm owner: `render/humanoid-base-renderer.js`
  - shared melee hand/elbow/torso displacement block inside `buildRig()`
- Weapon/trail owner: `render/humanoid-base-renderer.js`
  - `drawMeleeAutoAttackOverlay()`

## What caused the swing to be too shallow
The V0.16.87 keyframes used a short 0%–25% wind-up, a modest hand lift, and a strike endpoint only slightly below the default hand/chest zone. The weapon trail also used a smaller arc radius. This made the motion read as a small twitch/swoosh instead of a full wind-up and committed downward chop.

## Pose / angle / timing parameters changed
- `meleeSwingPulse()` was adjusted to the requested timing shape:
  - 0%–35% wind-up
  - 35%–58% downward strike
  - 58%–68% low impact/follow-through
  - 68%–100% recovery
- The impact phase remains `0.58`, preserving real hit alignment.
- The shared melee hand/elbow/chest pose now uses:
  - higher `lift`
  - stronger backward wind-up
  - larger `reach`
  - much larger downward `chop`
  - stronger torso commitment
  - lower follow-through hold
- The melee overlay now uses:
  - higher wind-up endpoint
  - lower strike endpoint
  - larger class-scaled trail arc
  - stronger low through-target impact position

## Classes updated
The correction applies through the existing shared melee auto-attack path from V0.16.87, which includes classes currently routed as melee by combat metadata. Expected affected player classes:

- Fighter
- Rogue
- Paladin
- Warden
- Bard only if the current metadata routes Bard as melee

Ranged/caster classes continue to be excluded by the existing auto-attack identity logic. Ranger remains protected from the shared melee overlay because Ranger owns bow rendering separately.

## Entity types updated
- Player melee classes using the shared humanoid renderer
- Melee bots/companions/mercenaries using the same humanoid auto-attack visual state

Separate pet, animal, skeleton, monster, ranged, caster, and bow renderer ownership was not changed.

## Impact alignment
The real auto-attack hit remains aligned around phase `0.58`. `startActorMeleeAutoAttackSwing()` and `refreshPlayerAutoAttackVisualState()` still compute visual start time from the real attack interval and real remaining attack timer. Only the visual pose curve and amplitude were changed.

## Trails / impact cues
No new particle system or duplicate hit event was added. The existing lightweight weapon trail overlay was enlarged and retimed to match the higher wind-up and lower strike. Existing hit/VFX behavior remains owned by combat/effects systems.

## Protected behavior confirmation
- Auto-attack damage unchanged.
- Auto-attack speed unchanged.
- Combat range unchanged.
- Hit/crit/threat unchanged.
- Class resource behavior unchanged.
- Spell animations unchanged.
- No duplicate animation or auto-attack system added.
- No cascade patch, wrapper, setTimeout, or bottom-of-file override added.
