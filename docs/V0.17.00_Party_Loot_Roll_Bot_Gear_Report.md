## Ownership audited

- Loot drop and corpse/chest grant ownership lives in `systems/loot-system.js`.
- Party eligibility is calculated from the existing party roster helpers in `systems/party-system.js`, primarily `getPartyCombatMembers()`, `partyMercIncluded`, and `isBotInParty()`.
- Player inventory/equipment ownership remains in `systems/inventory-system.js`.
- Bot inventory/equipment ownership remains in `systems/bot-player-system.js`.
- Mercenary gear ownership remains in `systems/mercenary-system.js`.
- Save/load persistence for bot and mercenary equipment already existed; this pass added mercenary inventory persistence for old gear displaced by loot-roll auto-equip.

## Roll session ownership

Roll sessions were added directly to the existing loot owner in `systems/loot-system.js`. No duplicate loot/inventory/equipment/party/bot system was created.

Primary added functions include:

- `isLootRollGearItem()`
- `shouldStartLootRollForItem()`
- `getLootRollParticipants()`
- `evaluateGearUpgrade()`
- `beginLootRollForItem()`
- `submitLootRollChoice()`
- `updateLootRollSessions()`
- `resolveLootRollSession()`
- `awardLootRollItemToParticipant()`

## Item rarity / gear detection

Gear roll eligibility is determined by:

- item is equippable gear
- item rarity rank is above white
- more than one eligible local party participant exists

Grey and white gear continue through existing loot behavior. Non-gear drops, materials, quest items, consumables, stackables, currency, fish, herbs, and ore do not trigger gear rolls.

## Need / Greed / Pass resolution

- Need and Greed roll random 1-100.
- Pass produces no roll.
- Highest Need beats any Greed.
- Highest Greed wins only when there are no Need rolls.
- Tied top rolls reroll tied participants up to a safety cap, then fall back deterministically.
- Player timeout after 30 seconds auto-passes.
- Bot and mercenary decisions occur after a short randomized delay for readability.

## Bot / mercenary gear evaluation

`evaluateGearUpgrade(entity, item)` returns compatibility, slot, current item, scoreCurrent, scoreCandidate, upgradeScore, shouldNeed, and reason. It uses existing compatibility helpers where available:

- player: `canEquip()` / `resolveEquipmentSlotForItem()`
- bot: `botCanUseItem()` / bot equipment state
- mercenary: `canMercEquipItem()` / `mercSlotForItem()`

Scoring uses available item stats only. Unsupported future stats such as specialized poison/execution/pet-power tags are not invented in this pass; the scoring function uses existing flexible stat metadata when present.

## Bot / mercenary auto-equip

When a bot wins a valid Need roll, the item is equipped into `bot.botEquipment[slot]`; the old item is moved into `bot.botInventory`. Bot gear score and party UI dirties are updated.

When a mercenary wins a valid Need roll, the item is equipped into `merc.equipment[slot]`; the old item is moved into `merc.inventory`. Mercenary stats are recalculated and mercenary inventory is now serialized/restored.

If a winner receives an item through Greed or cannot equip at award time, the item goes into that actor inventory where available rather than being deleted.

## Duplicate item award prevention

Roll-managed items receive `lootRollSessionId` while rolling and `lootRollAwarded` when resolved. The runtime grant path bypasses duplicate awards by routing eligible green+ party gear into one session instead of directly adding it to the player.

## Solo behavior

Solo play with no active bot/merc/party participant skips the roll system and uses existing loot behavior.

## Active roll save/load behavior

Active roll sessions are not persisted. Awarded player items, awarded bot equipment, and awarded mercenary equipment persist through existing save paths. The mercenary displaced-item inventory is now serialized/restored.

## Known limitations

- Remote human party members are not independently prompted in this local single-player implementation; local player, bots, and mercenaries are handled.
- Active unresolved roll prompts do not survive a full save/reload; awarded gear does.
- Gear scoring uses current stat vocabulary and does not invent new specialized stat pipelines.
