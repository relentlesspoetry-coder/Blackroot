changes** — only documentation, and the mandatory version/backup bookkeeping.

## 3. Systems Audited

Dark Woods zone definition, cave system, mob spawn system, resource/fishing/profession systems, collision
system, terrain/prop systems, waypoint/travel, VFX/animation, rendering/culling, minimap, save/world-state
persistence. Full findings below.

### 3.1 Zone definition
- Descriptor: `data/default-zones.js:16-23` — `dark_woods` zone object (`id/name/width:200/height:200/generation`). Five cave zones are attached as `parentZone: 'dark_woods'` (lines 24-41).
- **The descriptor's `width`/`height` are display metadata only.** The actual generator size is the single frozen global `DR.CONFIG.MAP_SIZE = 200` (`core/config.js:75`), read by `Game.prototype.generateMap` (`systems/world-system.js:99,105`). Changing the descriptor alone does nothing.
- Map storage: single 2D array `this.map[y][x] = {type, elev, blocked, waterDepth}` plus a parallel `this.objects[y][x]` prop layer — not chunked/flat.
- Generation: noise-based biome fill (`smoothNoise`/`seededNoise`) plus heavy **hand-authored absolute-coordinate** landmark/road/camp placement (`placeCamp`, `placeWorldLandmarks`, `ensureDarkWoodsAtmosphereLandmarks`, `ensureDarkWoodsTreeWallBoundary` — `world-system.js:258-624`). Camp center is `{100,100}` = `CONFIG.START_X/START_Y`.

### 3.2 Cave system
- `systems/cave-system.js`, `DR.CAVE_DEFINITIONS` (lines 8-114): **6 caves currently attached to Dark Woods** — `mossfang_cave`, `silk_web_cavern`, `ashroot_hollow`, `crystal_grotto`, `forgotten_mine`, `blackroot_catacombs`. All entrance coordinates and connecting trails (`makeTrail(...)`) are hardcoded absolute tile points.
- Entrance registration (`ensureDarkWoodsCaveEntrances`), proximity-based enter/exit warps (`checkZoneTransition`, `enterCave`, `exitCave`), and cave interior generation (`buildCaveFloorState`/`buildCaveMap`, organically carved, seeded per-cave) are all sound and reusable — the risk is purely that **4 of the 6 existing caves are not part of the target design** (only Silk Web Cavern and a not-yet-built Hidden Tree Cave are allowed per the plan's Cave Count Rule). This is a **Phase 5 (Cave Count Enforcement)** concern, flagged here so Phase 2/3 don't build around the wrong cave count.

### 3.3 Mob spawn system
- `systems/mob-spawn-system.js` + `data/enemies.js` + `data/npcs.js`. Existing families already cover the plan's needs: wolves (Gloom Wolf), spiders (Thorn Widow, Webling Skitterer, cave-elite variants), boars (Briar Boar), wisps (Grave Wisp, Duskwisp), plus Rotling, Hollow Stag, Ashroot Horror, named rares. **No bandit family or crow/raven ambient family currently exists** — new content for Phase 9/16, not a system rewrite.
- Placement is three hardcoded absolute-coordinate layers (`DEFAULT_MOB_SPAWN_PLACEMENTS`, `DARK_WOODS_MOB_CAMPS`, `DARK_WOODS_SOLO_ROAMERS`/`DARK_WOODS_ROAMING_GROUPS`).
- Level scaling is distance-from-spawn, not a static table: `overworldProgress` normalizes distance from `START_X/Y` by a hardcoded divisor of `96` (tuned for the current ~140-tile max radius). At 360x360 (max radius ~250) this constant **must be recalibrated** (e.g. proportionally to ~173) or the level curve flattens to 10 far before the new edge.
- Visual models for every needed family already exist (`render/dark-woods-mob-procedural-model.js`, `wolf-procedural-model.js`, `boar-procedural-model.js`, `rotling-procedural-model.js`, `silk-web-spider-procedural-model.js`) — no new art pipeline needed, only new bandit/crow visual entries later.

### 3.4 Resource / gathering / fishing / profession systems
- Resource schema (`data/resources.js`, `DR.RESOURCE_TYPES`): `id, name, category, tool, skill, profession, levelRequired, gatherTimeSeconds, xpReward, color, label, respawnSeconds, drops[], rareDrops[], lootTableId, note`. **No zone/region field on the definition** — zone restriction today is purely by *placement* (which zone's grid a node instance lives in), not by data. This matches what the plan's `allowedZone`/`allowedRegion`/`allowedBiome` fields will need to formalize in Phase 19-23.
- Only **5 hardcoded Dark Woods overworld resource nodes** exist today (`ensureDefaultResourceNodes`, `systems/resource-gathering-system.js:495-517`) — a real content gap, not just a scaling risk. None of the plan's 16 new resource IDs exist yet.
- Gathering/Mining EXP curve: `nextLevelXpFor(level) = max(50, floor(100 * level^1.45))` (`resource-gathering-system.js:41-43`). Fishing and Crafting each have their **own independently copy-pasted copy of the same formula** (`systems/fishing-system.js`, `systems/crafting-system.js`). This is flagged per the project's anti-duplication rule as three copies of one constant, not a duplicate *system* — worth consolidating when Phase 19 touches the EXP tables, but out of scope for Phase 1.
- Respawn timing is timestamp-based (`respawnAt = now + respawnSeconds*1000`), persisted via `serializeState()`/`importState()` and mirrored to `localStorage` — sound foundation for Phase 24's anti-farming rules.
- Fishing spots are found dynamically by scanning live water tiles per cast (not a hardcoded coordinate list) — naturally scale-agnostic. Minor pre-existing bug noted: `isValidHotspotPosition` hardcodes the safe-zone exclusion as `hypot(x-100,y-100)<10` instead of referencing `CONFIG.START_X/START_Y` — will misalign if spawn moves; flagged for whoever eventually touches that function, not fixed here (out of Phase 1 scope).
- Profession system (`systems/profession-system.js`) is a pure UI aggregator with no independent zone logic.

### 3.5 Collision system
- Tile-flag based (`tile.blocked`) plus an object-layer check (`hasBlockingObjectAt`) that treats an explicit `obj.blocked` flag as authoritative and otherwise infers blocking from type-name substrings (`tree`, `wall`, `rock`, `boulder`, `cliff`, `building`, `fence`, `stump`). Small decorative props are explicitly whitelisted as non-blocking. This model already matches the plan's "only large props block" rule — no redesign needed.
- **Blocker:** `activeMapSize()` (`collision-system.js:38-45`) correctly derives size from the live map array and is used by `isWalkable`, but `tryMove` (lines 21-22) clamps player position using the **hardcoded global** `CONFIG.MAP_SIZE - 1.5` directly. Must be switched to `this.activeMapSize()` before Dark Woods can safely differ in size from `CONFIG.MAP_SIZE`.

### 3.6 Terrain / prop systems
- `data/tiles.js` is a pure type→style dictionary, zone-size agnostic.
- Terrain variant/overlay placement (`render/terrain-renderer.js`) is **already deterministic and seeded by absolute tile coordinate** (`seededNoise`/`smoothNoise`), not randomized per frame — leaf litter, twig/root debris, moss tinting, and boundary grass runners are **already implemented** and will generalize automatically to new tile coordinates in a larger map. Phase 13's "terrain overlay" work is refinement/density tuning, not new infrastructure.
- Terrain renders through a fixed 16x16-tile chunk cache with per-chunk viewport culling and a dirty/rebuild signature cache — already scale-agnostic; a 360x360 map is just more chunks, not an architecture change.
- Props are placed once at world-gen time into `this.objects[y][x]` and rendered from that array (not regenerated per frame).

### 3.7 Waypoint / travel system
- **No fast-travel/waypoint network exists anywhere in the codebase.** `systems/portal-system.js` is a Mage recall-spell system (personal return-to-camp portal with a fully built animated vortex renderer) — a good visual reference for the future waypoint aura, but a different system; there is no existing waypoint implementation to duplicate or conflict with. Phase 10/11 will be building this fresh, which is expected per the plan.

### 3.8 VFX / animation system
- `systems/effects-system.js` provides a pooled, generic effect pipeline (`addEffect`, `updateEffects`) with reusable spawners (`spawnRing`, `spawnStatusPulse`, `spawnCastCue`) and a precedent for a looping ambient aura (`bardMeditationAura`). `render/effects-renderer.js` dispatches by `effect.type` string through one central switch. **This is the correct, idiomatic extension point for the Phase 12 waypoint aura** — a new `'waypointAura'` case, not a new render loop. No duplicate VFX systems exist.

### 3.9 Rendering / culling
- Terrain chunk culling is sound (see 3.6). Object/entity/effect culling is pixel-space viewport-pad driven via `systems/adaptive-performance-system.js` (frame-time-adaptive quality levels 0-3), which scales fine with a larger world.
- **Blocker:** the main per-frame visible-tile window in `game.js` (~line 3618-3621) clamps against the hardcoded `CONFIG.MAP_SIZE` rather than the live map array length — identical failure mode to the collision-system blocker in 3.5. Must be fixed or the outer ~80 tiles of a 360-wide Dark Woods will never be drawn.
- No hard cap on total prop count exists; density is controlled at placement time and by mob density caps, both of which are already tunable parameters, not architecture.
- Dead config flagged: `core/config.js:92` `ENEMY_COUNT: 230` has zero references anywhere in the codebase — stale/unused, noted for whoever owns `core/config.js` next; not removed in this phase per the "prove dead code" rule (documenting rather than deleting, since it is a config value someone may intend to wire in later and removing it is out of this phase's scope).

### 3.10 Minimap / map system
- `render/minimap-renderer.js` has two inconsistent code paths: the generic live minimap builder correctly derives size from the actual map array, but the primary runtime path (`buildStaticMinimap`/`drawMinimap`, called after every `generateMap()`) **hardcodes `CONFIG.MAP_SIZE`** for canvas allocation and loop bounds. This is the same class of blocker as 3.5/3.9 and must be fixed to read `this.map.length` before a differently-sized Dark Woods can render its minimap correctly (and before any other zone can safely coexist if `MAP_SIZE` were instead bumped globally).

### 3.11 Save / world-state persistence
- `systems/world-serializer.js` already has real schema/version migration infrastructure (`SCHEMA`, `SAVE_FORMAT_VERSION`, `migratePayload()`, `migrations[]`/`warnings[]` log, shape validators) — a solid, proven foundation.
- **Critical gap:** dimensions are tracked via a single scalar `mapSize` on the whole save payload, stamped identically onto every zone (Dark Woods and every cave share one `size` variable at serialize and load time). Supporting a Dark Woods that is a different size than its caves will require a genuine save-schema version bump (e.g. v2 → v3) to store real per-zone width/height — a moderate, well-precedented change, not a from-scratch build.
- Resource/mob runtime respawn state persists separately, keyed by zone+coordinate+type strings — not sensitive to map dimensions, should survive a resize without migration.
- Player-position load clamps in `systems/save-system.js` (two call sites) also hardcode `CONFIG.MAP_SIZE` directly — same fix pattern as 3.5/3.9/3.10.

## 5. Hard Blockers for the 360x360 Resize (priority order — for Phase 2)

1. `DR.CONFIG.MAP_SIZE` (`core/config.js:75`) is a single frozen global used identically for the overworld and every cave, across ~78 call sites. There is currently no per-zone size concept anywhere in generation, collision, rendering, minimap, or save code.
2. `collision-system.js` `tryMove` and the main render visible-window clamp in `game.js` (~3618-3621) both hardcode `CONFIG.MAP_SIZE` instead of the already-proven `activeMapSize()`/`this.map.length` pattern used elsewhere in the same file. Must fix or movement/rendering silently clips at tile 199.
3. `render/minimap-renderer.js` `buildStaticMinimap`/`drawMinimap` hardcode `CONFIG.MAP_SIZE` for canvas size and loop bounds — will truncate or crash depending on which side of the resize is done first.
4. `world-serializer.js` has only one global `mapSize` scalar for the whole save payload — a genuine schema bump (v3) is required to give Dark Woods a real, independent size from its caves. The migration mechanism to do this safely already exists.
5. All Dark Woods hand-authored content (cave entrances, mob spawn anchors/camps/roamers, camp/road/landmark placement, boundary tree wall) uses absolute tile coordinates tuned to the current 200x200 canvas. None of it is relative/proportional and none of it will "spread out" automatically — it must be re-authored or programmatically rescaled in Phase 2/3/4.
6. The mob level-scaling distance-normalization constant (`96` in `overworldProgress`) is tuned to the current map radius and needs recalibration for the new ~250-tile radius (Phase 3).
7. Six caves currently exist under Dark Woods; the target design allows exactly two (Silk Web Cavern + a new Hidden Tree Cave). Four existing caves (`mossfang_cave`, `ashroot_hollow`, `crystal_grotto`, `forgotten_mine`) will need to be removed or reassigned to a different zone in Phase 5 — flagging early so Phases 2-4 don't build road/landmark layouts around cave entrances that will later be removed.

## 6. Proposed 360x360 Region Layout (blueprint for Phase 2/3 — not implemented yet)

Proposal keeps the existing concentric distance-from-spawn level-scaling model (already implemented in
`overworldProgress`) rather than introducing a new directional model, since "danger increases outward from
town" is naturally a concentric relationship and reuses proven code with a recalibrated constant. Named
subregions are laid out as directional pockets at increasing radius bands, matching the plan's level table.
All coordinates are proposed, to be finalized/tuned in Phase 2 (resize) and Phase 3 (region metadata) —
nothing below is authored yet.

Assume town/spawn stays proportionally centered, scaled from `{100,100}` in the 200x200 grid to
approximately **`{180,180}`** in the 360x360 grid (center of map, preserving the current relative position).

| Region | Approx. radius band from center | Approx. bounding box (x, y) | Level | Notes |
|---|---|---|---|---|
| Dead Lantern Outskirts | 0–45 | ~135–225, 135–225 | 1–3 | Town + all four main road spokes; Dead Lantern Trail motif starts here. |
| Bramblefen Thicket | 45–100, SW arc | ~90–170, 190–280 | 3–5 | Dense brush, narrow trails, hidden nodes. |
| Bandit's Fall | 90–150, NE pocket | ~230–310, 70–150 | 4–7 | Ruined outpost, off the main NE road spoke. |
| Stone Hedge Clearing | 90–140, N pocket | ~150–220, 40–110 | 5–8 | Waypoint + wisps; kept off main road for a "discovered" feel. |
| Gloamroot Depths | 150–250 outer ring, N/E/S | outer band beyond the above pockets | 7–10 | Ancient trees, deepest danger; wraps the far side of the map. |
| Silk Web Approach + Cavern | 150–250, SW far corner | ~20–110, 250–340 | 6–10 | Furthest from town along the Bramblefen axis; only 1 of the zone's 2 caves. |
| Hidden Tree Cave (concealed) | inside Gloamroot Depths | ~250–290, 260–300 (indicative) | 4–8 | Small secret cave, deliberately away from main paths; exact spot to be chosen for concealment quality, not formula. |

This layout is a starting proposal only. It preserves every existing distance-scaling formula's *shape*
(closer to spawn = safer) while giving each named region a distinct compass direction so they read as
separate places rather than concentric rings. Phase 3 should treat the boxes above as a first draft to
adjust once real terrain/road generation is visible at the new scale.
