# V0.20.47 — Walk Speed & FPS Counter

Two requested changes: halve the player's walk speed, and bring back the FPS counter.

## 1. Walk speed halved

Added `CONFIG.PLAYER_WALK_SPEED_SCALE` (0.5) and a single `Game.playerWalkSpeed()` helper. There are
**three** player movement paths and they had each read the speed stat independently:

| path | file |
|---|---|
| keyboard / camera-relative | `game.js` |
| click-to-move | `game.js` |
| controller | `systems/steam-deck-system.js` |

All three now go through `playerWalkSpeed()`, so they can't drift apart if one is edited later.

**The speed *stat* is deliberately untouched.** The character sheet, gear bonuses and speed buffs all keep
reading the real value — no itemisation or buff maths changes. Only the ground actually covered per
second halves.

**One trap caught:** the click-to-move path had a hardcoded `Math.max(0.8, ...)` floor. Left alone, that
floor would have silently cancelled the slowdown for any character at or below it. The floor now scales
with the knob.

**Verified through the real movement call** (`tryMoveActorSubstepped`, not just the number):

- stat speed 4.55 → walk speed 2.275, ratio **exactly 0.500**
- actual travel over one simulated second: **4.55 → 2.275 tiles**
- speed stat unchanged ✅

## 2. FPS counter restored

The logic was **never removed**. `updateFpsCounter()` computes a real presentation-cadence FPS (500ms
window, EMA-smoothed) and is called every frame. What happened is recorded in its own comment:

> `// V0.17.06: FPS is no longer rendered in the Mini-Map HUD.`

That change dropped the `#fpsCounter` element, so for many versions the code has been faithfully
computing a number and writing it into an element that didn't exist.

Restored the element — including the `.fpsValue` span the code looks for — with styling, positioned under
the mini-map, and added to the `body:not(.gameStarted)` hide list so it follows the same convention as the
rest of the HUD.

**Verified by driving the real updater** (not by inspecting state):

| simulated frame rate | rendered text |
|---|---|
| 60 fps | **"60 FPS"** |
| 20 fps | **"20 FPS"** |

So the readout tracks reality rather than just existing.

### Why it reads 0 in a headless pane

`updateFpsCounter` intentionally bails when `document.hidden` is true — you shouldn't measure the frame
rate of a page that isn't presenting. The Browser pane I test in is permanently hidden, which is why the
counter reads 0 there. That's correct behaviour, not a bug, and it's also the reason my earlier FPS
figures in this session were unearned.
