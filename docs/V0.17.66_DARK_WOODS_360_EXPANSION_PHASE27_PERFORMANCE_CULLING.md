# Dark Woods 360x360 Expansion — Phase 27: Performance + Culling

**Phase name:** Performance + Culling (Phase 27 of 28)
**Previous version:** V0.17.65
**New version:** V0.17.66

---

## Goal

Ensure the expanded 360x360 zone plus all the new props, resource/fishing nodes, overlays, atmosphere,
and dungeon content stays performant: audit per-frame work, camera culling, and the "avoid per-frame
random generation / duplicate render loops / thousands of DOM nodes" rules; optimize genuine hotspots.

## Audit (the expansion already follows the rules)

- **No per-frame random generation added.** Every `Math.random` in the resource/fishing paths is
  harvest-time (`resolveDirectDrops`, success roll), per-catch (`weightedPick`, catch chance), or
  spawn-time (`createFish`/`findWaterCandidate` on `ensureAquaticLife`) — none run in a per-frame render
  loop. Atmosphere/fog/ambient life use deterministic `seededNoise`. The dungeon generator uses only 6
  `Math.random`, all at gen time.
- **No duplicate render/update loops added.** Everything hooked existing pipelines: atmosphere and
  resource/fishing rendering go through `updateExternalSystems`/`renderExternalSystems`; the waypoint aura
  goes through the effects system; there is no new `requestAnimationFrame`/interval anywhere in the
  expansion's work.
- **Culling is in place.** Resource-node `render()` iterates only the ~18 overworld + few dungeon nodes
  and culls any node >34 tiles from the player; atmosphere/fog/ambient clamp to a render range with bounded
  cell counts (`MAX_FOG_CELLS`, `MAX_FIREFLY_CELLS`, `MAX_AMBIENT_CREATURES`); the waypoint aura is
  camera-culled by the generic effect-culling pass. Terrain uses cached chunk baking (Phase 13); props are
  sparse dressing in the existing gen pass. Fishing pools are plain water tiles (terrain render).

## Optimization applied

**Per-frame memo for `getWorldLightState()`** (`systems/time-cycle-system.js`). It is a pure function of
`(minuteOfDay, zone, weather)` — all stable within a single frame — yet it was called ~17 times per frame
**plus once per on-screen lantern** (Phase 17 added the lantern glow), once per wisp, and once for the
waypoint aura, each recomputing `computeLight` + the weather merge with **no caching**. Now:

- A frame token (`_worldLightFrameToken`) is bumped once per frame in `updateWorldClock` (which runs in
  `update()`, before `render()`).
- `getWorldLightState` caches `{ token, zone, light }` and returns the cached light when the token + zone
  match, recomputing only when the token changes (next frame) or the zone changes.

So the light is computed **exactly once per frame** and every other caller reuses it — eliminating the
redundant per-lantern/per-wisp recomputation the expansion introduced. No stale light (time and weather
still advance every frame; the token freezes correctly when paused/not-started, when time is frozen too),
no new loop/timer, and callers already treat the light object as read-only so sharing it for one frame is
safe.

## Validation

### Results
- **JS/JSON:** all pass. Clean boot at V0.17.66, **zero console errors**.
- **In-browser (no-cache server):**
  - Within one frame token, 3 `getWorldLightState()` calls return the **identical** cached object (1
    recompute, 2 hits).
  - Bumping the token recomputes (new object); changing the zone recomputes.
  - Light values remain correct and advance: `nightStrength` 1.0 at 22:00, 0 at noon (`phaseKey='day'`) —
    no stale light.
  - `updateWorldClock` bumps the token by exactly 1 per call (3→4→5→6), including at `dt=0` (before the
    early return), so the memo refreshes once per frame in the live loop.

## Known limitations
- The `getWorldLightState` savings are modest in absolute terms (it was cheap arithmetic); the change is a
  correctness-preserving precompute-once that removes the redundancy the expansion introduced, per the
  phase's "precompute where possible" rule.
- FPS was not numerically profiled at the menu (no in-world render there); stability is validated by the
  audit (culling + no per-frame random + no duplicate loops) and left to the manual play-session checks.
