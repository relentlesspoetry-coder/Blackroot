# V0.16.87 - Melee Auto-Attack Swing Animation Report

## Scope
Implemented the Melee Auto-Attack Swing Animation Pass only. This phase is animation and visual timing only.

## Auto-attack animation ownership
The real auto-attack cadence and hit resolution remain owned by:

- `systems/combat-system.js`

The visual melee swing state is now scheduled by that same combat owner through:

- `refreshPlayerAutoAttackVisualState()` for countdown/wind-up scheduling
- `startActorMeleeAutoAttackSwing()` for actual impact-frame alignment when a melee auto attack resolves

No separate auto-attack system was created.

## Character/class renderer ownership
The procedural humanoid pose and class renderer are owned by:

- `render/humanoid-animation-system.js`
- `render/humanoid-base-renderer.js`

Renderer changes were made there so the existing shared humanoid rig raises the weapon, swings downward, holds the impact frame, and recovers smoothly. The existing class-specific renderers still own class identity and equipment silhouettes.

## Detected melee classes
The patch uses existing combat metadata instead of forcing class roles. The existing melee auto-attack metadata marks these player classes as melee:

- Fighter
- Rogue
- Paladin
- Warden

Bard is not forced into this phase. If Bard is melee-flagged by existing runtime metadata, the shared lighter flourish swing path supports it; otherwise Bard remains unchanged.

## Entity types affected
The visual state applies to:

- Player melee auto attacks
- Bot/companion melee auto attacks using `botAutoAttack()`
- Melee mercenary-style bot actors if they share the same humanoid/class renderer and bot auto-attack owner

The patch does not target pets, skeletons, animals, ranged attackers, casters, or monsters with separate model/render ownership.

## Impact-frame alignment
The combat system keeps existing attack timers and hit resolution. The new melee swing state schedules wind-up during the existing auto-attack countdown. When the real auto-attack resolves, `startActorMeleeAutoAttackSwing()` forces the visual phase to the configured impact phase (`0.58`), so the weapon is visually at the strike/contact portion when damage is actually applied.

The animation duration is derived from the current auto-attack interval. If future haste/slow systems alter the interval through the existing attack cadence owner, the visual swing compresses or stretches with that cadence.

## Swing stages
The renderer uses staged easing:

- 0%–25%: weapon raise / wind-up
- 25%–55%: downward swing / strike
- 55%–65%: impact hold / follow-through
- 65%–100%: recovery

The animation is non-linear, using cubic easing-style curves rather than robotic linear interpolation.

## Class-specific styling
- Fighter: heavier two-handed cleave with broader arc and stronger follow-through.
- Rogue: fast compact dagger/short-blade slash with small wind-up.
- Paladin: controlled disciplined melee strike while preserving defensive off-hand posture.
- Warden: heavier grounded strike using the same heavy tank timing family.
- Bard: only supported if existing metadata routes Bard as melee; no Bard role conversion was added.

## Weapon trails / impact cue
A lightweight procedural weapon arc/trail is drawn by the existing humanoid base renderer. No blur filters, CSS filters, extra canvases, particle spam, or duplicate hit VFX were added.

The existing combat impact VFX remains the hit cue after damage resolution. No fake hit event was added.

## Protected behavior
The following were not changed:

- auto-attack damage
- auto-attack speed
- combat range
- hit chance
- crit chance
- threat math
- stamina, mana, energy, or focus behavior
- cooldowns
- class balance
- spell animations
- spell behavior
- class model identity
- action bar, inventory, bank/stash, settings, map, quest, dialogue, spellbook, party, mercenary, pet, save, or unrelated renderer systems
