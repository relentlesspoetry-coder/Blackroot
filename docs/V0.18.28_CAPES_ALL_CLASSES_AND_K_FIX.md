# V0.18.28 — Capes on all classes + K regression fix

Two issues from the user:

> "We need to fix capes on all class models, also K doesn't open the professions
> and skills window anymore."

## 1. K stopped opening the window (regression from V0.18.27)

### Root cause
There were **two** K handlers:
- `game.js:1027` — the central keybind handler: `if (this.isActionKey(e, 'skills')) this.toggleSkillsPanel?.();`
- `systems/profession-system.js` — its own duplicate `keydown` listener calling `this.toggle()`.

Before V0.18.27 they fired **independently** (skills toggle + profession toggle) — that was the original "opening one at a time" desync. V0.18.27 unified the windows and routed *both* handlers to the same `profession.toggle()`. So a single K press now toggled the one window **twice** → net zero → K appeared to do nothing.

### Fix
Removed the profession system's duplicate `keydown` listener. The central
`game.js` binding is now the single K handler: `toggleSkillsPanel` →
delegates to `toggleProfessionPanel` → one toggle. Verified K opens/closes the
unified window cleanly (open → close → open).

## 2. Cape draws over back gear — for every class, not just the Bard

### Root cause (general)
The humanoid base renderer (`render/humanoid-base-renderer.js`) draws the cape —
the class `backCape` hook, or a **default cloth cape** when a class has none — in
`drawBackLayers`, which runs **early** (before the torso and the `chest`/belt-gear
hook). For front/side facings that's right (cape behind the body). But when facing
**away**, the cape is on the near side and must **drape over** the back-worn gear,
so drawing it early left belt pouches, quivers, instruments and weapons painted on
top of it. This affects **all** caped classes (Paladin `shortCape`, Ranger/Assassin
`splitCloak`, Wizard/Enchanter/Necromancer mantles, Warden, Cleric, Bard, the Druid
leaf-mantle, camp NPCs, and the default cape).

V0.18.26 fixed only the Bard, by hand, in its own model.

### Fix (one mechanism for all classes)
In the base renderer:

```js
// The cape (class hook or default cloth cape), drawn at a facing-dependent depth.
function drawBackCapeLayer(ctx, rig, palette, hooks) {
  if (hooks.backCape) { hooks.backCape(ctx, rig, palette); return; }
  if (rig.pose.action === 'swim') return;   // hidden under the waterline
  /* … default cloth cape poly … */
}

function drawBackLayers(ctx, rig, palette, hooks) {
  // Front/side: cape behind the body here. Back facing: drawn late (below).
  if (!rig.dir.backVisible) drawBackCapeLayer(ctx, rig, palette, hooks);
  if (hooks.backProp) hooks.backProp(ctx, rig, palette);   // props stay behind the cape
}

// …main draw sequence, after hooks.front:
if (rig.dir.backVisible) drawBackCapeLayer(ctx, rig, palette, hooks);  // drape over the back gear
```

So for any back-facing humanoid, the cape now paints **after** the belt gear
(`chest` hook), the instrument/props (`front` hook) and held weapons — over all of
it. Front/side facings are unchanged (cape behind the body). Back props (quivers,
stowed weapons) intentionally stay behind the cape. The Bard's V0.18.26
model-specific hack was reverted so the base renderer is the single mechanism.
