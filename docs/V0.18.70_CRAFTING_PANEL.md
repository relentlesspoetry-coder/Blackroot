# V0.18.70 — Crafting Panel Polish (Item 3 E) + effectCategory compile fix

Two things: the crafting-window UI polish the roadmap asks for, and a root-cause bug I found while building it.

## 1. Crafting panel — grouping + result info (`systems/crafting-system.js`)
The campfire crafting window (`refreshWindow`) now:
- **Groups recipes by category** with headers, in order: **Armor / Weapons / Cooked Meat / Cooked Fish / Meals / Health Potions / Mana Potions / Buff Potions / Materials / Other**.
- **Shows what each recipe makes** — armour shows `Armor +N · +stats`; consumables show `Restores N HP/MP · Buff: Name (Ns) · Cooldown Ns`.

Three new helpers do the work:
- `recipeOutputDraft(recipe)` — resolves the primary output item.
- `recipeCategory(recipe)` — classifies it (armour / weapon / potion by policy category / cooked fish / meat / meal / material).
- `recipeEffectText(recipe)` — formats the stats or consumable effect.

## 2. Root-cause fix — `effectCategory` was dropped on compile (`systems/item-compiler-system.js`)
While classifying recipes I found the item compiler was **dropping the top-level `effectCategory`** (and `armorType`) when compiling a draft into a runtime item. So at runtime `item.effectCategory` was `undefined`, which meant:
- the V0.18.66/68 consumable **cooldown routing** (`item.effectCategory → effectPolicy.cooldownGroupFor → shared cooldown group`) **never resolved** — potions were quietly getting *per-item* cooldowns instead of the intended shared potion cooldown; and
- the crafting panel couldn't tell a potion from a meal.

**Fix:** `compileOne` now carries `effectCategory` + `armorType` onto the compiled item, and `createRuntimeItemInstance` copies `effectCategory` onto consumable instances and `armorType` onto equipment instances.
