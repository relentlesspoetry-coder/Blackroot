# V0.20.49 — Humanoid Render Caching (Roadmap Item 21, first slice)

Your FPS counter read **12**, which matched the harness prediction and put the entire remaining gap in
**render**: ~70ms of an ~83ms frame (update is ~11ms after V0.20.46).

## The profile

Counting real canvas operations per frame:

| op | calls/frame |
|---|---|
| `beginPath` | 6,407 |
| `stroke` | 4,297 |
| `fill` | 3,135 |
| **`drawImage`** | **92** |

Ablation attributed **~12ms and ~2,612 of those strokes to entities** — 19 drawn actors at roughly **140
vector operations each**, because humanoid models were re-drawn limb-by-limb every frame.

## Root cause — not missing technology

`tryDrawEntityAtlasFrame` depends on a **pre-baked** atlas (`assets/atlases/entities.json`). It loads its
manifest but contains **zero pages and zero frames**, so `isReady()` is permanently false and **100% of
fallbacks reported `atlas_not_ready`** — every actor dropped to the procedural path.

Meanwhile the **runtime** sprite cache — which bakes a procedural model to a bitmap on demand — was
already healthy (**95% hit rate**) but had only ever been wired into the NPC path.

## The fix

Humanoid entities now route through that same runtime cache (`drawCachedHumanoid`). Two details make it
safe:

- The draw methods render into the module-scoped `window.ctx` rather than a passed context, so the bake
  **temporarily repoints `window.ctx`** at the cache's offscreen canvas and restores it in a `finally` —
  the identical pattern this file already used to cache nameplates.
- The cache **re-applies `_nameplateAnchor` on every draw including hits** (stored as a delta from the
  bake origin), so nameplates, health bars and rod/VFX anchors keep tracking.

If the cache declines for any reason (per-frame miss budget, disabled, bake failure), the procedural draw
remains the fallback — nothing can go unrendered.

## Result

| metric | before | after |
|---|---|---|
| render | 31.4 ms | **18.3 ms** (−42%) |
| `beginPath` | 6,407 | 3,713 (−42%) |
| `stroke` | 4,297 | **1,685 (−61%)** |
| entities cached | 0/16 | **15/16** |
| bake failures / evictions | — | **0 / 0** |

The one uncached actor is a *Briar Boar* — a non-humanoid mob on a different renderer, correctly
untouched. **Nothing was removed**: same enemies, same NPCs, same props.

## Correctness verified, not assumed

A bad cache key is worse than a slow frame, so each discriminator was tested:

| check | result |
|---|---|
| 4 facings → distinct cache entries | **4 keys** ✅ (a shared key would leave actors facing the wrong way) |
| walk vs idle differ | ✅ |
| **player** gear invalidates + reverts | ✅ (gear lives on `game.equipment`, not the entity) |
| **bot** gear invalidates + reverts | ✅ |
| nameplate anchors | **16/16** ✅ |
| suite / console errors | ok / **0** |

**That last check caught a regression I introduced.** `resolveActorVisualEquipment` returns an *empty
object* for bots — which is truthy — and assigning it to `visualEquipment` **shadowed `botEquipment`** in
`buildKey`'s `(visualEquipment || equipment || botEquipment)` fallback. A bot changing gear would have
kept a stale sprite. It's now only assigned when a real gear map resolves.

## Still outstanding

Render remains the dominant half:

- **World objects** — ~11.4ms, 1,746 paths, still drawn procedurally every frame; wants per-chunk baking.
- **Effects** — ~6.3ms, allocating 308 gradients per frame.

And the standing caveat: the browser pane I measure in never composites, so my absolute render numbers
**understate** the real cost. Real FPS must be re-read in game.
