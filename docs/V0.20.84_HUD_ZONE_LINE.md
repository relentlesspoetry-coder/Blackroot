# V0.20.84 — The HUD Zone Line Was Static HTML

Screenshot showed the mini-map panel reading:

```
Blackroot / Dark Woods · 200×200
```

## Two faults in one string

1. **The size had been wrong for many versions.** Dark Woods was resized from 200 to 360 back in the
   V0.17.4x expansion; this label never followed.
2. **The zone name could never follow the player** — because `#minimapZoneLine` is literal markup in the
   HTML with **no JavaScript writing to it anywhere.**

Confirmed rather than assumed:

```
grep -rn "minimapZoneLine" --include=*.js .   →   (no matches)
```

Zero references. It was decorative text that happened to be true on the day it was typed, and had been
quietly lying ever since. Worth noting this is a *different* label from the zone-map panel header fixed
in V0.20.83 — same symptom, unrelated code, which is why the previous fix didn't touch it.

## The fix

`syncMinimapZoneLine()` derives the text from live state:

- the product name,
- **the same `mapZoneLabel()` the zone-map panel uses** — so the two surfaces can never disagree again,
- the actual map dimensions.

### Where it is called from mattered, and the first attempt was wrong

I first hooked it into `drawMinimap`, on the assumption that runs per frame. The label then showed the
**previous** zone after every crossing. Measured:

```
drawMinimapCallsPer10Frames: 0
```

`drawMinimap` only fires on certain HUD events — it is not part of the per-frame render path at all. So
the label was updating on whatever unrelated event happened next, which is exactly the one-step lag
observed.

It is now driven from `setActiveOverworldZone` — the only thing that can change this text. That makes it
correct *and* free, rather than correct and per-frame.

The write is still guarded by a cached string, so an unchanged label never touches the DOM. Verified: **0
DOM writes across 30 identical frames.** This project has spent many versions fighting for frame time and
a per-frame DOM write for unchanged text is pure waste.

The stale markup in the HTML was corrected as well, so even the first paint before any script runs is
right.

## One deliberate choice

The label uses the literal `'Blackroot'`, **not** `DR.DEFAULT_WORLD.worldName` — that field still reads
`'Dream Realms'` because it is part of the persisted save schema, and using it here would have regressed
the HUD branding to the old name. A "cleaner" data-driven read would have been a visible downgrade.

## Verified

| check | result |
|---|---|
| at boot | corrects the stale markup → **Blackroot / Dark Woods · 360×360** |
| after travel to Ashen Valley | **Blackroot / Ashen Valley · 450×450**, immediately, no render required |
| after walking the road through | tracks correctly |
| back home | **Dark Woods · 360×360** |
| zone-map panel agreement | both read "Ashen Valley" |
| fog key | `ashen_valley` |
| 30 identical frames | **0 DOM writes** |

Suite ok, 0 console errors.
