# V0.20.38 — Colourblind Status Markers (Roadmap Item 20)

The **3rd of 12** accessibility features — the spec's *"avoid colour-only status."*

## Audit first — most of it was already handled

Status effects were **already mostly colourblind-safe**:

- Every family renders a distinct **glyph, always** — ☠ poison, ❄ frost, ✚ heal, ▣ ward, ✹ bleed…
- The player's buffs and debuffs sit in **separate trays** (positional, not colour).
- Every pip has an **aria-label** (`"Buff:"` / `"Debuff:"`) for screen readers, plus a timer ring.

**The one real colour-reliant gap:** on a **mixed** tray (a *target's* effects, where buffs and
debuffs appear together), valence — beneficial vs harmful — is two round pips distinguished mainly by
colour, and a few glyphs collide (✦ is shared by curse/stun/mana).

## The fix — an opt-in shape marker

`colorblindStatusMarkers` pref (off by default, its own Accessibility row, persisted via `uiPrefs`).
When ON, each pip gets a **shape** valence marker:

- **▲ up-triangle** = buff
- **▼ down-triangle** = debuff

Top-left corner, white with a dark text-shadow outline so it reads over **any** pip colour. `renderTray`
reads the **live** pref (trays re-render each tick, so toggling is instant) and injects the marker. The
family glyphs and aria-labels are **untouched** — this only *adds* the valence shape.

## Design decisions

| decision | why |
|---|---|
| **opt-in toggle**, not always-on | ▲/▼ on every pip is clutter for players who don't need it; the always-on glyphs already carry family identity. This adds valence for those who want it |
| **shape**, not a second colour | the whole point is not relying on colour — a triangle direction works in any colour vision |
| glyphs + aria-labels **unchanged** | they were already good; don't disturb what works |
| off by default | a default must never silently change the HUD for someone who didn't ask |

## The lesson

**Audit before building.** The instinct was "status effects rely on colour, add symbols" — but reading
the code showed the symbols (glyphs) were already there and always on. The actual gap was narrow
(valence on mixed trays), so the fix is narrow (a ▲/▼ marker) rather than a redundant re-do of the icon
system. The same discipline that keeps finding *"this is already handled"* (one-shot avoidance, pack
behaviour) kept this change small and correct.
