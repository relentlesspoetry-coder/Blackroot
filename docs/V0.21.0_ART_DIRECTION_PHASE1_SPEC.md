# V0.21.0 — Art Direction Phase 1, Done To Specification

V0.20.96 claimed this phase but deviated on its two hardest requirements. This version implements them
and corrects the reasoning I gave for skipping them.

## The correction

V0.20.96 refused to change `TILE_W` / `TILE_H` / `ELEV_STEP`, stating they were *"load-bearing for
collision, prop anchors, every authored landmark coordinate and every saved world."*

**That was wrong.** The claim was repeated in `core/config.js`, `PATCH_NOTES.md`, `VERSION.txt`,
`README.txt` and `docs/V0.20.96_ART_DIRECTION_PHASE1.md`.

Auditing every read site found 28, and all are projection or presentation:

| file | use |
|---|---|
| `game.js` | `worldToScreen` / `screenToWorld`, render-list projection |
| `render/terrain-renderer.js` | isometric basis vectors |
| `systems/render-backend-system.js` | isometric basis vectors |
| `render/entity-renderer.js` | jump-lift offset |
| `systems/fishing-system.js` | cosmetic line-length clamp |

Nothing in collision. Nothing in the save system. Nothing in stored coordinates. The world grid is in
**tile** space; these constants only decide how that grid projects to pixels.

The lesson matches this project's recurring one: **reading the code gave a different answer from
auditing it.** I asserted a blast radius instead of measuring it, then shipped that assertion as
documented fact across five files.

## What changed

| | before | after |
|---|---|---|
| `TILE_W` | 104 | **96** |
| `TILE_H` | 58 | **48** |
| ratio | 1.79 : 1 | **2.000 : 1** |
| `ELEV_STEP` | 28 | **32** |
| zoom default / min / max | 1.28 / 0.72 / 1.9 | **1.0 / 0.85 / 1.35** |
| camera | `snap-isometric-8` | **`snap-isometric-4`** |

**Camera.** The spec's first choice is a fixed camera, naming `snap-isometric-4` as the fallback *"if
Q/E rotation must stay."* Rotation is established here — four versions of terrain corner work
(V0.20.92–95) went into making walls read at every angle — so the sanctioned fallback ships.
V0.20.96's *eight* angles were not on the spec's list at all. `SNAP_ANGLES = 1` makes it fully fixed.

**`applyArtDirectionProfile()` is now actually called**, from `game.js` at boot. It never was: the
profile shipped in V0.20.96 and nothing invoked it, so the camera mode and snap count were never
applied and CONFIG/profile agreement was never checked.

It also no longer *writes* the metrics the way the spec's sample code does. `DR.CONFIG` is frozen, so
`this.CONFIG.TILE_W = p.tileWidth` would have been silently discarded — leaving the profile and the
real projection disagreeing with no symptom. `core/config.js` is the single source of truth; the
profile verifies against it and warns loudly on drift.

## Honest consequence

**The world is ~30% smaller on screen.** Effective tile edge fell from **76.2px** (104/58 at zoom 1.28)
to **53.67px** (96/48 at zoom 1.0) — a **29.6%** reduction.

This contradicts the spec's own *"Tile footprint: larger visual tiles than your current low-detail
pass."* That is an internal inconsistency in the document, not an implementation error. Its literal
numbers are what shipped.

Restoring the previous apparent size at a true 2:1 ratio:

| option | metrics | result |
|---|---|---|
| A (shipped) | 96 / 48 / 32 @ zoom 1.0 | spec-literal, 29.6% smaller |
| B | **136 / 68 / 45** @ zoom 1.0 | exact previous scale, true 2:1 |
| C | 96 / 48 / 32 @ zoom 1.35 | 5% smaller, hits the spec's zoom ceiling |

This matters **now** rather than later: Phase 2 authors the waypoint shrine against a tile scale, and
art produced at the wrong scale has to be redrawn.
