## Wall-face texture: written and reverted

Embedded stonework on cliff and cave faces — lit tops, shaded lower lips, grit between — positioned by
bilinear interpolation across the face quad, so it followed the wall at any camera heading rather than
being screen-aligned.

**Reverted, and not because it looked wrong.** `drawWallFaceDetails` runs **live — 138 calls per frame**
in elevated terrain — unlike the ground pass, which bakes into the chunk once and costs nothing
thereafter. Texturing cliff faces is still the right goal; it belongs in `buildTerrainChunk`.

## The performance numbers were wrong three ways

Recorded because it nearly caused a good change to be reverted.

| attempt | reported | reality |
|---|---|---|
| synthetic render loop | 114 ms/frame | artifact |
| in-game HUD | 0 FPS (earlier: 26, 13, 20) | artifact |
| selective stubbing | savings of 3.00 / 4.95 / 6.26 / 8.74 ms | all inside drift |

**Artifacts (1) and (2):** the browser preview pane is always `document.hidden`, so its
`requestAnimationFrame` is throttled and any FPS derived from it is meaningless. This is already
recorded in project memory and I failed to apply it.

**Artifact (3):** re-measuring the *untouched* baseline at the end gave **14.02 ms against the opening
19.75 ms** — a 29% swing with no code change. Every reported saving was smaller than that band. The
drift check must run **between every arm**, not once at the end.

### What can honestly be said

- Frame time is roughly **14–20 ms warm** (≈50–70 FPS), not 8 FPS and not 0.
- The ground texture pass is **cached**; its stub "saving" of 3 ms sits inside the noise, consistent
  with a near-zero per-frame cost.
- **The true frame cost of this session's work is still unknown** and needs a reading taken in the real
  game window, outside the preview pane.
