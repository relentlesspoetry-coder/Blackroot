## 4. Root Cause / Implementation Notes

**Reused rather than duplicated where possible.** Before adding anything new, existing props were checked
for reuse: `bones` and `caveWeb` (`drawBones`/`drawCaveWeb`) already exist and were already used outdoors
near this exact cave (the pre-existing "spider marsh landmark" in `placeWorldLandmarks`), so they're reused
as-is for "bones" and "web overlays" rather than rebuilt. `silkCocoon` (`drawSilkCocoon`) already exists as a
large vertical web-wrapped-body shape for the Silk Web Cavern dungeon interior (Phase 7's prop catalog is
already extensive — `webAnchor`, `eggCluster`, `webBones`, etc. were all found already implemented) and was
reused unmodified for "wrapped corpses" outdoors by passing `cocoonType: 'corpse'` (the function already
supports an arbitrary `cocoonType` string, rendered as a label).

**What needed new code, and why:** no existing object type covered "thorn bush," "web-covered bush," or
"broken weapon," so three small new draw functions were added. Each reuses the same
`fillPoly`/`drawPropContactShadow` helpers every other prop in the file already uses — no new rendering
pipeline. `drawDeadTree` already accepted an `obj.variant` number for color alternation; a `webbed` boolean
now optionally overlays a few web strands on the same tree silhouette, since the object grid holds exactly
one prop per tile (there is no way to "stack" a separate web object on top of a tree object at the same
coordinate — confirmed by checking `this.objects[y][x]` is a single slot, not an array).

**Placement mechanism:** `ensureSilkWebApproachDressing` reads the `silk_web_approach` region box directly
from `DR.DEFAULT_WORLD.zones.dark_woods.regions` (Phase 3's data — no new region/lookup system introduced)
and iterates its full `x1..x2, y1..y2` extent. For each tile it skips: non-existent tiles, already-occupied
tiles (`this.objects[y][x]` truthy — never overwrites an existing prop, including the pre-existing spider
marsh landmark dressing), water tiles, already-blocked tiles, and any tile within a 10-tile radius of the
actual Silk Web Cavern entrance (read from `DR.CAVE_BY_ID.silk_web_cavern.entrance`, not a hardcoded
duplicate literal, so it can't drift out of sync with the real entrance coordinate). Placement is
deterministic, seeded by tile coordinate via the same `seededNoise(x, y, seed)` helper the rest of the
codebase's terrain/prop dressing already uses — not randomized per frame or per session.

**Call ordering matters here:** the function runs *after* `ensureDarkWoodsAtmosphereLandmarks` (which is
what places the cave-mouth object and clears the ground around it) specifically so this dressing pass's own
occupied-tile check sees the entrance's clearing already applied and never overwrites it, and *before*
`normalizeOverworldCollisionAndDensity` so that pass's `decorOnly`/`solid` handling still applies uniformly
to whatever this phase places.

**Collision:** all 4 dressing types were added to `decorOnly` (matching how `caveWeb`/`bones` were already
registered) so the density-normalization pass explicitly keeps them non-blocking even if something else ever
places one with `blocked: true` by mistake. Every placement call in this phase already passes `blocked: false`
except the webbed dead tree, which — consistent with the existing plain `deadTree` type already being in the
`solid` set — blocks, matching how every other tree in Dark Woods already behaves.

## 5. Validation

Commands run:

```bash
find . -name "*.js" -print0 | xargs -0 -n1 node --check
find . -name "*.json" -print0 | xargs -0 -n1 python3 -m json.tool > /dev/null
```

**Results:** both passed with zero errors.

**Functional validation (executed in Node, not just traced):** re-implemented the exact placement-eligibility
and threshold logic from `ensureSilkWebApproachDressing` in a standalone script using the project's real
`seededNoise` function (loaded from `core/utils.js`), run across the full `silk_web_approach` box
(`x:10–100, y:105–195`):

```
Eligible cells (excluding the 10-tile entrance clear radius): 7976
Total placed (ignoring water/already-occupied, which a static script can't simulate): 626 (7.8%)
  webbedBush: 272   thornBush: 118   deadTree(webbed): 73
  bones: 58         brokenWeapon: 39  silkCocoon(corpse): 32   caveWeb: 34
Violations inside the entrance clear radius: 0
```

This confirms the entrance-clearing exclusion genuinely works (zero placements within 10 tiles of the cave
mouth) and that overall density (~7.8% of eligible tiles) is comparable to, not denser than, the existing
sparse forest dressing pass in `placeWorldLandmarks` (which places at up to ~16% density with its own
threshold stack). The 626 figure is an upper bound — the live game additionally skips water and any tile
already occupied by pre-existing objects (roads, the spider marsh landmark, lanterns), which this static
script couldn't simulate without the full world array.

**Not validated (no browser available, same limitation as Phases 2–5):** the actual rendered appearance of
the new props, and whether the visual density reads as appropriately "spider territory" rather than
cluttered or sparse — that judgment requires seeing it rendered.

## 7. Known Limitations

- Dead grass and dark dirt patches (also named in the plan's Silk Web Approach identity) were not added in
  this phase — they would require either a new tile type or a terrain-tint mechanism, which is explicitly
  Phase 13 (Terrain Tile Detail + Overlays) scope. This phase covers props only, per its own scope list.
- Density constants (the `r > 0.9xx` thresholds) are a first pass and may need visual tuning once seen
  rendered in a browser.
- `silkCocoon`'s "corpse" label renders as on-screen text ("CORPSE") the same way its dungeon-interior states
  do (`'OPEN'`/`'SURVIVOR'`) — this may read fine as environmental storytelling or may look out of place as
  ambient outdoor scenery; flagging for visual review rather than presuming either way.
- No in-browser visual/boot confirmation was possible in this environment (see Section 5).
