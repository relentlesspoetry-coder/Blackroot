# V0.18.23 — Bard/Druid fishing-line rod-tip anchor fix

User (fishing on a Bard) reported, with a screenshot:

> "Fishing line is broken again, it's not attached to the fishing pole!"

The line hung from the character's mid-body with the bobber, detached from the
visible rod tip.

## Root cause
The fishing line is drawn by `systems/fishing-system.js` from
`player.fishingRodTipScreen` (a live screen-space anchor) down to the bobber.
That anchor is published in `render/entity-renderer.js` (~L1105):

```js
if (e.fishing && e.fishingRodTipLocal /* finite */) {
  e.fishingRodTipScreen = {
    x: visualFoot.x + Number(e.fishingRodTipLocal.x) * scale,
    y: visualFoot.y + Number(e.fishingRodTipLocal.y) * scale
  };
  ...
}
```

i.e. it only publishes when the model draw has recorded `e.fishingRodTipLocal`
(the rod-tip offset in model space). The **shared** class rig
`drawClassFishingRig` (class-identity-procedural-model.js, used by
Fighter/Rogue/Cleric/Enchanter/Summoner/Necromancer) records it (added in
V0.17.68):

```js
if (source && typeof source === 'object') {
  source.fishingRodTipLocal = { x: tip.x, y: tip.y };
}
```

But **Bard and Druid draw their rod through their own `drawFishingRod`**
(`render/bard-procedural-model.js`, `render/druid-procedural-model.js`). Those
functions compute `tip` with the *exact same formula* as `drawClassFishingRig`
but were never given the recording line. So for a bard/druid:

- `e.fishingRodTipLocal` stayed unset →
- the entity-renderer anchor block was skipped →
- `fishingRodTipScreen` was never published →
- the fishing system fell back to a world-space guess near the body →
- the line detached from the pole.

The V0.17.68 rod-tip fix simply never propagated to the two classes with a
bespoke rod. (Unrelated to V0.18.21/22, which only touched object scaling and the
terrain depth-wash — this is a pre-existing bug that surfaced now.)

## Fix
Added the one missing line — identical to `drawClassFishingRig` — right after the
`tip`/`mid`/`butt` computation in **both** `drawFishingRod` functions:

```js
// Expose the live rod-tip offset (model space, relative to the foot anchor) so
// entity-renderer publishes the fishing-line screen anchor at the visible pole
// tip. Without this the <bard|druid>'s custom rod recorded no tip, so the line
// detached and fell back to a world-space guess (mirrors drawClassFishingRig).
if (source && typeof source === 'object') source.fishingRodTipLocal = { x: tip.x, y: tip.y };
```

`source = actor.sourceEntity || actor` in both functions (same as the shared
rig), and the `tip` formula is byte-identical to `drawClassFishingRig`, so the
recorded offset lands on exactly the pole tip that is drawn.

## Verification
- `node --check` on both files — OK.
- Boots with **0 console errors**.
- Drove both models with a synthetic fishing actor
  (`model.draw(ctx, { action:'fishing', sourceEntity, ... }, now)`): each now
  records `fishingRodTipLocal = { x: 126, y: -109 }` (model space — up and out at
  the visible rod tip) with **0 throws**. Previously the field was never set.
- Full in-world visual is the user's sign-off (they are actively fishing on a
  bard; the canvas page can't be screenshotted by the preview tooling).

## Files touched
- `render/bard-procedural-model.js` — `drawFishingRod` records
  `fishingRodTipLocal`.
- `render/druid-procedural-model.js` — `drawFishingRod` records
  `fishingRodTipLocal`.
