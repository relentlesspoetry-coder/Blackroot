# V0.18.79 ‚Äî Save Migration: New Content Reaches Existing Saves (Roadmap Item 10)

## Symptom
Gear added in V0.18.75‚Äì78 (belts, bags, chest, legs) did not appear for players who had already
been playing. In the preview, a session that had auto-loaded a saved world showed a **stale runtime
item catalog** (`compiledItemById` lagging behind `DR.ITEM_BY_ID`) even though the item **data** was
correct and current.

## Root cause
This is a save-shadowing bug, not a data bug:

1. `systems/world-serializer.js` `serialize()` snapshots **every shipped-content catalog** ‚Äî items,
   spells, loot tables, recipes, quests, npcs, mobs, bosses, dungeons, puzzles, professions, crafting
   stations, resource types ‚Äî **into the world save** (e.g. line ~395 `items: cloneJson(game.editorItems || DR.ITEM_BY_ID)`).
2. `Game.prototype.autoSaveCharacterState()` (`systems/save-system.js:1841`) calls `saveWorldState()`
   on the **character-autosave timer** ‚Äî so a world save (with the full catalog snapshot) lands in
   `localStorage` after a normal player plays for one autosave interval.
3. On boot, `game.js:606` `loadSavedWorldState({ duringBoot:true })` restores it, and
   `DR.WorldSerializer.apply()` used `cloneObjectOr(migratedPayload.items, DR.ITEM_BY_ID)` ‚Äî **"use
   the saved catalog if present, else the shipped one."** With a saved catalog present, the stale
   snapshot **fully shadowed** the shipped `DR.ITEM_BY_ID`, and the runtime catalog rebuilt from it.

Net effect: any content added in a newer build was invisible to every player who had autosaved once.

## Fix (surgical ‚Äî in the owning module)
New helper in `systems/world-serializer.js`:

```js
function mergeShippedCatalog(savedValue, shipped) {
  const base = objectOr(shipped, {});
  const saved = objectOr(savedValue, null);
  if (!saved) return cloneJson(base);
  return cloneJson(Object.assign({}, base, saved)); // shipped underneath, saved wins on collision
}
```

`apply()` now uses `mergeShippedCatalog(migratedPayload.X, DR.X_BY_ID)` instead of `cloneObjectOr`
for **all 14 shipped-content catalogs** (quests, resourceTypes, professions, craftingStations,
recipes, npcs, dungeons, puzzles, mobs, bosses, mobSpawnGroups, items, lootTables, spells).

- Every current **shipped id is present** ‚Üí new content appears in old saves.
- Any **saved entry wins on id collision** ‚Üí editor edits and custom (save-only) ids are preserved
  untouched.
- **Purely additive** for new shipped ids ‚Äî it never mutates or drops a saved entry, so it cannot
  regress an existing save or an editor customization.

### Deliberately left untouched (still `cloneObjectOr`)
These are the player's **world/progress**, not shipped catalogs, and must not have shipped data
merged over them:
- Zone placements: `editorNpcs`, `editorMobSpawns`, `editorAttributes`, `editorResources`,
  `editorEvents`, `editorDungeonMarkers`, `editorSpawnPoints`
- Runtime state: every `pending*RuntimeState`
- `zoneProperties`, `classSpellSlots`

## Out of scope (safe, non-destructive default)
Balance changes to an **existing** shipped id, and new drops added to a **pre-existing** loot table,
still do **not** retroactively override the saved copy (saved wins on collision). This is intentional:
it guarantees zero regression to saved data. New items remain fully obtainable regardless, because:
- **Vendor** stock (`shopItemIds`) reads the live `DR.SHOP_ITEM_LISTS` const ‚Äî always current, so new
  gear is buyable at the camp store on any save.
- **New recipe ids** merge in ‚Äî so new gear is craftable at the campfire.

A "shipped-wins refresh" (pushing balance/loot changes onto old saves, at the cost of overwriting
editor edits to shipped ids) would be a separate opt-in decision.

## Verification
- `node --check` clean on all JS; JSON valid; boots with **0 console errors**.
- **Simulated the exact bug:** serialized the current world, stripped all `item_legs_*` from the
  snapshot's `items` (a pre-legs save), then ran `DR.WorldSerializer.apply()`:
  - `staleSaveLegsCount: 0` ‚Äî the save had no legs.
  - `legsAfterMerge: 10`, `legsCompileAfter: 10` ‚Äî all 10 shipped legs merged back in and compile as
    `slot: legs` (equippable). **Bug fixed.**
  - `savedOverrideWins: true` ‚Äî a saved item override still wins on id collision. **No regression.**
  - `customEntryPreserved: true` ‚Äî a purely-custom save-only id is preserved (`313 = 312 shipped + 1
    custom`).

## Impact
Unblocks the value of the Item 4 itemization work: the belts (V0.18.75), bags (V0.18.76), chest
(V0.18.77) and legs (V0.18.78) already shipped now reach players who started before those versions.
