# V0.20.68 — Mount/Taming Debug Tools (Roadmap Item 23)

## Why these, rather than a snippet

Asked for a test mount "here and now". A throwaway console snippet would have worked once. Item 23
already specifies these exact tools — *"unlock mount"*, *"remove mount"*, *"spawn beast"*, *"start taming
attempt"* — so they are roadmap work that happens to answer the request permanently.

The debug file's own header and `help()` still read *"mounts/taming … arrive with those systems (Roadmap
items 7, 6 and 16)"*. That stopped being true when Item 7 landed in V0.20.64. Both corrected — stale
help is worse than no help.

## Quick start

```js
DarkWoodsGame.debug.on()
DarkWoodsGame.debug.unlockMount('mount_gloom_wolf')   // omit the id for all ten
```

Then **Y** for the stable, **X** to ride.

To exercise the minigame instead:

```js
DarkWoodsGame.debug.on()
DarkWoodsGame.debug.giveBait('gloomWolf')
DarkWoodsGame.debug.spawnBeast('gloomWolf')
DarkWoodsGame.debug.tameTarget()
```

## Commands

| command | does |
|---|---|
| `mounts()` | every beast: level, rank, bait, owned |
| `unlockMount(id?)` | grant one, or all ten |
| `removeMount(id?)` | remove one, or clear the stable |
| `spawnBeast(beastKey)` | spawn pre-weakened to its own threshold, and target it |
| `giveBait(beastKey?, qty=3)` | grant that beast's bait, or every bait |
| `tameTarget()` | begin an attempt, or explain why not |

## Design decisions worth recording

**Routed through the owning systems.** `unlockMount` calls `mountSystem.unlockMount` — the same function
a successful tame calls. A debug grant is therefore indistinguishable from an earned one, persists
identically, and **cannot produce a save state the game could not have created itself**. That is Item 23's
explicit safety rule, and it is also why a bug found while testing this way is a real bug rather than an
artefact of the test path.

**`spawnBeast` pushes to both `game.enemies` and `game.entities`.** A runtime spawn missing from
`entities` is AI-active but invisible — a trap this codebase has hit before.

**The tools explain refusals.** `tameTarget()` returns the evaluation, not a bare boolean:

```js
{ started: false, reason: "Requires level 5." }
{ started: false, reason: "You have already tamed a Gloom Wolf." }
```

A tool that answers `false` when the character is merely under-levelled is not a debugging aid — it just
moves the mystery. `spawnBeast` likewise reports `requiresLevel` / `yourLevel` / `requiresRank`.

## Safety (unchanged)

Development-only. Every command refuses until `debug.on()`, a red **DEBUG** badge shows the whole time,
and every command is logged to both the console and the in-game log.
