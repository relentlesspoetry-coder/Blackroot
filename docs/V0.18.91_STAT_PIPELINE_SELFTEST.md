# V0.18.91 — Stat Pipeline Regression Self-Check (Roadmap Item 14 / Item 22)

Item 14 requires regression tests for the stat pipeline (equip / unequip / death / respawn / zone /
save-load); Item 22 wants automated validation beyond syntax checks, covering gameplay regressions.
This version delivers the first slice: an isolated, always-on self-test of the stat pipeline's core
invariants, and the seed of a broader gameplay-regression suite.

## The self-test — `runStatPipelineSelfTest()`

New `Game.prototype.runStatPipelineSelfTest()` (`systems/inventory-system.js`, after
`getPlayerStatBreakdown`) asserts the roadmap's pipeline rules:

| # | check | why it matters |
|---|-------|----------------|
| 1 | baseline recalc is idempotent | proves no temporary bonus is written into base stats (`computePlayerDerived` always reads `baseX`) |
| 2 | equipping a stat item increases the stat | equipment feeds the aggregator |
| 3 | contribution is not double-counted on re-recalc | re-running recalc must not accumulate |
| 4 | unequipping removes the contribution exactly | back to baseline, no residue |
| 5 | a full armour set is detected | set membership works |
| 6 | set stat bonus flows through the pipeline | isolated by toggling `bonus.stats` on the same equipment |
| 7 | set bonus drops when a piece is removed | added once / removed when conditions fail |

## Safe by construction

- Runs on a **scratch player + equipment**; the real `this.player` / `this.equipment` are saved and
  restored in a `finally` block.
- The whole body is wrapped in `try/catch`; a failure is recorded as a failed check, never thrown.
- Returns `{ ok, total, failed, checks }` and stashes it on `game._statPipelineSelfTest`.

## Boot integration

Invoked once during init (`game.js`, after `bindEvents`), advisory-only:

- pass → `console.info('[Dream Realms] stat pipeline self-test: N/N checks passed.')`
- fail → `console.warn(... FAILED ..., [failed checks])`

mirroring the registry obtainability-audit boot log. A bare `try/catch` around the call guarantees it
never blocks the game loop. Callable manually via `DarkWoodsGame.runStatPipelineSelfTest()`.
