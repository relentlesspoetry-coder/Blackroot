# V0.20.74 — Side-On Arms, and the Whole Head Forward

Two follow-ups from screenshots.

## 1. Arms splayed in every view

`buildRidingRig` placed both hands left and right regardless of view. Side-on, the far arm reaches across
the rider's own body and belongs **behind the torso** — showing both was the remaining "sitting sideways"
tell after V0.20.73 fixed the legs.

| view | near hand | far hand |
|---|---|---|
| side | `nearSign * 13` | `nearSign * 5`, 2px higher |
| front / back | −11 / +11 (unchanged) | |

The `camView` / `frontOn` / `sideOn` resolution was **hoisted above the arm block**. It previously sat
below it, so only the legs could see it — which is why V0.20.73 fixed the legs and left the arms wrong.

## 2. The head box stopped at the shoulders

`mountOverlayBox` ended the head where the outline widens — but **a muzzle hangs below that line.**

Measured silhouette profile, black wolf, front-facing:

| dy | width | |
|---|---|---|
| −88 | 23 | head top |
| −72 | 33 | |
| −64 | 41 | widest head |
| −60 | 41 | head bottom |
| **−56** | **67** | body begins (shoulder jump) |

The cut landed at **−54**, just past the head, leaving the lower half behind the rider — exactly what the
screenshot showed. Height is now **×1.55**, covering **dy −91 to −34**.

**Over-reaching is safe by construction:** the box only clips a redraw of the *mount*, so a taller box can
reveal more of the animal's own front and nothing else. It cannot show ground, and it cannot draw
anything that was not already the mount.

## Known limitation — recorded rather than glossed

**I still cannot pixel-verify the finished image.** `worldToScreen()` does not return the same coordinate
space `drawEntity` draws in — the world camera transform is applied *inside* the draw path — so every
sample window aimed with it landed off-target.

That was finally exposed when a head-band probe returned **zero dark pixels with the overlay both on and
off**. Zero in *both* arms means the window was not over the character at all; it was never a render
fault.

So: geometry, call rate and rig anchors are verified. **How it looks is the user's confirmation.**

If a future pass needs real pixel verification, the fix is to capture the transformed `foot` from inside
`drawEntity` (wrap `drawRiddenMount` and record its argument) rather than recomputing position outside
the transform.
