# V0.18.5 — Meditation system review: safe cleanup

A full review of the meditation system was requested. This release applies **only
the safe fixes** — nothing that changes meditation mechanics, balance, or the core
state machine. The tick model (15s tick), the L1–20 progression chart, the regen
math, environment/Bard/group/Great-Stillness bonuses, and every interrupt path are
untouched.

## Fixed

### 1. Removed dead code
- **`Game.prototype.meditationRestoreAmount`** (game.js) — defined, never called
  anywhere in the codebase.
- **`meditationSafetyOk`** — the function in `systems/meditation-system.js`, its
  entry in `DR.MeditationHelpers`, and the `Game.prototype.meditationSafetyOk`
  wrapper. Defined and exposed but never invoked; bots and mercs gate their own
  meditation safety through `meditationDesireScore` plus their existing combat
  checks (`entities/bot-player.js`, `entities/mercenary.js`), so this helper was
  pure dead weight.

Removing `meditationRestoreAmount` also removed the **only non-tick caller** of
`calculateMeditationRegen`. That function has progress side-effects
(environment-visit tracking, and the `shared_dream` / Bard-duet achievement hooks).
With the dead caller gone, those side-effects can now only run on a real 15-second
meditation tick — closing the pathway by which they could have fired off-tick.

### 2. "Already fully rested" guard (player, UX only)
`updateMeditationForEntity` completes a session instantly when the actor is already
at full HP and resource. For the player that produced a jarring pair of log lines:
*"You sit down and begin settling into meditation."* immediately followed by
*"Meditation complete."*

`startMeditation` now checks, **for the player only and only when not force-started**,
whether HP and resource are already full and, if so, logs a single
**"You are already fully rested."** and returns without entering meditation. This
matches the current effective behavior (you could not meaningfully meditate at full
anyway — it insta-completed) with cleaner messaging. It is scoped to
`entity === this.player && !options.force`; bots and mercs pre-check need before
calling `startMeditation` and never start at full, so companion behavior is
unchanged.

## Deliberately left alone (would change behavior / the core system)
These were flagged in the review but not touched, because fixing them would alter
gameplay behavior or a shared contract — outside the "no core changes" scope. Each
is a decision for the owner:

- **Passive mana regen stacks with meditation ticks (player).** `combat-system.js`
  `regen(dt)` restores the player's mana every frame with no meditation guard, so a
  meditating player gets passive regen *plus* meditation ticks. This looks like an
  intended baseline (bots/mercs are excluded from `regen` and recover only via
  meditation); suppressing it would nerf player mana recovery. Left as-is.
- **`actorIsMeditating` is intentionally broad** (matches several flags and free-text
  state strings). This drives movement-freeze via `resolveActorMoveSpeed`, so a stray
  `isResting`/`isRecovering` flag elsewhere could theoretically freeze an actor.
  Nothing triggers it today, and narrowing a predicate this many systems depend on
  risks real regressions for a non-triggered latent case. Left as-is.
- **V0.17.21 player-XP revert safeguard** in `updateMeditationForEntity` — reverts any
  `player.xp` change during a meditation frame. It could in theory revert a legitimate
  XP gain that lands on the same frame, but it guards a real historical leak bug and
  removing it is riskier than keeping it. Left as-is.
- **`hpPerSec` / `resourcePerSec` chart-field naming.** The chart values are applied
  as a flat amount per 15-second tick (the deliberate V0.17.23 fix), so the field
  names read as per-second but aren't. Renaming would change the
  `calculateMeditationRegen` result contract and every reader — real breakage risk
  for a cosmetic gain — so the names stay with their existing explanatory comment.

## Verification (in-browser, port 8899, 0 console errors)
- `game.meditationRestoreAmount` and `game.meditationSafetyOk` (and the helper export)
  are all `undefined`.
- Player at full HP+resource: `startMeditation` returns `false`, does **not** enter
  meditation, logs one line.
- Player hurt (40% HP/resource): `startMeditation` returns `true`, `actorIsMeditating`
  is `true`, `calculateMeditationRegen` returns hp/resource per tick, and a forced
  recovery tick restored HP (L1 Cleric: 2 HP). Meditation mechanics intact.
