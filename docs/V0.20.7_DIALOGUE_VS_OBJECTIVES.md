# V0.20.7 — Ask Me What I Want (Roadmap Item 5, final buildable piece)

The spec's layout list requires a **"clear visual distinction between spoken dialogue and objective
summaries"** — and the window had **no objective summary at all** to distinguish.

## The gap

`openNpcWindow` rendered:

- `dialogueText(npc)` — a single spoken line
- an action button reading **`Accept: Light the Way`**

**You agreed to a quest without ever being told what it asked of you.** Its authored task labels — 38
across 26 quests — sat unread in the data at the exact moment the decision was made. You found out
afterwards, by opening the journal.

## The build

`objectiveSummaryHtml(npc)` renders a briefing from those authored labels. The two halves are now
**different objects**, not one blob of text:

| | treatment | reads as |
|---|---|---|
| **spoken dialogue** | italic, warm, behind a speaker's rule | a person talking |
| **objective summary** | upright, own frame, monospace counts, uppercase heading | a job |

That *is* the distinction the spec asks for. Rendering a character's words and the game's instructions
identically is exactly how a person's line becomes a system message — which is the thing Item 5 exists
to kill.

### Details that matter

- **An offered quest shows the required count only — not `0/5`.** A zero against work you haven't
  started reads as *failure*; you haven't failed anything, you've been *asked*.
- **An accepted quest** switches to live `n/required` with a tick on finished objectives, and the
  heading changes off the pick mode: *"What this asks of you" → "Underway" → "Ready to report"*.
- **The summary reads the same `chooseQuestFromSource` pick the action button uses**, so the briefing
  and the button can never disagree about which quest is on the table.
- **An NPC with no quest — or a quest with no tasks — renders nothing.** An empty briefing frame is
  worse than none.

## Also — the NPC's title

The subtitle printed the **raw data key** as the NPC's title: `quest_giver`, `fisher_provisioner`. The
spec asks for *"NPC title or role where applicable"*.

Humanised **for display only** — `Quest Giver` — with the underlying value untouched and
`mercenary_master`'s existing special case preserved.

## Verification — driven through the real window

| check | result |
|---|---|
| quest **offered** | lists objectives, **no progress counts** |
| same NPC, quest **accepted** | `2/8`, heading **"Underway"** |
| NPC with **no quest** | renders `''` — **no empty frame** |
| speech | computes **italic** + left rule |
| objectives | compute **upright** + own border |
| they can overlap? | **no** — separate elements |
| all 10 distinct roles | humanised, **no underscores** reach the player |

The finished window was **screenshotted**: portrait, name, role, italic speech behind its rule, and the
framed `WHAT THIS ASKS OF YOU · Relight lanterns on the trail`. It reads as intended.

**Suite PASS:** statPipeline 15/15, items 396/396, 0 data/economy/dialogue/spellShape issues,
registryWarnings 2, **0 console errors**.

### A typo, caught

A corrupted colour literal (`#c9b<CJK>98`) was typed into the first draft — a stray character in a hex
value. Caught and fixed before it shipped; `node --check` wouldn't have flagged it, since it's a valid
string.

**Two requirements need content, not code:**
- **Alternate expressions** — no NPC authors one. The portrait renderer takes a descriptor; an
  `expression` field would slot straight in.

  until one is written; `openNpcWindow` already repaints per speaker, so the mechanism is there.
