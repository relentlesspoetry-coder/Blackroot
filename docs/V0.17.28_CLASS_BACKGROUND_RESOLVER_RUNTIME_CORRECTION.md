## Root Cause
The previous pass did not complete the runtime binding. `ensurePlayerHudClassVisuals()` updated a dataset key, then recursively called itself with `force=true` instead of applying `applyClassHudVisualStyle()` to the HUD. That prevented the fast HUD path from pushing class background variables into the actual `.hudClassBackdrop` surface.

The Character Model preview also rebuilt its DOM after the first class visual application, so newly created preview surfaces could miss the active class background variables. CSS also relied on background shorthand with fallback layers, making the fallback green/dark panel win when the class art variable did not resolve at the exact surface.

## Resolver Changes
- Kept `classBackgroundVisualSpec()` as the canonical source of class background paths.
- Expanded `applyClassHudVisualStyle()` to set both existing variables and explicit image aliases: `--party-class-bg-image` and `--pd-class-bg-image`.
- Synced rebuilt HUD/preview child surfaces without adding panel-specific class maps.

## Player HUD Changes
- Replaced the recursive `ensurePlayerHudClassVisuals()` call with direct canonical resolver application.
- Preserved the current HUD size, text, bars, class icon placement, and resource logic.

## Character Preview Changes
- Re-applied class background styling after the Character window rebuilds its markup.
- Changed preview CSS to use explicit `background-image`, `background-size`, `background-position`, and `background-repeat` layers so the class image remains visible behind the model.
- Reduced overlay opacity so class art is visible but the model remains readable.

## Fallback Behavior
If class art cannot resolve, the preview/HUD uses the existing dark fallback. Missing class background asset warnings remain gated behind the debug/dev path.
