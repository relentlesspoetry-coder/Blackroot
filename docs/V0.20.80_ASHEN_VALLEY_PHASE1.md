# V0.20.80 — Ashen Valley Phase 1: a Second Overworld Zone

Roadmap Item 26. This phase ships **the land and the road to it**. The city, mobs, items, quests and the
Ashen Valley Crypt are later phases.

## Scope, stated honestly up front

The Ashen Valley specification is **3–5× the entire existing game**:

| | in the game today | the spec asks for |
|---|---|---|
| items | **397 total**, all level <10 | 420 class-specific **+** ~420 general |
| quests | **26 total** | 35–40 + 8–10 repeatable |
| enemy types | **28** | **100 variants**, 500 abilities |
| overworld zones | **1** | +1 at 450×450 |

It is a multi-version programme, and the spec's own §28 says as much. Phasing agreed with the user:
**foundation → city → subregions/mobs → items → quests → crypt.**

## The core problem: there was only ever one overworld

`game.overworldMap` / `overworldObjects` / `overworldEnemies` were plain fields holding Dark Woods.
`portal-system.js:261` restores the overworld with a literal `this.map = this.overworldMap`. Caves swap
maps, but through a different mechanism entirely. A second *walkable region* had nowhere to live.

### Why this is not a cascade patch

The three names have **~70 call sites across 20+ files.** Renaming them to a keyed collection would have
meant touching all 70 — precisely the cascade-patching this project forbids.

But counting the split changed the answer:

```
writes: 8      reads: 62
```

So `core/overworld-zone-registry.js` makes the three names **prototype accessors** onto the active zone's
slot. Every existing read works unchanged. Every existing write lands in the active zone. Switching zones
is one assignment, in one place.

**Install order is load-bearing.** The registry installs *first* in `game.js`, before `new Game()`,
because the constructor assigns all three fields — and an assignment with no setter present creates an
**own property that permanently shadows the prototype accessor**. That is the same shadowing trap that
broke ctx patching during the V0.20.4x profiling work. Verified explicitly:
`ownPropertyShadowing: false`.

## What Ashen Valley is

450×450, levels 10–20, nine subregions:

| subregion | levels | tiles owned |
|---|---|---|
| Abandoned Cemetery | 17–20 | 2,296 |
| Abandoned Mine | 15–17 | 2,576 |
| Sunken Marsh | 14–16 | 3,366 |
| Old Ruins | 15–17 | 4,026 |
| Ashen Riverlands | 11–13 | 4,014 |
| Ashwood Forest | 12–15 | 4,918 |
| Valley Farmland | 10–12 | 4,850 |
| Ashenfall City | 10–20 | 5,396 |
| Rocky Highlands | 16–18 | 5,495 |

Terrain lives in `systems/ashen-valley-world-system.js` — a river spine from the highlands into the
marsh, forest in the west, rising stone north and east, a flattened city plate, and roads reaching every
subregion so the zone is traversable before any content exists in it.

It is a **separate file from `world-system.js`** deliberately: that file is 1,600 lines of Dark
Woods-specific geography, and growing a second zone inside it would make both harder to change.

The tile contract is matched **exactly** — `{ type, elev, blocked, waterDepth }`, `blocked` derived from
`TILE_DEF[type].walk` — so collision, pathfinding, swimming, fishing, minimap and the renderer all work
on the new zone with no changes at all.

**Generation is lazy.** 202,500 tiles is not worth paying at boot for players who never travel; it costs
**42ms** on first entry. This project has spent many versions fighting for frame time and boot cost is
not free.

## Fast travel

`systems/waypoint-system.js` had *already* set the condition for this work, in its own header:

> "with exactly one waypoint in the game so far, a fast-travel menu/teleport UI would have nowhere
> meaningful to travel between yet, so that is intentionally left for whenever a second waypoint exists."

A second waypoint now exists. Travel is implemented to the spec's rules: discovery is **per character**
and personal, undiscovered destinations are refused, and **every refusal returns a specific reason** —
a silent failure at a waypoint strands the player with no feedback.

## Four bugs found by testing — all of them mine

1. **The Ashenfall waypoint was nearly placed in Dark Woods.** `Game.placeObject` writes into
   `this.objects`, the *active* zone, and Ashen Valley is generated while Dark Woods is still active.
   Same class as the Silk Cavern runtime-spawn list mismatch: a helper that silently targets the wrong
   collection. Now written directly into the new zone's own array, and verified absent from Dark Woods.

2. **A new character inherited the previous character's zone** and would have begun at level 1 in the
   level 10–20 region. Fixed with a mirror in `resetCharacterOwnedState` — the same belongings-leak
   shape as the V0.20.2 mount bug.

3. **That mirror initially set only the zone ID, not the live map.** A fresh character stood on Ashen
   Valley's 450×450 terrain while every readout said `dark_woods`. Nastier than the original bug,
   because it looks correct from the outside — caught only by asserting on `map.length` as well as the
   id.

4. **Subregions were authored largest-first.** Box regions are tested in array order and the first match
   wins, so **Sunken Marsh was entirely hidden behind Ashen Riverlands** and **Abandoned Mine behind
   Ashwood Forest** — neither could resolve anywhere in the zone. Found by sampling every subregion
   anchor rather than a couple. Now ordered smallest-area-first, with the reason recorded at the data.

## Deliberately not in this phase

The **city is an empty plate.** Its ground is cleared and flattened so it can be built on known-good
terrain, but nothing is placed on it. Per the agreed plan, Ashenfall's render density must be
**measured before it is populated**: Dead Lantern Camp already costs ~23,700 draw ops at 15 FPS, and a
capital with districts, crowds and market stalls is strictly denser. Building it first and optimising
afterwards is the approach that failed three times already (the sprite atlas and two caching passes,
all reverted).
