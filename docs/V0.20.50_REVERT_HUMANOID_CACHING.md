# V0.20.50 — Revert of V0.20.49 (humanoid sprite caching)

My V0.20.49 optimisation **made performance worse** — 12 FPS down to 7 — and **broke the snare-root foot
anchors**. Both were my fault. Both are undone.

## Why it failed

The runtime sprite cache key *correctly* includes the animation bucket and `moveBlend`. That's right for
correctness — but it means an actor who is **moving** invalidates their key every frame.

The cache has a per-frame miss budget (96 bakes). Real motion blows it instantly — **2,514 budget skips
measured in a single sampled frame** — so `drawModel` **declined nearly every call**.

But by the time it declined, each actor had already paid for:

1. an entity object spread (`Object.assign({}, e, …)`),
2. an equipment resolution,
3. a long cache-key string build (with `equipmentSignature`),

…and then **the procedural draw ran anyway**. I added pure overhead on top of the original work.

## Why my verification missed it — the real lesson

I benchmarked by calling `render()` in a loop on a **frozen frame**. No `update()`, so nothing moved,
every cache key was stable, and the cache reported a **92% hit rate**.

That number was real and completely meaningless. **A render-only benchmark is not a benchmark** for a
cache whose key depends on motion. The measurement has to contain the thing that breaks it.

## The root anchor bug

Humanoid feet draw ~38px below the body origin, so `_rootVfxFootAnchor` sits at the **hips**. The true
foot line is published by the humanoid renderer as `_lastGroundAnchor.groundY` (V0.18.36), and the
effects renderer prefers it when present.

On a cache **hit**, no draw runs — so `_lastGroundAnchor` was never published, and the roots fell back to
the hip anchor. Roots at the waist, exactly as reported.

This is a general hazard with render caching: **any state a renderer publishes as a side effect of
drawing goes stale the moment you skip the draw.** The cache handled `_nameplateAnchor` (it stores and
re-applies it), which is precisely why I checked that one and wrongly assumed the rest were fine.

## Verified after revert

| check | result |
|---|---|
| humanoids publishing `_lastGroundAnchor` | **15/15** ✅ |
| player foot line vs hip anchor | **+33px below** ✅ (ankles, not waist) |
| entities still on the cache path | **0** ✅ |
| suite / console errors | ok / **0** |

The `drawCachedHumanoid` helper is deleted rather than left as dead code.

## Where Item 21 goes next

Not abandoned — but the approach has to change, and so does the measurement:

1. **Benchmark with `update()` running** so actors move. Every future perf claim gets measured under
   motion.
2. **Target costs that don't churn per frame** — static world objects (~1,746 paths/frame) are genuinely
   cacheable because they don't animate; effects allocate 308 gradients/frame that can be reused.
3. **Or reduce work outright** rather than cache it — fewer strokes in the procedural models, or fewer
   pixels via render-scale, both of which help regardless of motion.
