# V0.20.59 — The Actual Number: ~44,000 Canvas Ops Per Frame

Everything before this version was optimising blind.

## What settled the regime

The user dropped Render Resolution to 50% — a ~75% cut in pixels — and got **"lows 8, stable 12, highs 13"**
against a baseline of 9.

That is the whole diagnosis: **draw-call bound, not fill-rate bound.** The cost is the *number* of
drawing instructions, not the pixels they cover.

It has a second consequence that unblocked this pass: **this software-rendering preview is draw-call
bound too**, so for the first time it is a fair proxy — and **counting canvas operations is reliable here
in a way that timing them never was.**

## The number

**~44,000 canvas operations per frame.** A browser canvas2d sustains roughly **5,000–10,000 per frame at
60 FPS**. At 44,000 you get single digits, which is exactly what was being reported.

| phase | ops/frame | share | per item |
|---|---|---|---|
| **`drawEntity`** | **21,811** | **49%** | 18 entities → **1,212 each** |
| `drawObject` | 12,058 | 27% | 151 props → 80 each |
| `drawTile` | 3,491 | 8% | 58 tiles → 60 each |
| `npcRuntime` | 3,478 | 8% | 27 NPCs → **129 each** |
| `drawEffect` | 732 | | |
| prop contact shadows | 630 | | |

## The finding

The 1,212-vs-129 gap is the entire story. Camp NPCs cost **129 ops** because they blit from the sprite
cache. Nearby entities cost **1,212** because they are drawn live at full procedural detail.

`shouldRenderModelLive()` was rejecting them: **`spriteCacheLiveRadius: 18`** — set by V0.18.4 to fix
"animations only play fully in Debug mode" — **covers the entire visible screen.** Instrumented:

```
fallbackCalls: 20/frame    rejectedAsLive: 18/frame    cacheAllowed: 2/frame
```

The cache existed, worked, and was being handed almost nothing to do.

## The fix and the knee

`spriteCacheLiveRadius` **18 → 6**. Drift-checked (r=18 arm repeated last: 41,793 vs 43,954):

| live radius | ops/frame |
|---|---|
| 18 (before) | 43,954 |
| 10 | 28,842 |
| **6 (now)** | **26,697** |
| 4 | 25,122 |
| player only | 25,344 |

**6 is where the curve flattens.** Below it you surrender animation fidelity for almost no further gain —
and note player-only is no better than r=4, so swinging to the extreme buys nothing.

**The trade, plainly:** actors beyond 6 tiles animate more coarsely (cached buckets). Anything in combat
or conversation range still animates fully, which is what V0.18.4 was actually protecting. Set it back to
18 in `core/config.js` to prefer animation over frame rate.

## Two measurement traps found — both had already produced wrong answers

**1. `performanceSettings()` is memoized (V0.17.98).** Mutating `DR.CONFIG.PERFORMANCE` mid-run does
*nothing*. An earlier liveRadius A/B reported "no difference" (22,191 / 21,350 / 21,467) purely because
the config change never reached the reader. **Stub `shouldRenderModelLive` directly instead** — doing so
immediately produced the real 8× difference.

**2. `runtimeSpriteCache.beginFrame()` only runs from `loop(now)`.** Synthetic loops leave the cache
permanently budget-skipped (documented in V0.20.58).

Both traps share a signature: **the knob appeared to do nothing, which read as "not the lever."** When a
change produces *exactly* no effect, suspect the plumbing before the hypothesis.

## What is still left

`drawObject` — **12,058 ops across 151 props, of which 68 are measured off screen** (screen-y −350 to
−411). Same world-radius-instead-of-viewport pattern fixed for NPCs in V0.20.56. It needs an **asymmetric
pad** (generous below the viewport, tight above) because props draw *upward* from their ground anchor.

Rough remaining budget after this version: ~26,700 ops. Culling off-screen props should take ~5,000 more.
Reaching ~10,000 will also require reducing per-prop cost (80 ops for a tree) or the prop count itself.
