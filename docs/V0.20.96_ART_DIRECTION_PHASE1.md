# V0.20.96 — Art Direction Phase 1

First step of the requested migration to a high-detail dark-fantasy isometric presentation.

## Before anything else — what this migration can and cannot be

The reference image is a **painted or pre-rendered asset**. Phases 11–15, 17 and 18 of the supplied plan
(player/NPC sprite sheets, monster atlases, environment atlases, terrain tile families, UI art, remaining
zones) are **asset production**. They require an artist or a render pipeline. I can build every part of
the engine that *consumes* such art — metadata, loaders, validation, anchors, emissive masks, the
lighting passes that light it — and I can push the procedural renderers considerably closer to the
reference. I cannot draw the sprites.

Agreed direction: **build the pipeline, art arrives later.** That is what these phases deliver.

## Two plan assumptions this codebase already contradicts

Recorded because both change what "correct" means here.

**1. There is already 99MB of baked entity atlas on disk — and it was reverted for being slower.**
V0.20.53 shipped it; V0.20.54 pulled it after measuring **5–6× slower** rendering, because 4096² atlas
pages blit **6.6× slower** than small ones. The plan treats "move to atlases" as straightforwardly
correct. Here it has already failed once, and the next attempt has to start from that evidence, not from
the assumption.

**2. The plan requires a fixed camera; four versions of work just went into free rotation.**
V0.20.92–95 fixed terrain corner joins, directional face shading and material crush *specifically so
terrain reads at any angle*. An 8-angle snap is the compromise: rotation survives as a feature, and
per-angle authored art remains possible.

## What shipped

### 1. `systems/art-direction-system.js` — one source of truth

Style constants were scattered across four renderers. **That is precisely how the V0.20.93 regression
happened**: the per-face shade was computed in `terrain-renderer`, consumed nowhere, and when I finally
applied it I crushed four materials to black because nothing tied the values together.

The profile carries `ambient`, `terrain`, `shadows`, `lighting`, `actors`, `vfx`, plus:

- `artProfileValue(path, fallback)` for safe dotted lookup
- `terrain.wallFloorLuma` / `wallHeadroomDivisor` — the V0.20.95 fix, now a named knob rather than a
  magic number inside `drawLayeredWallFace`
- `lighting.boostExempt` — the elements that clip to white if night-boosted, named rather than
  rediscovered (the V0.20.89 lesson)

### 2. Eight-angle camera snap

```js
if (Math.abs(cam.yawVel) > 0.06) return;   // still turning — leave it alone
cam.yaw += (snapTarget - cam.yaw) * Math.min(1, dt * 9);
```

Easing rather than clamping, and only below a velocity threshold, so rotation feels continuous while the
key is held instead of notching between steps.

## Tile metrics deliberately not changed

The plan proposed `TILE_W 96 / TILE_H 48 / ELEV_STEP 32` against the shipped `104 / 58 / 28`.

Those three numbers are load-bearing for **collision, camera framing, prop anchors, every authored
landmark coordinate, and every saved world.** Changing them is its own measured migration with its own
regression pass — not a line in a style profile. `DR.CONFIG` is also frozen, so a profile that wrote to
it would fail silently.

The profile therefore **mirrors them read-only** and records the target:

```js
proposed: { tileWidth: 96, tileHeight: 48, elevStep: 32, status: 'not applied - needs its own migration' }
```
