# V0.16.94 - Meditation System Major Feature Revamp Report

## Phase source

- Design document used: uploaded `Dream Realms - Meditation System.txt`, adapted into BlackRoot's existing architecture, naming, save system, renderer, HUD, and bot/mercenary systems per the phase brief's adaptation rule.
- Baseline used: `Blackroot V0.16.93`
- Target produced: `Blackroot V0.16.94`

## Pre-existing state (audited before any changes)

BlackRoot already had a large, organically-grown meditation implementation before this phase: a working R-key toggle, an interrupt system with 13+ call sites across 8 files routed through a single `cancelMeditation` choke point, a 2-second settle delay, level-scaled tick/restore formulas (levels 1-60), a flat `+3` HP/MP Bard aura, and 9 of 14 classes with bespoke meditation auras (Fighter, Paladin, Rogue, Cleric, Enchanter, Summoner, Necromancer, Bard, Druid). It had no true state machine (booleans/strings only), no centralized balance config, no achievement/title system, no world-event scheduler, and no Bard tier/Duet/group-synergy/environmental-bonus mechanics. This phase builds the requested feature set on top of that base rather than replacing it.

## Ownership

A new owning module, `systems/meditation-system.js`, was added (registered in `_script_order.json` and the HTML's script list right after `systems/combat-system.js`, and installed in `game.js`'s `installExternalSystems`-adjacent sequence via `window.DreamRealms.MeditationSystem?.install(Game)`). It owns:

- `MEDITATION_BALANCE` (the spec's exact object) and a separate `MEDITATION_TUNING` table for implementation constants (settle seconds, grace/rush windows, synergy radii/percentages, etc.).
- The Level 1-20 XP/HP/mana chart (`MEDITATION_LEVELS`).
- State machine constants (`MEDITATION_STATES`, `MEDITATION_AI_STATES`).
- Environmental bonus table (`MEDITATION_ENVIRONMENTS`), achievement/title table (`MEDITATION_ACHIEVEMENTS`), and per-class emote data (`MEDITATION_EMOTES`).
- The central `Game.prototype.calculateMeditationRegen(entity, context)` function.
- Bard tier/Duet/group-synergy/environment/Great-Stillness/desire-score/save-migration helper functions.

Player/bot/mercenary meditation **state transitions** (`startMeditation`, `cancelMeditation`, `updateMeditationForEntity`, `actorIsMeditating`) remain owned by `game.js`, which already had every interrupt call site wired through them; their bodies now call into the new central module for math/config instead of using inline/duplicated formulas. This avoids creating a second meditation-state owner while still centralizing balance.

## Where things live

- **Player meditation input/state**: `game.js` - `defaultKeyBindings()` (R key, unchanged), `toggleMeditate` (`systems/spell-system.js`, unchanged), `startMeditation`/`cancelMeditation`/`updateMeditationForEntity` (`game.js`, rewritten bodies).
- **Central regen calculation**: `Game.prototype.calculateMeditationRegen` in `systems/meditation-system.js`. Returns `{ hpPerSec, resourcePerSec, xpPerTick, multiplier, visualModifiers }` per the spec, capped by `MEDITATION_BALANCE.maxTotalRegenMultiplier`.
- **Meditation XP/leveling**: `systems/meditation-system.js` (`grantMeditationXp`, `MEDITATION_LEVELS`). `systems/skills-system.js` (the pre-existing Skills-panel owner) now mirrors this state into the legacy `meditationSkill` shape instead of running its own independent (uncapped, level-60-style) XP formula.
- **Class aura definitions**: `render/humanoid-base-renderer.js` (`meditationClassId`, `drawSharedMeditationAura` dispatch, per-class `drawXMeditationAura` functions). Five new classes were added here: Warden, Ranger, Assassin, Wizard, Shaman (`drawWardenMeditationAura`, `drawRangerMeditationAura`, `drawAssassinMeditationAura`, `drawWizardMeditationAura`, `drawShamanMeditationAura`), using the same lightweight screen-composite/gradient/particle technique as the existing 9. A generic `drawMeditationAscensionRing` fires at Level 20 for every class, and a light additional ring layers on at Level 5 and Level 15 (aura upgrade tiers) - see Known Limitations for how literal the per-class iconography is.
- **Pet mirroring**: `entities/pet.js` (`update()` sets a cosmetic-only `meditationMirror` flag, distinct from real `meditating`, and halts the pet's movement/target while the owner meditates) and `render/entity-renderer.js` (`drawMeditationMirrorAura`, a simplified 50%-opacity glow using the owner's class color, drawn under `drawPetSprite`).
- **Bot/mercenary meditation AI**: `entities/bot-player.js` (`restIfNeeded`) and `entities/mercenary.js` (`needsRest`, `handleMeditation`) remain the owning AI loops (they already had working HP/mana-threshold + safety checks). `Game.prototype.meditationDesireScore` (`systems/meditation-system.js`) computes a 0-1 weighted score from the spec's high/medium/low-weight factors and is used to nudge (not replace) each companion's existing recovery thresholds.
- **Mercenary Rest/Stand/Follow commands**: `systems/mercenary-system.js` (`Game.prototype.setMercCommand`) - `rest` is normalized to the pre-existing `meditate` command (same plumbing, no second command system); `stand` is a new distinct command that cancels meditation and suppresses auto-rest until a new command is issued; `follow` already existed and already breaks meditation. `entities/mercenary.js` (`handleMeditation`) was given a small `commandStand` branch. The legacy merc command panel (`systems/ui-system.js`) now includes a `stand` button.
- **Bard Harmonic Meditation**: `Game.prototype.calculateMeditationRegen` + `bardHarmonicBonusPct` (`systems/meditation-system.js`). Stage 1 (levels 1-9) grants the recipient's own "+1 effective level" rate delta (computed per-recipient from the Level 1-20 chart, not a flat percentage); Stage 2 (level 10+, Meditation Melody) grants a flat +15%; Stage 3 (level 20, Crescendo) grants the spec's `bardCrescendoMultiplier` (regen tripled) before the global cap. The Bard receives its own bonus too. The pre-existing flat `+3` `bardMeditationAuraBonus` helper in `game.js` was removed (see Protected/Removed below) since its only call site is now this central calculation; the existing `bard_meditation_aura` buff-status/VFX pipeline (`game.js`, `render/effects-renderer.js`) is kept as the visual indicator and its description text was generalized since the bonus is no longer a fixed number.
- **Bard Duet**: `bardDuetBonusPct` + `Game.prototype.trackBardDuetPartner` (`systems/meditation-system.js`), using the same 15-yard nearby-meditating-Bard detection as the aura, gated on two Bards both meditating with at least one at Meditation Melody+.
- **Group synergy**: `groupSynergyBonusPct` (`systems/meditation-system.js`), built on the existing `Game.prototype.getPartyCombatMembers` (`systems/party-system.js`) - no duplicate party tracking was added.
- **Environmental bonuses**: `DR.MeditationHelpers.detectEnvironment` (`systems/meditation-system.js`) reads (in order) an explicit `meditationGarden`/`meditationEnvironment` editor attribute, then zone-id substring matches (graveyard/crypt, cave), then existing tile terrain types (`data/tiles.js`'s `TILE.CAMP`/`TILE.RUIN`/`TILE.WATER`/forest tiles). Categories with no current BlackRoot map signal (inns, temples, ley lines, battlefields, musical venues, mountain peaks) only read the explicit attribute hook - see Known Limitations.
- **Meditation Gardens**: `systems/editor-system.js` gained a new "Meditation Garden" attribute/tool (alongside the pre-existing "Safe Zone" tool, same `setAttributeAt` mechanism), and `meditationGardenBonusPct` (`systems/meditation-system.js`) computes the `+1%`-per-meditator (max +20%) Communal Stillness bonus using the same party-member helper as group synergy.
- **Great Stillness**: `updateGreatStillness` (`systems/meditation-system.js`), scheduled off the existing world clock (`Game.prototype.getWorldTimeInfo`, `systems/time-cycle-system.js`) rather than a second clock - ticked once per frame from `game.js`'s main loop alongside the existing `updateBardMeditationAuraRuntime`. It never touches combat/threat/AI state; it only flags `isGreatStillnessActive()`, which `calculateMeditationRegen` reads for the regen/XP multiplier, and tracks per-entity uninutterrupted-hold time for the reward.
- **Save migration**: `DR.migrateMeditationSaveData` / `Game.prototype.normalizeMeditationState` (`systems/meditation-system.js`), called from `entities/player.js` (construction), `systems/save-system.js` (load), and lazily from `startMeditation`/`updateMeditationForEntity`/`grantMeditationXp` for any entity (so bots/mercs always have a valid object before first use even though their progression isn't persisted - see Known Limitations). Save/load additions are in `systems/save-system.js` (`meditation: cloneJson(...)` alongside the pre-existing `meditationSkill` field).
- **Achievement/title hooks**: `MEDITATION_ACHIEVEMENTS` + `unlockMeditationAchievement`/`checkLevelAchievements`/`trackEnvironmentVisit` (`systems/meditation-system.js`). BlackRoot has no achievement/title UI, so unlocks are migration-safe save flags (`entity.meditation.achievements`, `.titlesUnlocked`) plus a chat log line, per the phase's "hook-only" instruction.

## Class aura additions (BlackRoot-only classes not in the uploaded doc)

Per the phase brief, Paladin/Warden/Ranger/Assassin/Wizard/Shaman needed class-appropriate mappings. Paladin already had a bespoke "Premium" aura; the other five were added this pass:

- **Warden**: bark/stone/root guardian aura - green/brown palette, creeping roots on the ground, earth-tone motes.
- **Ranger**: leaf/feather/wilderness aura - green/parchment palette, drifting leaf shapes, a bow silhouette line.
- **Assassin**: red-black execution sigil - onyx/crimson palette, dagger-glint lines, faint smoke ellipses.
- **Wizard**: arcane blue/purple rune circle - ground glyphs plus fire/frost/arcane-colored motes.
- **Shaman**: storm/earth/spirit totem aura - teal/brown palette, a totem stake with crackling storm-lines.

## Save data

`entity.meditation` matches the spec's structure exactly: `level, xp, totalSecondsMeditated, unlockedEmotes, selectedEmote, achievements, titlesUnlocked, activeTitle, environmentalLocationsVisited, fullPartyMeditationCount, bardDuetPartners, lastMeditationBreakTime, lastMeditationStartTime, lastGreatStillnessParticipation, stillnessMoteUnlocked`. Migration clamps level to 1-20, floors xp at 0, and drops unknown emote/achievement ids with a `console.warn` rather than corrupting the save. No XP/level-up events fire during load (fields are assigned directly, not through `grantMeditationXp`).

## Known limitations

- **Meditation progression persistence is scoped to the player character.** Bots and mercenaries get a valid, lazily-initialized `entity.meditation` object at runtime (so AI/regen math always works) but it is not written to the save file, consistent with the pre-existing behavior where only the player's `meditationSkill` was ever saved and companion "is meditating" runtime state was already intentionally discarded on load (`systems/bot-player-system.js`'s save-load cleanup). The spec's example save shape (titles, duet partners, achievements) is also inherently player-shaped.
- **Environmental bonus detection is heuristic, not hand-authored.** Forests/water/caves/ruins/graveyards/campfires are detected from existing tile terrain types and zone ids; Inns, Temples, Ley Lines, Battlefields, Musical Venues, and Mountain Peaks have no current BlackRoot map data to key off, so they only activate via an explicit `meditationEnvironment` editor attribute (new hook, not yet painted anywhere) - this matches the phase brief's "do not hardcode every tile manually / add hooks and document editor follow-up" instruction.
- **Meditation emotes have no bespoke per-emote visual/animation.** BlackRoot's `F` key is already bound to fishing and there is no radial emote UI, so activation is via the new `/meditate emote` chat command (unlocked at Level 10, per spec) plus a `cycleMeditationEmote` hook; feedback today is a chat log line and an `entity.activeMeditationEmote` data flag a future pass could animate.
- **Aura level-tier "upgrades" are intensity-based, not fully bespoke re-illustrations.** Level 5/15 add an extra faint outer ring; Level 20 adds a generic orbiting-icon ascension ring keyed by a per-class color/shape table (`ASCENSION_RING_ICONS`) rather than hand-modeled geometry (e.g. literal orbiting longsword/greataxe/warhammer meshes for Fighter). This keeps the renderer's "no blur filters, lightweight particles" constraint and reuses the existing shared aura pipeline.
- **The supplemental aura layer in `render/class-identity-procedural-model.js`** (a second, lighter effect drawn for the same classes) was left as-is for the five new classes - it already falls back to each class's own `palette.glow` tint rather than the old default cyan, so it isn't broken, just undecorated with extra glyphs like the 5 original entries.

## Removed as proven dead/duplicate (documented per phase rule)

- `systems/ui-system.js`: a stale, unused duplicate `nextMeditationXp(level)` (confirmed no callers).
- `game.js`: the flat `+3` `bardMeditationAuraBonus(entity)` helper - its only call site was replaced by the central `bardHarmonicBonusPct` calculation; keeping both would have double-defined the Bard aura bonus in two places.
