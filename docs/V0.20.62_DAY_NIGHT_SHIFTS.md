# V0.20.62 — The Camp Works Shifts

The biggest single frame-rate win of this effort, and notably a **content** fix rather than a code trick.

## Why

Profiling isolated the camp: **8 FPS standing in camp against 20 FPS away from it.** The cause was the
sheer **number** of procedural characters drawn at once — **45–46**, at 129–1,212 canvas ops each.

Three separate attempts to make each character *cheaper* via bitmap caching all regressed and were
reverted:

| version | attempt | result |
|---|---|---|
| V0.20.49 | humanoid render cache | 12 → 7 FPS |
| V0.20.53 | full sprite atlas | 5–6× slower |
| V0.20.59 | live radius 18 → 6 | 12 → 10 FPS |

With "make each one cheaper" exhausted, the only lever left was **draw fewer of them.**

## Measured at camp

| | before | after |
|---|---|---|
| characters present | 46 | **23** |
| canvas ops/frame | 40,603 | **30,631 (−25%)** |

Measured interleaved, with shift state re-applied each frame (the first attempt at this A/B was invalid —
it forced everyone awake but then only called `render()`, so `update()` never re-applied the shifts and
both arms measured the same thing).

## How it works

Every NPC and every ambient bot gets a fixed day or night shift.

**Off-shift NPCs** walk to the nearest tent and set the **existing `_asleep` flag** — already honoured by
the renderer, both interaction pickers and the minimap. "Off shift" therefore inherits all four
behaviours for free instead of needing four new checks. This reuses the V0.20.44 night-sleep plumbing
rather than adding a parallel system.

**Off-shift bots** set `_offShift`, gated in exactly two places: the render-queue build in `game.js`
(so the work is skipped at queue time) and the entity update loop (so they cost no AI either).

## Shift assignment: sort and alternate, not hash

The first implementation hashed each id independently. It is stable across sessions — but **not
balanced**. On the real rosters it produced:

- NPCs: **11 day / 19 night** (wanted 14/14)
- bots: **7 / 10** (wanted 8/8)

Sorting the ids and alternating is *equally* deterministic across sessions and saves — same roster, same
shifts — while guaranteeing an even split. It recomputes only when the roster actually changes, so adding
an NPC does not reshuffle everyone (it re-derives from the same sorted order).

## Bug found while testing

An off-shift NPC that could not **reach** a tent — blocked path, crowded doorway, or no tent at all —
stayed `_asleep = false` indefinitely and appeared in **both** phases. **2 of 28 did exactly that.**

Fixed with an 8-second grace period: if still not bedded down, stand down where standing. NPCs stationed
away from camp (the bandit fence, the silk-web quartermaster) and the ethereal spirit healer have no tent
to walk to and now stand down in place directly, rather than marching across the map.

## Verified across a full day and night

| | day | night |
|---|---|---|
| NPCs awake (of 28) | **14** | **14** |
| bots active (of 16) | **8** | **8** |
| appearing in both phases | **0** | |
| appearing in neither phase | **0** | |

Complete coverage, mutually exclusive, exactly half.

## Gameplay consequences — accepted deliberately

Trainers, merchants and quest givers are **included**, as requested. If your class trainer works nights,
you come back after dark. Shopping, healing, resurrection and hiring each rest for half the cycle.

**One deliberate exception:** a bot already in the player's party ignores its shift. A party member
blinking out of existence at dusk mid-fight would be a bug, not atmosphere.

To carve out exceptions later, the cleanest hook is `npcIsOffShift` in `systems/npc-system.js` — return
`false` for whichever roles should always be on duty.
