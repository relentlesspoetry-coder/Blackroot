# V0.20.46 — Performance: the update loop ~5× faster, nothing removed

Reported symptom: everything laggy. Explicit constraint: **do not reduce enemy counts or any content** —
make it run smoother.

## I profiled instead of guessing, and my first guess was wrong

My prime suspect was my own V0.20.45 NPC animation work (turning on `action:'walk'` for ~28 NPCs, which
moves them to denser sprite-cache bucketing). **Measured: 7.8ms of a 53.8ms update.** Not the cause.

Selective stubbing gave real attribution:

| stubbed | frame ms | cost |
|---|---|---|
| `game.hasStatusTag()` | 41.8 | **26.9 ms** ← largest single item |
| `updatePatrol()` | 53.1 | 15.6 ms |
| entire enemy `update()` | 38.5 | 30.2 ms |
| removing all 444 enemies | 4.5 | 49.3 ms |

## Root cause

```js
hasTag(entity, tag) {
  if (!entity || !Array.isArray(entity.buffs)) return false;  // checks isArray, never length
  const key = slug(tag);                                      // two regex replaces + allocations
  return entity.buffs.some(...);                              // over an empty array
}
```

An **empty** `buffs` array passes the guard, so `slug()` ran anyway — for a constant literal like
`'stun'` — then `.some()` iterated nothing, to return `false`.

Live measurement on a camp frame: **517 `hasTag` calls, 100% of them against an empty buffs array.** All
424 enemies carry a `buffs` array; not one had a single buff.

## Fixes (all behaviour-preserving)

1. **Early-out on empty buffs** in `hasTag`, `hasCurableTag`, `statusSpeedScalar` — *before* `slug`. An
   empty buff list could never have matched a tag, so behaviour is identical.
2. **Memoised `slug()`** (bounded to 512 entries) — it's called with a tiny fixed vocabulary of constant
   literals hundreds of times per frame.
3. **My own O(n²) NPC neighbour scan** (V0.20.44) — replaced the `hypot` storm with an axis reject plus a
   squared-distance compare.

## Result — same scene, nothing removed

| | before | after |
|---|---|---|
| **update** | **53.8 ms** | **13.3 ms** (~5×) |
| `hasStatusTag` | 26.9 ms | 2.3 ms |
| `updatePatrol` | 15.6 ms | 5.4 ms |
| enemy update | 30.2 ms | 4.3 ms |
| NPC ambient life | 7.8 ms | ~0 ms |

Still present: **444 enemies, 27 camp NPCs, 42 camp props** — nothing culled, no counts reduced.

## Honest limits

- **Render is now the dominant half (~44ms) and is untouched.** Profiling showed no single hot spot —
  it's diffuse canvas2d work (canopy 3.1ms, effects 4.8ms, minimap 1.4ms, occlusion 1.8ms ≈ 11ms of
  37ms). Getting to 60 FPS needs a render pass: render-scale, or the WebGL backend (present but in safe
  mode), or batching. That's Roadmap Item 21 work.
- **I do not trust my render numbers.** The Browser pane page is permanently `document.hidden`, so it
  never composites and `requestAnimationFrame` is throttled to zero. That same limitation is why my
  earlier FPS claims were unearned — they came from a synchronous harness, not observed frames.
- **Real FPS must be re-measured in-game**: `DarkWoodsGame.startPerformanceBenchmark()` then read
  `perfStats.benchmarkAvgFps`.
