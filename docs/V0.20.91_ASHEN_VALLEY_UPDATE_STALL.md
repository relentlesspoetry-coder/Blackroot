# V0.20.91 — Ashen Valley Was Burning 35ms a Frame on an Empty Zone

The city phase was agreed to open with a **density prototype and measurement, not construction**. The
first measurement never got as far as density.

## The baseline came out backwards

| | draw calls / frame | ms / frame |
|---|---|---|
| Dark Woods camp | 3,691 | ~14 |
| **Ashenfall plate — empty** | **738** | **~38** |
| Ashen wilderness | 4,057 | ~43 |

The empty city plate draws **five times fewer** things than the camp and runs **three times slower**.
That immediately invalidates the plan as written: Ashen Valley is not draw-call bound, so prototyping
prop density there would have measured the wrong thing entirely.

## Located by splitting the frame

| | update | render |
|---|---|---|
| Dark Woods camp | 1.44 ms | 4.26 ms |
| **Ashenfall plate** | **34.88 ms** | 1.31 ms |
| Dark Woods (repeat) | 1.64 ms | 7.21 ms |
| Ashenfall (repeat) | **35.71 ms** | 1.36 ms |

Reproduced on repeat arms, because this project's synthetic timing drifts and a single sample proves
nothing. Rendering in Ashenfall is *faster* than Dark Woods — there is nothing there. The cost is
entirely in `update`.

Stubbing each external system's `update` in turn:

```
fishing        32.84 ms saved      ← of a 34.06 ms baseline
npcRuntime      0.65
taming          0.63
questRuntime    0.61
(with the whole external loop stubbed: 1.21 ms)
```

**96% of the frame, in the fishing system, in a zone with no player fishing.**

## Two compounding faults — the same root assumption

Both are the hardcoded *"there is only one overworld"* assumption that already broke the zone-map label
and the fog key in V0.20.83. This is its third appearance.

### 1. Every overworld was called `dark_woods`

```js
function zoneKey(game) { ... return 'dark_woods'; }        // for ANY overworld
function activeMap(game, zone) {
  if (zone !== 'dark_woods') return game.caveMap || ...;   // ← Ashen Valley took the CAVE branch
}
```

So Ashen Valley inherited Dark Woods' ambient fish, whose stored coordinates are dry land here. Every
fish failed its position check and asked for a new target — every frame.

`zoneKey` now returns the active overworld zone, and `activeMap`/`activeObjects` test for *"a zone with
no `parentZone`"* — a real property of the data — rather than one hardcoded id.

### 2. The water search only looked at the first 200×200 tiles

```js
const size = Math.min(map.length || 0, CONFIG.MAP_SIZE || 200);
```

`CONFIG.MAP_SIZE` is **200**. That constant predates both the Dark Woods 360 expansion *and* this zone.
On a 450×450 map it searched only the top-left corner — and **Ashen Valley's river lies entirely east of
x=280**, so that corner contains no water at all. Every fish failed all 520 random probes, fell through
to a full grid sweep, found nothing, and did the whole thing again on the next frame.

**Dark Woods survives the identical bug purely by luck** — its creek happens to sit inside its first
200×200. The constant has been wrong there for many versions and never once showed a symptom. Worth
recording: a latent bug that only manifests when the data moves is invisible to every test that uses the
old data.

A failed search now sets a **4-second per-zone backoff**. That guard exists for the zones not yet built —
a future region without water would otherwise hit exactly this.

## Verified

| | before | after |
|---|---|---|
| Ashenfall update | 34.88 ms | **1.93 ms** (18×) |
| Ashenfall whole frame | ~38 ms | **3.69 ms** |
| Ashen wilderness | 43.31 ms | 11.10 ms |
| Ashen riverlands | — | 9.92 ms |
| Dark Woods | — | unchanged within drift |

**Fishing still works in both zones** — both enter the active state on cast — and each zone now keeps its
own fish: 34 in Dark Woods, 10 in Ashen Valley.

*(A test assertion here initially read `false` because it checked `startFishing()`'s return value, which
is not a success flag; the system's `active` state showed both zones behaving identically.)*

## Noted for the content phase

Ashen Valley has **no fish table of its own** and falls back to Dark Woods' fish. Functional, but the
spec asks for 6 new fish for the region. Not fixed here because this version is a performance fix and
adding content to it would muddy both.

## The wider point

The agreed plan was to prototype city density before building. Had I skipped straight to construction,
the city would have gone into a zone already spending 35ms a frame on an empty-map fishing sweep — and
the resulting frame rate would have been blamed on the buildings.
