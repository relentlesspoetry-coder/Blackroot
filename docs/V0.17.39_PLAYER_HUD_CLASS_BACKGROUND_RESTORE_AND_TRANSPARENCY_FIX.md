## Root Cause of Player HUD Transparency / Missing Class Background
The class background resolver was producing valid class art paths, but the visible Player HUD card did not own the class art. Previous passes applied the art to the outer HUD wrapper or to an absolute `.hudClassBackdrop` layer. The visible top player card content lived in `.playerHudCore`, while the HUD container's black panel could still visually dominate or hide the separate backdrop layer.

## Player HUD Background Render / Compositing Change
`.playerHudCore` is now the actual class-background owner. It receives the resolved class background image, class emblem watermark, readable dark overlay, and inset frame treatment directly. The legacy `.hudClassBackdrop` is disabled so no hidden or covered background layer competes with the visible top-card surface.

## Class Background Resolver Usage
The existing canonical class visual resolver still supplies:
- normalized class key
- class background path
- class emblem path
- class color

The Player HUD applies those variables and the direct background style to `.playerHudCore`.

## Party HUD Regression Verification
Party HUD card layout and class-background treatment were not changed. Companion frames, party slot count, HP/resource bars, action/status text, and command buttons remain owned by the existing Party HUD renderer.
