# V0.20.53 — The Atlas Is Installed, and the Bug That Was Hiding It

The V0.20.52 baker ran locally. This version installs its output and fixes the one thing that would have
made it look like a failure.

## Installed

19 atlas pages, **7,969 frames**, into `assets/atlases/` — replacing the placeholder manifest that had
been sitting there since V0.13.69.

Validated **before** copying anything in:

| check | result |
|---|---|
| all 19 pages present | ✅ none missing |
| PNG signature + IHDR dimensions vs manifest | ✅ all 4096×4096, all match |
| frames referencing an unknown page | **0** |
| manifest shape vs loader expectations | ✅ `pages[].image` + per-frame `page` |

The loader has two paths — `fetch()` of `entities.json`, and a `<script>` fallback for `file://` where
fetch is blocked. `entities.manifest.js` registers itself under the key `'entities.json'`, which is
exactly the key `loadScriptManifest` falls back to, so **offline play from `file://` works too**.

Verified **after** install, in a running game:

```
manifestsLoaded 1   pagesRequested 19   pagesLoaded 19
framesLoaded 7969   invalidFrames 0     pageErrors 0    ready true
```

## The bug installing it exposed

`inferModelId()` (`systems/sprite-atlas-system.js:54`) carries explicit branches for spirit healers,
boars, wolves, rotlings, spiders, skeletons and bandits — then falls through to:

```js
return normalizeToken(renderer || family || actor.kind || 'Unknown', 'Unknown');
```

Three creature families have no branch. So they asked the atlas for:

| runtime asked for | baker had written |
|---|---|
| `wisp.duskwisp.idle.…` | **`Wisp.duskwisp.idle.…`** |
| `stag.hollowStag.idle.…` | **`HollowStag.hollowStag.idle.…`** |
| `deadroot.ashrootHorror.idle.…` | **`AshRootHorror.ashrootHorror.idle.…`** |

The frames existed, correctly baked, and were never found. Those creatures went on being redrawn stroke
by stroke every frame.

### Why this mattered so much

Resolved every live entity through the real runtime path (`buildEntityAtlasActor` → `getEntityFrame`):

| | before |
|---|---|
| resolved | **133 / 400** |
| missed | **267** |

Those three families are **265 of ~400 actors in the opening area**. A perfectly good atlas would have
delivered almost nothing, and the obvious conclusion would have been "the atlas didn't help" — the wrong
conclusion, drawn from a real measurement.

## The fix

Three branches in that same function, matching the pattern of the seven already there. Nothing else
needed touching: the **variant** tokens already lined up exactly — `duskwisp`, `lumenWisp`, `hollowStag`,
`thornCrownedStag`, `ashrootHorror` and `ashrootElder` all arrive via `mobVisualKey` through the default
path at `inferVariant` line 102. Only the model name was wrong.

| | after |
|---|---|
| resolved | **404 / 404** |
| missed | **0** |

## What I could not measure

The preview browser has no save file, so the game sits on the splash screen. Instrumenting the canvas and
calling `render()` there produces **1 `drawImage`, 53 `fillRect`, 0 path ops** — it is painting the menu,
not the world. **In-world FPS and `atlasDraws` cannot be read from this environment.**

That number has to come from your machine:

1. Load your character.
2. Watch the FPS counter.
3. `DarkWoodsGame.spriteAtlasSystem.getStats()` — `atlasDraws` should climb every frame, and
   `proceduralFallbacks` should stay flat.

If `proceduralFallbacks` does climb, the `fallbackReasons` map names the reason per actor, which points
straight at whatever is still missing.

## Backup command changed (decided and applied)

The baked pages are **87MiB of already-compressed PNG** — zip cannot shrink them, and they are a
reproducible build artifact rather than source. `RELEASE_CHECKLIST.md` now specifies:

```
zip -r -q "backups/Blackroot <PREV>.zip" . -x "blender/*" -x "backups/*" -x "assets/atlases/*.png"
```

Measured against the real tree:

| | size |
|---|---|
| with the exclusion | **165 MiB** — identical to the pre-atlas V0.20.51 backup |
| without it | ~252 MiB |

The pattern deliberately excludes only the **PNG pages**. `entities.json` and `entities.manifest.js`
compress well and stay in, so a restored tree still records exactly which 7,969 frames it expects — the
gap is obvious rather than silent.

**Consequence to know:** a restored backup has no atlas pages. It boots and plays, but with `pageErrors`
and procedural fallback (i.e. slow). Rebake per the instructions above, or copy the pages back — the
originals from this bake run are still in `~/Downloads/entities_*.png`, verified byte-identical to the
installed copies.
