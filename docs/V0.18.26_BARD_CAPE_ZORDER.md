# V0.18.26 — Bard cape draws over back gear (z-order fix)

User, viewing a Bard from behind (with belt pouches, an instrument and a weapon
visible on top of the cape):

> "none of these items should show through your cape."

## Root cause
The humanoid base renderer draws each model as an ordered sequence of layer
hooks (`render/humanoid-base-renderer.js`). The relevant order:

```
... hooks.back ...            (step 1)
drawBackLayers -> hooks.backCape   (step ~4, BEFORE the torso)   <- Bard cape here
drawTorso / paperdoll torso        (step ~6-7)
hooks.chest                        (step 9)   <- Bard belt, beltGear, sash
hooks.front                        (step 13)  <- Bard instrument (front layer)
hooks.effects                      (step 14)
```

The Bard draws its cape in the **`backCape`** hook, which fires **early** (before
the torso). Its belt pouches/sash draw in **`chest`** (step 9) and the instrument
in **`front`** (step 13) — both **after** the cape. For front/side facings that's
correct: the cape hangs behind the body and the gear sits on the front. But when
the actor faces **away**, the cape is on the near side and should **drape over**
the back-worn gear — yet it was still drawn early, so every belt item, the
instrument and weapons painted on top of it.

(The paperdoll `capeDepthLayer` already flips *equipped* capes to a late pass when
`backVisible`; the Bard's procedural cape simply didn't do the equivalent.)

## Fix (`render/bard-procedural-model.js`)
Draw the cape at a facing-dependent depth:

```js
backCape(ctx, rig, palette) {
  // Front/side: cape hangs behind the body (early), as before.
  // Back facing: drawn late in `front` (below) so it drapes over the back gear.
  if (!rig.dir.backVisible) drawCape(ctx, rig, palette);
},
front(ctx, rig, palette) {
  drawBootAndTrouserDetail(ctx, rig, palette);
  drawPerformerHandAccessories(ctx, rig, palette);
  if (pose.action === 'fishing') {
    drawFishingRod(ctx, rig, palette);
  } else {
    if (!hideDefaultLute && pose.action !== 'swim' && instrumentLayer(rig) === 'front') drawInstrument(ctx, rig, palette, 'front');
    if (pose.action === 'cast') drawCastHandGlow(ctx, rig, palette, true);
  }
  // Back facing: cape drapes over all back gear so none of it shows through.
  if (rig.dir.backVisible) drawCape(ctx, rig, palette);
},
```

Now, for a back-facing Bard, the cape (drawn in `front`, step 13) paints over the
belt gear (`chest`, step 9), the front-layer instrument, and any held paperdoll
weapon (which already flips to the back pass / step ~12 when back-facing). The
idle instrument is drawn in the `back` hook (step 1) when back-facing and is
behind the cape as well. The cape geometry stops at the shoulders, so the head/hat
(drawn earlier) stay visible.

## Scope — why only the Bard
`backCape` is a shared hook name but means different things per model, so a blanket
base-renderer change would misfire:
- **class-identity** (`drawBackSilhouette`) — the back **body silhouette**, not a
  cape; moving it late would paint the body over the gear.
- **Druid** (`drawLeafMantleBack`) — a leaf-mantle **underlayer** with different
  layering intent.

Only the Bard's `backCape` is an actual outer cape, so the fix is scoped to it.

## Verification
- `node --check render/bard-procedural-model.js` — OK.
- A back-facing Bard renders with **0 throws** across facings; boots with **0
  console errors**.
- Sampling the central back/belt band of a back-facing Bard: the region is now
  **cape-dominant** (cape purple + its dark lower gradient vs. leather-brown gear),
  i.e. the cape sits over the central back gear instead of the gear over the cape.
- Full in-world visual is the user's sign-off (turn the character to face away).
