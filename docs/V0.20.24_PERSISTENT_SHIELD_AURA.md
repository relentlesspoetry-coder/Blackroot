# V0.20.24 — A Shield You Can See For As Long As It Protects (Roadmap Item 6 fidelity)

## The spec's example, structurally impossible before this

> *"A shield must remain visible for the actual protection duration."*

It couldn't. A shield fired a **~0.68s cast flash** and then nothing — Guardian Aura lasts **1800s** and
was visible for 0.68 of them — because **the entity renderer had no per-frame pass over active buffs at
all.** Confirmed: `entity-renderer.js` drew from `statusEffects`/`buffs` zero times; every buff visual
was a sub-second `spawnStatusPulse` flash.

This isn't a units bug like the AoE ring. There was no persistent visual to fix — **there was no
system.** This builds one.

## The design property that makes it correct by construction

`drawEntityStatusAuras(e)` is called from `drawEntity` **after the body** (so the aura envelops rather
than occludes), and it draws **straight off `actor.buffs`**:

> An aura's **visual lifetime IS the buff's gameplay lifetime.** It appears the frame the buff is
> applied and vanishes the frame `remaining` hits 0 — because there is no *second* lifetime that could
> drift from the first.

That's the whole point. A timed animation would be a duration guessing at another duration — the exact
class of bug this session keeps finding. Driving the draw from the buff list itself makes drift
impossible.

## Scope — the two categories the spec names, and only those

`isProtectiveBuff(b)` matches a buff as protective when **any** of:

- `tags` ∩ {shield, ward, barrier, bulwark, aura, absorb, bubble, guard, protect, aegis, sanctuary}
- `absorbRemaining > 0`
- `damageTakenMultiplier` (or magic variant) in `(0, 1)`

**Ground-truthed against real casts before writing the predicate:**

| shield | how it's caught |
|---|---|
| Shield of Faith | tags `shield`/`ward`, dtm 0.85 |
| **Guardian Aura** | **only tag is `aura`** — caught by the `dtm 0.97 < 1` arm |
| Radiant Bulwark | `absorbRemaining: 156` |
| Mana Shield | `absorbRemaining: 108` |

The `dtm < 1` arm exists specifically because Guardian Aura would slip past a tag-only check.

**DoTs are deliberately not drawn.** They're the spec's other named case and the same mechanism, but
they're on nearly every enemy in a fight — a persistent haze on all of them needs its own clutter/perf
pass. Honestly deferred, not forgotten.

## The near-miss, caught by testing

`age = duration − remaining`, so a **freshly-cast** buff has `age = 0 → fadeIn = 0 → invisible`. My
first test **froze time**, so `age` stayed 0 and **every case drew 0 ellipses — the shields *and* the
negative controls.** A vacuous pass that would have "confirmed" a feature drawing nothing.

Re-run with the buff aged past the 0.3s fade-in:

| case | ellipses | |
|---|---|---|
| shield-tag / aura+dtm / absorb-only | 3 / 3 / 3 | draws |
| haste buff / hostile DoT / no buffs | 0 / 0 / 0 | **negative controls that can fail** |
| two stacked shields | **3, not 6** | one dome |
| expiring at 0.2s remaining | 3 | still visible, fading out |

The fade-in is intentional — the shield *forms* as the cast flash fades — and a shield loaded from a
save mid-duration has `age > 0` and shows instantly.

## Verified through the live render loop, not a manual call

The half that killed V0.20.20 was a resolver that never ran, so I proved the **real pipeline** invokes
this:

| | |
|---|---|
| player-aura draws in ~1s **while shielded** | **218** |
| player-aura draws **the instant the shield was removed** | **0** |
| ally bot Calia's Guardian Aura | also drew a dome — predicate keys on the buff, not the actor kind |
| live renderer after instrumentation removed | **clean original, no orphan refs** |

Appears with the shield, vanishes with the shield, works for allies. A visual pixel-diff was attempted
and abandoned — the canvas is 964px at a ~1.2 devicePixelRatio while the anchors are in logical space,
so `getImageData` sampled the wrong band; the live-loop invocation count is the sounder proof, and a
screenshot shows the blue Mana Shield dome clearly enclosing the caster.

## Still open

- **DoT persistence** — the spec's other named case, deferred for a clutter/perf pass (DoTs are on
  nearly every enemy in combat).
- **Front/back dome split** — the dome is drawn additively after the body, which reads as enveloping;
  a true behind-and-in-front split would need a second draw hook before the body.
- Absorb shields (`absorbRemaining`) could **weaken visibly as they deplete** — a "power" signal the
  spec lists — but that needs a captured initial value and was kept out of v1 to avoid mutating buff
  state.
- Unchanged from before: `combatImpact`/`combatSpark` generic; projectile origin at torso-center, not
  the hand.

## The lesson

The AoE fix was a units bug — a number that meant two things. **This was an absence** — a whole visual
that was never drawn, hidden behind a cast flash that *looked* like the feature. Grep finds a wrong
number; it doesn't find a missing render pass. Only asking *"what draws while this buff is alive?"* —
and finding the answer was "nothing" — surfaced it.
