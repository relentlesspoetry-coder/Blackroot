# V0.20.8 — Two Men In One Pair Of Boots (Roadmap Item 2)

## The bug Item 2 names twice

> *"Prevent NPCs from spawning or standing on top of one another."*
> *"No overlapping NPC coordinates."*

`systems/world-system.js` → `ensureStartingCampRevamp` created **Quartermaster Brann** at
`(campX + 6, campY)`.
`systems/npc-system.js:242` had already placed **`npc_camp_merchant`** at `(106, 100)`.

`campX = 100, campY = 100`. **The same tile.** Two vendors occupying one square.

Found by **auditing every camp NPC pair for identical coordinates** — not by looking at the camp, where
a crowd of people looks like a crowd of people.

**Brann moves**, because Brann is the newer arrival: the revamp added him *onto* a merchant who was
already standing there. He moves to `(campX+7, campY+2)`, at the head of his own supply line — which
satisfies the same rule's other half: *"vendors and service NPCs should be positioned near objects
relevant to their role."*

**Both** branches that set his position were updated — the create branch **and** the `Object.assign`
branch that re-asserts position on an already-saved camp. Fixing only the first would leave every
existing save with Brann still inside the merchant.

## The camp wasn't much of a camp

The revamp placed **16 objects**: 5 tents, a fire, a cooking spit, 2 log seats, **one** bedroll, 4
lanterns, flowers, a stash.

Item 2's environmental targets ask for crates and barrels, weapon racks, storage piles, training areas,
tables and benches, medical supplies, food and water. The camp had **zero** of them.

**Recon explains why.** A previous pass deliberately *shrank* the camp to a "compact outpost", and its
`removeCampObjectTypes` set is still there, stripping exactly the objects Item 2 wants back:

```js
'largeCampTent', 'campLeanTo', 'vendorStall', 'campStall', 'crate', …
'weaponRack', 'supplyStack', 'studyTable', 'herbalistTable', 'forge', 'loom'
```

## What was added — 26 placements, none of it invented

**Every type was audited against `render/object-renderer.js`'s own `case` list first.** Placing an
object the renderer can't draw would be creating the very dead data this session keeps deleting.

**The herbalist table pays off an old debt.** V0.18.68's own note: *"the Herbalist Table is removed from
the camp by ensureStartingCampRevamp and never re-placed… deferred (ties into the Dead Lantern Camp
overhaul, Roadmap Item 2)."* This is that item.

## What was NOT added

**Hitching / animal areas.** The renderer has no such type — and the spec says *"where appropriate"*.
Inventing one means new art, not placement. A horse rail nobody can see isn't appropriate.

## Verified by measurement

| check | result |
|---|---|
| NPC exact-overlaps | **1 → 0** (Brann `7,2`, merchant `6,0`) |
| objects on an NPC tile | **0** |
| all 18 requested types actually placed | **yes** — a type that silently failed would be invisible |
| blocking objects among camp props | **only tents / lean-to** |
| each new tent's neighbours walkable | **4/4** — they're islands, not walls |
| **flood fill from the campfire** | reaches **all 27 camp NPCs** and **all 4 exits** |

That last one is Item 2's *"maintain clear walking paths between tents, NPCs, quest areas, and exits"* —
**proven rather than asserted**, by walking the camp tile by tile from the fire outward.

Screenshotted in-world: the camp reads as inhabited.

**Still open:**
- **Hitching/animal areas** — needs a new object type (art), not placement.
- **The `removeCampObjectTypes` stripper still runs.** It's now working *against* Item 2 rather than
  for it: any future camp content using those types will be deleted on load. It was left alone because
  it also cleans up genuinely stale saved camps, and untangling that is a separate, riskier change than
  this item needed.
- **NPC distribution** — Item 2 wants idle NPCs "distributed naturally around workstations, fires,
  tents, and social areas". The 27 are deliberately placed and non-overlapping, but they're still
  arranged in rings rather than clustered at the things they'd actually be doing.
