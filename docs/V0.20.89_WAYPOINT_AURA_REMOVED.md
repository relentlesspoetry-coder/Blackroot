# V0.20.89 — Old Waypoint Aura Removed From On Top of the New Waypoint

Reported with a screenshot: a pale blue blob sitting on the centre of the new dais.

## It was not the new art

It was the **`waypointAura` effect**, added in V0.17.51 to give the *old* flat grey disc a pulsing glow,
rotating ring and rising motes. Effects render **after** world objects, so it drew straight over the
rebuilt stonework.

The V0.20.88 waypoint already draws its own core, ground pool, glyph rings, beam and motes — the aura had
become pure duplication, and the only thing it could do was cover the art underneath it.

Worth noting how easy this was to miss: the previous version's verification measured `drawWaypoint`'s own
output in isolation, on a blank backdrop. That is the right way to check the art, and it cannot possibly
see a *second* system drawing on top in the live scene.

## Removed in full

- the spawner (`ensureWaypointAuraEffects`) in `systems/waypoint-system.js`
- the dispatch branch and `drawWaypointAuraEffect` in `render/effects-renderer.js` (3.2KB)

The system now **purges** any `waypointAura` still present instead of creating them. Those effects carry
`life: 1e9` — they never expire — so a session already running, or any world carrying one, would have
kept an orphan on the dais indefinitely. The purge runs on init and on the same ~1s cadence the old
refresh used, so anything that re-introduces one gets cleaned up rather than becoming permanent.

## One behaviour was worth keeping

V0.17.56 item 3 scaled the aura at night so a waypoint reads as a beacon in the dark. Deleting the aura
would have silently taken that with it — a real feature loss hidden inside a cleanup. It moved into
`drawWaypoint`, driven by the same world-light `nightStrength` everything else uses.

### Getting it right took three measured attempts

Each corrected by pixels, not by eye:

**1. Scaling `pulse` did nothing.** Measured: 2225 vs 2223 blue pixels — a rounding error, not an effect.
`pulse` already peaks at 1.0, so multiplying and clamping can only lift its troughs. The original aura
scaled **alphas**, which is the thing that actually controls brightness. Now so does this, through an
`L()` helper applied to 13 light alphas.

**2. That worked, and brought the blob back.** +22.7% brighter — and the saturated centre run went
**10px → 25px** at full night, because additively boosting something already near-white just clips it.
The core bloom and the beam **root** are now exempt from the boost. That is precisely the *"capped so it
never blows out"* note V0.17.56 wrote about its own aura; I had to rediscover it.

**3. Still 16px — so I located the runs by position.** All seven long runs were at the **dais centre**,
none at the brazier flames. That mattered: a hot core in a flame is correct, a hot blob on the dais is
the reported bug, and the aggregate number cannot tell them apart. The core disc's alpha is now divided
by `nightBoost`, compensating for the boosted pool and rings beneath it. Day is unchanged, because
`nightBoost` is 1.

## Verified

| check | result |
|---|---|
| brighter at night | **+33.9%** blue pixels |
| widest saturated centre run, day | **8px** |
| widest saturated centre run, midnight | **12px** |
| aura effects after 330 live frames | **0** |
| injected legacy aura | purged, 1 → 0 |
| renderer still handles the type | **no** — `drawWaypointAuraEffect` is `undefined` |
| stray effect of that type in the world | renders without throwing |

## Lesson

Two, and they are different:

1. **Verifying a prop in isolation cannot see what draws over it.** The art check was sound and still
   missed this entirely, because the fault was compositing order between two systems. A live-scene check
   was needed, and the screenshot was the thing that provided it.
2. **An aggregate brightness number cannot tell a hot flame from a blob.** Locating saturated pixels *by
   position* turned an ambiguous "16px" into a specific, fixable answer.
