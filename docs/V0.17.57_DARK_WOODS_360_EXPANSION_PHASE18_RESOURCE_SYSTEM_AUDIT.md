# Dark Woods 360x360 Expansion — Phase 18: Resource System Audit

**Phase name:** Resource System Audit (Phase 18 of 28)
**Previous version:** V0.17.56
**New version:** V0.17.57

---

## Goal

Read-only audit (like Phase 1) of the gathering, fishing, profession-EXP, resource-respawn,
item, inventory, vendor, and save-state systems, before adding the Dark Woods resource set in
Phases 19–23. Deliverables: an ownership map, a list of safe files/functions to modify, and a
clear statement of the architectural gaps/risks Phases 19–23 must plan for. **No functional
game-code changes this phase** (version/docs only).

## Key findings (the important ones for Phases 19–23)

### 1. The plan's proposed schema does not match the existing one — reuse, don't invent
Confirmed by grep: **none** of the plan's field names (`professionXp`, `firstDiscoveryXp`,
`respawnSecondsMin/Max`, `yieldMin/Max`, `allowedZone/allowedRegion/allowedBiome`, `nodeVisual`,
`questTags`) exist anywhere in the codebase (0 files each). The existing schema already covers
the same concepts under different names:

| Plan field | Existing equivalent |
|---|---|
| `requiredLevel` | `levelRequired` (the code already reads **both** — `requiredLevel()` checks `levelRequired ?? requiredLevel`) |
| `professionXp` | `xpReward` (note: actual granted XP = `xpReward + min(12, itemCount·2)`) |
| `respawnSecondsMin/Max` | single `respawnSeconds` (no randomized range today) |
| `yieldMin/yieldMax` | `drops[].min` / `drops[].max` |
| `vendorValue` | `sellValue` **on the item** (`data/items.js`), not on the node |
| `lootTable` | `lootTableId` |
| `rarity` | `rareDrops[]` chance + item `rarity` |
| `allowedZone`/`allowedRegion` | **implicit** — nodes are physically placed per-zone grid; region via placement coords + `getDarkWoodsRegionAt` |

**Recommendation for Phase 19–23:** extend `DR.RESOURCE_TYPES` and `data/items.js` using the
existing field names; optionally add the *new* concepts (`firstDiscoveryXp`, rare-node bonus) as
additive fields **only if wired cleanly** (see #4). Do NOT create a parallel resource registry.

### 2. Region→resource spawn hooks already exist (dormant, like Phase 16's signpostProps)
`data/default-zones.js` region metadata already declares `identity.resourceIds`:
- `dead_lantern_outskirts` → `['herb_lantern_moss']`
- `bramblefen_thicket` → `['herb_thornberry']`
- `stone_hedge_clearing` → `['herb_wispbloom']`
- `gloamroot_depths` → `['herb_blackroot']`
- `bandits_fall` → `[]`, `silk_web_approach` → `[]`

Four of the five overworld herb IDs are already mapped to their regions (the fifth,
`herb_gloomcap_mushroom`, is region-agnostic "shaded woods / fallen logs"). **But** these
`resourceIds` are currently read **nowhere** (dormant metadata). Phase 20 should consume them
(the same pattern Phase 16 used for `signpostProps`) to place nodes in-region, plus the region
box + `getDarkWoodsRegionAt` from Phase 3 for the region-agnostic ones. The actual resource
**definitions** (`herb_lantern_moss` etc. in `data/resources.js`) and **items** (in
`data/items.js`) do NOT exist yet — they are what Phases 20–23 create.

### 3. Fishing is a fundamentally different architecture from gathering
`fishing-system.js` is **fish-anywhere-near-water** (press F near any water tile), NOT placed
nodes:
- Fish are chosen from hardcoded `FISH_TABLES` keyed by zone (`dark_woods`, `cave`,
  `crystal_grotto`, `blackroot_catacombs`). Each entry: `{itemId, name, weight, xp, rarityKey, junk?}`.
  (`crystal_grotto`/`blackroot_catacombs` tables are **orphaned** — those caves were removed in Phase 5.)
- **No per-fish level requirement and no level gate** — any level can catch any fish in the
  table (weighted random). The plan's "low-level players cannot catch high-level fish" is **not
  enforceable today** without adding a `levelRequired` field to fish entries AND gating the
  weighted pick / catch resolution.
- **No respawn/depletion** — water is infinite; there are no depletable fishing nodes. The plan's
  per-fish respawn timers don't map to this model. "Rare fish in deeper/dangerous water" *can* be
  honored via the existing `waterDepth >= 9/10 → spot.rare` + `deepWaterBonus` hook.
- **Dual XP tracking (divergence risk):** `fishing-system` keeps its own `state.level/xp` with
  curve `max(65, floor(80·level^1.32))` AND also calls
  `resourceGatheringSystem.gainSkillXp('Fishing', xp)` (curve `max(50,100·level^1.45)`). Two
  Fishing levels with different curves that can diverge; the profession panel shows the
  resource-gathering one, catch-chance uses the fishing-system one. **Phase 19 must pick one as
  authoritative** (recommend: keep both fed but treat the resource-gathering "Fishing" skill as
  the level shown/gated on, and adjust the fishing-system to gate on that).

### 4. What already exists vs. what Phases 19/25 must add
- **Hard level gate + "requires" message: EXISTS.** `startHarvestNode` blocks with
  `"<node> requires <skill> <level>."` when `skill.level < requiredLevel`. Locked nodes render
  dimmer and the profession UI already says nodes show `!` when level is too low. Phase 25 just
  needs to align the wording to the plan ("Requires Gathering Level X").
- **Soft-failure: EXISTS.** `completeHarvest` uses a success chance
  (`0.78 fishing / 0.84 mining / 0.88 gather + skillAdvantage·0.035`, cap 0.98) with a consolation
  XP on fail. This is NOT the plan's exact 35/75/90/100% table, but the plan says "only if the game
  already supports failure chances" — it does. Phase 19 can optionally retune toward the plan's
  curve; the hard "cannot attempt below required level" rule is already enforced separately.
- **First-time discovery bonus: DOES NOT EXIST.** Would be a new additive concept in Phase 19
  (track discovered resource IDs in `state`, grant `firstDiscoveryXp` once). Feasible cleanly.
- **Rare-node XP bonus: DOES NOT EXIST.** `gainSkillXp` counts `rareFinds` but grants no extra XP
  for them. Would be a small additive change in Phase 19/20/22.
- **XP formula vs. plan table:** the current power curve does NOT match the plan's Gathering/
  Fishing table (100/150/225/325/475/675/950/1300/1750). Phase 19 must decide: replace
  `nextLevelXpFor` with the plan's explicit per-level table, or keep the curve. **Recommend
  replacing with the plan's table** since Phase 19's stated goal is "add/align the level 1–10
  progression"; verify no other system hard-codes the old curve first (profession UI and fishing UI
  both call `nextLevelXp`, so a single change propagates).

### 5. Silk Web Cavern (Phases 21 & 23) is the hard part — it is a DUNGEON
- Entering Silk Web Cavern routes to `dungeonSystem.enterDungeon` (`currentZone==='dungeon'`),
  because its cave def has `directDungeon`. It has its own map, floors, per-run state
  (`currentRunId`), and object system (`silkQuestObjective`, `silkCocoon`, `webAnchor` node types).
- The **fishing system explicitly disables itself in dungeons** (`render()`/aquatic update
  early-return on `currentZone==='dungeon'`). So **Phase 23 (Silk Web cave fish) cannot use the
  current fishing system as-is** — it needs either a dungeon-water fishing path or a different model.
- The **resource-gathering `zoneKey()` returns `'dark_woods'` for any non-cave zone (including
  dungeons)**, and only `editorResources.dark_woods`/`mossfang_cave` grids are serialized. So
  **Phase 21 (Silk Web herbs) cannot simply place nodes** without either (a) extending the
  gathering system to be dungeon-aware (new zone key + serialization), or (b) using the
  dungeon-system's own node system. The dungeon already has a gatherable-objective node
  (`silkQuestObjective` → `grantEditorItem(node.itemId)`), but it is **one-time** (persisted in
  `interactedObjects`, no respawn) — a respawning variant would be new work.
- **Recommendation:** Phases 21/23 should be scoped/estimated with this in mind. Cleanest options:
  place Silk Web herbs/fish as **dungeon-system-owned nodes** (extend the dungeon's existing node
  types with a respawning resource node + a dungeon fishing pool), rather than bolting dungeon
  awareness onto the overworld gathering/fishing systems. This keeps each owning system intact.
  Note: Hidden Tree Cave (`mossfang_cave`) IS a real cave (`currentZone==='cave'`) and DOES support
  `seedCaveResourceNode` + fishing — but the plan puts no exclusive resources there.

### 6. Anti-farming / persistence (Phase 24) is largely already handled
- Depleted nodes are stored as `state.depletedNodes[zoneKey:x,y:type] = { respawnAt: Date.now()+ms }`
  — an **absolute wall-clock timestamp**, persisted to both localStorage
  (`dream-realms.resource-runtime.v2`) and the game save (`resourceRuntime`). Leaving/re-entering a
  zone or reloading does **not** reset the timer. `pruneRespawnedNodes` clears expired records.
- This means Phase 24's "no instant respawn on reload/zone-change/off-screen" is **already true for
  gathering nodes**. The gaps Phase 24 must still address: fishing has no depletion model at all
  (see #3), and Silk Web dungeon resources (if added) would follow dungeon per-run state, which
  needs its own anti-farm reasoning.

## Known limitations / open decisions handed to Phases 19–23
1. **Fishing dual-level divergence** — pick an authoritative Fishing level (Phase 19).
2. **XP curve vs. plan table** — decide replace vs. keep (Phase 19; recommend replace).
3. **Silk Web Cavern is a dungeon** — Phases 21 & 23 need a dungeon-owned resource/fishing path;
   the overworld systems do not reach it. This is the single biggest scoping risk downstream.
4. **Fishing has no depletion/respawn model** — the plan's fish respawn timers don't map to
   fish-anywhere; reconcile in Phase 22/24.
5. **Plan schema field names differ** — map onto existing fields, don't invent a parallel registry.
