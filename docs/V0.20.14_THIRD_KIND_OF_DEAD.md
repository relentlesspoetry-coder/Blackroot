# V0.20.14 — The Third Kind Of Dead

**No game change, deliberately.** The correct action on this candidate was *none*, and taking none is
the finding.

## The candidate

`allowedZone` / `allowedRegion` / `allowedBiome` — 8× each in `data/resources.js`, **0 reads**. By now
that pattern sounds exactly like the last several problems.

**It isn't one.** The data file documents itself:

> *"allowedZone/allowedRegion/allowedBiome are **declarative spawn-scope metadata** (placement already
> enforces exclusivity **physically** via the dark_woods resource grid; **a later phase can read these
> to hard-guard**). Placed by `Game.prototype.ensureDarkWoodsOverworldHerbs`."*

The author **knew** they were unread, **wrote down why**, and the rule **is** enforced — just
physically, by the placement function only putting each herb in its own region, rather than
declaratively.

- **Retiring them** would delete a documented plan.
- **Implementing them** would build a fence around a field that's already fenced.

## Three kinds, identical in a list

| kind | what it is | action | examples |
|---|---|---|---|
| **BROKEN** | claims something and never delivers | **implement** | `coneDegrees` drew a circle · 33 NPC portraits never shown · `interactLabel` had a hardcoded twin |
| **EMPTY** | true of everything, or duplicated by a field that IS read | **retire** | `requiresHeavyWeapon` on every Fighter attack incl. the lv1 basic · loose footstep fields copying `footstepProfile` |
| **DELIBERATE** | documented metadata awaiting a phase | **leave alone** | `allowedZone`/`Region`/`Biome` |

**86 declarations retired** across V0.20.12–13. **Zero** retired here — correctly.

## The tool now says so

`tools/check-dead-fields.js` prints all three categories with their examples on every run, and ends
with the only instruction that has actually worked:

> *Open the file. Read the comments. Ask what it would MEAN if it were read.*

## Why that instruction is written from experience

The scan has now been wrong in **every available direction**:

| error | version |
|---|---|
| missed **quoted** keys (`"classMechanic":` ≠ `classMechanic:`) | V0.20.11 |
| reported a profile's own **nested contents** as dead tile fields (`terrainName`) | V0.20.13 |
| my triage grep counted a **default literal in code** as a read | V0.20.13 |
| **I** called an **empty** flag a **broken gate** and recommended implementing it | V0.20.12 |

Four errors, four different kinds. **The tool's value is that it points. The judgement was always the
human part** — and the tool now says that in its own output rather than leaving the next reader to
learn it the way I did.

## The remaining 24

Unexamined: `xpCurve` (crafting), `rendererKey` / `equipmentAnchorProfile` (races), `mobFamilies`,
`entranceZoneId`, `requiredZone`, `aoeRadius` (npcs), `resourceShortName`, `combatResourceRegen`,
`shatterSetup`, `pulseLimit`, `fieldDuration`, and ~13 others.

Each needs the same three questions, in this order:

1. **Is it documented?** — if the file explains it, believe the file. *(DELIBERATE)*
2. **What would it mean if it were read?** — if the answer is "nothing", it's empty. *(EMPTY)*
3. **Does something claim it and fail?** — only then is it broken. *(BROKEN)*

Asking them in the wrong order is how V0.20.12 happened.
