# V0.20.85 — Mini-Map Regression

Reported: *"Mini Map doesn't work anymore."*

**A regression I introduced in V0.20.80 and did not catch for five versions.**

## Root cause

`setActiveOverworldZone` cleared the cached mini-map image so it would be redrawn at the new zone's size:

```js
this.staticMinimap = null;
```

But nothing ever rebuilt it, and `drawMinimap` opens with:

```js
if (!this.staticMinimap) return;
```

So the mini-map went black and stayed black.

**And it was worse than a travel-only bug.** `resetCharacterOwnedState` calls `setActiveOverworldZone` —
so this fired for **every new character**, with no travel required. That is exactly the reported
screenshot: a fresh Paladin standing in Dark Woods with a black mini-map.

Reproduced before fixing:

```
atLoad_staticMinimap:        false
afterStart_staticMinimap:    false
after30Frames_staticMinimap: false
afterTravel_staticMinimap:   false
```

## The fix

```js
this.staticMinimap = this.buildStaticMinimap?.() || this.staticMinimap;
```

The exact idiom `portal-system.js` already uses. The `|| this.staticMinimap` fallback keeps the previous
image if a rebuild ever fails, rather than blanking the HUD — failing to a stale map is far better than
failing to nothing.

## How this should have been caught

Every **other** map-swapping path in the codebase rebuilds the static minimap right where it swaps:

| file | line |
|---|---|
| `systems/portal-system.js` | 276 |
| `systems/combat-system.js` | 2200 |
| `systems/save-system.js` | 2314 |
| `systems/world-serializer.js` | 636 |
| `systems/world-system.js` | 286 |

**Mine was the only one that did not.**

The lesson is specific and worth keeping: when adding a code path that does what several existing paths
already do, the correct check is to **diff against those paths** — not to reason from first principles
about what the new path needs. Four independent call sites all did the same extra step; that is a
contract, not a coincidence. The diff would have taken a minute and saved five versions.

It also explains why the earlier verification passed: I tested that travel worked, the map swapped, the
arrival tile was walkable, and regions resolved. Every one of those was true. I never checked the HUD,
because the HUD was not what I had changed — but nulling a cached image *is* changing the HUD.

## Verified by pixels, not by the field being non-null

An earlier version of this check would have asserted `!!game.staticMinimap` and passed the moment the
field was set. Sampled the actual canvases instead:

| check | result |
|---|---|
| on-screen mini-map canvas, Dark Woods | **98.3% non-black** |
| on-screen mini-map canvas, Ashen Valley | **98.3% non-black** |
| static image, Dark Woods | 360×360, 100% non-black |
| static image, Ashen Valley | 450×450, 100% non-black |
| present at load | ✅ |
| present after character creation | ✅ (the failing case) |
| present after travelling out | ✅ |
| present after travelling back | ✅ |

Suite ok, 0 console errors.
