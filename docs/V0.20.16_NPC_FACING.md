# V0.20.16 — Look At Me When I'm Talking To You (Roadmap Item 16, ambient)

## The bug I shipped

V0.20.15's patrol ended with:

```js
npc.setFacingFromDelta?.(nx - npc.x, ny - npc.y);
```

**A placed NPC is a plain grid node, not an Entity.** It has no `setFacingFromDelta`, and no
`facingX`/`facingY` at all — the node has 29 keys and neither field is among them.

The optional chain **swallowed the call in perfect silence**. The Mossfang Scout and Road Warden Cael
walked their new rounds for an entire version **sliding** — never turning to face their direction of
travel.

**The `?.` that made the call safe is exactly what made it invisible.**

Same pattern as V0.20.9's `findLootableCorpseNear` — a guard calling a method that never existed —
**written by me one version after I documented the pattern** in `V0.20.11_DEAD_FIELD_MAP.md`.

| | |
|---|---|
| probe before | `facingX: undefined, facingY: undefined` — `hasFacingFields: {}` |
| my call's effect | **none, silently, every frame, for a version** |

## What found it

Asking a different question: **do NPCs turn to face the player when spoken to?**

They never have. `interactWithNpc` went straight to `openNpcWindow`, so **Cael would tell you the road
is safer than the trees while facing away from you, down it.** Nobody in camp has ever looked at the
person speaking to them.

Item 16 asks for NPCs that *react* rather than stare, and this is the cheapest possible instance — **no
authored data needed**, the player's position is right there.

## The fix

`faceNodeToward(node, dx, dy)` sets **`facingX`/`facingY` directly** — the fields the renderer actually
reads (`entity-renderer` `standingDirectionProfile`, V0.19.9).

- **normalised** — the renderer's direction maths expects a unit vector
- **a zero delta is refused**, not written as `(0,0)` — V0.19.7 spent a whole version on quietly-wrong
  facing vectors, and **`(0,0)` is not a direction**

Applied in two places: the patrol (fixing the dead call) and `interactWithNpc` (the new behaviour).

## The uncomfortable tally

Dead calls and dead data **I have written myself**, in a session spent removing other people's:

| version | mine | how it hid |
|---|---|---|
| V0.20.9 | `findLootableCorpseNear?.(4.5)` — a guard calling a method that never existed | `?.` → `undefined` → falsy → never guarded. **Caught before shipping.** |
| V0.20.11 | reported `requiresHeavyWeapon` as "a gate that never gates" | didn't ask what it gated. **Corrected in V0.20.12.** |
| **V0.20.15** | **`setFacingFromDelta?.()` on a non-Entity** | **`?.` → silent. Shipped, and lived for a version.** |

The common thread is **optional chaining on a method I assumed existed**. `?.` is protection against a
*missing* object; used against a *misunderstood* one, it converts a crash into silence — and silence is
the thing this entire session has been about.

`node --check` cannot see it. The duplicate-method checker cannot see it. The dead-field scanner cannot
see it (it reads data, not calls). **The only thing that caught it was asking what the code was
supposed to do and then measuring whether it did.**
