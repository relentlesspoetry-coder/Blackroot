# V0.20.71 — Black Wolf Starter Mount, Palette Resolver Fix, Head Layering

Requested: a black wolf built from the existing wolf mobs, as a level-1 mount, slower than tamed mounts —
and the mount's head hidden behind the class model when facing the camera.

## The black wolf

- **`black` coat** added to `wolf-procedural-model` `BASE_PALETTES` — the module that owns wolf colouring.
- **`blackWolf`** added to `DARK_WOODS_MOB_VISUALS` — a *real authored visual key*, exactly as
  `mossfangAlpha` and `thornCrownedStag` are, so the mount references real art rather than an invented
  creature (Item 7.J).
- **`mount_black_wolf`** replaces `mount_camp_packbeast` as the vendor starter; the Drover's Whistle now
  calls it.
- **Speed 1.14×** — the slowest mount in the game (tamed range 1.18–1.85×). It gets a level-1 character
  moving without making taming pointless.

## Bug found doing it: wolf coats were unreachable by name

```js
let base = BASE_PALETTES.gloom;
if (key.includes('mossfang') ...) base = BASE_PALETTES.mossfang;
else if (key.includes('cave'))   base = BASE_PALETTES.caveGloom;
else if (key.includes('dire'))   base = BASE_PALETTES.dire;
```

A hardcoded if-chain that **never consults `BASE_PALETTES` by key**. The new coat rendered identically to
gloom — measured, both at avg RGB **51, 60, 59**.

This is the same failure shape as the sprite-atlas `inferModelId` chain from V0.20.53: **a lookup table
that the lookup ignores.** Adding an entry to the table does nothing, silently, and looks like an art
problem rather than a code one.

**Fixed** with a lower-cased name index consulted first; the heuristics stay as fallback.

| coat | luminance before | luminance after |
|---|---|---|
| black | 57 *(rendering as gloom)* | **29** |
| gloom | 57 | 57 (unchanged) |
| mossfang | 85 | 85 (unchanged) |

Any coat added later now works with no code change.

## Front-facing head layering

`drawRiddenMountHeadOverlay` re-draws the mount **clipped to its head region, after the rider** — but
only when `entityFacingView` reports the `front` view. From behind or side-on the head is not in front of
the rider, and drawing it over them would be wrong.

The head box is **measured** per mount from its front-facing silhouette — the top of the mass down to
where the outline widens at the shoulders — and cached. Measured rather than authored because the beasts
are different shapes, and for the same reason the saddle height is measured (V0.20.69).

Note the view test is *screen-space*, not compass: world-south projects to a **side** view in this
isometric projection; **southeast** is what faces the camera. That is the V0.20.65 parity table, and
getting it wrong here would have made the overlay fire on the wrong headings.

## Verification status — stated honestly

**Geometry checks pass:**

| check | result |
|---|---|
| head box contains mount pixels (not empty/misplaced) | **1063 px** |
| rider body band vs head band | y108–174 vs y139–176 — **overlap** |
| overlay called, front view only | 1/frame, `view.mode === 'front'` |
| regression | none observed |

**The pixel A/B was inconclusive**, and I am not going to dress that up:

- with live time, the world animates between the two captures — 714 "changed" pixels turned out to be
  scenery, appearing *outside* the clip rect
- with `performance.now()` frozen, the second render appears to be skipped entirely — a **0-pixel diff**
  either way

So the layering is geometrically sound and confirmed to run without regression, but **not proven by a
before/after image**. The user's eyes settle this one. If the head still hides, the next step is to
instrument the clip rect in the live camera transform rather than assuming `worldToScreen` space matches
it.
