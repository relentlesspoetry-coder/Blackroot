# V0.20.51 — Snare Roots Anchor at the Feet (class-specific fix)

Reported repeatedly: snare roots wrap the **waist** instead of the ankles. I twice claimed a fix and was
twice wrong — because I tested with a **Fighter**, and the bug is **class-specific**.

## Mechanism

Ground-anchored VFX prefer `target._lastGroundAnchor.groundY` (V0.18.36), because a humanoid's model
origin sits at the **hips**, roughly 33–44px above the rendered feet.

The shared humanoid base renderer derives the foot line correctly:

```js
const groundOffsetY = max(legs.near.foot.y, legs.far.foot.y);
const groundAnchor = { x, y, groundY: y + groundOffsetY };
```

But the **bespoke class models** published:

```js
const groundAnchor = { x, y, groundY: y };   // the model ORIGIN — the hips
```

So any class routed through its own premium model reported its waist as the floor.

## The measurement that found it

Sweeping **all 14 classes**:

| classes | foot line vs hip anchor |
|---|---|
| Fighter, Warden, Rogue, Ranger, Cleric, Druid, Bard, Wizard, Shaman, Summoner, Necromancer, Assassin, Enchanter | **+33px** (below — correct) |
| **Paladin** | **−5px** (above — broken) |

The reporter's character is a Paladin. That single row is the whole bug.

## Fix

Both bespoke publishers — `drawPremiumPaladinModel` and the fighter-bruiser variant — now derive
`groundY` from their own `legs.left/right.foot.y` anchors, exactly as the shared base renderer does. A
scan confirms no `groundY: y }` publishers remain.

## Verified end-to-end, not by proxy

| check | result |
|---|---|
| all 14 classes report a positive foot line | ✅ Paladin **+44px**, others +33px |
| classes still broken | **0** |
| **live rooted Paladin**: effect draw Y vs true foot Y | **682 vs 682** — dead on the feet ✅ |
| offset below hips | 44px ✅ |
| console errors | 0 |

## The lesson

**"I verified it" is worthless if the sample cannot exhibit the bug.** My two earlier checks —
that `_lastGroundAnchor` was published, and that a Fighter drew correctly — were both *true* and both
*irrelevant*. Neither could have failed, because neither touched the broken class.

A per-class defect needs a per-class sweep. The fix took minutes once the sweep existed; the two wasted
versions were entirely a measurement failure.

## FPS status (deliberately not bundled)

No performance edits shipped alongside this bug fix. Current state:

- update ~11ms after V0.20.46; **render dominates**.
- Per this project's own V0.17.95 notes, **this preview renders canvas2d in software** and is
  draw-call-bound and **resolution-insensitive** — fill-rate wins *cannot* be demonstrated here, only on
  real GPU-composited canvas.
- A **Render Resolution slider (50–100%) already exists** in Settings → Graphics. Verified live: 75% cuts
  backing-store pixels by **44%**, 50% by **75%**, restoring cleanly.
- **Flagged:** `adaptivePerformance` is enabled but reported **0 downshifts** while the game sat at ~12
  FPS. A working governor should have dropped to `performance` (renderScale 0.85) or `rescue` (0.72).
  Worth investigating next.
