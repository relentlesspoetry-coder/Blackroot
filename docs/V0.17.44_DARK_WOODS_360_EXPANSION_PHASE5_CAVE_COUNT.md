# Phase 5 — Dark Woods 360x360 Expansion: Cave Count Enforcement

**Phase:** 5 of 28 (Dark Woods Zone Expansion + Resource Identity master plan)
**Previous version:** V0.17.43
**New version:** V0.17.44

---

## 1. Goal

Reduce Dark Woods from its current 6 attached caves down to exactly the 2 allowed by the design rule: Silk
Web Cavern (kept as-is) and Hidden Tree Cave (new identity). No dungeon interior content was built — that
is explicitly Phase 8 (Hidden Tree Cave) and already-existing Phase 7 territory (Silk Web Cavern).

## 2. A Judgment Call Made With the User Before Implementing

The plan's cave list ("The only caves are: Silk Web Cavern, Hidden Tree Cave") strictly excludes the
existing Mossfang Cave. But Mossfang Cave is hard-required by `systems/world-serializer.js` as Dark Woods'
second persisted zone (`validateNormalizedPayload`, `migratePayload`, and `serialize()` all reference a zone
literally keyed `mossfang_cave` in multiple places). I asked the user how to resolve this conflict before
touching anything; they chose: **repurpose Mossfang Cave into Hidden Tree Cave, keeping the same
`mossfang_cave` save-schema key/id and only changing its display identity** (name, theme, level range).
This was chosen over (a) keeping Mossfang Cave and adding a third zone slot (violates the exact-2 rule) or
(b) fully renaming the zone key (would force a save-schema v3→v4 migration for no gameplay benefit). This
report documents that decision so it's traceable independent of the chat transcript.

## 4. Root Cause / Implementation Notes

**Why this was safe to do with only 2 files:** `ensureDarkWoodsCaveEntrances` (`cave-system.js`) already
places cave-mouth objects generically via `for (const def of CAVE_DEFINITIONS) placeEntrance(def)` — removing
entries from the array automatically stops those 4 caves from being placed, with no separate removal code
needed. The same function also filters `this.caveEntrances` (runtime state, e.g. from an old save) down to
only entries matching a currently-known cave id/name, so any stale entrance records for removed caves are
self-cleaning on the next `ensureDarkWoodsCaveEntrances()` call rather than needing an explicit migration.

**What did need explicit removal:** 4 hardcoded `makeTrail(fromX, fromY, toX, toY, width)` calls in that same
function existed *outside* the generic per-definition loop — direct reinforcement trails hand-carved toward
each cave mouth's specific coordinates. These aren't tied to `CAVE_DEFINITIONS` content, so removing cave
defs alone would have left 4 dirt trails carving a path to now-empty forest. The 2 trails for the surviving
caves (Hidden Tree Cave, Silk Web Cavern) were kept; the 4 for removed caves were deleted.

**What did NOT need to change:** the general road network built by `carvePath` inside `generateMap` (Phase 2)
was deliberately left untouched. Those roads lead to the same ruin/stone terrain and existing
`ruinWall`/`ruinPillar`/`crystalNode`/`mineSupport` landmark props that were placed by `placeWorldLandmarks`
regardless of whether a cave portal sits there — removing the cave doesn't remove the terrain or props
around it, it only removes the interactive cave-mouth object and floor-generation hook. This terrain is not
wasted: Phase 3 already placed the `bandits_fall` region box specifically to cover the Ashroot Hollow/
Blackroot Catacombs ruin cluster, and Phase 4's new road spurs already branch from the same road endpoints
that used to lead to Crystal Grotto and Forgotten Mine. Nothing from Phases 3–4 needed rework.

**Project-wide reference sweep before removing anything:** searched every `.js` file for
`ashroot_hollow|crystal_grotto|forgotten_mine|blackroot_catacombs` before editing. Found:

- `systems/fishing-system.js` — `FISH_TABLES.crystal_grotto` / `.blackroot_catacombs` entries. These become
  unreachable (the zones can no longer be entered) but are harmless to leave — they simply stop being
  looked up. Not deleted.
- `data/loot-tables.js` — `loot_cave_ashroot_mobs`, `loot_cave_crystal_grotto_mobs`,
  `loot_cave_forgotten_mine_mobs`, `loot_cave_blackroot_catacombs_mobs` (mob-death loot tables tied to
  `caveMobFamily` sources for the removed caves) and their appearance in an unrelated aggregate
  slot-loot-weighting array (`data/loot-tables.js:~700-708`, confirmed to be a generic vendor/slot loot pool
  registration, unrelated to zone reachability). Harmless to leave; not deleted.
- `data/dungeons.js` — `dungeon_glooms_crypt`'s `entranceZoneId`/`caveZoneId` reference `blackroot_catacombs`
  — this is the one real content consequence, covered in detail below.
- `core/registry.js:187` — a code comment using `"blackroot_catacombs_f4"` as an example zone-id string; not
  functional code, no change needed.
- No editor UI, zone-list, or dropdown anywhere in the codebase hardcodes the set of 6 caves separately from
  `CAVE_DEFINITIONS`/`CAVE_BY_ID` (confirmed by grep) — everything that iterates caves does so generically.

**The one real content consequence — Gloom's Crypt is now orphaned:** `data/dungeons.js`'s
`dungeon_glooms_crypt` ("Gloom's Crypt") is a fully-built 3-floor, level 9–12 dungeon with 3 unique bosses
(`boss_hollow_warden`, `boss_rootbound_matriarch`, `boss_gloom_king`), 3 puzzles, a key item, and its own
loot table. Its *only* access point (per an explicit code comment in `systems/dungeon-system.js`:
"dungeon progression is owned by named cave final-floor hooks... do not inject standalone overworld shortcut
markers") was Blackroot Catacombs' floor-4 `dungeonHook`. Removing that cave entrance, as the plan requires,
leaves this dungeon with no current way to reach it. Per the project's rule against removing existing
gameplay without proving it dead, **none of Gloom's Crypt's data was touched or deleted** — it still fully
exists in `data/dungeons.js`, simply unreachable until a future phase decides how (or whether) to give it a
new access point. This is flagged prominently here and in the chat response rather than silently dropped.

## 7. Known Limitations

- **Gloom's Crypt (`dungeon_glooms_crypt`) is now unreachable** (see Section 4). Its data is intact and
  undamaged, but there is currently no way to enter it. This needs an explicit decision in a future phase:
  re-attach it to a different final-floor hook, redesign it as part of Silk Web Cavern's or Hidden Tree
  Cave's content (the plan explicitly forbids turning Hidden Tree Cave into a second dungeon, so that option
  is likely off the table), or accept it as permanently cut content.
- Hidden Tree Cave's `enemyNames`/`resourceTypes`/`lootTableId`/exterior concealment still reflect the old
  Mossfang Cave identity (wolves/rotlings/thorn widow, moss variant) — only the name, theme label, and level
  range changed in this phase. Full interior content and exterior tree/bush/vine concealment matching the
  plan's Hidden Tree Cave description is explicitly Phase 8 scope.
- Ashroot Hollow/Crystal Grotto/Forgotten Mine's unique resources (`ore_crystal`, `ore_blackiron`,
  `herb_ashroot_moss`, etc.) and fish tables are no longer obtainable via Dark Woods now that their caves are
  gone. No audit was done of whether any crafting recipe or quest requires these specifically — flagging as
  an open question rather than claiming it's been checked.
- No in-browser boot/interaction test was possible in this environment (see Section 5).
