# V0.17.96 — Nameplate render cache + GPU-check tool

## The correction that led here

Earlier versions assumed the in-world slowness was "software vs GPU rendering" (a
headless-sandbox artifact). That was **wrong**. The `Blackroot GPU Check.html` tool
(added this version) confirmed the test machine has a real GPU (AMD RX 7900 XT) with
hardware-accelerated canvas2d — and the game was **still** ~5 FPS in-world. So the
slowness is real game render/update cost, not GPU availability.

## Draw-phase profiling (real GPU)

In-world (`perfStats` + wrapping the draw functions), render ≈ 48 ms/frame split as:

| phase | per-frame | note |
|---|---|---|
| **drawEntity** | **33 ms** | **1.43 ms per entity** (23 drawn) |
| lighting/weather/fog | ~14 ms | full-screen passes |
| drawObject | 2.4 ms | 105 calls, cached (0.023 ms each) |
| drawTile | 1.6 ms | cheap |

Splitting `drawEntity` further: the sprite blit is cheap (sprite cache ~0.12 ms), but
**`drawNameplate` was ~2.1 ms per plate → ~24 ms/frame** for ~12 plates — over half of
render. Canvas2d **text** (`fillText`/`measureText`) plus `shadowBlur`, a per-frame
gradient, and 5 rounded-rect frame strokes are CPU-bound (a GPU can't accelerate them)
and were redrawn from scratch every frame for every visible entity.

## The fix — per-entity nameplate image cache

`render/entity-renderer.js` `drawNameplate`:

- Build a `plateKey` from everything that changes the plate's pixels: `name`, `level`,
  `actionText`, `kind`, `elite`, `classKey`, quest flag, `w`, and **bucketed** HP,
  threat% (+ primary), spider-web, and meditation progress.
- Keep a per-entity offscreen canvas `e._nameplateImgCache`.
- **Cache hit** (key unchanged): `ctx.drawImage(cache.canvas, x0-PAD, plateY-PAD)` and
  return. No text/shadow/gradient/stroke work.
- **Cache miss**: retarget the global `window.ctx` to the offscreen (translated so the
  plate's absolute screen coords land in a padded buffer), run the **unchanged** drawing
  code into it (wrapped in `try/finally` so the real canvas is always restored), cache
  the image, then blit.

Because `ctx` is the global `window.ctx` and the drawing code is reused verbatim, the
cached plate is **pixel-identical** — no visual change, nothing removed. The layout
(anchor/stacking/position) still runs each frame so plates track their entities and
stack correctly; only the expensive rendering is skipped on hits.

Dynamic plates (HP/threat/meditation changing in combat) re-render when a bucket
boundary is crossed — so they stay live with no staleness, while idle plates (the
common case, e.g. a camp full of NPCs) hit the cache every frame.

## Results

- Per-plate: **2.1 ms → 0.68 ms** in this (draw-call-bound, software) sandbox.
- Render: **~48 ms → ~23 ms median** (min ~22 ms).
- On real GPU+CPU the win is **larger** than the sandbox shows — cache-hit blits are
  ~0.1 ms there, so drawNameplate effectively goes from ~24 ms to a few ms.
- Verified in-browser: nameplates render identically (before/after screenshots), 0
  console errors, high hit rate on idle plates, correct live updates on dynamic plates.

## `Blackroot GPU Check.html` (new, root of project)

A standalone, double-click diagnostic for players: WebGL GPU name + software-vs-hardware
verdict, a canvas2d speed test, display/DPR/pixel-load, and plain-English fix steps
(enable browser hardware acceleration / update drivers). It never touches the game.

## Remaining bottleneck (next)

With nameplates cached, the largest remaining per-frame cost is the **enemy-AI update
loop** (`update` ≈ 90 ms at low FPS; distance-banded but each full AI run does
pathfinding/target-scan). That's CPU work (unaffected by the GPU), and it's the next
optimization target on the road to a constant 60.
