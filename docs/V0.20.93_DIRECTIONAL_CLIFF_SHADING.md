# V0.20.93 — Cliff Faces Lit by Direction

The actual cause of *"the corners still do not render correctly at every angle"*.

## What it really was

A zoomed screenshot settled it in one look: **nothing was missing, and nothing was misplaced.**

Every wall face of a raised block was drawn in the **same flat shade**. Two perpendicular walls meeting
at a corner therefore had **zero contrast** between them — so the corner had nothing to show it, and a
stepped stone cliff read as folded paper rather than solid rock.

That is why it looked wrong "at every angle": it was wrong at *all* of them, equally, and rotating the
camera just changed which flat facets you were looking at.

## Root cause

Each face already carried a per-direction shade, set in `terrainEdgeDefinitionsForTile`:

```js
n: { ..., shade: -28 },  e: { ..., shade: -20 },
s: { ..., shade: -32 },  w: { ..., shade: -38 }
```

`tallFaces` propagated it faithfully:

```js
return { ...face, shade: isCaveWall ? caveShade : face.shade, ... };
```

And then `drawLayeredWallFace` **never took it as an argument**:

```js
drawLayeredWallFace(a, b, aBottom, bBottom, def, isCaveWall, variant, faceKey)   // no shade
gradient.addColorStop(0, colorShade(top, -2));    // identical for all four sides
```

The value was computed, carried through two layers, and thrown away at the point of use. It *is*
consumed at one other site — an outline stroke at line 2071 — which is exactly why the wiring looked
complete on a grep.

## The fix

The shade is passed in and applied **relative to its set's midpoint** rather than absolutely:

```js
const MID = isCaveWall ? -11.5 : -29.5;
const dir = clamp((faceShade - MID) * 1.6, -26, 26);
```

Applying it raw would have darkened every cliff in the game by 20–38 points. Relative application makes
the faces differ *from each other* while preserving the existing average brightness. The ×1.6 widening
keeps the distinction readable underneath the vertical gradient layered on top.

Both call sites updated — overworld and cave wall.

## Verified

| check | before | after |
|---|---|---|
| perpendicular face contrast | **0** — identical | **10–19** luminance points |
| per-direction luminance | all equal | e 115, n 102, s 96, w 86 |
| distinct stone shades on screen | — | 20–21 at every 90° of yaw |
| highlands frame time | — | 5.65 ms / 7.18 ms repeat |
| all zones × four yaws | — | no errors |

## Three wrong hypotheses came first

Each was measured and discarded rather than shipped, but it still took too long:

1. **Yaw-fixed screen corners.** The naming (`corners.east` for the *north-east* world corner) made this
   look certain. Measured: corner tops are exact to **0.00px** at every yaw, with an identical 250 joins
   at each angle. Wrong.
2. **Per-tile jitter separating shared edges.** `drawLayeredWallFace` uses the passed corner points
   untouched. Wrong.
3. **Thin dark seams varying with angle** — I measured 74→144 across yaws and briefly took it as
   evidence. Reading the code, those are the *intentional* contact-darkening (`rgba(0,0,0,0.56)`) and
   ambient-occlusion strokes drawn along every face bottom. My metric was counting a deliberate feature.

## The lesson

After the second wrong hypothesis I stopped and asked for a zoomed screenshot instead of guessing a
third time. That was the single highest-value action in this whole sequence — the answer was visible
immediately once the image showed the *surfaces* rather than the silhouette.

The prior audits weren't wasted (V0.20.92 fixed 433 genuinely missing corners), but they all measured
**geometry**, and the fault was in **shading**. An audit can only find the class of fault it is written
to look for, and I kept writing geometric ones because the first real bug had been geometric.
