# V0.21.2 — Art Direction Phase 2: The Waypoint Shrine

The spec's Phase 2 is the vertical slice: *"build one fully authored Dark Woods waypoint … verify it
reads correctly in-world."* It is explicit that the waypoint **"must be rendered as one authored shrine
asset, not a small procedural circle with generic glow"** and that it should feel *"ancient, powerful,
permanent … rather than resembling a small glowing puddle."*

## How this is "authored" without a PNG from an artist

`systems/waypoint-shrine-asset.js` draws the shrine **once** at high detail into an offscreen canvas,
then registers it through the authored-atlas pipeline built in V0.20.97 with real `anchor` /
`occlusionHeight` / `contactShadow` / `emissive` / `light` metadata.

That is how pre-rendered 2.5D assets actually work — a source is rendered down to a sprite. The runtime
path is now the **authored** path, not per-frame procedural drawing. When a hand-painted PNG replaces
the bake, only the page source changes; every consumer already reads it as an atlas frame.

It is also the **first real asset through the Phase 10 pipeline**, which is the only way to discover
whether that pipeline works. It does — registered with 0 errors and 0 warnings.

## Baked vs live

Following the spec's own module separation:

| baked — static, object-renderer | live — animated, effects-renderer |
|---|---|
| raised platform + side wall | cold blue flames |
| 26 fitted masonry blocks, individual tone / mortar / chipping | bright centre core |
| 20 radial paving wedges, 3 concentric joint rings | vertical beam |
| 90 grime & moss patches, 12 crack paths | drifting motes |
| 4 ritual pillars with courses and capitals | |
| ruined backplate arch | |
| **emissive page**: 2 segmented rune rings, 16 glyph ticks, central sigil | |

Baking the animated parts would freeze the thing that sells the effect. The bake is **deterministic**
(seeded), so every build produces an identical asset.

Sized in **tile units** so it survives a metric change: a 320px frame spanning ~2.35 tiles at 136×68,
measured in-world at **1.93 tiles wide × 203px tall**, against the old dais's 58px radius.

## A double shadow retired

`drawWaypoint` drew its contact shadow **inline** while `waypoint` is also in the `shadowSize` table
added in V0.20.99 — so every attuned waypoint has stood on two stacked shadows since that version.

The V0.20.99 probe counted `drawPropContactShadow` calls and **structurally could not see an inline
one**. A blind spot in the measurement, not in the reasoning. Gating the procedural block behind the
authored path retires it.

The procedural dais is **kept as a complete fallback** rather than deleted: if the bake or its
registration ever fails, the waypoint degrades to the previous look instead of to bare ground.

## Two measurement failures, recorded because both nearly shipped a false conclusion

**1. `object-renderer.js` destructures `ctx` once at install time.** Swapping `DR.runtime.ctx` to a
recording canvas does not redirect it. A probe built that way saw the shrine blit — which reads
`DR.runtime.ctx` live — but *not* the procedural dais, and reported the fallback broken at 385 pixels.
It was never broken; the dais was drawing to the real canvas the whole time. Re-measured on the real
canvas, both paths draw substantially with blue light in every run.

**2. A shadow-shape classifier measured itself.** It reported 26 props with doubled shadows — including
`tree` and `crate`, never touched, and `barrel`, just fixed. It was counting `drawPropContactShadow`'s
*own* ellipse as a second shadow. Uniform results across props I had and hadn't modified is what gave
it away.

## Verified

| check | result |
|---|---|
| atlas registration | **0 errors, 0 warnings** |
| colour page | 36,366 px painted (35.5% coverage) |
| emissive page | 3,082 px |
| frame metadata | anchor (160,268), occlusionHeight 118, contactShadow, light |
| rendered in isolation | 33,647 px, **1.93 tiles wide**, 203px tall |
| authored + fallback on real canvas | both draw with light, 4 alternating runs |
| boot suite | PASS — 15/15, 397/397, 1 atlas / 1 frame, 0 atlas errors |
| frames | 20 clean, 0 console errors |

In-world pixel counts are **not** repeatable run to run — the live world animates (torches, day/night,
particles), the same noise that defeated pixel-diffing in V0.20.98. The stable figures above come from
isolated rendering; the in-world runs establish only that both paths draw, which is what they were for.
