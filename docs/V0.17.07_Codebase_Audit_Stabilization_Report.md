# V0.17.07 Codebase Audit + Stabilization Report

## 1. Project structure summary
BlackRoot is a browser game built from one HTML shell, `game.js`, ordered modules in `core/`, `data/`, `entities/`, `systems/`, and `render/`, plus assets and documentation. `_script_order.json` is the canonical modular load manifest; the HTML script list mirrors it.

## 2. Files audited
All JavaScript and JSON files were syntax checked. The load manifest, HTML script list, version files, patch notes, all owning UI/input/save/loot/render modules, registries, and data modules were statically inspected. Asset binaries and historical reports were inventoried but not interpreted as executable code.

## 3. Systems audited
Boot/load order, game loop, canvas rendering, paperdoll/cape/weapon rendering, Player HUD, unified Party/Companions HUD, minimap, menu, Settings, Patch Notes, loot rolls, input/keybindings, save/migration, inventory/equipment, party/bot/mercenary state, and data registries.

## 4. Bugs found
- The Player HUD opening container, portrait, and identity header markup were missing even though the HUD children and closing tag remained, leaving `getElementById('hud')` null and breaking Player/Party HUD ownership.
- Settings controls had three competing paths: the canonical delegated controller, duplicate listeners recreated on every Settings render, and a late wrapper module around the Settings renderer/toggle.
- Missing or non-boolean HUD preferences could be interpreted as hidden instead of using visible defaults.
- The V0.17.07 metadata/report still described the earlier narrow HUD recovery phase rather than this requested audit.

## 5. Bugs fixed
- Restored the missing canonical `#hud` opening markup from the last known-good structure without changing HUD layout or creating another renderer.
- Kept one delegated Settings input owner and removed the duplicate per-render listeners.
- Moved `ensureSettingsList()` into the owning UI module and removed the late Settings wrapper.
- Normalized missing/invalid minimap, mercenary HUD, and Party HUD values to visible; explicit `false` remains respected.
- Synchronized filename-facing metadata, runtime build metadata, Patch Notes, and this report to V0.17.07.

## 6. Bugs intentionally not fixed and why
No speculative balance, AI, map, art, or architecture changes were made. Historical performance/release systems were not deleted merely because their filenames describe old phases; they remain loaded, expose active runtime diagnostics, and were not proven dead. Runtime-only gameplay scenarios remain listed for manual testing.

## 7. Dead/duplicate code removed
- Removed duplicate Settings control listeners from `systems/ui-system.js`.
- Removed `systems/settings-panel-regression-fix-system.js` after retaining its useful DOM recovery behavior in the UI owner.

## 8. Patch/hotfix/cascade remnants found
The Settings regression-fix module was a loaded late wrapper around canonical UI methods. Other `setTimeout` uses found were legitimate object-URL cleanup, audio sequencing, long-press input, or queued UI work rather than repair cascades.

## 9. Patch/hotfix/cascade remnants removed
Removed the Settings wrapper file, its load-manifest/HTML references, its installer call, stale feature flags, and its duplicate fallback bindings.

## 10. Canvas/render state issues found
No confirmed active canvas-state leak was found. World rendering resets transform, alpha, composite mode, and filter before composition; world-camera save/restore is closed through the owning transform methods. Clip calls inspected are contained by save/restore scopes.

## 11. UI/HUD issues found
The Player HUD owner markup was structurally incomplete: its opening container and identity block were absent while its bars, unified companion HUD, and closing tag remained. Menu Logout/Exit ownership and minimap FPS removal were already correct. Invalid HUD preference defaults and duplicate Settings ownership were also confirmed.

## 12. Input issues found
No active racial binding remained. Loot roll uses the canonical choice submitter, rejects disabled Need, immediately dismisses valid player prompts, and guards resolution by session state. Settings had duplicate ownership; this was consolidated.

## 13. Save/default/migration issues found
Save normalization already guards inventory/equipment and migration paths. Loot-roll sessions are runtime-only. HUD preference objects are now normalized defensively in the UI owner so missing/invalid visibility fields default visible without touching player save data.

## 14. Data registration issues found
No missing load-manifest files, duplicate manifest entries, or unloaded executable modules remained after cleanup. Static registry/reference inspection found no confirmed registration defect requiring a data change.

## 15. Loot roll issues found
No remaining root-cause defect was confirmed. Need/Greed/Pass share one submit path, valid choices dismiss immediately, unusable Need is disabled and rejected, and resolved sessions cannot resolve twice.

## 16. Patch notes issues found
Recent entries V0.16.97 through V0.17.07 were already embedded in both panels. The V0.17.07 title/content required synchronization to the full audit phase and was updated.

## 20. Validation results
All JavaScript syntax, all JSON syntax, `_script_order.json`, and the modular validation loop passed. The 130 HTML script references exactly match `_script_order.json` in content and order, with no missing files. HTML structural parsing passed with balanced tags, no duplicate IDs, and all required HUD/menu IDs present. The V0.17.06 backup and V0.17.07 build archives passed integrity testing. Browser gameplay testing remains manual.

## 21. Manual tests performed
Automated/static validation and archive integrity testing were performed. Full interactive gameplay testing was not claimed.

## 22. Known remaining risks
Save compatibility, party persistence, loot distribution under full bags/ties, renderer direction variants, and every viewport/input combination need interactive testing with representative existing saves. Static save/renderer review cannot prove those stateful scenarios.

## 23. Recommended next phase
A focused manual regression QA pass using existing saves and deterministic loot/render test fixtures. Do not begin feature work until the checklist passes.
