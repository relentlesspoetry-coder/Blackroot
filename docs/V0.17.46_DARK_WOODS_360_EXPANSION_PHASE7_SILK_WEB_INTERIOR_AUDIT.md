# Phase 7 — Dark Woods 360x360 Expansion: Silk Web Cavern Dungeon Interior

**Phase:** 7 of 28 (Dark Woods Zone Expansion + Resource Identity master plan)
**Previous version:** V0.17.45
**New version:** V0.17.46
**Type:** Audit/verification. No gameplay, prop, or enemy changes.

---

## 1. Goal

Implement/refine Silk Web Cavern as the only dungeon cave (entrance area, middle/deep area, final chamber,
spider enemy placement, webbed dungeon props, path readability, level 6–10 progression, named/boss chamber).

## 2. Why This Phase Produced No Code Changes

Phase 6's report flagged that a large Silk Web Cavern prop catalog already existed in
`render/object-renderer.js` and recommended auditing `systems/dungeon-system.js` before building anything
new. That audit was done first, and it found the entire scope of this phase already implemented and wired
to real runtime logic — not draft data awaiting a build pass. Building new versions of any of this would
have been duplicate-system work against the project's explicit anti-duplication rule. This phase's actual
deliverable is the verification itself, so a future phase (or reviewer) has a documented, checked baseline
instead of having to re-discover this from scratch.

## 3. Audit Findings (Verified Against Source, With File:Line Citations)

**Entry routing (`directDungeon: true`):** `systems/cave-system.js:872-889` (`enterCave`) special-cases
`silk_web_cavern` and calls `this.dungeonSystem.enterDungeon(marker, 1)` directly, bypassing the generic
cave-floor pipeline entirely — a deliberate, already-documented V0.13.30 fix (see the code comment at
`cave-system.js:867-871`) so the authored 3-floor dungeon loads instead of a generic cave.

**Dungeon draft data:** `data/dungeons.js` has two entries. The one actually used is `id: 'silk_web_cavern'`
(`data/dungeons.js:182-231`, matching the cave entrance's own `dungeonHook.dungeonId`) — 3 floors, 3 boss
ids, 9 miniboss ids, 3 floor-specific puzzle ids, a quest-chain id, 3 quest-giver NPCs, a spirit healer, and
dedicated loot tables. The other, `dungeon_silk_web_depths` (`data/dungeons.js:30-49`, 2 floors,
`bossIds:['boss_silk_mother']`), is legacy/superseded — its own notes field describes an older design, and
no code path reaches it (see Section 4).

**Floor generation:** `generateSilkWebCavernFloor` (`dungeon-system.js:1896-1962`) is a real, bespoke
generator — not the generic room-plan-based one used by other dungeons — driven by `SILK_WEB_FLOOR_CONFIG`
(`dungeon-system.js:654-793`), which hand-authors all 3 floors' room graphs (12-14 rooms each, with x/y/w/h,
room links, and `visualTags` per room, e.g. floor 1's `boss` room at `dungeon-system.js:671`:
`{id:'boss', kind:'boss', name:'Broodwarden Skirr Den', x:118,y:34,w:54,h:34, visualTags:['boss','brood_sacs','arena']}`).
Every object type found in Phase 6's renderer sweep is placed by real code: `placeSilkDressingObjects`,
`placeSilkHighFidelityEnvironmentObjects`, `placeSilkReadabilityObjects`, `placeSilkWebObjects`,
`placeSilkVenomSacks`, `placeSilkStructuredObjectives` (all in `dungeon-system.js`, ~1300-1750) place roughly
30 distinct prop types (`webAnchor`, `eggCluster`, `chitinColumn`, `bossCocoonPrison`, `silkFloorSigil`,
`bossArenaMark`, and more) — confirmed by grepping every one of Phase 6's catalogued type names against
`dungeon-system.js` directly.

**Level progression 6–10:** confirmed directly in the floor config — floor 1 `levelMin:6, levelMax:7`
(`dungeon-system.js:656`), floor 2 `levelMin:8, levelMax:9` (`:701`), floor 3 (level 10, per
`SILK_WEB_COMBAT_TUNING`, `:803-823`, which also scales elite/miniboss/boss HP/attack/defense upward per
floor, e.g. elite HP ×1.92/×2.04/×2.12 across floors 1/2/3 via `applySilkEnemyBalance`, `:834-852`).

**Spider enemy placement, theme purity:** `spawnSilkWebCavernEnemies` (`:1853-1894`) reads each floor's
`encounters` array (10-11 hand-placed groups per floor) and spawns named elite spiders via
`createSilkDungeonEnemy` (`:1750-1800`), 2-4 minibosses per floor via `createSilkMiniBoss` (`:1802-1830`),
and the floor boss via `createSilkBoss` (`:1832-1851`) — entirely separate from the generic overworld
mob-spawn system. Directly confirmed the floor 1 `elites` list (`dungeon-system.js:696`) —
`['Webbed Cave Skitterer Elite','Silkfang Stalker Elite','Venom Sac Spider Elite','Cocoon Crawler Elite','Webguard Spinner Elite']`
— matches exactly the `enemyNames` already declared on the cave's `CAVE_DEFINITIONS` entry
(`cave-system.js`), and contains no non-spider names (no wolves, bats, husks, bandits) — **"no non-spider
theme pollution" is satisfied by the data itself, not just by review.**

**Boss/final chamber:** 3 fully-implemented named bosses with distinct ability kits
(`data/enemies.js:122-164`: `boss_broodwarden_skirr`, `boss_matron_velyra`, `boss_queen_arakhzel`), driven by
real runtime logic — `updateBossAbilities` (`dungeon-system.js:2756-2775`, HP-phase-gated ability cadence)
and `castSilkWebBossAbility` (`:2778-2865`, implements summon-adds, cocoon-prison grabs requiring the player
to destroy a placed object to free the target, shields, heals, retreat, and frenzy at low HP). Bosses are
dormant until the player enters their arena (`silkWebBossArena`/`silkWebBossEncounterEngaged`, `:984-1103`).
`boss_silk_mother` (the legacy draft's boss, `data/enemies.js:100-118`) has a stat block but is never spawned
by any code — orphaned, consistent with its parent draft being superseded.

**Exit/collision:** dungeon-specific, not the generic cave `caveExit`/`caveStairsDown`/`caveStairsUp`
mechanism (which only applies when `currentZone === 'cave'`, per `cave-system.js:848-854`). Instead,
`activateDungeonTravelNode` (`dungeon-system.js:2691-2743`) handles `dungeonExit` (calls `exitDungeon()`),
`dungeonStairs` (blocked with a "Boss gate sealed" toast while `floorBossesAlive()` is true, otherwise loads
the next floor), and `dungeonTreasure` (boss-gated loot roll). `exitDungeon()` (`:2887-2923`) restores the
player to whichever zone/map they entered from and saves.

**Quest chain, puzzles:** a 5-quest chain (`data/quests.js:136-179`,
`quest_silkweb_threads_in_the_dark` → ... → `_the_looming_queen`) with matching NPC quest givers
(`data/npcs.js:524-610`) and 3 floor-specific web-anchor gate puzzles (`data/dungeons.js:234-269`, tied to
defeating specific minibosses via `SILK_WEB_GATE_REQUIREMENTS`, `dungeon-system.js:796-800`) are all real and
consistent with each other.

## 4. Small Gaps Found, Documented But Not Changed

- **`poisonDrip`** (`render/object-renderer.js:159`) has a render case but is never instantiated by any
  placement code anywhere in the codebase (confirmed by grep across `dungeon-system.js`, `cave-system.js`,
  `mob-spawn-system.js`). Left in place rather than removed — it's a harmless, working, unused capability
  that a future phase (e.g. Phase 6-style exterior dressing elsewhere, or Phase 15's environmental
  storytelling scenes) could reasonably use later; deleting a working render function because it's
  momentarily unreferenced isn't proven-dead in the sense the project's rules mean (it's not broken or
  orphaned, just not yet called).
- **`dungeon_silk_web_depths`** (`data/dungeons.js:30-49`) and its boss **`boss_silk_mother`**
  (`data/enemies.js:100-118`) are genuinely unreachable — no code path resolves this draft id at runtime
  (the live cave hook resolves to `'silk_web_cavern'`, not `'dungeon_silk_web_depths'`). This is legacy data
  superseded by the `silk_web_cavern` draft. Left in place rather than deleted in this phase, since deleting
  data files carries a small risk of an untraced dependency (e.g. an editor dropdown listing all
  `DUNGEON_DRAFTS` generically) and this phase's validation criteria don't require cleanup — flagging as a
  safe candidate for a future maintenance pass rather than acting on it unprompted.
- **`quest_chain_silk_web_cavern`** (referenced as a `questChainId` field on the live dungeon draft,
  `data/dungeons.js:210`) has no corresponding chain-container object defined anywhere. Cosmetic only — the
  5 quests already chain correctly via their own `prerequisiteQuestIds`.

**Results:** both passed with zero errors (expected — no gameplay files were touched this phase).

**Functional spot-checks (executed in Node, not just traced):**

```
$ grep poisonDrip across dungeon-system.js/cave-system.js/mob-spawn-system.js -> no matches (confirmed unused)

$ node -e "... load cave-system.js ... console.log(CAVE_DEFINITIONS.map(c => ({id, directDungeon})))"
```

this phase's "no extra dungeon caves created" criterion using the actual current codebase, not the
pre-Phase-5 state.
