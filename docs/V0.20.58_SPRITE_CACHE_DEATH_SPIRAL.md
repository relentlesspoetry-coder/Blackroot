# V0.20.58 — The Slow-Frame Death Spiral

## The finding

**25 of 27 sprite-cache keys changed every single frame.** Diffing the key components showed the only
difference was the trailing animation time bucket:

```
...::west::walk::alive::atk0,...,sc8:::_d8a87e:short:...::noeq::t2
...::west::walk::alive::atk0,...,sc8:::_d8a87e:short:...::noeq::t3
                                                              ^^
```

Identical actor, identical direction, identical action, identical equipment. Only the clock moved.

## Root cause — the spiral

A moving actor's cached bitmap is bucketed to last `spriteCacheMovingFrameMs` = **100ms**.

At 9 FPS, **one frame takes 111ms.**

So the bucket expires before it can ever be reused. Every frame lands in a new bucket → the key never
repeats → every lookup misses → the cache **re-bakes** the model (a full procedural draw into a fresh
offscreen canvas) rather than reusing anything.

**Re-baking costs strictly more than just drawing the model live.** So:

> the slower the game runs → the more frames fall outside their bucket → the more it re-bakes →
> the slower it runs

The cache was designed as a performance net and, below ~30 FPS, had inverted into a performance tax.

## The fix

`systems/runtime-sprite-cache-system.js`, `actionFrameMs()`: floor the bucket duration at **3× the
measured `perfStats.frameMs`**, so the quantization actually quantizes.

**Provably a no-op when healthy.** At 60 FPS the floor is 50ms — below *every* configured bucket
(66 / 100 / 125 / 250ms), so it cannot alter behaviour. It only starts to engage below ~30 FPS:

| frame rate | floor | effect |
|---|---|---|
| 60 FPS | 50ms | none — under all buckets |
| 30 FPS | 99ms | none — still under the 100ms moving bucket |
| **9 FPS** | **333ms** | **engages: buckets outlive frames again** |

## Measured — by counters, not timings

Timings in this preview are noise (documented in V0.20.55). Counters are not.

| arm | hit rate | re-bakes/frame |
|---|---|---|
| OLD (no floor) | 55% | **12.2** |
| **NEW (~9 FPS)** | **88%** | **3.2** |
| OLD again — **drift check** | 54% | 12.4 |

The drift check reproduces arm 1, so the difference is the change and not the environment.

**~9 fewer full procedural humanoid re-bakes per frame**, in a frame where humanoid drawing was ~29ms of
~44ms total.

## A correction I owe

`runtimeSpriteCache.beginFrame()` is called from **exactly one place**: `loop(now)` in `game.js` — the
real animation-frame loop.

Driving `update()` + `render()` directly — which is how I measured all session — **never resets
`missesThisFrame`**. After the first 96 misses the cache sits permanently in budget-skip mode and
silently draws everything live.

Every cache measurement I took before this ran through that broken loop. That is why V0.20.57 reported
*"the runtime sprite cache is NOT the lever"* with a confident-looking 10ms-in-both-arms result — **both
arms were measuring the same uncached path.** The conclusion was an artifact and it was wrong.
`docs/V0.20.57_NAMEPLATE_TEXT_MEMO.md` has been corrected in place rather than left standing.

**Rule for any future cache profiling: call `cache.beginFrame()` once per emulated frame.**

## Context: what the user's own test established

Dropping Render Resolution to 50% (a ~75% cut in pixels) moved 9 FPS to "8 to 12" — i.e. **not fill-rate
bound, draw-call bound.** That is what redirected this pass away from resolution and into per-draw cost,
and it is also why the software-rendering preview is a *reasonable* proxy here: it is draw-call bound too.
