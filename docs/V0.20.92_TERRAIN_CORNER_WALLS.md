# V0.20.92 — Corner Walls on Raised Terrain

Reported from a screenshot of the Ashen highlands: notches along the cliff edges where a slope steps
diagonally.

## Root cause — one line

`resolveHalfBlockExposure` opened its corner loop with:

```js
if (!sideFaces.length) continue;
```

That drops any corner exposed **only diagonally**, and that is the common staircase case: a tile level
with *both* cardinal neighbours but higher than the diagonal one has no side faces at all, so no tile
ever emitted its corner column.

**Measured:** 433 of 1,579 interior exposed corners in the Ashen highlands — **27.4%** — had zero owners.

## It is pre-existing, not an Ashen Valley bug

The same audit on Dark Woods: **34.6%**. It has never shown because Dark Woods has almost no raised
ground — my scan found **0 raised tiles** in a 110×110 sample of it. Ashen Valley's elevation didn't
introduce this fault; it made a long-standing one visible.

Worth stating plainly given the last several versions: not everything that surfaces in the new zone is a
new-zone bug. Two of them were (fishing, save serialisation); this one is not.

## Two measurement errors of my own

Both caught before they became conclusions, and both worth recording because they are the same shape:
**an audit whose sampling excludes the thing it is measuring.**

1. **Sampling every 3rd tile reported a 50% orphan rate.** A corner is shared by four adjacent tiles; by
   stepping 3, I excluded three of the four possible owners by construction. The number was meaningless.
2. **The contiguous rescan still counted boundary corners.** Corners on the edge of the scan window have
   sharers *outside* it. All six of my first "orphan" samples turned out to sit on that boundary row — I
   nearly diagnosed the `singleOwner` rule from them. Only corners whose four sharers all lie inside the
   window are counted now.

## The fix, scoped to a genuine elevation drop

First attempt allowed **any** diagonal exposure. Result, measured:

| | orphans | double-drawn |
|---|---|---|
| before | 433 | 8 |
| any diagonal exposure | **0** | **16** |
| elevation drops only | **0** | 8 (Dark Woods back to its baseline 4) |

`isLowerTerrainExposure` is also true for **water** and **cave-wall** adjacency, so with the loose rule
both diagonal pairs around a corner could claim it. Restricting to `tileElev > diagonalElev` keeps the
staircase fix and leaves water corners to the shore pass that already owns them.

**Honest residual:** the Ashen highlands still carry 8 double-drawn corners above baseline — 0.5% of
1,579, which reads as a slightly darker seam — against 433 corners that previously had no wall at all.
That is the right trade, but it is a trade, not a clean win.

## No change needed downstream

`drawHalfBlockCornerJoins` already computes `full = twoSidedJoin || sideFaces.length > 1`, so a
diagonal-only join automatically gets the lighter single column — which its own comment describes as the
treatment for *"one-sided diagonal joins… which repairs zig-zags and bridge endpoints"*. The intent was
already there; the corners just never reached it.

## Verified

| check | result |
|---|---|
| elevation corners orphaned — Ashen highlands | **0** (was 433) |
| elevation corners orphaned — Ashen basin | **0** |
| elevation corners orphaned — Dark Woods | **0** |
| double-drawn — Dark Woods | 4, baseline |
| highlands frame time | **6.06 ms / 6.02 ms** repeat (165 FPS) |
| render across highlands, riverlands, Dark Woods | no errors |

Adding 433 corner columns cost no measurable frame time, confirmed on a repeat arm because this
project's synthetic timing drifts.
