# V0.20.66 — Taming Discoverability and Unobtainable Bait

Both bugs surfaced from one question — *"how do I actually get a mount?"* — answered by walking the
player's path rather than re-reading the code.

## Bug 1 — pressing T told the player nothing

`beginTaming()` computed a precise rejection reason and passed it to `notify()`. But `notify()` only
sets `lastMessage`, and `lastMessage` is rendered **nowhere except inside an active session**.

So the one moment the explanation was needed — *before* an attempt could start — was the exact moment it
could not appear.

```js
const check = this.evaluate(entity);
if (!check.ok) { this.notify(check.reason); return false; }   // reason computed, then discarded
```

Measured: pressing T on a healthy wolf produced `whatThePlayerSees: []`.

**The feature was undiscoverable.** None of its rules — weaken first, bring bait, clear other hostiles,
level and rank gates — could be learned by playing.

**Fix:** log the reason. Verified the player now sees `Requires level 4.` and
`You need Darkwater Fish to approach it.`

## Bug 2 — three of the ten beasts could never be tamed

I chose each beast's bait to suit the animal, without checking the item could be obtained.

| bait | wanted by | actual status |
|---|---|---|
| `item_thornberry` | Briar Boar, Old Tusk Briarback | appears **only** as a crafting input and a quest delivery target — nothing drops, sells or crafts it |
| `item_queens_silkroot` | Thorn-Crowned Stag | defined in `data/items.js` and **referenced nowhere else in the game** |

**Briar Boar, Old Tusk Briarback and the Thorn-Crowned Stag were untameable by anyone.**

**Fix:** boars now want `item_gloomleaf` (drops *and* sold in camp); the stag wants `item_mooncap`
(drops). Both verified present in loot/vendor data.

### Why nothing caught it

`validate.sh` verifies that **loot references resolve to items** — the opposite direction. It is
structurally blind to an item that *nothing points at*. The existing `itemsSourced` figure counts items
with sources; it does not check that a *consumer* of an item can reach it.

### Guard added

A `mountBaits` section in `runValidationSuite` fails the suite if any bait lacks a loot / vendor / craft
source, reported in `summary.mountBaitIssues`.

**Tested with a negative control**, not merely observed passing:

| arm | result |
|---|---|
| current baits | `ok: true`, 0 issues |
| `item_queens_silkroot` re-injected | **`ok: false`** — `mount_gloom_wolf: bait item_queens_silkroot has no loot/vendor/craft source` |
| real bait restored | `ok: true` |

A check that has never been seen to fail is not a check.

## Flagged, not silently patched

`item_thornberry` and `item_queens_silkroot` remain **unobtainable items in the catalogue**, and a Dark
Woods quest asks the player to **deliver a thornberry** — meaning that quest is currently uncompletable.

These are pre-existing content gaps belonging to **Roadmap Item 12** (loot and acquisition audit), not to
the mount system. Recorded here rather than papered over, because fixing them inside Item 7 would hide a
quest bug that deserves its own attention.
