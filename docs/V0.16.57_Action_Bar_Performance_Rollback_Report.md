# V0.16.57 - Action Bar Performance Rollback Report

## Summary

Rolled back only the action-bar PNG asset integration from the active in-game HUD. The runtime `#hotbar.abilityBar` now uses the existing lightweight CSS/DOM renderer again instead of the `Action Bar UI` PNG shell.

Gameplay behavior was preserved: EXP updates, hotkeys, slot clicks, spell icons, cooldown overlays, Auto Attack state, and the Meditate button all continue to use the existing action-bar markup and owning systems.

## Root Cause of the Lag

The V0.16.53/V0.16.55 action bar integration made a large PNG shell the active hotbar background:

- `#hotbar.abilityBar[data-ui-asset="Action Bar UI"]`
- `background: url("assets/ui/Action Bar UI.png") center / contain no-repeat`
- fixed asset aspect-ratio sizing against a large source image
- percentage-positioned overlay and hitbox mapping for EXP, slots, cooldowns, and Meditate
- asset-specific responsive rules that bypassed the prior lightweight layout

That forced the browser to keep a large transparent PNG in the active HUD paint/composite path. During combat, the hotbar has frequent DOM state changes for cooldowns, Auto Attack, EXP, and slot labels, so the PNG-backed shell became an avoidable paint/composite cost.

## PNG Action Bar Code Removed/Disabled

- Removed the active `data-ui-asset="Action Bar UI"` and `data-ui-asset-file="Action Bar UI.png"` attributes from the `#hotbar` markup.
- Removed the CSS block that made `assets/ui/Action Bar UI.png` the runtime action-bar background.
- Removed the PNG-specific absolute overlay mapping for EXP, slots, labels, cooldowns, Auto Attack, and Meditate.
- Removed PNG-specific responsive sizing rules for the action bar.
- Removed the `ACTION_BAR_UI_ASSET` runtime registration from `systems/ui-system.js`, so no active UI asset registry path points the hotbar at the PNG shell.

No `Action Bar UI` asset file was deleted. This rollback only removes the asset from the active runtime HUD path.

## Lightweight Renderer Restored

The existing lightweight `#hotbar.abilityBar` CSS/DOM renderer remains active:

- bottom-centered fixed `#hotbar`
- ten-slot `.abilitySlots` grid
- CSS gradient panel shell
- CSS-rendered EXP meter
- DOM-rendered spell icons and labels
- DOM cooldown fill/readout overlays
- CSS Auto Attack active/cooldown outline states
- existing Meditate button placement

No duplicate action bar renderer was added. The old renderer is restored by removing the newer PNG-specific override from the owning stylesheet path.

## Action Bar Input / Hitbox Verification

Static verification confirms the same action-bar DOM and input ownership remains intact:

- `systems/spell-system.js` still binds input through `Game.prototype.bindActionBarInput()` on `#hotbar`.
- Click/pointer/keyboard activation still delegates to `.actionSlot` elements inside `#hotbar`.
- Slot 1 keeps `data-action-slot="autoAttack"` and calls `playerAttack()`.
- Spell slots keep `data-action-slot="spell"` and `data-spell-index`, then call `useClassSpell(index)`.
- `systems/ui-system.js` still updates cooldown bars/readouts through `renderHotbarCombatState()`.
- `#steamDeckMeditateBtn` markup is unchanged and remains bound by the existing Steam Deck/UI input path.
- `#xpFill` and `#xpText` remain in the action-bar EXP meter and are still updated by the UI system.

## Performance Notes

- The active hotbar no longer references or renders the large PNG shell.
- The hotbar is back to static CSS gradients and small DOM elements.
- The rollback avoids per-frame large-image scaling/compositing on the action bar.
- No new canvas filters, shadows, composites, wrapper hotfixes, timers, or duplicate render paths were introduced.
- Expected runtime result: FPS should improve whenever the action bar is visible, especially during combat cooldown/Auto Attack updates.
