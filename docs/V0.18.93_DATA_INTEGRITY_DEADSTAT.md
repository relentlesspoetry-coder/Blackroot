# V0.18.93 — Data-Integrity Validation + Dead-Stat Fix (Roadmap Item 22)

A new item-stat data-integrity check (Item 22, "cover data integrity") immediately surfaced a
widespread, pre-existing bug: a lot of gear's attribute bonuses were **silent dead stats**. This
version adds the check and fixes the root cause.

## The new check — `validateItemData()`

`Game.prototype.validateItemData()` (`systems/inventory-system.js`) scans every `DR.ITEM_BY_ID`
entry's `stats` for keys not in the valid set, and validates armour-set integrity (piece ids resolve,
set-bonus stat keys valid). It is wired into `runValidationSuite` as a **hard-fail** section — an
unknown stat key is a real bug, not advisory. The boot log gains `N data issue(s)`.

## The bug it found

The game **displays** the attributes under different names than its internal keys:

| internal key | displayed as |
|---|---|
| dexterity | **Agility** |
| intelligence | **Intellect** |
| wisdom | **Spirit** |

(`systems/inventory-system.js:1459-1460`.) Item/data authors naturally wrote bonuses under the display
names — **`intellect` (45 uses in items.js), `agility`, `spirit`** — but `core/stats.js`
`mergeAttributes` only read the canonical keys, and **nothing read the display-name spellings as
values**. So every such bonus did nothing: caster gear (including the pre-existing Wispflame Focus and
the V0.18.75-88 caster pieces) and the silk-cavern venom set granted attributes that silently had no
effect.

Confirmed live before the fix: equipping the Runespun Cowl (`intellect: 4`) changed Intelligence by
**0**.

## The root fix (surgical, one place)

`core/stats.js` gains an alias table and folds aliases into the canonical attribute during merge:

```js
const ATTRIBUTE_ALIASES = {
  str: 'strength', dex: 'dexterity', agility: 'dexterity', sta: 'stamina',
  int: 'intelligence', intellect: 'intelligence', wis: 'wisdom', spirit: 'wisdom'
};
// mergeAttributes: for each attribute, add bonusStats[canonical] + any alias spellings
```

Exposed as `DR.ATTRIBUTE_ALIASES`. `validateItemData` accepts `STAT_KEYS` + those aliases + legitimate
non-combat fields (bag **`slots`**) + one tolerated-unimplemented stat (**`poisonResist`** — an
intended silk-cavern resistance not yet wired into combat; whitelisted so a real typo still stands
out, and spawned as a follow-up task to implement or retire).

## Balance note

This restores intended functionality, so affected caster and silk-cavern gear now grants its
Intellect / Agility / Spirit (a buff to previously-dead stats). That is the correct behaviour — the
items always displayed these bonuses.
