# V0.20.23 ‚Äî The Blast You See Is The Blast That Hits You (Roadmap Item 6 fidelity)

## The spec's own example, failing on every AoE in the game

> *"A ground-targeted spell must show the correct affected area."*

It didn't. `spawnRing`'s radius argument is **screen pixels** ‚Äî its `|| 18` default proves the intent ‚Äî
but the resolvers passed **`spell.radius`, which is world tiles**. Two units in one expression:

```js
enemiesIn(p.x, p.y, spell.radius || 4)   // gameplay: WORLD TILES
this.spawnRing?.(p.x, p.y, color, 22);   // visual: PIXELS, unrelated
this.spawnRing?.(p.x, p.y, color, spell.radius || 18);   // tiles fed to a pixel default
```

**Measured, not reasoned** ‚Äî one world radius projects to **73.54px** on x:

| spell | damages | drew | |
|---|---|---|---|
| Fireball | 3 tiles = 179px | 20px | ~9√ó small |
| Flame Wave | 4 tiles = 238px | 20px | ~12√ó small |
| **Meteorfall** | **7 tiles = 515px** | **46px** | **~11√ó small** |

**The game has been killing players ~11√ó further out than the graphic showed**, and no player could
learn a spell's true reach by looking at it. This is the identical bug the **V0.19.x cone fix**
addressed ‚Äî *"a cone attack must visibly use the real cone dimensions"* ‚Äî still alive for rings.

## Fixed at the funnel, not the ~99 call sites

`resolveClassSpell` already publishes the caster (V0.20.21), so it now also publishes the **area
radius**; `spawnRing` carries it as an optional `effect.worldRadius`, and the renderer sizes from it.
One change, every AoE in the game, and every future one free.

### The trap the data revealed

**`spell.radius` does not mean one thing.** On an `aoe` kind it's the blast; on a **buff** it's the
**ally search range** ‚Äî Mana Shield, Arcane Intellect and Blink all author `radius: 14`. A blanket
conversion would have ringed the caster in an **834px halo**.

Published for `aoe`/`aoeDebuff`/`aoeHeal`/`aoeMelee` **only**. **Verified: 0 buff rings leaked.**

### Two things measuring revealed that reasoning would have missed

1. **The ring was the wrong shape too.** `worldRingMetrics` measures the ellipse a ground circle
   actually projects to (rightmost point at world `(R/‚àö2, ‚àíR/‚àö2)`; a 360-sample sweep agrees to 2dp).
   The engine's true y:x ratio is **0.558**; the stylised ring hardcodes **0.42**. World rings now use
   the true ratio; **legacy pixel callers keep 0.42, untouched.**
2. **The sizing target is the peak, not p=1.** `easeOutBack` **overshoots to ~1.1**, so `r` peaks at
   `base √ó 2.098`, not `base √ó 1.96`. Sizing to p=1 would let the graphic claim **~7% more reach than
   the spell damages** ‚Äî a lie in the dangerous direction. Sized to the peak it may under-draw for an
   instant; **it never over-promises.** `ringPeakMult()` measures that from the live easing rather
   than hardcoding it, so changing `easeOutBack` cannot silently desynchronise the drawing from the
   damage.

## My first verification was tautological

```js
const peakDrawn = (truePx / g.ringPeakMult()) * g.ringPeakMult();   // ‚â° truePx
```

I divided by a constant and multiplied it straight back, then asserted equality. **A check that cannot
fail** ‚Äî and the sizing is exact by construction anyway, so it proved nothing. **The fourth vacuous
probe of this session.**

Re-done by instrumenting `drawExpandingRing` and driving the effect across its whole life through the

## Still open

- `combatImpact`/`combatSpark` remain generic (damage path ‚Äî class context would give a Wizard an
  undrawable `rune` and alter melee)
- `decorateEffectVisualPayload` pooling defaults for `kind`/`style`/`school`
- The hardcoded-pixel `spawnRing` calls on **non-area** spells are still arbitrary numbers. They're
  not *wrong* ‚Äî nothing claims they represent an area ‚Äî but nothing ties them to anything either.

## The lesson

The cone got this fix in V0.19.x. **The rings didn't, and nobody noticed for the same reason nobody
noticed `drawRuneFragments` was never called: the code looked right.** A radius was passed, a ring was
drawn, and both halves were individually plausible. Only *measuring the drawn thing against the
gameplay thing* exposed an 11√ó gap.

**Ask what the number means, not whether it's there.** `spell.radius` was present, non-zero, and
authored ‚Äî and it meant tiles to one function and pixels to the next.
