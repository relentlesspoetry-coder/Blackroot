## What was built

### (a) Completed history

`renderQuestLog` put `Completed ${n}` in the summary line and rendered **active quests only**. Seven
finished stories behind a digit.

`state.completed` stores only a key, so the fix reads the authored descriptor back out of
`DR.QUEST_BY_ID`. Completed entries render as a **compact history row** deliberately ‚Äî they have no live
task progress and no claimable rewards, so reusing the active card would show empty meters and dead
buttons.

### (b) Authored fields that were never shown

The card's meta line carried only `folder` and the turn-in. It now also shows **recommended level**, the
**giver** (distinct from who you return to), and a **Repeatable** badge.

These read from the **descriptor**, not the active-state copy, so they can't go blank if that copy ever
drops a field.

### (c) Search / filter / sort

- **Search** ‚Äî name, giver, place. `"Cael"` finds everything the road warden asked of you.
- **Filter** ‚Äî Active / Completed / Repeatable / All.
- **Sort** ‚Äî level / name / place.

View state (`logSearch`/`logFilter`/`logSort`) is deliberately **not** part of quest `state`: that
round-trips through every character save, and a search box is a view preference, not progress.

## What was deliberately NOT built

The spec asks for three things the data cannot support:

| | why not |
|---|---|
| **Main / Side category split** | no quest carries any main/side field ‚Äî `folder` is region-based |
| **Quest level** distinct from `minLevel` | no such field on any quest |
| **Repeatable reset timing** | no `resetHours`/`cooldown`/`dailyReset` anywhere |

Building those would mean **inventing the data**, or shipping a control with nothing behind it ‚Äî the
exact dead-data anti-pattern this session has spent its time removing, only *created* rather than
inherited. A "Main Quests" tab that is always empty is worse than no tab.

**They need authoring first.** When someone decides which quests are main quests, the journal can say so.

### A vacuous test, caught

The first place-search and place-sort tests **passed while proving nothing**. The fixture took the first
5 quests ‚Äî which all share one folder ‚Äî so the Bramblefen search found **0** (correctly), and
`.every()` on an empty array returns `true`. Sort-by-place was equally trivial across five identical
values.

Re-tested with a fixture spanning **all 6 distinct folders**: place search matched 1 of 6 and excluded
the rest; place sort ordered 6 distinct regions. That's the third time this session a test has lied ‚Äî
worth the habit of asking what a pass actually proved.
