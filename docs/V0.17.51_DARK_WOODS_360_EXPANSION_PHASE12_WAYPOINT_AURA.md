**Phase:** 12 of 28 (Dark Woods Zone Expansion + Resource Identity master plan)

## 1. Goal

Animate the Stone Hedge waypoint: a soft glowing/pulsing aura, subtle radial light expansion, faint rising
motes, an optional rotating rune ring, and a slight ground glow — using the existing VFX/Canvas system, with
no new render loop, low particle count, and the emblem staying readable.

## 3. Root Cause / Implementation Notes

**No new render loop — confirmed, not assumed.** Before writing anything, I traced exactly how effects reach
the screen: `game.js`'s main per-frame loop already calls `updateExternalSystems(dt)`, which calls `.update(dt)`
on every registered external system (`systems/chest-system.js`, and — since Phase 11 — `systems/waypoint-system.js`,
via the shared `registerDreamRealmsSystem`/`installExternalSystems()` pattern). Phase 11 had already left
`update(dt)` as an empty stub on the waypoint runtime specifically so this phase could fill it in. That means
adding the aura required **zero** new hooks into `game.js` — only implementing a method on an object that was
already being called every frame.

**Reused the existing ambient-aura pattern, not invented a new one.** `systems/effects-system.js`'s
`bardMeditationAura` is the established precedent for a persistent, non-expiring ambient visual effect (as
opposed to the far more common one-shot combat effects). Its actual creation/refresh logic lives in `game.js`
(`ensureBardMeditationAuraEffect`) and follows a specific shape: track the effect in a side `Map` keyed by a
stable id, and on each periodic check, either reuse the still-live tracked effect or recreate it if it was
ever removed from `game.effects` (e.g. by the effect budget cap dropping old effects under heavy load).
`ensureWaypointAuraEffects()` follows this exact shape, simplified because a waypoint (unlike a meditating
actor) has no "is the source condition still active" check to make — it's always present, so the only thing
being verified is "does the tracked effect still exist," not "should it still exist."

**Why `life: 1e9` instead of the bard aura's short-refresh-cycle approach.** The bard aura uses a short life
(0.95s) and resets `t = 0` every ~0.3s specifically because it needs to naturally disappear soon after its
active condition stops (a player stops meditating) without a separate removal call. A waypoint has no such
"stop" condition within a play session, so rather than replicate that refresh-and-reset dance, the waypoint
aura effect is given an effectively infinite `life` and is only re-verified (not reset) every ~1s purely as a
resilience check against the shared effect budget cap. This is a deliberate simplification, not a shortcut —
it does the same job with less code because the underlying problem is simpler (no on/off condition to track).

**Camera culling is free, confirmed by tracing the render pipeline, not assumed.** `game.js`'s render loop
already applies a generic per-effect screen-space cull (`insideScreenFast(ef.x, ef.y, elev, effectCullPad)`,
~line 3793) before any effect — including `waypointAura` — is added to the frame's renderables list. No
culling logic needed to be written for this phase; using the standard `game.effects`/`addEffect()` path is
what makes this automatic.

**Animation is driven by elapsed real time, not effect-relative `t`.** Since `life` is huge, `e.t / e.life`
(the fade fraction other effects use) stays at ~0 essentially forever — appropriate, since this should read
as persistent, not fading. All actual motion (pulse brightness, ring rotation, mote rise, radial expansion)
instead uses `time = runtimeNowMs() / 1000` directly with `Math.sin`/modulo, the same technique already used
by `drawLanternPost`, `drawRuneStone`, and `drawMagicResidue` from earlier phases — giving smooth, seamless
looping with no reset/restart bookkeeping required.

**Readability and particle count.** The aura draws at low alpha (peak ~0.4–0.7 depending on pulse phase) and
is layered on top of, not instead of, the static emblem `drawWaypoint` already renders underneath — the
emblem's cross/rune strokes remain visible through the glow. Total per-frame draw work for the whole aura is
4 rising motes + 6 rotating rune-ring tick marks + 3 simple shapes (ground glow, aura glow, expanding ring) —
comparable to or lighter than the bard aura's own particle count (5 orbiting glyphs + 2 wave rings), and
there is exactly one waypoint in the game, so the total cost is negligible.

## 4. Validation

Commands run:

```bash
find . -name "*.js" -print0 | xargs -0 -n1 node --check
find . -name "*.json" -print0 | xargs -0 -n1 python3 -m json.tool > /dev/null
```

**Results:** both passed with zero errors — **after fixing a real syntax bug caught by this exact check**: a
stray `#` character was accidentally left at the start of a comment line in `render/effects-renderer.js`
(should have been `//`), which `node --check` immediately flagged. Fixed before proceeding; this is a direct
example of why this validation step is run before every phase is considered done, not just at the end.

**Functional validation (executed in Node, not just traced):** re-implemented the exact
`ensureWaypointAuraEffects()` lifecycle logic in a standalone script against a fake `game.effects` array:

```
After 1st call, effects.length: 1
After 2nd call (should not duplicate): 1
After effect removed + 3rd call (should recreate): 1
```

This confirms, by running the real logic rather than reading it, that the aura effect is created exactly
once, is never duplicated by repeated `update()` ticks, and correctly self-heals if it's ever dropped by the
effect budget cap — the three behaviors this phase's design depends on.

**Not validated (no browser available, same limitation as Phases 2–11):** the actual visual appearance and
smoothness of the pulse/rotation/motes, whether the aura is visible in dark lighting as required, and
real-world frame-rate impact — none of these can be confirmed without rendering a frame.

- Check visibility specifically at night/in dark lighting conditions if the game has a day/night cycle

  active nearby (e.g. combat).
- Confirm attuning the waypoint (Phase 11's interaction) doesn't interrupt or duplicate the aura effect.

## 6. Known Limitations

- Only one waypoint exists, so this phase's "recreate if dropped" resilience logic has only been exercised
  in a single-waypoint synthetic simulation, not under real heavy multi-effect load in a live session.
- No in-browser visual or performance test was possible in this environment (see Section 4).
- The aura's exact visual tuning (pulse speed, ring rotation speed, mote count/rise speed) is a first pass
  and may need adjustment once actually seen rendered.
