# V0.20.41 — Armor Proficiency (Roadmap Item 8, first slice)

The first slice of the Unified Skill Progression System: the **armor-proficiency mechanic**. Four armor
skills (cloth / leather / chain / plate), level 1–20, earned from taking real enemy hits while wearing
that armor, raising the armor DR those pieces provide.

## Built on existing conventions

- **Data on the GAME** — `game.armorProficiency = { cloth:{level,xp}, … }` (belongings live on the
  game, per V0.20.2), lazy-initialised.
- **Bonus through the stat pipeline** — added to the derived-stats aggregator inside
  `recalculatePlayerStats`, so it flows through `computePlayerDerived` and is **recomputed on every
  equip, never written to item base** (the spec's explicit requirement).
- **XP hook** — `damageEntity`'s player-took-enemy-damage branch.
- **Persistence** — `serializeCharacterState` / `applyCharacterState`, reset in
  `resetCharacterOwnedState`.
- **Anti-exploit** — a trivial-damage floor (`< 1% max HP` grants nothing) + a 1.5s cooldown, so idle
  chip damage and rapid hits don't farm it. Max-level (20) enforced.

## Verified through the real paths — and the verification caught two traps

| check | result |
|---|---|
| dominant armor type from equipment | leather (3 leather pieces) ✅ |
| XP → leveling | 50 gains → level 18, 17 level-ups ✅ |
| anti-exploit | trivial damage → 0 XP; cooldown holds 2nd gain ✅ |
| max level | capped at 20 ✅ |
| **armor DR bonus (mechanical benefit)** | leather **22.2% → 24.6%**, high-armor gear **53.7% → 56.2%** at L20 ✅ |
| save/load round-trip | level 12 preserved ✅ |

**Two traps the real-path verification caught before shipping wrong:**

1. **I first measured `player.defense` — which didn't move.** `defense` and `armor` are **separate
   stats**, and damage mitigation uses **`armorReductionFor` (armor), not defense**. Measuring `armor`
   showed the bonus lands (23 → 27).
2. **A flat "damage taken 37" at both levels looked like the bonus did nothing** — it was the V0.20.27
   one-shot cap masking the small delta. Measuring `armorReductionFor` **directly** proved the benefit
   (the DR% rows above).

Both are the session's recurring lesson: verify the *thing that matters* (the DR the player actually
gets), through the real path, not a convenient adjacent number.

## Honest gaps — documented, not hidden

1. **44 starter armor items lack `armorType`.** All loot/shop/crafted armor has it, so proficiency
   works once real armor is equipped — but a **brand-new character in starter gear trains nothing**.
   Tagging the starter items needs a **class → armor-weight mapping** that neither the class defs nor
   the items encode yet. That's a **design decision** (what weight each class wears), not something to
   invent 40 versions deep, so it's flagged for a deliberate call.
2. **No skills-progress UI yet.** The spec wants a progress display in the character/skills interface;
   today there's the level-up log line and the visible armor/DR increase. A proper panel is the next
   slice.

So this is the **mechanic/foundation**, complete and verified, not the finished Item 8 feature.

## The lesson

The build was straightforward; the *verification* was the work. Every one of the four integration
points looked right in the source and had to be proven through the real path — and two of them (the
armor-vs-defense stat, the cap masking the delta) would have shipped as "done" on a lazier check. A
feature is only as verified as the specific number you measured, on the path the player actually takes.
