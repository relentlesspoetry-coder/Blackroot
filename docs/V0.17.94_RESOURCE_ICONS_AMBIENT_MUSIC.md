# V0.17.94 — Resource Item Icons + Ambient Music Overhaul

Two independent asset-level fixes, no gameplay/system logic changed.

## 1. Resource / tool / fish item icons (61)

The final batch of item art (delivered in `Item Icons/`) was bound to the game's
existing PNG-icon pipeline, completing icon coverage for the whole
resource/gathering economy (ores, bars, herbs, mushrooms, fish, pelts, gems,
essences, gathering tools, and a few quest pieces).

**How the icon pipeline works** (unchanged): an item shows a PNG when its
`item_*` id is listed in `UPLOADED_ITEM_ICON_IDS` (`data/items.js`). That list
populates `item.iconKey` (= id minus the `item_` prefix), `DR.ITEM_ICON_KEYS_BY_ID`,
and `DR.ITEM_ICON_ASSET_KEYS`. At render time `Game.prototype.resolveItemIconKey`
(`systems/inventory-system.js`) returns `hasAsset=true` only when the key is in
`ITEM_ICON_ASSET_KEYS`, and the image path is `assets/item-icons/<iconKey>.png`.
Items not in the list fall back to their procedural `{family,color,glyph}` badge.

**What changed:**
- Copied 61 PNGs → `assets/item-icons/<iconKey>.png` (all 128×128, matching the
  existing set).
- Appended the 61 item ids to `UPLOADED_ITEM_ICON_IDS`.
- Updated the records manifest `assets/item-icons/manifest.json` (157 → 214
  entries; the manifest is documentation only — it is not fetched at runtime).

**Name-matching notes** — 58 icons matched their item by exact (normalized) name.
Three matched by intent despite a label difference:
- `Rotting Root.png` → `item_rotling_root` ("Rotling Root" — spelling variance).
- `Gloom Wolf Pelt.png` → `item_wolf_pelt` (item name is "Gloom Wolf Pelt").
- `NightCrystal.png` → `item_night_crystal`.

Four ids already had an unused PNG on disk but were **not** in the uploaded list
(so they were never actually shown): `basic_fishing_rod`, `crude_pickaxe`,
`herbalist_knife`, `worn_hatchet`. Their PNGs were replaced with the supplied art
and the ids added to the list, so they now render.

## 2. Ambient music overhaul

**Root cause of the "buzzing".** The four `music/*.wav` loops were bare
additive-synth drones, confirmed by spectral analysis:
- `town_lantern_loop.wav` — a near-**pure 196 Hz sine** (all other partials ≤2%),
  temporally static (RMS coefficient-of-variation 0.05).
- `combat_pulse_loop.wav` — a static 220 Hz tone (cv 0.06).
- `dark_woods_loop.wav` — stacked octave-A drones (440/220/880 Hz).
- `cave_ambience_loop.wav` — the only one with real texture.

A continuous static low-frequency tone reads to a listener as a hum/buzz. The
**playback code was already correct** — `systems/audio-system.js` fetches, decodes,
and loops the WAVs via WebAudio, and the procedural fallback (off by default) is
sine/triangle, not the source of the buzz. So this is an **asset** fix, not a code
fix; `audio-system.js` is unchanged.

**Fix.** `tools/generate-ambient-music.py` (new, dev-only, pure stdlib)
regenerates all four loops as real ambient music:
- detuned **stereo pads** (3 slightly-detuned partials per note, panned L/C/R),
- a slow **chord progression** with equal-power crossfades between bars,
- a soft lowpassed **noise air-bed** (stereo-decorrelated),
- per-zone accents: town **arpeggio**, cave **bells**, combat **heartbeat pulse**,
- a **seamless equal-power loop crossfade** (renders `loop+xf`, folds the tail
  continuation back over the head so the end→start wrap is continuous),
- tanh soft-clip + normalize to ~55–75% peak (headroom, no clipping).

Re-run with `python3 tools/generate-ambient-music.py` from the project root.

## Verification (in-browser, port 8899)
- `DR.ITEM_ICON_ASSET_KEYS` = 214 keys; all sampled new keys present;
  `resolveItemIconKey` returns `hasAsset=true` for the new items incl. the three
  variance matches; PNGs load 128×128.
- All four new loops fetch (200) and `decodeAudioData` as stereo with the expected
  durations (24 / 20 / 16 / 8 s); measured peaks 54–61% (no clipping); loop-wrap
  discontinuity 0.1–1% of peak (seamless); RMS cv 0.13–0.21 (evolving, vs the old
  0.05–0.06 static drones).
