# V0.17.95 — Performance: boot recursion fix + render-resolution scaling

A full profiling review of the running game plus two concrete changes: one real
bug fix and one new performance lever. No systems, features, or graphics removed.

## Profiling summary (what the review found)

Measured live in-browser (menu backdrop + in-world Fighter at camp, 338 mobs):

- **Render pipeline is already well-optimized.** Screen-space culling drops ~347
  of 355 entities (only ~8–16 actually drawn); renderables use reused object
  pools (zero per-frame allocation); objects are chunk-indexed; the runtime sprite
  cache runs ~88% hit with 0 re-bakes/frame and 0 failures; there is **no**
  per-frame `shadowBlur` / `ctx.filter` / `getImageData` in the canvas2d path.
  CPU-side render logic (queue build + sort) is ~0.4 ms/frame.
- **Enemy AI is distance-banded and wall-clock budgeted** (near/mid/far/sleep with
  staggered per-real-second intervals), so total AI work is bounded per second —
  roughly 6–7 ms/frame at 60 FPS.
- **Sandbox caveat:** this headless preview renders canvas2d in **software**, which
  is draw-call-bound (≈48 ms to draw ≈88 items, and *resolution-insensitive* — the
  same ≈55 ms at 100% and 50% scale). Absolute FPS here does **not** represent real
  GPU-composited hardware. This is why the review leaned on code inspection +
  targeted checks rather than absolute frame numbers.

## 1. Boot recursion / stack-overflow storm (bug fix)

`systems/performance-stable-release-lock-system.js`

`PerformanceStableReleaseLockSystem`'s constructor ran side-effecting init
(`applyReleaseDefaults` → `setHybridDefaultRolloutPolicy` → `renderSettingsPanel`).
The settings-panel snapshot chain re-enters `ensurePerformanceStableReleaseLockSystem()`,
but the singleton was only assigned **after** the constructor returned — so the
re-entrant call saw a falsy field and constructed the system **again**, recursing
until the V8 stack overflowed. The first allocation to fail was a trivial regex in
the QA telemetry (`/[|,]+/g`), producing the misleading
`Invalid regular expression: /[|,]+/g: Stack overflow` spam. Worse: the overflow
happened *before* the `defaultsApplied='1'` guard was written, so the guard never
persisted and the storm **re-fired on every page load**.

**Root-cause fix:** `ensurePerformanceStableReleaseLockSystem()` now stores the
singleton on the game **before** running the deferred init (`applyReleaseDefaults`
+ `syncPerfStats`), so the re-entrant call reuses the instance. The constructor no
longer performs those side effects. Verified: forcing the exact path
(`performanceStableReleaseLockSystem = null; ensure…()`) now constructs cleanly with
0 errors, and a settings-panel render produces 0 errors.

## 2. Render-resolution scaling (adaptive cap + user slider)

The canvas backing store was sized `cssPixels × devicePixelRatio` with only a DPR
cap and **no absolute pixel budget**. On a large/high-DPI display run maximized,
that fills millions of pixels every frame, and every full-screen pass (lighting,
weather, fog) plus every entity blit pays per backing pixel — the classic cause of
low FPS in a 2D-canvas game even on strong hardware.

**Design — one funnel.** Everything derives from `Game.displayPixelRatio()`, so the
scale is applied there once:

```
dpr = clamp(devicePixelRatio, 1, maxDprCap)          // existing behaviour
    × effectiveRenderScale()                          // user slider × adaptive level
then clamp so cssW*dpr * cssH*dpr <= maxRenderPixels   // absolute budget
floored at ~minRenderScale
```

- `CONFIG.PERFORMANCE.maxRenderPixels` (default **5,000,000**) and `minRenderScale`
  (default 0.5). 5.0M leaves 1080p/1440p-native untouched and caps 4K to ~1800p.
- `Game.userRenderScale` persists in `localStorage['dreamRealms.renderScale']`,
  **default 1.0** (no visible change for anyone). `Game.setRenderResolutionScale()`
  clamps, persists, and calls `resize()`.
- `AdaptivePerformanceSystem.effectiveSettings()` now emits `renderScale` per level
  (balanced/quality 1.0, performance 0.85, rescue 0.72). The existing governor
  already downshifts under sustained frame pressure (>24 ms) and upshifts on
  recovery, requesting a resize — so the world resolution now auto-drops under load
  and restores when smooth.
- **Settings → Graphics → Render Resolution** slider (50–100%), wired through the
  existing settings-input delegation (`data-render-scale`), reusing the audio-slider
  styling.

**Why it's safe (no input/layout impact).** The world renders in **CSS-pixel space**
(`getViewportMetrics()` uses `window.innerWidth/Height`) and the DPR is applied only
as the final `ctx.setTransform` step. Pointer input maps via `getBoundingClientRect`
in CSS pixels. So changing the render scale changes **only** the backing-store
sharpness — never layout, HUD/DOM, or coordinate mapping. Verified empirically:
`screenToWorld(400,300)` returns identical world coords at 100% and 50% scale.

## Verification (in-browser)

- Recursion path constructs with 0 errors; settings panel renders with 0 errors.
- Slider present (Settings → Graphics), default 100%; dragging it fires
  `setRenderResolutionScale`, updates the label, persists to localStorage, and
  resizes the canvas backing store (100%→full, 75%→0.75×, 50%→0.5× linear).
- Pixel budget caps a simulated 4K window (3840×2160) to ≤5.0M backing px (dpr 0.78).
- Adaptive `renderScale` present in effective perf settings (1.0 at balanced).
- `screenToWorld` identical across scales (input mapping unaffected).

**Not verifiable in this sandbox:** the actual FPS gain, because the headless
software rasterizer is draw-call-bound (resolution-insensitive). On real
GPU-composited canvas2d the full-screen passes are fill-bound, so lowering the
backing resolution reduces them proportionally. The feature is off by default and
the slider/adaptive give the player direct control — the real-world gain is the
player's to confirm on their hardware.
