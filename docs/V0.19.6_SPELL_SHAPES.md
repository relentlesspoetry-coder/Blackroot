# V0.19.6 — The Cone That Wasn't (Roadmap Item 6)

## The violation

Item 6's own first example is: **"a cone attack must visibly use the real cone dimensions."**

`spell_assassin_fan_of_knives` declares itself a cone **three times**:

| surface | value |
|---|---|
| name | **Fan** of Knives |
| category | `"AoE Ranged / Cone"` |
| description | `"Throw multiple blades in a short cone…"` |

…and hit a **full 360° circle**, including enemies standing **behind** the player.

## The root cause — structural, not a typo

The angle test was a local `inFront` closure defined **inside** `Game.prototype.resolveFighterRevampSpell`,
and its single call site hardwired the angle:

```js
.filter(enemy => inFront(enemy, radius, id === 'cleave' ? Number(spell.coneDegrees || 100) : 360))
```

Two consequences:

1. **`coneDegrees` was inert for all 279 other spells.** Only `cleave` declared it — a **1/280 field that
   looked plumbed and wasn't**.
2. **No other class could reach the test.** The generic `kind === 'aoe'` handler (in the *assassin*
   resolver) was a pure radius check with no notion of facing:
   ```js
   if (!enemy?.alive || distFn(enemy, p) > (spell.radius || 3.2)) continue;
   ```

This is the **same dead-data pattern** as the attribute aliases (V0.18.93), `poisonResist` (V0.18.96),
and the silver/gold money paths (V0.18.99): a field the data author reasonably believed was doing
something.

## The fix (root cause, owning module)

- `inFront` lifted to **one** shared `Game.prototype.spellShapeContains(enemy, radius, coneDegrees, actor)`
  — a single implementation for every class, instead of each resolver growing its own copy.
- Fighter filter is now **data-driven**: `Number(spell.coneDegrees) > 0 ? Number(spell.coneDegrees) : 360`.
- Generic `aoe` handler honours a declared cone, engaging **only when `coneDegrees` is present**, so every
  existing aoe spell is untouched.

## The one intentional behaviour change

`fan_of_knives` now declares `coneDegrees: 180` — a **forward fan**. "Short" in its description qualifies
the **reach** (`radius: 3.2`), not the arc. It still hits everything in front; only targets at the
assassin's **back** are spared.

This is a modest nerf, and deliberate. The design intent was **unambiguous** — name, category *and*
description all say cone — so implementing it is **honouring the stated design, not guessing at it**.
That is the distinction from the economy scale (V0.18.99), which had two valid readings and was
escalated to the user instead.

## A second bug, found by testing rather than reading

`Math.cos(Math.PI / 2)` is **`6.12e-17`**, not `0`. So `dot >= Math.cos(halfAngle)` **excluded** an enemy at
exactly 90° from a 180° cone — a target standing straight out to your side, skipped on a rounding
artifact. Harmless at cleave's 100°; **exactly on the boundary** for a half-plane. Now carries a `1e-9`
epsilon.

## Verified, not assumed

| check | result |
|---|---|
| **Scope check** | caught that `inFront` was **not in scope** at the assassin call site — `resolveFighterRevampSpell` (831) closes before `resolveAssassinRevampSpell` (1062). `node --check` **cannot see this**; it would have been a `ReferenceError` the first time anyone cast Fan of Knives. |
| **Negative test** | stripped `coneDegrees` and confirmed the validator **catches** it — rather than trusting a first-run pass (the false-pass trap from V0.19.4). |
| **No-regression** | `cleave`/`groundbreaker`/`whirlwind_cleave` proven identical **angle-for-angle** (100°/360°/360° before and after). |
| **Geometry** | 15/15 synthetic cases, including **both** 90° boundaries, out-of-radius, and dead targets. |
| **Compiler** | `coneDegrees` confirmed carried through compilation (cleave's 100 survives) — the **item** compiler is known to drop custom fields. |

## New validator section — `validateSpellShapes()`

Holds the line so this can't silently return. Across **280 spells**, flags:

- `cone-without-dimensions` — claims a cone/arc, declares no `coneDegrees`
- `area-without-radius` — area spell with no radius
- `dot-without-tickrate` — `tickDamage` with no `tickRate`
- `no-cast-animation` — no `animation.cast`

Field names were confirmed against **live data** before the checks were written (`animation.cast` present
280/280), not assumed.
