## Rotling Root Anchor Fix Details
The foot/ground-contact anchor itself (`entity._rootVfxFootAnchor` / `entity._groundContactAnchor`, set every frame in `render/entity-renderer.js: drawEntity`) was already correct - it was added in a prior pass (V0.17.22) and `drawRotlingRootEntangleEffect` already reads it in preference to the entity's raw world position. No anchor-ownership change was needed or made; the anchor was not the remaining bug.

## Rotling Root Render-Layer Fix Details
**Root cause**: `game.js` builds one combined `worldRenderables` list per frame containing entities, effects, tiles, etc., each tagged with a `depth` value, then paints them in ascending-depth order (so a higher depth value paints later, i.e. in front). Entities queue at `depthFast(x, y, elev) + 0.95`. The Rotling root effect queued at a single depth bias of `+0.86` - always numerically less than the entangled entity's own `+0.95` - so at the same world position, the entity's own sprite always sorted after (in front of) the root effect. The entire root VFX, including the ankle-wrapping tendrils meant to read as "you are rooted," was therefore always painted before and then fully covered by the character model. This is a straightforward off-by-direction bug in one bias constant, not a missing feature.

**Fix**: `game.js`'s effect-queueing loop now special-cases `rotlingRootEntangle` and pushes the *same* effect object as two separate render-queue entries with two different depth biases:
- `layer: 'base'` at bias `+0.86` (unchanged) - stays behind the entity, like a normal ground decal.
- `layer: 'wrap'` at bias `+1.02` - now numerically greater than the entity's `+0.95`, guaranteeing the tendril/ankle-wrap half paints after (in front of) the entangled entity's sprite every frame, regardless of other nearby effects/entities.

`makeRenderable()` was extended with a `layer` field (reset to `null` alongside its other pooled fields so stale values can't leak onto unrelated renderables across frames), and `drawEffect(e, layer)` now forwards `layer` to `drawRotlingRootEntangleEffect(e, layer)`, which draws only the ground decal when `layer === 'base'`, only the tendrils/motes when `layer === 'wrap'`, and both (the pre-fix behavior) if `layer` is omitted entirely - defensive behavior for any future direct caller, though the one real call site always supplies a layer. The WebGL "visible effect" hybrid-renderer batch path was checked and confirmed to never intercept this effect type (`rotlingRootEntangle` has no sprite-atlas entry, so `candidateForEffectRenderable` always returns null for it), so both new render-queue entries always fall through to this same Canvas 2D path - no hybrid-renderer changes were needed.

**Visibility polish**: while re-reading the tendril-drawing loop, the vertical reach was increased (from ~7-13px to ~11-19px above the ground anchor) and both the dark-strand and pale-green highlight strokes were brightened/thickened slightly, so that now that the tendrils are guaranteed to render in front of the lower legs, they read clearly as wrapping the ankles instead of being a thin, easy-to-miss line.

**Duration/cleanup**: unchanged. `syncRootEntangleVfx`/`retireRootEntangleVfx` (`systems/status-effect-system.js`) still own the effect's lifetime tied to the actual `root`-tagged status buff's `remaining` value, and still run for every actor (`updateGame` iterates `actorList(game)`), so bots/mercs/pets that get rooted get their own correctly-anchored, correctly-layered effect the same way the player does.

## Character Preview Background Root Cause
**Root cause**: three separate elements/rules independently painted the *same* class background image on top of each other, each adding its own semi-transparent dark-gradient overlay:
1. `.dollCenter::before` (outer, via `var(--party-class-bg)`, set on `#characterPanel` by `applyClassHudVisualStyle`)
2. `.dollCenter .paperDollPreview` / `.paperDollPreview.classModelPaperDoll` (middle, via `var(--pd-class-bg)`, set inline by `paperDollMarkup()`)
3. `.paperDollClassBackdrop` (innermost, directly behind the model canvas, also via `var(--pd-class-bg)`)

These are not competing/conflicting rules fighting over one property (that part of the cascade was already resolving correctly to a real class-art `url(...)` with a sane dark fallback) - they are three *different elements*, nested inside each other, each independently drawing the full class-background image with its own darkening gradient. Stacked on top of each other, three compounding semi-transparent dark passes over the same image desaturate and darken it far more than any single pass was tuned for. For the nature-toned classes (Druid especially, whose real background art - verified by opening `assets/ui/class-backgrounds/druid.png` directly - is a detailed dark-green forest/antler emblem, not a placeholder) this compounding reads as a flat, muddy, monochrome-leaning green wash rather than legible art, matching the reported bug.

**Class background resolver usage**: `DR.creatorClassBackgroundImagePath(className)` (`systems/ui-system.js`) - resolving to `assets/ui/class-backgrounds/{classKey}.png` via the shared `classEmblemMeta`/`classEmblemKey` class-name normalizer - was confirmed correct for all 14 classes (all 14 files exist on disk, one per class) and was **not** the bug; it was already the single resolver feeding all three layers.

**Fix**: consolidated to a single owning layer. `.paperDollClassBackdrop` (innermost, sitting directly behind the model canvas, already the most specifically-tuned of the three) is now the sole layer that paints `var(--pd-class-bg, ...)`. The other two were changed to plain neutral dark gradients (no class-art reference) so they still provide their vignette/depth framing without a second or third copy of the same image:
- `.dollCenter::before`: now a plain radial dark vignette.
- `.dollCenter .paperDollPreview` / `.paperDollPreview.classModelPaperDoll`: now a plain dark linear-gradient panel background (box-shadow unchanged).

The green fallback panel treatment is gone; the actual class background image now renders once, behind the model, at a single, intentional darkening pass. Verified for at least two classes by reading their real art assets (`fighter.png` - dark red/gold, `druid.png` - dark green/antler) and confirming the resolver path/selector chain is identical for both (class-agnostic - no hardcoded class). The existing dark fallback (`linear-gradient(180deg, #12111b, #050509)`) still applies via the surviving rule's `var(--pd-class-bg, ...)` default if a class's background image were ever missing.

**Not touched**: the player HUD's own class-background treatment (`.unifiedHudClassBackdrop`, `.partyClassBackdrop`, using the separate `--party-class-bg`/`--party-class-color` pair) was left exactly as-is - it is a single-layer treatment already and was not part of the duplicated stack.

## Character Stats Header Class Icon Sizing Changes
The Character Stats header's class emblem was rendered via `classEmblemMarkup(className, { size: 92, className: 'sheetClassEmblem largeSheetClassEmblem', ... })` (inline `width/height: 92px`), further forced to `92px !important` by a combined CSS rule `.sheetClassEmblem.largeSheetClassEmblem, .sheetClassEmblem { width/height: 92px !important; ... }`.

That combined selector was the wrong scope: `.sheetClassEmblem` alone (no `.largeSheetClassEmblem`) is also used by the bot-inspector debug panel (`systems/bot-player-system.js`, requesting its own inline `64px`), so bumping the shared rule would have resized that unrelated panel too. The rule was split:
- `.sheetClassEmblem.largeSheetClassEmblem` (the Character Stats header only) is now `132px !important` (up from 92px, a ~43% increase).
- `.sheetClassEmblem` alone is left at its existing `92px !important`, preserving the bot-inspector panel's current appearance exactly.

`systems/ui-system.js: renderCharacterPanel`'s `classEmblemMarkup(className, { size: 132, ... })` call was updated to match, so the inline size and the CSS `!important` size agree (no confusing mismatch between the two). `.statHeader.ornateStatHeader`'s grid was updated in step: its first column (`92px` -> `132px`) and `min-height` (`104px` -> `140px`) both grew to fit the larger icon without clipping or pushing the Gear Score column out of alignment. `align-items: center` (unchanged) keeps the icon vertically centered against the name/level text.

## Gear Score Alignment Changes
`.playerGearScoreBadge` was `text-align: right`, which right-aligned both the "GEAR SCORE" label and the big number flush to the box's inner edge - visually offset rather than centered under the label. Changed to a small flex column (`display:flex; flex-direction:column; align-items:center; justify-content:center;`) with `text-align: center`, so both the label and the number center themselves horizontally, and the column centers vertically within the box. `justify-self: end` (unchanged) keeps the whole badge anchored to the header's right side. No change to the Gear Score formula, its calculation, or the markup - only the badge's own internal alignment.
