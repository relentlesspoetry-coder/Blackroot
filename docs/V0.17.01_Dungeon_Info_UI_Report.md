# V0.17.01 - Dungeon Info UI Reposition + Visual Cleanup Report

## Phase source

The uploaded phase brief requested a dungeon information overlay cleanup and reposition pass. It named V0.16.97 -> V0.16.98, but the current baseline provided in this workspace was V0.17.00, so this implementation advances the project to V0.17.01.

## Ownership audited

- Dungeon information overlay ownership lives in `systems/dungeon-system.js`, specifically `renderDungeonLabels()`, which previously drew a rough fixed-position canvas rectangle at screen coordinates near the upper-left/upper-center.
- Minimap/HUD layout ownership lives in the HTML/CSS `#minimapWrap` HUD panel and the minimap renderer in `render/minimap-renderer.js`. The minimap itself remains unchanged.
- Dungeon state data remains owned by `systems/dungeon-system.js` through `this.state.active`, `game.activeDungeon`, `game.dungeonFloorMeta`, `game.dungeonNavigation`, and the existing Silk Web summary helpers.

## Old overlay replacement

The old rough canvas overlay block inside `renderDungeonLabels()` was removed from the live render path. Room world labels remain because they are separate world-space dungeon readability labels, not the rough HUD summary overlay.

## New panel placement

A new `#dungeonInfoHudPanel` element is inserted directly after the `#minimap` canvas inside `#minimapWrap`. This means the panel naturally anchors under the minimap and inherits minimap wrapper positioning, fullscreen behavior, HUD visibility, and responsive scaling. No duplicate minimap renderer or camera-space positioning was added.

## Visibility and update behavior

`DungeonSystem.updateDungeonHudPanel()` updates the card from the existing dungeon state. It hides the panel outside a dungeon or when no active dungeon state exists. It shows the panel while in a dungeon and updates:

- dungeon name
- floor count
- floor/sub-area name
- objective or mini-boss count
- current room name
- current room hint / safe transition message

The panel refresh is called from the existing dungeon update/render ownership and uses the existing 0.18s dungeon status tick where applicable.

## 1440p and responsive behavior

The panel is anchored to `#minimapWrap`, not fixed canvas coordinates. Since `#minimapWrap` already owns top-right placement, mobile/Steam Deck scaling, and minimap visibility, the panel follows those same rules at the 2560x1440 target and smaller responsive layouts.

## Protected systems

No changes were made to dungeon data, floor generation, transitions, boss/miniboss tracking, combat, loot, party behavior, save/load mechanics, map data, or minimap drawing.
