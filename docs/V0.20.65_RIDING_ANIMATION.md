# V0.20.65 — Riding Animation and Mount Facing Parity

Follow-up to Roadmap Item 7. Two defects in mounted play, both found by taking the request literally and
measuring rather than eyeballing.

## Bug 1 — the mount faced the wrong way on half the compass

`drawRiddenMount` passed the rider's raw `facingX` / `facingY` / `facingName` straight to the mob model.

But the two render paths resolve facing **differently**:

| path | function | applies camera yaw? |
|---|---|---|
| humanoid (the rider) | `entityFacingView()` | **yes** |
| mob model (the mount) | `facingProfile()` | **no** — reads the compass name |

Measured across all 8 headings:

| heading | rider view | mount view (old) | agreed? |
|---|---|---|---|
| east | side | side | ✅ |
| southeast | front | front | ✅ |
| **south** | **side** | **front** | ❌ |
| **southwest** | **side** | **front** | ❌ |
| west | side | side | ✅ |
| northwest | back | back | ✅ |
| **north** | **side** | **back** | ❌ |
| **northeast** | **side** | **back** | ❌ |

**Wrong on 4 of 8.** A rider heading *northeast* projects to a **side** view on screen, but the compass
name contains "north", so the beast drew its **back** while the rider showed its side. Any camera
rotation would have widened the gap, because only one of the two paths rotates.

**Fix:** derive the mount's facing from `entityFacingView(rider)` — the same camera-aware function the
rider's own model uses — then express it as a compass name the mob model parses to the identical view:

```
back  → north / northwest      front → south / southwest      side → east / west
```

Result: **8/8 headings agree**, and they agree *by construction*, because both now read the same source.

## Bug 2 — the rider ran on the spot

The walking animation kept playing in the saddle: legs striding as though jogging while the beast carried
the character.

`getStandingPuppetPose()` is the **single place** humanoid posture is derived, so the riding pose belongs
there — all 14 class models inherit it without 14 separate edits (and without a parallel "mounted
renderer", which Item 6's technical rules forbid):

- stride suppressed — `moving: false`, `legSwing: 0`
- `legSpread: 6.5` — knees out over the barrel of the animal
- `legLift: 4.5` — feet ride higher than they would on the ground
- `bob` driven by the mount's gait at the **same 8.5 rate** the quadruped models use, so rider and beast
  rise and fall together rather than drifting in and out of phase

`legSpread` and `legLift` are `0` for every non-riding pose, and both leg draws read them via
`Number(p.legSpread) || 0`, so **walking and idle geometry is byte-identical to before.**

## Verified

| check | result |
|---|---|
| mount/rider facing agreement | **8/8 headings** (old behaviour: 4/8) |

| suite | ok, 396/396, 0 console errors |

## Note

The mob models resolve to three views (front / back / side, with a horizontal flip), which is the *same*
scheme the humanoid models use via `entityFacingView` — so "the same directional renders as the player"
is now literally true: identical resolution function, identical view set, single source. The fix was
parity, not adding a fourth view the beast art does not have.
