# V0.20.98 — Art Direction Phase 16: Spell/Combat VFX

## The audit that had to come first

Before writing any Phase 16 code I checked who actually **read** the profile Phase 1 shipped:

```
grep -rn "artProfileValue|ART_PROFILE|renderProfile|applyArtDirectionProfile" --include=*.js .
```

One consumer: `applyCameraYawSnap` at [game.js:3546](game.js:3546). `applyArtDirectionProfile()` was
**never called**. The `ambient`, `terrain`, `shadows`, `lighting` and `vfx` blocks were read by
**nothing**.

Phase 1 shipped a file that was ~90% dead — the exact "computed in one place, consumed in another"
failure its own header says it exists to prevent. Recorded rather than quietly fixed, because the
pattern has now produced three separate faults in this project (V0.20.0 dead VFX styles, V0.20.93 dead
face shading, this).

## The change

Additive spell VFX — `lighter` and `screen` passes — draw brighter by `PROFILE.vfx.additiveBoost`
(1.35 nominal). Structural `source-over` drawing is deliberately untouched: boosting that washes out
silhouettes rather than glow, which is the opposite of the reference.

## Why a context intercept rather than editing the renderer

Canvas `globalAlpha` is **absolute, not multiplicative**. A wrapper that sets it and calls through is
overwritten by the first of ~200 branches in `drawEffect` and does nothing.

Editing 200 call sites is the cascade-patch the golden rules forbid, and every effect added later would
need the same edit. Shadowing the accessor for the duration of one call is **one place** that every
branch respects — including branches not yet written.

Verified *before* writing it, rather than assumed:

| property | result |
|---|---|
| shadow intercepts assignment | 0.8 → 0.4 at scale 0.5 ✅ |
| `save`/`restore` sane through the shadow | ✅ |
| `delete` restores native accessor | ✅ |
| cost | 200k assignments: **4.3ms native vs 14.6ms shadowed** ≈ 0.05µs each |

Restoration is in a `finally`. A leaked shadow would boost the entire rest of the frame — terrain,
entities, UI.

**Install point:** a separate `installVfxPass(Game)` called after `EffectsRenderer.install` at
[game.js:4594](game.js:4594), *not* from the systems block at 4553 — `drawEffect` does not exist yet at
4553 and the wrapper would have been silently overwritten. The same class of fault this version opens
by reporting.

## Two measurement methods I threw away

| method | why discarded |
|---|---|
| whole-scene pixel diff, one baseline | repeat of arm 1 drifted **20%** — same magnitude as the claimed effect |
| per-arm baseline diff | within-arm spread still **188%** (values swung 9,270–31,300 inside one arm) |

Both failed for the same reason: these renderers animate off `this.time`, so no two frames match, and
the noise swamps the signal. The drift check is what caught it — an uncontrolled version of this would
have reported "21.4% brighter" and been wrong.

Switched to the deterministic op-recording method from V0.20.0: hand the real `drawEffect` an
instrumented context and record the alpha actually in force at each drawing operation.

## Result

| type | additive ops | boost | source-over |
|---|---|---|---|
| wizardElement | 12 | **1.21×** | unchanged |
| bolt | 5 | **1.18×** | unchanged |
| casterMotif | 17 | **1.10×** | unchanged |
| ring | 1 | **1.00× — no gain** | unchanged |

Effective gain sits below the nominal 1.35 because clamping to 1 absorbs the remainder.

### Stated plainly

**`ring` gains nothing.** It emits a single additive op already at full alpha, and with 99 `spawnRing`
call sites it is likely the most common effect in the game. This is **not** a uniform brightening and
must not be described as one.

My first deterministic test used `ring` alone and returned a boost ratio of exactly 1.00 — the change
looked completely dead. Sampling the wider population showed **71.4% of additive ops do have headroom**
(median alpha 0.343). `ring` was the single worst test case available.

## Controls

| control | result |
|---|---|
| source-over alpha, boosted vs not | **byte-identical** in every arm |
| additive alpha ceiling | never exceeds 1 |
| `lighting.boostExempt` arm (coreBloom/beamRoot/flameCore) | **byte-identical** — V0.20.89 clip-to-white protected |
| `uiPrefs.hdVfx = false` | restores original behaviour exactly |
| context after the call | **zero own properties**, native alpha restored |
| repeat runs | deterministic, identical sums |
| perf, 120 simultaneous effects | **+1.1ms (1.8%)**, arms overlapping (off 60.4–63.2, on 60.9–65.0) |
