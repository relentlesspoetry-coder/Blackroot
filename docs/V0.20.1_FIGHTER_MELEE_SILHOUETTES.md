# V0.20.1 ‚Äî Ten Swings, Ten Silhouettes (Roadmap Item 6)

V0.20.0 fixed the VFX styles the game **asked for** (`slam`/`crush`/`cast` ‚Äî passed and never drawn).
This fixes the ones it **never thought to ask for**.

## The gap

`systems/spell-system.js:1016` hardcoded:

```js
id === 'titan_chop' || id === 'perfect_masterstroke' ? 'slam' : 'slash'
```

So **eight** other single-target Fighter melee spells fell through to the same generic sweep:

| spell | what it does | what it drew |
|---|---|---|
| Crushing Blow | applies Crushed Armor | level-1 sweep |
| Reckless Strike | self-debuffs the caster | level-1 sweep |
| Heavy Riposte | a **counter** off a dodge/parry | level-1 sweep |
| Armor Splitter | applies Split Guard | level-1 sweep |
| Bonebreaker | a snap that slows | level-1 sweep |
| Final Swing | an **execute**, gated below 30% HP | level-1 sweep |
| Shatter Guard | breaks a shield | level-1 sweep |
| Titan Chop | (already had `slam` via the id check) | slam |

## This one was different ‚Äî and worth saying plainly

**No validator could have caught this.** Nothing was claiming otherwise. No field was inert. Nothing was
malformed. The spells did exactly what they said they did.

Every previous find in this thread was falsifiable ‚Äî `category: "AoE Ranged / Cone"` vs a 360¬∞ hit,
`coneDegrees` read for one spell in 280, `'slam'` passed and never drawn. A machine can check those.

This was a **design gap**: ten abilities that were all honest and all looked identical. The only way to
find it was to read the spells and ask what each should look like. That is the long-tail work flagged at
the end of V0.20.0, and it is not automatable.

## The fix ‚Äî data-driven, same shape as `coneDegrees`

A per-spell **`vfxStyle`** authored in `data/spells.js`, beside the damage and the cone. The resolver
reads `spell.vfxStyle || 'slash'` instead of naming two spell ids.

Six new silhouettes, each chosen to communicate what the spell **does** ‚Äî not to look different for its
own sake:

| style | reads as | why |
|---|---|---|
| `split` | vertical rend, halves parting | armour being **opened** |
| `thrust` | straight counter-stab + tip flash | a riposte is **not a swing** |
| `wild` | two crossing off-balance arcs | it debuffs **you**; it shouldn't look controlled |
| `execute` | decisive downward arc + glint | a **decision**, not another hit |
| `shatter` | shards from the block point, no sweep | the shield **gave way** |
| `break` | hard zigzag crack | a **snap**, not an edge |

**Crushing Blow reuses `'crush'`** (V0.20.0's blunt pound) ‚Äî semantically exact, no new code.

**Heavy Swing authors nothing** and stays the baseline sweep. It *is* the plain swing; a bespoke look
there would be decoration pretending to be information.

## Two checks, each at the layer where it's knowable

**(1) Runtime ‚Äî because authored styles are DATA.** `validateSpellShapes()` rejects any `vfxStyle` the
renderer can't draw, checked against `DR.SLASH_STYLES`.

**This one can actually fail**, unlike the `validateVfxStyles()` written and deleted in V0.20.0: the
registry is *declared* in `render/effects-renderer.js` beside the branches, and *requested* by
`data/spells.js` ‚Äî **two independent sources**, not a list compared against itself.

**(2) Static ‚Äî because the registry is hand-written.** `tools/check-vfx-styles.js` now asserts
`DR.SLASH_STYLES` matches the real `e.style === '...'` branches **in both directions**, so the registry
can't silently become the next piece of dead data.

That checker's extraction was **also too broad** and had to be scoped: it scanned the whole renderer and
flagged `'arrow'` as a stale entry ‚Äî but `'arrow'` belongs to `drawHighQualityBoltEffect` and has nothing
to do with `spawnSlash`. It now reads only the `e.type === 'slash'` block.

| check | result |

| registry drift ‚Äî an undrawn registry entry | **exit 1** |

The drift test needed doing twice: the first version read **`head`'s** exit code through a pipe rather
than **node's**, and falsely reported `0` for both failure cases. The check was right; the test was
lying.

## Item 6 status

**The Fighter is done** ‚Äî cone, styles, and per-spell silhouettes all honest, and the mechanism is now
data-driven, so the remaining classes are authoring work rather than engineering.

Still open:
- **6 classes unaudited** (Assassin had its pass in V0.18.29; Fighter is now done). Each is the same
  shape of work: read the spells, author `vfxStyle`.
- The Fighter's `buff`-kind spells (Momentum, Weapon Flow, Bloodrush, Berserker's Roar, Unstoppable
  Footwork, Savage Momentum) share `applyFighterSelfBuff` and are not covered by `vfxStyle` ‚Äî they'd
  need the same treatment on the self-buff path.
- `spell_summoner_mass_dismissal` declares `radius: 4` on a single-target spell.
- The sprite-cache backend (off by default) doesn't implement the new styles ‚Äî it's a simplified LOD
  path, consistent with how it already approximates other effects.
