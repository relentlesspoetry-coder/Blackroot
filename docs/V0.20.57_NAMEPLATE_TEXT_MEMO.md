# V0.20.57 — Nameplate Text Measurement Memo

Small, real, and deliberately not oversold.

## The finding

Profiling inside `drawEntity`: **`drawNameplate` cost ~2.7ms/frame** across ~20 plates — *despite* the
V0.17.96 nameplate image cache working correctly at a **91% hit rate** (16.4 hits vs 1.6 misses/frame).

A cached blit should be nearly free. It wasn't, because **the cache is checked too late**.

By the time it is consulted, the layout block above has already run, per plate:

- 3 × `ctx.measureText()` (name, level, action text)
- 2 × `ctx.font` assignment

and the `ctx.restore()` immediately below **discards that font state**. On a cache hit every one of those
calls produced nothing — about **66 `measureText` calls per frame** of pure waste.

## The fix

A module-level memo in `render/entity-renderer.js`, keyed by `(font, string)` — a text width is a pure
function of exactly those two — touching `ctx.font` only on an actual miss.

Verified the drawing path still sets its own fonts (`12px`/`10px`/`bold 7px` assignments remain further
down); only the measurement-phase assignments were removed, and `ctx.restore()` already guaranteed
nothing downstream depended on them.

## Measured

| metric | before | after |
|---|---|---|
| `measureText` calls/frame | ~66 | **14** |
| `drawNameplate` ms/frame | 2.7 | **0.38** |

## What I could not show — stated plainly

**Total render time did not visibly move: 43.5ms vs a 41.4ms baseline.** ~2ms sits inside this preview's
noise band, and `update()` drifted 12.9 → 32.2ms during the same run — the same environment instability
documented in V0.20.55.

The function-level win is solid and repeatable. **The frame-level win is not demonstrable here, so it is
not being claimed.** This is ~2ms off a ~42ms frame. It is not what takes 9 FPS to 60.

## Ruled out this pass — recorded so none of it gets re-investigated

| hypothesis | result |
|---|---|
| Runtime sprite cache is the lever | ~~No~~ **THIS ROW IS WRONG - CORRECTED IN V0.20.58.** The test drove `update()+render()` directly, which never calls `runtimeSpriteCache.beginFrame()` (only `game.js loop(now)` does), so `missesThisFrame` never reset and the cache sat permanently in budget-skip mode drawing everything live. Both arms were measuring the same uncached path. The cache **is** a lever - see `V0.20.58`. |
| A per-frame leak is degrading FPS | **No.** Entities stable at 435, cache entries stable at 96, across 60 frames. The earlier "degradation" was caused by my own repeated config changes mid-measurement. |
| Canvas misconfigured / software-forced | **No.** Created with `{alpha:false}`; `willReadFrequently` is not set anywhere (it would force software rendering). |
| Update loop is the real cost | **No.** Render 41.4ms vs update 12.9ms — render is the right place to work. |
| Nameplate image cache is thrashing | **No.** 91% hit rate. The waste was *before* the cache check, not in it. |

## Where this leaves the frame

From the same profile, after the V0.20.56 NPC cull:

| phase | ms | calls |
|---|---|---|
| `drawEntity` (via `drawCachedModelOrFallback` → `drawClassIdentityModel`) | ~10 | 17–22 |
| `drawObject` | ~7 | 151 |
| `npcRuntime` (post-cull, away from camp) | ~4 | — |
| `drawTile` | ~3.3 | 58 |
| `drawNameplate` | **0.38** | ~20 |

The remaining bulk is **procedural character drawing** (~0.4ms per humanoid × ~15) and **151 world
objects**. Neither has a cheap fix left: the sprite cache is already ruled out for characters, and the
V0.20.53 atlas attempt made things worse for unrelated packing reasons.

**The untested lever is fill rate**, which this software-rendering preview is structurally incapable of
measuring (per V0.17.95's own note). `maxRenderPixels` is 5.0M by default. On a high-DPI display that is
a large per-frame pixel bill that would not show up in any measurement taken here.
