# V0.16.96 - Class Buff Spell Icon Linking Pass Report

## Phase source

- Phase brief: user-supplied "Class Buff Spell Icon Linking Pass" spec, including the exact root-cause diagnosis, recommended `DR.SpellIcons`/`decorateStatusEffectFromSpell` design, and a validation list of ~150 class buffs/debuffs across all 14 classes.
- Baseline used: `Blackroot V0.16.95`
- Target produced: `Blackroot V0.16.96`

## Root cause (confirmed by audit)

The action bar/spellbook already resolved real spell icons via `Game.prototype.spellIconDescriptor(spell)` / `spellIconHtml(spell)` in `systems/spell-system.js`, including bound imported PNGs (`CLASS_SPELL_ICON_PATHS`, one map per class) and Necromancer's themed-icon fallback. The active buff/debuff tray, however, rendered exclusively through `DR.StatusEffects.iconHtmlFor(effect)` in `systems/status-effect-system.js`, which only ever called `statusTheme(effect)` - a generic keyword/tag-matcher that returns one of a handful of glyphs (`▲ ▾ ☠ ✦ ♫` etc.). Status effects created from class spells never carried a reference back to their source spell, so the tray had no way to ask for the real icon. This matches the brief's diagnosis exactly.

## How spell icons are propagated to status effects

1. **Shared descriptor** - `DR.SpellIcons.descriptorForSpell(spell, fallbackClassName)` was added at the top level of `systems/spell-system.js` (same closure as the pre-existing `resolveBoundSpellImagePath`/`themedNecromancerIcon`/`CLASS_SPELL_ICON_PATHS`, so nothing was duplicated - just exposed). `Game.prototype.spellIconDescriptor` is now a 2-line delegate to it, so action bar/spellbook rendering is byte-for-byte unchanged.
2. **Decoration helper** - `Game.prototype.resolveClassSpellId(spell)` and `Game.prototype.decorateStatusEffectFromSpell(spell, effect)` were added in `systems/spell-system.js`, exactly matching the brief's recommended shape. `decorateStatusEffectFromSpell` calls `spellIconDescriptor` once and stamps `sourceSpellId`, `sourceSpellName`, `sourceClass`, and a serialized `spellIconDescriptor` onto the effect object (without overwriting any of those fields if a call site had already set them explicitly - several Necromancer/Assassin effects already set `sourceSpellId`/`sourceClass` by hand, and those values are preserved).
3. **Call sites** - every one of the **105** `applyStatusEffect` calls in `systems/spell-system.js` (across all 14 class resolver functions, `resolveXRevampSpell(spell, options)`) now wraps its effect argument with `this.decorateStatusEffectFromSpell(spell, ...)`. This was applied mechanically with a verified source-transform script (paren/brace/template-literal-aware, not a blind regex) rather than 105 individual hand edits, to guarantee no call site was missed or malformed; every transformed line was validated with `node --check` and spot-checked against the original source.
4. **Two call sites had no `spell` in their own local scope** and needed small supporting fixes rather than a blind identifier swap (which would have thrown `ReferenceError` at runtime - not caught by `node --check`):
   - **Fighter Momentum** (`Game.prototype.addFighterMomentum`) creates the `fighter_momentum` status from several different weapon abilities, so it never received a single `spell`. It now looks up the real "Momentum" spell via a new small helper, `Game.prototype.findClassSpellById(className, wantedId)` (searches the class's own compiled spellbook by resolved id - not a hardcoded map), and decorates with that.
   - **Queued Wizard/Shaman impact spells** (`Game.prototype.updatePendingSpellImpacts`, used for Meteorfall/Archwizard's Cataclysm/Ancestral Cataclysm's delayed AoE) apply their DoT/debuff after a delay, by which point the original `spell` parameter was out of scope. `Game.prototype.queuePendingSpellImpact` payloads now also carry `spell`, and the delayed handler decorates with `impact.spell`.

## Which status renderer now owns buff/debuff icon output

`systems/status-effect-system.js` remains the sole owner of the tray/tooltip renderer - no second buff/debuff system was created.

- `normalizeEffect()` now also preserves `sourceSpellName` and a normalized `spellIconDescriptor` (in addition to the `sourceClass`/`sourceSpellId` fields it already preserved).
- A new local helper, `normalizeSpellIconDescriptor(icon)`, sanitizes whatever descriptor shape arrives (matching the brief's spec exactly).
- A new local helper, `spellIconDescriptorForStatus(effect)`, prefers the effect's own `spellIconDescriptor` (the fast path, set by `decorateStatusEffectFromSpell` at cast time) and only falls back to searching `DR.CLASS_SPELL_BOOK` by `sourceSpellId`/`sourceSpellName`/`sourceClass` when that metadata is missing (e.g. an older save's status object) - this fallback calls `DR.SpellIcons.descriptorForSpell`, it does not re-implement icon resolution.
- `iconHtmlFor(effect)` now resolves `spellIconDescriptorForStatus(effect) || statusTheme(effect)` and renders an `<img class="spellIconImageAsset">` inside the existing `.statusIcon` element when an `imagePath` is present, otherwise renders the original glyph markup unchanged.
- `tooltipHtmlFor(effect)` already built its icon by calling `iconHtmlFor(effect)` and swapping the CSS class (`statusIcon` -> `statusTooltipIcon`), so it automatically inherited the fix with no separate logic; its two card-color CSS variables were switched from `statusTheme(effect).color` to a new `statusCardColor(effect)` helper so the tooltip border color also matches a resolved spell icon's color when one exists.
- The circular timer ring, stack count, and time label (`.statusPip`, `.statusTime`) were not touched at all - they read `effect.remaining`/`effect.duration`/`effect.stacks`/`effect.color` exactly as before.

## CSS

`.spellImageIcon .spellIconImageAsset` (absolute-fill, `object-fit: cover`, `z-index: 1`) already existed generically in the main HTML and required no change. One rule was added because it did not already exist outside the hotbar's `.actionSlot`-scoped version: `.statusIcon.spellImageIcon .iconCore/.iconGlyph` and `.statusTooltipIcon.spellImageIcon .iconCore/.iconGlyph` are now hidden, matching the existing hotbar behavior, so the (usually empty, but not always) fallback glyph text never renders on top of a resolved spell image.

## Validation performed

- `node --check` on every `.js` file and `python3 -m json.tool` on every `.json` file (incl. `_script_order.json`) - zero errors.
- A standalone Node harness (loading `data/spells.js` + `systems/spell-system.js` + `systems/status-effect-system.js` with a minimal `window` stub, no browser needed since icon resolution is pure data logic) exercised the full pipeline end-to-end for 9 of the brief's validation cases (Fighter Momentum, Fighter Battle Rush, Paladin Shield of Faith, Bard Song of Mending, Druid Rejuvenating Vine, Necromancer Bone Armor, Necromancer Grave Rot, Ranger Hunter's Mark, Enchanter Rune of Focus): every one resolved its real spell PNG through `DR.SpellIcons.descriptorForSpell`, survived `normalizeEffect`, and rendered `spellImageIcon` markup with the correct `<img src>` via `iconHtmlFor`. The fallback name/class search path (no `spellIconDescriptor` present) was also verified to resolve correctly, and a plain non-spell environmental status effect was verified to still render via the generic `statusTheme` path without throwing.
- Traced two of the brief's "debuff should use a different-named spell's icon" cases directly in source: `Crushed Armor` (`fighter_crushed_armor`) is created inside the `id === 'crushing_blow'` branch and decorated with that Crushing Blow `spell`, and `Grave Rot`'s DoT is applied via `Game.prototype.applyNecromancerDot(target, spell, ...)`, which already decorates with whatever `spell` it's given - both confirmed to end up carrying the correct source spell identity.
- Because this project has no automated browser test runner, in-browser rendering (image actually painting inside the circular tray pip) was not visually captured this pass - see Manual Tests Needed below.

## Assumptions made

- Where a single spell cast produces a secondary, differently-named side-effect through a local `applyBuff(target, data)` closure (e.g. Paladin's `Turn the Fallen` also applying a separate `Ward Against the Fallen` buff), the decoration uses the *casting* spell's icon for all of that cast's side-effects, since the closures capture only the outer `spell`. This is a minor fidelity tradeoff, not a crash risk (the identifier is always in scope), and matches the same "wrap with `spell`" pattern the brief itself specifies.
- Fighter Momentum's resolved spell falls back to a small inline `{ name: 'Momentum', className: 'Fighter', ... }` object if `findClassSpellById` can't find a compiled "momentum" spell (e.g. a heavily customized editor spellbook) - in that case the icon falls back to the generic buff glyph rather than crashing.
- Scope was kept to `systems/spell-system.js`/`systems/status-effect-system.js` per the brief; `systems/spell-compiler-system.js` did not need changes (spell compilation was already producing the same spell objects `spellIconDescriptor` already consumed).
