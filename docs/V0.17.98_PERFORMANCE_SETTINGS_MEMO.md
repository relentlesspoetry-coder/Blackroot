# V0.17.98 — Performance: memoized `performanceSettings()` (the big one)

The largest single frame-rate win of the whole performance effort, from a one-function
fix. Engine-wide, not just AI.

## How it was found

Profiling the enemy-AI `update` loop (~90 ms/frame, the dominant frame cost) attributed
**~85 % of it to one trivial function**: `Enemy.aiUpdateBand` (a distance → near/mid/far/
sleep band check) measured at **~0.19 ms/call × ~460 calls = 88 ms/frame**. A band check
should be nanoseconds.

## Root cause — a hidden allocation storm

`aiUpdateBand` calls `game.performanceSettings()` **twice per call** (~920×/frame), and
`performanceSettings()` was not memoized:

- `performanceVerifier.effectiveBaseSettings(base)` does `Object.assign({}, base)` — a
  **fresh object every call**.
- That fresh object is then passed to `adaptivePerformance.effectiveSettings(base)`, whose
  cache is keyed on **object identity** (`this.settingsCacheBase === base`). A new `base`
  every call ⇒ the adaptive cache **never hits** ⇒ it re-allocates and re-derives its
  settings object too.

So each `performanceSettings()` call allocated 2–3 objects, and AI banding alone triggered
~2500 allocations/frame — a GC/CPU storm that *looked* like "AI cost" but was really
`Object.assign` + garbage collection. `performanceSettings()` has ~47 call-sites
(render cull pads, DPR, expensive-effect gating, UI cadence, AI), all paying this.

## The fix

`game.js` `performanceSettings()` is now memoized. It rebuilds only when the inputs that
actually affect the result change — `performanceVerifier.preset`, `lowSpecMode`,
`adaptivePerformance.level`, and adaptive `enabled` — via **4 scalar comparisons** on the
hit path; otherwise it returns the cached object.

Safe because the result is **read-only** at every call-site. (The `perf.x = …` writes in
the QA-telemetry systems are on `game.perfStats`, a different object — verified.)

```js
performanceSettings() {
  const verifier = this.performanceVerifier, adaptive = this.adaptivePerformance;
  const level = adaptive ? adaptive.level : -1;
  const preset = verifier ? verifier.preset : '';
  const lowSpec = verifier && verifier.lowSpecMode ? 1 : 0;
  const enabled = adaptive ? (adaptive.enabled !== false) : true;
  const c = this._perfSettingsCache;
  if (c && c.level === level && c.preset === preset && c.lowSpec === lowSpec && c.enabled === enabled) return c.value;
  let base = DR.CONFIG?.PERFORMANCE || {};
  if (verifier?.effectiveBaseSettings) base = verifier.effectiveBaseSettings(base);
  const value = adaptive?.effectiveSettings ? adaptive.effectiveSettings(base) : base;
  this._perfSettingsCache = { value, level, preset, lowSpec, enabled };
  return value;
}
```

## Measured (in-browser, software sandbox, 345 mobs)

| metric | before | after |
|---|---|---|
| `Enemy.aiUpdateBand` | 88 ms | **0.18 ms** |
| `Enemy.update` (all mobs) | 106 ms | **1.35 ms** |
| `update` (whole sim) | 82 ms | **17 ms** |
| `render` | ~23–48 ms | **~5 ms** |
| frame | ~114 ms | **~24 ms** (~4.7×) |

(~8 → ~40 FPS in this software sandbox; larger on real GPU+CPU where render is cheap.)
The remaining ~17 ms of `update` is legitimate AI work (patrol/target-pick/pathfinding),
dt-amplified at low FPS and smaller at 60.

## Verification

- Memo returns a **stable object** across calls within a settings state.
- **Invalidates correctly** on adaptive level change (renderScale 1 / 0.85 / 0.72 for
  levels 2 / 1 / 0), distinct objects per level.
- The V0.17.95 render-resolution slider still works; `displayPixelRatio`/mode correct.
- 0 console errors.

## Note

This memoization is the reason the earlier "render is a sandbox artifact / the engine is
well-optimized" reads were partly misleading: a huge chunk of *both* update and render was
this allocation churn, which is real on any hardware. Fixing it helps everywhere.
