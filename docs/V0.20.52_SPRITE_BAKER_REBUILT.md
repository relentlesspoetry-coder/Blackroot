# V0.20.52 — The Sprite Baker Is Back (the real cause of the frame rate)

Chasing the 60 FPS goal into the render layer turned up the actual cause. It is not a tuning problem.

## The finding

Blackroot was **designed to blit pre-baked sprites**. All the machinery exists and works:

- `SpriteAtlasSystem` — the loader
- `tryDrawEntityAtlasFrame` — the render hook
- `systems/sprite-bake-system.js` — the bake engine

But `assets/atlases/entities.json` is still the development **placeholder** (`"pages": []`,
`"frames": {}`) — because `tools/sprite-baker.html` and `tools/sprite-baker.js`, the exact tool that
`assets/atlases/README.txt` instructs you to use, **had gone missing from the repo**.

With no atlas, `isReady()` is permanently false, **100% of fallbacks report `atlas_not_ready`**, and
every actor and prop falls back to procedural vector drawing:

| per frame | measured |
|---|---|
| canvas path ops | **~10,000** |
| strokes | ~4,300 |
| `drawImage` | **~92** |

A build step was simply never run. That is the 12 FPS.

## What this version does

Rebuilds the missing tool as `tools/sprite-baker.html` + `tools/sprite-baker.js`.

It fetches the load order from `_script_order.json` and injects the renderer scripts, deliberately
**skipping `game.js`** — whose last line is `window.DarkWoodsGame = new Game()` and would boot the entire
HUD — because the bake engine builds its own minimal render context via `createObjectBakeGame()`.

It offers a one-model smoke bake (with thumbnails) before committing, then a full bake that downloads the
atlas pages plus `entities.json`.

## Verified end-to-end in-browser

| check | result |
|---|---|
| script load | **135 scripts clean, no game booted** ✅ |
| bake definitions found | **87** |
| Paladin smoke bake | **328 frames in 923ms** @176×192, 24 thumbnails ✅ |
| packing | real **4096×4096** page with actual pixels ✅ |
| manifest metadata | `x/y/w/h` + `anchor`, **`footAnchor`**, `nameplateAnchor`, `shadowAnchor`, `clickBounds` ✅ |

The baked `footAnchor` is the same foot line the **V0.20.51 snare-root fix** depends on, so that
behaviour carries through the atlas rather than regressing.

## What I could not do

The full bake saves **tens of megabytes** of PNG pages through the browser's download mechanism, which
cannot be driven from this environment. That final step is a local run.

### How to run it

1. From the project root: `python3 -m http.server 8899`
2. Open `http://localhost:8899/tools/sprite-baker.html`
3. **Load Renderers** → *(optional)* **Bake One Model** → **FULL BAKE & DOWNLOAD**
   (allow *automatic multiple downloads* for the site if only the first file saves)
4. Copy every `entities_*.png` **and** `entities.json` into `assets/atlases/`, replacing the placeholder.
5. Reload and check `DarkWoodsGame.spriteAtlasSystem.getStats()` — `framesLoaded` should be non-zero and
   `proceduralFallbacks` should stop climbing.

## Also established

The **adaptive performance governor is not broken**. Fed 83ms frames (12 FPS) it correctly downshifts
balanced → performance → rescue, reaching **render scale 0.72 within 2.5s**.

That matters: it means the pixel-reduction lever is *already being pulled automatically* and is
effectively spent. **Draw-call reduction is the only remaining lever** — which is exactly what the atlas
delivers, converting ~140 vector ops per actor into a single blit.
