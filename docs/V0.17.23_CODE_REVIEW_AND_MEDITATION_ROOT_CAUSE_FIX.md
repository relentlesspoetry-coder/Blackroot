## Codebase Review Findings

### System Ownership Problems
- **Meditation itself is a healthy example, not a problem.** A single module (`systems/meditation-system.js`) owns balance/config/math (`calculateMeditationRegen`, the Level 1-20 chart, environment/bard/group bonuses, Great Stillness), while `game.js` owns the actual state machine (`startMeditation`/`cancelMeditation`/`updateMeditationForEntity`/`actorIsMeditating`). Every caller - player (`game.js` main loop), bots (`entities/bot-player.js`, `systems/bot-player-system.js`), and mercenaries (`entities/mercenary.js`) - routes through the same `updateMeditationForEntity`/`applyMeditationRecoveryTick` entry points. No duplicate regen loop was found. Risk: none. Fixed: n/a.
- **Performance/release-gate ownership is foggy.** `systems/performance-verification-system.js`, `performance-settings-cleanup-system.js`, `performance-stable-release-lock-system.js`, `final-performance-qa-patch-system.js`, `stable-release-candidate-system.js`, `post-release-candidate-hardening-system.js`, `release-manifest-checklist-system.js`, `final-ship-cleanup-system.js`, and `launch-validation-package-audit-system.js` all install onto `Game.prototype` with overlapping "is this build ready" responsibilities and near-identical export shapes (`{version, build}` payloads, `dream-realms-v*-report.json` downloads). It is not obvious from naming alone which one is authoritative if two ever disagree. Risk: medium. Fixed: no. Future phase: a dedicated ownership consolidation/audit pass for the performance/release-gate system family.

### Duplicate / Patch-Cascade Code
- **Two independently hand-duplicated patch-notes HTML blobs.** `Blackroot V0.17.23.html` contains a full version-history block twice - once inside `#splashPatchNotesPanel` (main menu) and again inside `#menuPatchNotesPanel` (in-game escape menu) - each maintained by hand instead of one panel rendering from a single source. Both copies were already stale before this phase (last entry present was V0.17.18, four versions behind the V0.17.22 baseline), confirming they already drift independently. Risk: medium (informational only, no gameplay impact, but guarantees future staleness). Future phase: replace both with one panel that renders from `PATCH_NOTES.md` (or a shared JS array) at runtime.
- **`<title>` tag was 5 versions stale** (read `V0.17.18` while `core/config.js` correctly said `V0.17.22`), confirming in-game version metadata is not consistently touched on every release. Risk: low/medium (cosmetic, but a repeating drift pattern). Fixed: yes, corrected as part of this phase's required version-metadata step (title now matches the new version); the underlying "some surfaces get missed" process risk is still deferred for a future "single source of truth for version strings" pass.
- **The Level 1-20 Meditation chart's `hpPerSec`/`resourcePerSec` field names are misleading**, not duplicated code but a naming/documentation defect that caused this phase's core bug (see Root Causes below). Left the field names as-is (renaming would have rippled into bard-aura math, bot desire scoring, and save code for no behavior change) but corrected the surrounding comments so the actual per-tick semantics are documented at the source. - **`Game.prototype.meditationRestoreAmount`** (`game.js`) has zero callers anywhere in the project (verified via full-project grep). Risk: low. Fixed: no (dead-code removal was not required to fix Meditation and is out of this phase's scope; flagged as a cleanup candidate).

### Timing / Update Loop Risks
- **Root cause of this phase's bug**: `MEDITATION_TUNING.tickIntervalSeconds` was `1.0` (not the documented `15.0` from `docs/PASS_47_1_DARK_WOODS_MEDITATION_QOL.txt`), and `game.js`'s `applyMeditationRecoveryTick` multiplied the already-final per-tick chart amount by the tick length again. Together this made Meditation apply a full "per-tick" restore roughly every 1 real second instead of every 15, i.e. ~15x too fast. See Root Causes below for the full chain. Fixed: yes, this phase.
- The main `loop(now)` / `update(dt)` cadence itself (frame-driven, `requestAnimationFrame`, `dt` clamped by `CONFIG.MAX_DT`) is sound and was not changed. `updateMeditationForEntity`'s own `safeDt = Math.min(0.25, dt)` per-frame accumulation cap and 4-tick catch-up ceiling (for tab-backgrounding recovery) were preserved unchanged.
- A full per-system audit of every other system's own dt/tick handling (80+ files in `systems/`) was not performed this phase - out of scope. Future phase: a dedicated timing/update-loop audit across non-Meditation systems if similar symptoms are reported elsewhere.

### Skill / EXP System Risks
- **Meditation skill EXP was fully disconnected from the tick loop.** `systems/skills-system.js` already exposed a correct, isolated `game.gainMeditationXp(amount)` -> `game.grantMeditationXp(entity, amount)` path that only ever mutates `entity.meditation.xp/level` and the mirrored `entity.meditationSkill` - never `entity.xp`. But nothing in `game.js`'s tick loop ever called it (confirmed via full-project grep for `grantMeditationXp`/`gainMeditationXp` call sites before this fix). Risk: was high (a whole skill silently did nothing). Fixed: yes, this phase, gated to the player only and to one call per completed tick.
- **`systems/skills-system.js`'s `gainMeditationXp(amount)` has a latent `amount || 1` fallback** that silently floors a `0` argument up to `1` XP instead of granting zero. It does not affect current behavior (this phase's `xpPerTick` constant is always `5`, never `0`), but it is a footgun for any future change that intentionally wants a zero-XP tick (e.g., an environment with a `0` XP multiplier). Risk: low. Fixed: no. Future phase: change the fallback to `Math.max(0, Math.floor(Number(amount) || 0))`.
- **`MEDITATION_BALANCE.safeZoneXpMultiplier`, `greatStillnessXpMultiplier`, and each `MEDITATION_ENVIRONMENTS[x].xpMultiplier`** are defined config fields that remain unconsumed even after this fix. This phase intentionally restored Meditation skill XP as a flat, conservative, documented rate (see Root Causes) rather than guessing how these dormant multipliers were meant to combine with it, to keep the fix surgical. Risk: medium (incomplete feature, not a regression). Fixed: no. Future phase: "Meditation XP bonus wiring" to decide and implement how these multipliers should scale skill XP.
- The player-level-EXP safeguard in `updateMeditationForEntity` (snapshot `entity.xp` before the tick loop, restore it if anything mutated it, log once) is a defensive "catch the symptom" pattern rather than a structural guarantee that nothing can ever write to `entity.xp` during Meditation. It currently works correctly and was left untouched (rewriting it for no behavior change would be scope creep and a regression risk). Risk: low. Fixed: no.

## Meditation Root Causes

### Healing Too Fast Root Cause
`game.js`'s `applyMeditationRecoveryTick` computed `amount = data.hpPerSec * safeTick`, where `safeTick` was the tick length in seconds (clamped 0.05-5) and `data.hpPerSec` came from the Level 1-20 chart in `systems/meditation-system.js`. Those chart values (e.g. `2` at level 1, `37` at level 20) were themselves already the intended *per-tick* restore amount (matching `docs/PASS_47_1_DARK_WOODS_MEDITATION_QOL.txt`'s "Level 1 HP/Mana restore: +2"), not a per-real-second rate - but the V0.16.94 revamp renamed/reused them as `hpPerSec` and started multiplying by elapsed tick seconds, silently double-applying the restore amount every tick.

### Missing 15-Second Tick Root Cause
`MEDITATION_TUNING.tickIntervalSeconds` was `1.0`, and `game.js`'s `meditationTickRate()` carried a comment claiming this only drove "the cosmetic aura pulse cadence" and that "the actual regen math is per-second." Combined with the multiplication bug above, Meditation both ticked 15x too often and restored a full tick's worth of HP/resource on each of those over-frequent ticks - a compounding, not additive, timing bug.

### Missing Meditation Skill EXP Root Cause
V0.17.21 correctly fixed a real bug (Meditation was leaking into player-level `entity.xp`) by having `calculateMeditationRegen` always return `xpPerTick = 0`. But `xpPerTick` was the same signal meant to feed the *separate, legitimate* Meditation skill EXP path (`game.gainMeditationXp` -> `entity.meditation.xp`/`entity.meditationSkill`, rendered only in the Skills panel). Zeroing it as a blanket safeguard silenced both the bug and the correct behavior at the same time, and nothing in `updateMeditationForEntity`'s tick loop ever called `game.gainMeditationXp` regardless - the wiring for skill XP was missing entirely, independent of the zeroing.

### Player EXP Safeguard
Untouched and still active: `updateMeditationForEntity` snapshots `entity.xp` before the tick loop runs and restores it (logging once) if anything mutated it during the update. This phase's new Meditation-skill-XP call only ever writes to `entity.meditation`/`entity.meditationSkill`, never `entity.xp`, so this safeguard should never trip under normal operation; it remains in place unchanged as defense in depth.

## Meditation Fix Details

### Tick Timing
- `systems/meditation-system.js`: `MEDITATION_TUNING.tickIntervalSeconds` changed from `1.0` to `15.0`, restoring the documented 15-second cadence. `game.js`'s `meditationTickRate()` comment corrected to state this is the authoritative recovery-tick period (not cosmetic), with its fallback also updated from `1.0` to `15.0`.
- No change was needed to the tick-accumulation/catch-up logic in `updateMeditationForEntity` (`meditationTickTimer` accumulation, the `while (... >= tickPeriod && tickSafety < 4)` catch-up loop, or the per-frame `0.25s` accumulation cap) - these were already correct and delta-time-driven, just running against the wrong period.
- `startMeditation`/`cancelMeditation` already reset `meditationTickTimer` to `0` on start and stop; this was not changed and continues to guarantee no partial-tick credit survives an interruption.

### HP Recovery
- `game.js`'s `applyMeditationRecoveryTick` no longer multiplies `data.hpPerSec` by the tick length; the chart value (already inclusive of environment/bard/group/Great-Stillness multipliers computed in `calculateMeditationRegen`) is applied once per completed tick.
- `entity.hp = Math.min(maxHp, before + amount)` caps at max HP, unchanged. No change to the "never heal while dead / while not meditating / while in combat" guards at the top of `applyMeditationRecoveryTick`.

### Resource Recovery
- Same fix applied symmetrically to `data.resourcePerSec` against `entity.mana`/`entity.maxMana` (the shared field the class-specific UI labels as Mana/Focus/Stamina/Energy - that labeling was not touched). Capped at max, unchanged guard conditions.

### Meditation Skill EXP
- `systems/meditation-system.js`'s `calculateMeditationRegen` now returns `result.xpPerTick = MEDITATION_TUNING.skillXpPerTick` (a new, conservative, documented constant, `5`) instead of a hardcoded `0`.
- `game.js`'s `updateMeditationForEntity` now calls `this.gainMeditationXp?.(regen.xpPerTick || 0)` once per completed tick inside the existing catch-up `while` loop, gated to `entity === this.player` (bots/mercenaries meditate through the same loop but must not advance the player-only Skills-panel entry).
- This flows through the pre-existing, unmodified `systems/skills-system.js: gainMeditationXp` -> `systems/meditation-system.js: grantMeditationXp`, which only ever mutates `entity.meditation.xp/level` and `entity.meditationSkill` - never `entity.xp`.

### Interruption Rules
- No changes were needed. `cancelMeditation` already resets `meditationTickTimer`/`meditationProgress` to `0` and clears `meditating`/`isMeditating` on movement, attack, spell/portal cast, combat entry, and death, and `updateMeditationForEntity` already calls `cancelMeditation` when combat starts. Because the tick accumulator is discarded on any interruption, no partial-tick HP/resource/XP can ever be granted after an interrupt - this held true both before and after this fix and was verified by re-reading every call site, not just assumed.

### UI Feedback
- `render/entity-renderer.js`: corrected the stale comment on the nameplate recovery-progress bar (see UI Feedback Risks above). The bar's actual rendering logic (`meditationProgress`-driven fill, never an EXP value) was already correct and is unchanged.
- No changes to the Meditation aura VFX start/stop logic, class-specific aura renderers, or the Skills panel's own rendering code - all already correct and now simply receive real data.

## Known Limitations
- Meditation skill EXP now uses a flat, conservative per-tick rate (`5`) rather than reviving the dormant safe-zone/Great-Stillness/per-environment XP multipliers, which remain defined but unwired (see Deferred Bugs Found). This was a deliberate scope decision to keep the fix surgical rather than a guess at unrecoverable historical numbers.
- The very old (pre-V0.16.94) Pass 47.1 design scaled the tick rate down by level (15s at level 1 to 5s at level 60); the current V0.16.94-era Level 1-20 chart and this phase's fix both use a single flat 15-second tick at every level, per this phase's explicit requirement. Whether tick rate should ever scale with Meditation level again is a design question for a future phase, not assumed here.
- The two in-HTML patch-notes panels and the performance/release-gate system family ownership overlap were documented but not touched, per phase scope.
