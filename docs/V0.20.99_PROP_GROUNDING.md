# V0.20.99 — Art Direction Phase 19: Prop Grounding

> "They look like they're floating."

That complaint was about terrain, but the same fault ran through the props. A contact shadow is what
stops an object reading as pasted onto the map.

## The measurement changed the answer

Reading the source suggested **118 of 141** prop types had no shadow — only 24 appear in the
`shadowSize` table in `drawObjectShape`. That number was wrong. Many props call
`drawPropContactShadow` directly inside their own draw function and never appear in the table.

Instrumenting the helper and invoking `drawObject` once per dispatch type gave the real figure:

| | |
|---|---|
| dispatch types | 141 |
| types that draw anything | 105 |
| **already grounded** | **66** |
| **genuinely ungrounded** | **39** |

Acting on the source-read would have given dozens of props two stacked shadows. This is the second
time in three versions that reading the code gave a different answer from running it.

## 27 grounded, 12 deliberately not

Left ungrounded, with the reason recorded beside the table:

| type | why |
|---|---|
| `cave`, `caveExit`, `caveStairsDown/Up`, `dungeonStairs` | openings in the floor — a shadow over a hole reads as a lid |
| `stalactiteCluster`, `flowstoneDrapery`, `heavyWebCurtain` | ceiling-hung, never touch the ground |
| `miningVein` | embedded in a rock face |
| `fire` | a light source; darkness beneath a flame is backwards |
| `grassTuft`, `flower` | too small — a shadow only muddies the grass |

`stalagmiteCluster` **is** grounded — it stands on the floor, unlike its ceiling counterpart.

The landmarks are the point: `townHouse`, `shop`, `well`, `campStall`, `ruinWall`, `ruinPillar` and the
**waypoint shrine** — the prop this project rebuilt by hand across two versions — all sat on the ground
with nothing under them.

## A pre-existing bug the same probe found

`barrel` was listed in the `shadowSize` table **and** `drawBarrel` called the helper itself with the
identical `(16, 7, 0.24)`. Every barrel drew its shadow twice, stacking to an effective alpha near
**0.42** against an authored **0.24** — over half again as dark as intended.

The table entry was the duplicate, so that is what was removed rather than the call inside the draw
function. One source of truth, not a suppression.

**Reported rather than changed:** `giantWebLair` draws 4 contact shadows. It is a composite lair of
several mounds, so that is plausibly deliberate and I could not confirm the intent.

## Another dead profile block made live

`drawPropContactShadow` now honours `PROFILE.shadows.groundedProps` and `contactRadiusScale`. Both hold
their existing values (`true`, `1`), so this is a **no-op by value** today — it converts a dead block
into a real knob without changing how anything looks, continuing the V0.20.98 finding that five of the
profile's blocks were read by nothing.

The profile is frozen, so `groundedProps` is an edit-the-profile switch, not a runtime toggle.

## Verified

| check | result |
|---|---|
| grounded prop types | **66 → 93** of 105 |
| remaining ungrounded | **12, all on the intentional list, zero unexpected** |
| barrel shadow calls | **2 → 1** |
| 14 grounded props over flat grey | darken by peak **8–39 luma**, 42–2743 px |
| controls: `flower`, `caveExit`, `fire`, `stalactiteCluster` | **exactly 0** darkened px |
| frames | 20 clean |
| boot suite | PASS, 0 console errors |

Pixel-proven rather than call-proven: a call is not evidence of visible output. `ruinWall` is weakest
at peak 8 because the wall body covers most of its own shadow — correct for a wall flush to the ground.
