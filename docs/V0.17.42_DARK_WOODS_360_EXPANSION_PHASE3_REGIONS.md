# Phase 3 — Dark Woods 360x360 Expansion: Named Regions + Level Bands

**Phase:** 3 of 28 (Dark Woods Zone Expansion + Resource Identity master plan)
**Previous version:** V0.17.41
**New version:** V0.17.42

---

## 1. Goal

Add region-boundary and level-band metadata for the six named Dark Woods subregions, plus a lookup API,
without adding any terrain, mobs, resources, or props yet. This phase is pure infrastructure/data — later
phases (4, 6–16, 20–23) will consume the region data as they add real content.

## 4. Root Cause / Implementation Notes

No existing "named region" concept was found anywhere in the codebase before this phase (confirmed by a
project-wide search — the only other uses of the word "region" are unrelated: `weather-system.js`'s fog
rendering rectangles and `layered-map-editor-system.js`'s tileset-picker selection region). This phase adds
new, non-duplicating data and a new lookup function rather than extending or repurposing either of those.

**Where the data lives:** the region list was added as a new field on the existing `dark_woods` zone
descriptor in `data/default-zones.js`, rather than a new data file, since it is exactly zone geography
metadata and that file already holds the zone's `width`/`height`/`generation`/`description`. This keeps the
change additive and avoids introducing a parallel descriptor location.

**Anchoring the geography:** the Phase 1 report's first-draft region layout assumed the camp/spawn would be
re-centered to roughly `{180,180}` in the new 360x360 canvas. Phase 2 instead deliberately kept
`CONFIG.START_X`/`START_Y` at their original `{100,100}` to preserve all existing hand-authored content
unchanged. This phase's region boundaries were therefore redrawn from scratch against the **actual**
implemented geography, not the Phase 1 draft:

- `silk_web_approach` (box `x:10–100, y:105–195`) was sized specifically to contain the real, unmoved Silk
  Web Cavern entrance at `(52,139)` (`systems/cave-system.js` `CAVE_DEFINITIONS.silk_web_cavern.entrance`).
- `bandits_fall` (box `x:130–280, y:10–110`) was sized to encompass the existing Ashroot Hollow ruin
  landmarks already placed by `placeWorldLandmarks` (`(141,68)`, `(145,63)`, `(170,50)`, the Blackroot
  Catacombs ridge at `(164–178,44–57)`), since those are already-authored ruin/stone terrain that visually
  fits the "ruined bandit outpost" identity, even though the actual bandit mobs/props are a later phase
  (Phase 9). Note: several of the caves near this cluster (Ashroot Hollow, Blackroot Catacombs) are slated
  for removal in Phase 5's cave-count enforcement — this phase does not depend on them surviving, it only
  reuses their existing terrain footprint.
- `stone_hedge_clearing` (box `x:130–250, y:190–300`) was deliberately placed in the newly-generated outer
  band (south of the old 199 edge) rather than near any existing landmark, giving Phase 10/11 a clean,
  undisturbed canvas for the standing stones and waypoint.
- The three ring bands (`dead_lantern_outskirts` r0–42, `bramblefen_thicket` r42–100,
  `gloamroot_depths` r100–1000) are concentric around the real spawn point and act as the fallback for any
  coordinate not inside one of the three named boxes — together they cover every distance from 0 to 1000
  tiles with no gaps or double-coverage, so `gloamroot_depths` naturally blankets the vast new southeast
  wilderness created by the Phase 2 resize without needing its own explicit box.

**Lookup semantics:** `getDarkWoodsRegionAt(x, y)` checks the three `box` regions first (array order = 
priority — a point inside more than one box resolves to whichever is listed first) and falls through to the
`ring` regions, which are mutually exclusive by distance. `rMax` uses a finite sentinel (`1000`) rather than
`Infinity`, since `Infinity` does not survive a `JSON.stringify` round-trip (`DR.DEFAULT_WORLD` is read by
`world-serializer.js`'s `cloneJson` helper elsewhere in the codebase, which uses
`JSON.parse(JSON.stringify(...))`) — 1000 is comfortably larger than the map's actual maximum possible
distance from spawn (~366 tiles, the far corner at `(359,359)`), so this has no practical effect on coverage.

**Verified, not just traced:** unlike Phase 2, this phase's core logic is pure data + a pure function with no
DOM/canvas/window dependency, so it could actually be executed in plain Node. `data/default-zones.js` was
loaded and evaluated directly with a stubbed `window` global, and the lookup logic was run against 7 test
coordinates spanning both box and ring regions and the extreme map corner — every coordinate resolved to
exactly one region with the expected level band (see Section 6).

## 5. Deliberately Deferred

- Region `identity.mobFamilies` and `identity.resourceIds` are forward-looking labels only — the mob and
  resource IDs listed (e.g. `herb_wispbloom`, `herb_blackroot`) do not exist as real definitions yet
  (Phases 20–23) and nothing currently reads these fields at runtime. This is intentional: the plan calls
  this out explicitly as "Mob family hooks" / "Resource spawn hooks" for Phase 3, meaning the data should
  exist for later phases to wire in, not that the wiring happens now.
- The existing mob-spawn-system.js distance-based leveling (`overworldProgress`/`chooseLevel`) was not
  changed to consult these regions. It continues to use its own independent, already-recalibratable
  distance formula. Whether to make it region-aware is a judgment call for whichever phase actually reworks
  mob placement in the new areas (Phase 3's own validation criteria only ask for the lookup itself to work
  and for there to be no spawn explosion — both are satisfied without touching that system).
- There is a small, intentional level-continuity gap in the ring-only fallback area: `bramblefen_thicket`
  tops out at level 5 and `gloamroot_depths` starts at level 7, so a coordinate that falls in neither named
  box exactly at the r=100 ring boundary has no ring representing level 6. This is a minor first-draft
  smoothness issue, not a coverage bug (every coordinate still resolves to exactly one region) — noted for
  Phase 4 to consider when it draws real paths through this geography.
- Region boundaries are approximate first-draft boxes/rings, as flagged in the Phase 1 report; they are
  expected to be tuned once Phase 4 adds real paths and once Phases 6–16 add actual landmark content.

## 6. Validation

Commands run:

```bash
find . -name "*.js" -print0 | xargs -0 -n1 node --check
find . -name "*.json" -print0 | xargs -0 -n1 python3 -m json.tool > /dev/null
```

**Results:** both passed with zero errors across every `.js` and `.json` file.

**Additional functional validation (executed, not just traced):** loaded `data/default-zones.js` in a
plain Node script with a stubbed `window` global and ran the same box/ring lookup logic against 7
coordinates:

| Coordinate | Resolved region | Level band |
|---|---|---|
| (100,100) — spawn | `dead_lantern_outskirts` | 1–3 |
| (52,139) — Silk Web Cavern entrance | `silk_web_approach` | 6–10 |
| (170,50) — existing ruin landmark | `bandits_fall` | 4–7 |
| (359,359) — far map corner | `gloamroot_depths` | 7–10 |
| (0,0) — near-origin corner | `gloamroot_depths` | 7–10 |
| (130,190) — Stone Hedge box corner | `stone_hedge_clearing` | 5–8 |
| (250,250) — Stone Hedge box corner | `stone_hedge_clearing` | 5–8 |

Every coordinate resolved to exactly one region with the expected level band; no coordinate returned
`null`, confirming full-coverage lookup across the entire 360x360 map.

**Boot/console-error check:** still not possible in this environment (no browser, no jsdom/puppeteer — same
limitation as Phase 2). This phase's new code has no DOM/canvas dependency, which is why it could be
validated directly in Node above; the game's actual boot sequence still needs a manual browser check.

## 8. Known Limitations

- **Process deviation (repeated):** as in Phase 2, a ZIP backup of the previous version (V0.17.41) should
  have been created immediately before starting this phase's edits, and was not. This is the second time
  this step has been missed; a backup of the resulting V0.17.42 state is being taken immediately after this
  report so Phase 4 has a clean checkpoint, but this needs to actually happen *first* next time, not be
  caught after the fact.
- Region data is inert — nothing at runtime currently calls `getDarkWoodsRegionAt`/`getDarkWoodsRegionLevelRange`
  except the validation script in Section 6. This is expected for a metadata-only phase.
- The level-6 ring-fallback gap noted in Section 5 is unresolved.
- Region boundaries are a first draft and will likely need adjusting once real paths/landmarks exist.
