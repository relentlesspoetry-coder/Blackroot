## Implementation — reuse over invention

The key decision: **make the existing gathering system dungeon-aware** rather than build a parallel
dungeon-resource system. `updateExternalSystems`/`renderExternalSystems` are **not** zone-gated, so
the resource-gathering `update`/`render`/G-key already run in every zone (including dungeons) — they
just resolved the wrong grid via `zoneKey()`. So the whole Phase 19/20 machinery (level gate, cast
bar, respawn/depletion with wall-clock timestamps, first-discovery, rare-node bonus, node rendering)
is reused for free.

1. **Herb defs + items** — 3 entries in `data/resources.js` and 3 in `data/items.js`, identical schema
   to the Phase 20 overworld herbs (authored `drops`, `tool:'Hands'`, no `lootTableId` so the
   Phase-20 `tableForResource` fix yields the herb itself). `allowedZone:'silk_web_cavern'`,
   `allowedBiome:'cave'`.
2. **`zoneKey()` (resource-gathering-system.js)** — a dungeon now returns a floor-specific key
   `silk_web_cavern:F<n>`. Any dungeon returns a dungeon key (so overworld nodes never leak into a
   dungeon — a latent bug also fixed); only Silk Web floors are actually seeded. The floor is in the
   key so per-floor nodes/depletion don't collide.
3. **Placement — `Game.prototype.ensureSilkWebCavernHerbs(dungeon, floor)` (world-system.js)** — seeds
   herbs into `editorResources['silk_web_cavern:F<n>']` at walkable cave-floor tiles near the
   deterministic `SILK_WEB_FLOOR_CONFIG` rooms (ring-scan, skipping tiles that hold a dungeon object).
   Because the room layout is fixed, positions — and therefore the respawn/depletion keys — are stable
   across runs. **Idempotent per (type, room)**, so re-entering a floor never duplicates nodes and the
   wall-clock respawn timers stay meaningful. Webcap on floors 1–2 (entrance/mid), Widow's Veil deeper
   (floors 2–3), Queen's Silkroot only in the floor-3 boss room (the Queen's throne = final chamber).
4. **Call site** — `game.ensureSilkWebCavernHerbs(dungeon, targetFloor)` at the end of
   `loadDungeonFloor` (dungeon-system.js), after `currentZone`/`activeDungeon`/`dungeonMap`/`dungeonRooms`
   are all set. Runs on every floor entry; idempotent.

## Pre-existing bug fixed (discovered during verification)
`render/object-renderer.js` dispatched `case 'silkStrands': return this.drawSilkStrands(s, obj)` but
**`drawSilkStrands` was never defined**, while `placeSilkDressingObject` places `silkStrands` dressing
on every Silk Web floor — so entering Silk Web Cavern threw a render fault every frame per such object
(caught by the render-fault guard → no crash, but constant console spam). Unrelated to the herb work;
surfaced by the Phase 21 dungeon verification. Added the missing non-blocking `drawSilkStrands`
renderer (hanging web curtains, modeled on `drawCaveWeb`). Verified: 20 direct calls, 0 throws.

## Known limitations
- `respawnSeconds` is a single midpoint value; the plan's min–max randomization is deferred to Phase 24.
- Full re-entry anti-farming for the rare Queen's Silkroot relies on the deterministic room layout
  keeping positions (and thus depletion keys) stable; Phase 24 will verify/tighten anti-farming and
  persistence across dungeon runs and reloads.
- The dungeon herb resource grids (`silk_web_cavern:F<n>`) are in-memory (re-seeded on each floor
  entry); they are not added to `world-serializer` persistence. Depletion state (which is what matters
  for respawn/anti-farm) persists via the serialized `resourceRuntime`.
