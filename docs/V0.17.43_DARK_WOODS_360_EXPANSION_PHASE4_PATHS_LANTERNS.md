## 1. Goal

Phase 2 resize, and establish the Dead Lantern Trail motif (lantern variants that visually communicate
danger/territory) along existing and new roads, using the Phase 3 region boundaries to decide which variant
belongs where.

## 4. Root Cause / Implementation Notes

**New paths:** three `carvePath` calls were appended right after the existing 8-road network in
`generateMap`, branching from existing route endpoints (`(163,123)`, `(176,162)`, `(65,171)`) rather than
from town directly, since those endpoints already sit at the old map's edge and are the natural jumping-off
points into the new territory:

- `(163,123) → (182,158) → (196,200) → (200,240)` at width 1.7 (matches the existing main-road width) —
  terminates inside the `stone_hedge_clearing` box (Phase 3), giving that region a proper maintained road.
- `(176,162) → (220,210) → (260,250) → (280,280)` at width 1.3 (narrower — a rougher "side path") —
  deliberately routed to stay outside the Stone Hedge box and terminate well inside `gloamroot_depths`
  (confirmed: distance from spawn at the endpoint is ~254 tiles, ring band starts at 100).
- `(65,171) → (52,200) → (38,228)` at width 1.0 (narrowest) — an unmarked trail with **no lanterns placed
  along it at all**, which is the actual "hidden/overgrown" signal here (the plan's other overlay-based
  hidden-path dressing — broken dirt patches, grass blending — is Phase 13 terrain-overlay work, not part of
  this phase's scope).

**Dead Lantern Trail:** rather than hand-assigning a variant at each of the roughly two dozen existing
`lanternPost` placement call sites scattered across `placeCamp`, `placeWorldLandmarks`, and
`ensureDarkWoodsAtmosphereLandmarks`, a single new pass (`applyDeadLanternTrailVariants`) runs once at the
end of `generateMap`, after every lantern has been placed. It scans the full active `this.objects` grid for
`type === 'lanternPost'` and assigns `obj.variant` by looking up `getDarkWoodsRegionAt(x, y)` (Phase 3) and
mapping the region id to a variant:

```
silk_web_approach   -> webbed
bandits_fall        -> bandit
gloamroot_depths    -> wisp
bramblefen_thicket  -> broken
(anything else)     -> intact
```

Any lantern that already carries an explicit `obj.variant` (the two hand-placed examples below) is left
untouched by the pass, so manual overrides always win. This approach was chosen over editing every
placement call site individually because it is a single, auditable rule instead of ~24 scattered edits, and
it automatically produces correct results for the 4 newly-added road lanterns without needing to hand-compute
which region each one lands in.

Two lanterns were hand-placed with explicit variants to exercise the two motif variants the region-based
pass never assigns on its own (the plan doesn't tie `hanging`/`fallen` to a specific region — they're general
scenery variety): one `fallen` lantern just off the Gloamroot side path (`265,255`), one `hanging` lantern
just off the Stone Hedge spur (`192,205`).

**Rendering:** `drawLanternPost` (`render/object-renderer.js`) already drew the post/lamp/glow procedurally
in Canvas 2D with no sprite assets, so adding variants meant branching within the same function on
`obj.variant`, reusing the same base geometry (post rectangle, arm polygon, lamp box, glow arc) with
per-variant color/geometry tweaks — no new draw function, object type, or render pass was introduced.
`fallen` is the one variant with meaningfully different geometry (the whole post rotated ~90° and drawn low
to the ground, unlit); every other variant keeps the standing-post silhouette and only changes color, glow
color/intensity, or adds a small overlay (a red rag for `bandit`, faint web strands for `webbed`).

**Collision:** unaffected by design. `lanternPost` was already in `normalizeOverworldCollisionAndDensity`'s
`decorOnly` set (forces `tile.blocked = false` regardless of what object sits there) before this phase, and
every new/existing lantern placement call still passes `blocked: false` to `placeObject`. The `variant`
field carries no collision meaning.

## 5. Validation

mutual separation, but not rendered).

## 7. Known Limitations

- Variant assignment is purely by region id; `bandits_fall`'s `'bandit'` variant is not currently exercised
  by any placed lantern (no existing or new lantern happens to land inside that box yet) — it will trigger
  automatically once Phase 9 or a later pass places lanterns there, with no further code change needed.
- The "hidden path" identity relies solely on narrower width + absence of lanterns; the plan's fuller
  hidden-path dressing (broken dirt patches, grass blending) is Phase 13 (Terrain Tile Detail + Overlays)
  scope, not this phase.
- No in-browser visual/boot confirmation was possible in this environment (see Section 5) — the manual
  tests in Section 6 are required before this phase is fully verified.
- Region boundaries themselves are still the Phase 3 first draft; if Phase 3's boxes are later adjusted,
  the lanterns placed in this phase will automatically re-resolve to whatever region they then fall in the
  next time `generateMap`/`applyDeadLanternTrailVariants` runs (no separate update needed), but the *new
  path endpoints* were chosen by hand against the current boundaries and may need re-routing if those
  boundaries move substantially.
