# V0.20.12 — I Was Wrong About The Heavy Weapon

## Correcting V0.20.11

V0.20.11's new scanner found `requiresHeavyWeapon` (authored 13×, **0 reads**) and I reported it as:

> *"the coneDegrees pattern exactly: an authored gameplay gate that never gates"*

…and recommended implement-or-retire, pending a balance decision.

**Investigating it killed that framing.**

## What it actually is

| | |
|---|---|
| authored on | **13 of the Fighter's 20 spells** |
| the other 7 | **all buffs and a dashStrike** — every spell that does *not* swing a weapon |
| includes | **Heavy Swing, level 1** — the class's basic attack |
| two-handed weapons in the entire game | **7 of 42** — one of them the Blackroot Staff, a *caster* weapon |

So the flag isn't separating heavy hitters from light ones. It's separating **attacks from shouts** —
which the game already knows, because one is a sword swing and the other is a buff.

**Enforcing it would have locked a Fighter's entire kit, basic attack included, behind a handful of
weapons they may not own.** A Fighter would pick up their starting sword, swing, and be told no. That's
not a bug fix — that's breaking a class because a field told me to.

**The field discriminates nothing.** It says *"the Fighter's weapon attacks use the Fighter's weapon."*

A flag that is true of every member of a set carries **no information** — the identical fault to the
`Required` marker I refused in **V0.20.5** (no objective is optional, so the word would never vary).

## `classMechanic` — the same shape

| class | spells with the field | class total | distinct values |
|---|---|---|---|
| Shaman | 20 | 20 | `'primal_schools'` |
| Summoner | 20 | 20 | `'planar_summons'` |

**One value per class, stamped onto every spell of that class.** A class-level fact copied 40 times. It
belongs on the class, not the spell — and nothing has ever read it.

## Retired

**53 declarations deleted** from `data/spells.js` — 40 `classMechanic` + 13 `requiresHeavyWeapon`. This
is the **V0.18.96 `poisonResist` decision on the retire branch**: implement it *or* remove it, never
leave it looking alive.

**Behaviour-neutral by construction** (nothing read them) — and verified rather than assumed:

| check | result |
|---|---|
| spells still compile | **280** |
| Fighter / Shaman / Summoner | **20 / 20 / 20** |
| compiler errors | **0** |
| Heavy Swing keeps its real fields | `power`, `weaponDamageMultiplier` **intact** |
| suite | **PASS** — 15/15, 396/396, 0 issues |
| console | **0 faults** |
| scanner's own candidate list | **29 → 27** — the loop closes |

## The lesson — this session's own, turned back on me

**"Authored and never read" identifies a suspect, not a crime.**

Six times this session it meant something was broken: a cone that drew a circle, poses nobody could
reach, styles never drawn, 33 portraits never shown, a label with a hardcoded twin.

**Twice now it has meant the field was meaningless from the start.**

I reported "a gate that never gates" **before checking what it was supposed to gate** — which is the
same failure as trusting a green check without asking what it proved. The tool did its job: it pointed
at something. Drawing the conclusion was mine, and I drew it too fast.

That is exactly why `check-dead-fields.js` prints *"candidates, not verdicts"* — a caveat I wrote and
then, one version later, ignored.

"what would it mean if it were?"**
