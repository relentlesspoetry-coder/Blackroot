# V0.20.60 — Revert of V0.20.59, and Retiring the Caching Approach

## What happened

V0.20.59 cut `spriteCacheLiveRadius` from 18 tiles to 6, on the strength of a real, drift-checked
measurement: **canvas ops/frame 40,277 → 29,315.**

Reported result: **12 FPS → 10.**

## The measurement was true and insufficient

Cutting that radius pushes characters onto the sprite cache. Measured *after* the report:

| live radius | cache entries | offscreen canvas memory |
|---|---|---|
| 18 | 485 | 135 MB |
| **6** | **744** | **207 MB** |

Every distinct source canvas is a **GPU texture upload** when blitted. With hundreds cycling through
animation buckets, that upload churn costs more than the vector drawing it replaced.

**Canvas-op count is not a sufficient cost proxy when the ops are blits from many distinct source
canvases.** Identical failure mode to the V0.20.53 atlas revert: 19 pages, 1.28GB of texture, 5–6× slower.

## Third strike — caching is closed

| version | attempt | result |
|---|---|---|
| V0.20.49 | humanoid render cache | 12 → 7 FPS, reverted |
| V0.20.53 | full sprite atlas | 5–6× slower, reverted |
| V0.20.59 | live radius 18 → 6 | 12 → 10 FPS, reverted here |

Three independent attempts, three regressions, one shared mechanism: **trading vector draw calls for
bitmap blits loses on this renderer**, because the bitmaps are numerous and constantly changing.

Treat sprite/bitmap caching as a **closed avenue** unless someone first measures texture-upload cost on
real hardware.

## Standing diagnosis

**~40,000 canvas ops/frame against a ~5–10k budget for 60 FPS.**

| phase | ops |
|---|---|
| `drawEntity` | ~21,800 |
| `drawObject` | ~12,000 |
| `drawTile` | ~3,500 |
| `npcRuntime` | ~3,500 |

Reaching 60 needs one of:

1. **Draw fewer things** — `drawObject` alone is 151 props/frame, **68 of them measured off screen**
   (the V0.20.56 pattern, not yet applied to props). Beyond that: fewer props authored per screen, or
   simpler prop geometry (a tree is ~80 ops).
2. **A different backend** — `renderBackendMode` is `'canvas2d'`, and `systems/render-backend-system.js`
   already contains a WebGL path with sprite/terrain texture caches and an auto-fallback. A draw-call-bound
   canvas2d game is exactly the case WebGL solves, since it batches. This is the only avenue with a
   plausible path from ~10 FPS to 60; it is also the largest and riskiest, and would need its own
   investigation on real hardware.

It does **not** need another caching layer. I should have concluded that two versions ago.

## Method notes worth keeping

Three measurement traps produced confident wrong answers during this investigation:

1. **`performanceSettings()` is memoized** — mutating `DR.CONFIG.PERFORMANCE` mid-run does nothing.
   Stub the reading function instead.
2. **`runtimeSpriteCache.beginFrame()` only runs from `loop(now)`** — synthetic loops leave the cache
   permanently budget-skipped.
3. **Instance-level context patches shadow the prototype** — restoring by assignment leaves own
   properties behind, so a later prototype patch silently counts zero. Reload between instrumentation runs.

And the standing rule that caught every one of them: **always repeat arm 1 last as a drift check.**
