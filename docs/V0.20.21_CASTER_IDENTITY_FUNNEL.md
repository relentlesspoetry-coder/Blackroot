# V0.20.21 — Nobody Told The Ring Who Cast It (Roadmap Item 6, the caster half)

## Retraction first

My previous report told the user `spell.projectile` was **"double dead — authored zero times, and
passed under a key `spawnBolt` ignores."**

**Both halves were false.**

| my claim | the truth |
|---|---|
| "authored 0 times" | the **compiler generates it** — `spell-compiler-system.js:192`, `projectileDescriptorFor`, for `bolt`/`boltDot`/`boltSummon`/`drain` |
| "`spawnBolt` never reads it" | it reads it **20 lines below** the `style \|\| kind` line I stopped at, to derive `travelDuration` |

I grepped `data/spells.js` for the writer and the top of `spawnBolt` for the reader. **The writer was
in a third file; the reader was further down the same function.** Caster bolts already travel. The
field was never broken, and `pierce`/`homing` are documented opt-in metadata — V0.20.14's
**DELIBERATE** category: leave alone.

I have now written *"don't ask whether the data is there, ask whether anything reads it"* into four
consecutive reports **and then declared a live field dead without properly checking either end.**

## The real find — the inverse of dead data

Not data nobody reads. **A system fully built that nothing can reach.**

`vfxStyle()` (`render/effects-renderer.js:90`) keys a palette off
`sourceClass + school + kind + style + label` and maps it to **eight school identities**, each with its
own `secondary`/`dark`/`smoke` and a **`motion`**:

`rune` · `wisp` · `bubble` · `bloom` · `organic` · `wave` · `precise` · `impact`

`drawHighQualityRingEffect` branches on that motion into `drawRuneFragments` / `drawSmokeWisps` /
`drawArcSigil`. **All implemented. All shipped. All unreachable.**

> **Of 99 `spawnRing` calls in `spell-system.js`, exactly 1 passed any identity key.**

So the key resolved to `" bolt bolt "`, matched no regex, and fell to the generic default at line 100.
**The Wizard's rune fragments have never once been drawn.** Every caster threw the same anonymous
white ring — not for want of art, but because **nothing ever told the effect who cast the spell.**

## The fix is the funnel, not the call sites

Patching 99 sites is the **cascade the golden rules forbid**, and 99 chances to miss the 100th.

`resolveClassSpell` is the **single dispatcher all 13 classes route through**. It now publishes the
lowercased casting class; `decorateEffectVisualPayload` — which `spawnRing` already funnels through —
falls back to it. **One change: every existing call site fixed, every future one free.**

| decision | why |
|---|---|
| **restores** the prior value, doesn't clear | a spell that resolves another spell must not strip the outer cast's identity on the way out |
| **try/finally** | a throwing resolver must not leave the context stuck on |
| **never defaults from `effect.sourceClass`** | effects are **pooled** — that would inherit the *previous caster's* class. The identical reset hazard `spawnBolt` already documents for `travelDuration` |
| explicit payload still wins | a class with bespoke VFX keeps its designed look |

**`school` was the obvious key and it's a dead end:** `damageType` is only ever `physical`/`magic`
(line 3422), and neither string matches any palette regex. **`sourceClass` is the key that works** —
`wizard`, `necromancer`, `druid`, `bard` match directly.

**`label` was considered and rejected.** Matching is by *regex order*, not key position, so a Cleric
spell named "Shadow Ward" would match `/necro|…|shadow/` **before** `/cleric|holy/` and take the
necromancer palette. Spell names would have actively broken class identity.

## Scope corrected — a third time

"Nine caster classes need this" was wrong. **`config.js:100` records V0.18.31 already gave the Cleric
bespoke holy VFX**, as V0.18.29 did the Assassin. The per-class pass is an *existing program*; some
classes have already graduated off the generic path.

**Verified, not assumed, that both keep their looks:**

- **Cleric — unaffected.** `drawClericBlessEffect`/`drawClericSmiteEffect` **never call `vfxStyle`**.
- **Assassin — unchanged.** `drawAssassinProjectileEffect` does use it, but its `dart`/`knife`/
  `crossbow` kinds already match at **lines 92–93, before any class regex**. `sourceClass` cannot
  displace an earlier match.

**The change is strictly additive: only effects that previously matched _nothing_ now match their class.**

## The funnel wasn't enough — caught only by casting for real

The tests above proved the wire in isolation and I called it shipped. **The user pushed back: "make
sure it actually works."** Booting a real Wizard and casting real spells found a hole immediately.

```
Mana Shield -> statusPulse : "" : magic      <-- still generic
               ring        : wizard : rune   <-- fixed
```

**`spawnStatusPulse` sets its identity by hand and never calls `decorateEffectVisualPayload`**, so
the funnel didn't reach it. A Wizard's Mana Shield drew runes on the ring and a generic pulse on the
player, in the same cast.

**`arcane_intellect` hid this.** Its pulse showed `rune` — not from the fix, but because
`spawnStatusPulse` sets `effect.label`, and the name *"Arcane Intellect"* matches `/arcane/` by
accident. **One spell in three passed for the wrong reason**, which is exactly how a partial fix
survives review.

Fixed at `spawnStatusPulse` (the same fallback), and **`spawnBolt` was verified rather than assumed**:
it sets `sourceClass` manually at line 373 but calls the decorator at 398, which overwrites — so bolts
were already covered. Assuming that would have been the fourth `?.`-on-an-assumption bug of the session.

## Still generic — deliberately out of scope

`combatImpact` and `combatSpark` still resolve to `-:magic`. They're spawned by the **damage** path,
not the VFX path, and `drawCombatImpactEffect` only branches on `motion === 'impact'` — so pushing the
caster's class in would hand a Wizard `rune`, which that function cannot draw, while changing **melee**
impacts as a side effect. Out of scope, and honestly still broken.

It also suggests `drawCombatImpactEffect`'s `impact` smoke branch may be **unreachable the same way** —
a separate finding for whoever picks it up, not a claim I've verified.

## Turning the wire on exposed two shadowed terms — one live in the shipped game

`vfxStyle` tests its lines **in order** with **unanchored substrings**, so an earlier line can claim a
word meant for a later one. Making `sourceClass` reachable surfaced it.

| term | authored for | actually got | why |
|---|---|---|---|
| `warden` | `organic` (druid line) | **`bloom`** (cleric) | `"warden"` contains **`ward`** |
| `protect` | `bloom` (cleric line) | **`wisp`** (necro) | `"p`**`rot`**`ect"` contains **`rot`** |

**Both terms were authored deliberately and neither has ever been reachable** — `warden` is listed in
the druid regex and could never fire. That is `drawRuneFragments` again, one line away.

**`protect` was live in the shipped game and is not my doing:** `spawnStatusPulse` sets `label` from
the spell name and line 462 explicitly handles `protect` labels, so **Cleric protection spells have
been rendering as necromancy green**. `warden` only became reachable this version.

Fixed at the root — the unanchored substring, not the call sites:

- `ward` → `ward(?!en)` — keeps `wards`/`warding`, releases `warden`
- `rot` → `\brot` — word-start only, keeps `rot`/`rotting`/`rotten`, releases `protect`

**Then audited all 50 authored terms instead of fixing them one at a time:** every term now reaches
the palette its own line intends, **0 shadowed**, and the derived words each term exists to catch
still work (`rotting`→wisp, `wards`/`warding`→bloom, `roots`→organic, `fireball`/`snowstorm`→rune).

## Two mistakes I made in this version

**1. My first suite probe was vacuous.** It read `r.statPipeline` and `r.dataIssues` — neither
exists; the real shape is `r.summary.*`. Every `0 issues` was `undefined || []` and **meant nothing**.
Caught only because `statPipeline` printed `undefined/undefined`. The same artifact as the earlier
vacuous `.every()` on empty arrays, and **it would have read as a clean pass.**

**2. I broke `config.js`.** My build-name edit matched only the *prefix* of the old string, so the new
string closed and the old tail dangled as bare code. **`node --check` caught it immediately** — which
is the entire reason that rule exists. Repaired by replacing the line by index rather than by string
match.

## Item 6 status

**I first wrote here that eight classes — including Shaman, Summoner, Enchanter and Ranger — "now have
a school identity". That was false, and I never checked it before writing it.** Those four class names
match **no regex in `vfxStyle`** and still resolve to the generic default. Measured, per class:

| class | motion | |
|---|---|---|
| Fighter | `impact` | |
| **Rogue** | `precise` | **this version** — joined the blade line |
| **Paladin** | `bloom` | **this version** — joined the holy line |
| **Warden** | `organic` | **this version** — the shadow bug is why it never worked |
| Assassin | `precise` | V0.18.29 bespoke |
| Cleric | `bloom` | V0.18.31 bespoke |
| Wizard · Necromancer · Druid · Bard | `rune` · `wisp` · `organic` · `wave` | palettes existed; now reachable |
| **Shaman · Summoner · Enchanter · Ranger** | **`magic`** | ❌ **still generic — no palette exists** |

**10 of 14 classes have an identity. Four do not.** Giving those four one means **authoring new
palettes** — picking colours and a motion for each — which is a design decision, not a wiring fix, and
is not something to invent unilaterally at the end of a long session. `shaman`, `summoner`,
`enchanter` and `ranger` are simply absent from `vfxStyle`, and no amount of plumbing invents them.

**This is a floor, not a ceiling.** A bespoke pass (the Assassin/Cleric treatment) is still what gives
a class a look of its **own** — but for those 10 it is now an upgrade from a real baseline rather than
a rescue from nothing.

## The lesson, restated — and inverted

Four reports said: *don't ask whether the data is there, ask whether anything reads it.*

**This version adds the mirror image:** *don't ask whether the renderer exists, ask whether anything
can reach it.* `drawRuneFragments` was written, correct, and shipped. It had simply never been called.
**A feature nothing can reach is indistinguishable from a feature nobody built** — and grep finds the
first kind, not the second.
