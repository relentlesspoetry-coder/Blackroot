## 2. Bots sliding instead of walking

13 of 14 bots set their locomotion correctly. One was measured covering **1.09 tiles with `moveBlend 0`**
and no velocity, so `getEntityAtlasAction` resolved `idle` and it slid in a standing pose.

**Cause:** a bot can be repositioned while its own update has already declared it stationary â€” the
casting/busy branches zero `moveBlend` and `return` early. So a fix placed *inside* the bot update was
overwritten later in the same frame (I tried that first; it didn't hold).

**Fix:** `Game.deriveBotLocomotion()`, a post-pass that runs **after every entity update**, deriving the
gait from the ground actually covered. Ground truth, can't be clobbered, and covers movement paths added
later. It only raises the gait, then decays when no ground is covered, so a genuinely stationary bot
still reports idle.

**A measurement trap worth recording:** my first check compared position across 45-frame batches but
sampled `moveBlend` only at the final instant â€” so a bot that walked and then *stopped* looked like it
had slid. Measuring **per frame** gave the truth.

| metric | before | after |
|---|---|---|
| sliding while walking | sustained | **1.6%** of walking frames (single-frame derivation lag) |
| false "walking" while still | 655/871 (intermediate attempt) | intended blend-out tail only |

The intermediate attempt is worth noting: it only ever *raised* `moveBlend`, which traded sliding for
bots animating in place. The decay branch fixed it.
