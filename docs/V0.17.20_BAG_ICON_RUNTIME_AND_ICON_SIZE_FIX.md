## Root Cause For Bag Fallback Icons
Bag items could arrive as saved/runtime instances with generated rarity prefixes or generic copied `icon` metadata. The prior resolver could allow a generic copied icon key/family to outrank canonical uploaded-asset resolution, and exact-name matching did not account for runtime prefixes such as `Plain`. That left bag cards falling back to category glyph presentation even though the same canonical assets existed and equipment cards could display them.

## Resolver Changes
- Added shared runtime name aliasing for icon hydration.
- Added canonical lookup support for rarity-prefixed runtime names, including `Plain Briar-Silver Earring` -> `Briar-Silver Earring`.
- Reordered icon-key resolution to prefer real uploaded asset keys from item/canonical IDs before generic family/glyph metadata.
- Kept missing mapping / missing asset logging behind the existing debug/dev flag and once-per-key behavior.

## Bag Renderer Changes
- Bag slots continue to call the shared `itemIconHtml()` path.
- Final CSS owner enlarges filled bag slot icons to 46px and keeps them in the dominant left column of the card.
- Stack badges, item names, rarity borders, hover tooltips, right-click equip/use, drag metadata, and bag slot counts are preserved.

## Equipment Card Layout / Icon Size Changes
- Filled Character equipment cards now use a `largeIconEquipSlot` class.
- Uploaded item art is displayed at 46px.
- Slot label, item name, and compact stats are grouped so the larger icon does not overlap text.
- Empty equipment slots retain their placeholder glyphs.

## Saved Item Hydration Behavior
Existing saves do not need a wipe. Inventory and equipment item instances continue to resolve by stable item IDs first. When a saved/generated item lacks full icon fields or has a rarity-prefixed display name, rendering resolves back to the canonical item definition and uploaded icon key at display time.
