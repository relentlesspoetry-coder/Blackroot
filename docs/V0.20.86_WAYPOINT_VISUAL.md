# V0.20.86 — Waypoint Visual Rebuilt

Reported: *"The Waypoint needs revamped. Use the 2nd image as reference for the waypoint visual."*

## What was there

`drawWaypoint` was a ~24px grey disc, a lighter inner disc, one ring, and three scratched lines. In the
screenshot it read as a small blue puddle. It was also nothing like the spec it was supposed to satisfy —
Ashen Valley §5.4 asks for a *"large circular stone platform, engraved runes, regional symbols, subtle
magical light, rotating inner ring, particle wisps, faint vertical beam, activated rune segments."*

## What it is now

Back to front, which is the order that matters for an isometric prop:

1. ground shadow
2. **two-tier stone dais** built from **16 radial blocks with mortar lines** — this is what makes it read
   as masonry rather than a flat disc
3. inner platform tiers
4. engraved rune ring
5. **12 rune segments** that light in a running sequence
6. **two counter-rotating glyph rings**
7. radial core glow
8. **vertical beam**
9. rising motes, seeded deterministically per index so they don't flicker frame to frame the way
   `Math.random()` particles do
10. **the two flanking braziers last**, so they correctly occlude the beam

Radius **24 → 34**, so it reads as a landmark instead of a pebble.

## The three states, as specified

| | dormant | attuned |
|---|---|---|
| stonework | full | full |
| cut runes | present, unlit | lit, sequencing |
| glyph rings | — | rotating |
| beam | **none** | yes |
| brazier flames | **none** | yes |

§5.4 explicitly requires an undiscovered waypoint to be *"still visually recognizable"* — so dormant
keeps everything structural and loses only the light.

## The beam is additive, and that was a measured correction

The first version used ordinary source-over blue at 0.2–0.3 alpha. Over the brown ground that resolves to
roughly **(123, 119, 112)** — visually grey. It classified as **stone** in the pixel check, not as glow,
which is how I noticed.

`globalCompositeOperation = 'lighter'` makes it read as *light* over any ground. That matters concretely:
this same object stands on dirt in Dark Woods and on paved ground in Ashenfall. Verified on both:

| backdrop | blue pixels in the beam column |
|---|---|
| dirt `#6b5540` | **392** |
| stone `#4a4a4a` | **1176** |

## A state bug the new art exposed

`obj.attuned` is baked when the world is generated — when there may be **no player yet** — and
`tryAttuneWaypoint` returns early on the "already attuned" path **without re-setting it**.

So a waypoint you had attuned rendered **dormant after every reload**. Invisible with the old art, since
attuned and dormant looked nearly identical; obvious with the new one.

Attunement now reads `player.unlockedWaypoints` — the single source of truth (Item 9) — with
`obj.attuned` kept only as a fallback for objects that have no id.

## Safety

Every ellipse radius is clamped at draw time. A negative radius throws and takes the whole frame with it;
that is the exact failure that produced a black screen during the silk-critter work.

## Verified by rendering and reading pixels

Not by inspecting the code. Drawn to the real canvas under an identity transform, then sampled:

| | attuned | dormant |
|---|---|---|
| pixels drawn | **4360** | 2655 |
| blue glow | **1278** | 125 |
| bright core / flame | **365** | 0 |
| vertical extent | **121px** | 43px |

An ASCII render of the output confirms the stack visually — dais, both braziers with flames, rune ring,
bright core, and the beam rising through the middle.

### Note on the harness

The first attempt drew nothing at all: I created an offscreen canvas and swapped `DR.runtime.ctx`, but
`object-renderer.js` destructures `ctx` **once at install time**, so the swap had no effect and everything
drew to the real canvas. Drawing to the real canvas under `setTransform(1,0,0,1,0,0)` — where coordinates
map 1:1 to pixels — is the approach that works. Same family as the canvas-transform trap from the mount
work: verify where the pixels actually went before trusting a zero.
