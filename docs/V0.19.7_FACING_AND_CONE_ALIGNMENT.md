# V0.19.7 — Facing North (Roadmap Item 6)

Chasing V0.19.6's cone into the VFX layer turned up three things — one of them a one-line bug repeated
across the whole spellbook.

---

## 1. The facing bug — `0 || 1` is `1`

```js
const fx = Number(p.facingX || 1), fy = Number(p.facingY || 0);
```

`facingX` is **legitimately `0`** when an actor faces due north or south (`setFacingFromDelta` normalises
`dx / len`). And `0 || 1` is `1` — so this idiom **silently rewrote a north-facing actor's facing to
`(1, 1)`** and swung it **45° off-axis**.

It came from the original `inFront` closure and had been copied to **six sites**:

| site | consequence |
|---|---|
| `spellShapeContains` (was `inFront`) | **shipping Cleave has mis-aimed** for any fighter facing straight up or down, since it was written |
| `blink` | wizard teleports 45° off-course |
| `illusory_step` | teleports 45° off-course |
| `disengage_shot` | dodges diagonally instead of straight back |
| `evasive_step` | dodges diagonally |
| `isRogueBehindTarget` | **the whole backstab arc rotated 45°** |

The rogue check had the **mirror image** of the defaults — `target.facingX || 0, target.facingY || 1` —
so a target facing **due east** (`facingY === 0`) was read as facing 45°. `shadowstep`, which positions
you at a target's back, could land you at their **shoulder**.

**Fix:** one shared `Game.prototype.actorFacingVector(actor)` that defaults only a genuinely
absent/non-finite component, falls back to `facingAngle` when both are missing, and treats the zero
vector as directionless.

> `||` cannot supply a numeric default where **zero is a valid value**.

## 2. Fan of Knives is 108°, not 180° — correcting V0.19.6

**I picked 180 before I had found the VFX.** `spawnAssassinFanVfx` (`systems/effects-system.js:588`) has
drawn a `Math.PI * 0.6` = **108° fan since V0.18.29**, and the call site passed **no** spread.

So V0.19.6 shipped a **hit arc 72° wider than the drawn one** — it closed one visual/behaviour mismatch
by opening another.

**108° is not arbitrary**: it is the **authored width players have actually been watching**. It is the
single source of truth now — the resolver derives the VFX spread **from** `spell.coneDegrees`, so the two
cannot drift apart again.

## 3. One direction

The resolver centred the cone on `p.facing` but aimed the VFX at the **first enemy found**
(`atan2(aimY, aimX)`) — **up to half a cone apart**. Unlike the fighter's resolver, whose
`hostileTarget()` faces the target before swinging, this handler **never set facing at all**.

It now faces the selected target first (when one is in range), and that single facing drives **both** the
hit test and the visual. The VFX reach also comes from the same radius the test uses — it previously drew
to `range` (3.0) while killing to `radius` (3.2).

Fan of Knives is the **only** assassin `kind: 'aoe'` spell (verified — the other 14 belong to classes that
don't route through this resolver), so the change is contained to it.

## Also: two tooltips were lying

Neither resolver gates on `range`; it only feeds the tooltip's `Range: N`. Both cone spells advertised a
**shorter reach than they actually killed at** — display-only, caught by the new `cone-range-under-radius`
check:

| spell | was | now |
|---|---|---|
| `fan_of_knives` | range 3 / radius 3.2 | **3.2** / 3.2 |
| `cleave` | range 1.85 / radius 2.05 | **2.05** / 2.05 |

## Verification — the bug was found *by a test*, not by reading

An end-to-end run of the **real resolver**, told to target due north, **drew the fan at 45°** and hit
*both* the north and the east enemy. That is what exposed the `|| 1` idiom.

| check | result |
|---|---|
| fan aim/spread/reach, end-to-end through the real resolver | **4/4** |
| `actorFacingVector` (north, south, zero-vector, `facingAngle`-only, null actor) | **9/9** |
| `isRogueBehindTarget` vs axis-facing targets | **7/7** |
| `spellShapeContains` geometry | **15/15** |
| all four `validateSpellShapes` checks **negative-tested** | **4/4 proven to fire, then restore** |

The first negative test mutated `COMPILED_SPELL_BY_ID` — **the wrong object** — and falsely reported the
range check dead. The validator reads the raw authored `DR.SPELL_BY_ID`. Retested against the object it
actually reads.

## Item 6 status — still open

Closed: cone **targeting truth**, cone/VFX **alignment**, and the **facing** foundation both depend on.

Still open:
- **Cleave draws a full 360° ring** (`spawnRing`, `spell-system.js:946`) while hitting a 100° cone — the
  roadmap's example violated in the *visual* direction. The ring is baked through a **sprite cache**
  (`effectSpriteCacheKey`) keyed by `type:color:style:radius:progress` — **facing is not in the key**, so a
  directional wedge needs the key extended (quantised facing) or a non-cached draw path. This is the same
  trap as the spider `modelScale` cache bug.
- `spell_summoner_mass_dismissal` declares `radius: 4` on a single-target spell — may be legitimate for a
  `tempMinions` effect.
- The remaining 279 spells' VFX have not been audited against their descriptions.
