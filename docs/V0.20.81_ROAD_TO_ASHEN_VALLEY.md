# V0.20.81 — The Road to Ashen Valley

Asked directly: *"is Ashenfall connected to dark woods?"*

**No — and worse than not connected. It was deadlocked.**

## The deadlock

| step | requirement |
|---|---|
| fast travel to Ashenfall | the waypoint must be **discovered** |
| discover the waypoint | you must be **standing beside it** |
| stand beside it | you must **already be in Ashen Valley** |

Measured on a fresh character:

```
REACHABLE_BY_NORMAL_PLAY: false
legitimateRoutes: []
darkWoodsObjectsMentioningAshen: []
caveEntrancesToAshen: 0
```

Nothing in Dark Woods referenced the new zone — no border, no entrance, no neighbour data. The only
thing that reached it was `debug.travelZone()`.

**So V0.20.80's "you can travel to it" was true for me and false for a player.** I verified the travel
*mechanism* — the API, the refusal reasons, the arrival tile — and never asked the one question that
mattered: can a character actually invoke this? That is a verification gap, not a coding slip, and it is
the second time in this project that testing the mechanism instead of the player's path has produced a
false pass.

## The fix

- **`DR.ZONE_LINKS`** (`data/default-zones.js`) — where the walked routes are.
- **`systems/zone-link-system.js`** — the interaction. Deliberately mirrors the waypoint system's shape
  (same nearby-object scan, same interact-key binding) rather than inventing a second interaction idiom.
- **The two world generators** place the crossings.
- **`core/overworld-zone-registry.js`** still owns the actual zone change — one place, as before.

| side | location | region | levels |
|---|---|---|---|
| Dark Woods | 344, 250 (far east, ~157 tiles from spawn) | **Gloamroot Depths** | 7–10 |
| Ashen Valley | 34, 300 | **Valley Farmland** | 10–12 |

The Dark Woods crossing sits in the deepest, highest-level part of the woods on purpose — a genuine
difficulty step into a level 10–20 region, not somewhere a level 3 character wanders by accident. The
arrival is the valley's gentlest ground, with roads onward to the city so a player who walks in has
somewhere to walk to.

**The waypoint is not gifted.** Spec §5.5 forbids remote unlock, so the player walks there and attunes it
like any other. The road breaks the loop; it does not skip the rule.

**Symmetry is required and asserted** — every link must have a return leg, or a player can walk somewhere
they cannot walk back from.

## Four more bugs, each found only by walking a fresh character through it

1. **Placed before the object-pruning pass.** `normalizeOverworldCollisionAndDensity` strips objects, so
   the road tiles were carved and the interactable silently removed. Moved after it — where the overworld
   herbs already sit, for exactly this reason, with a comment that said so.

2. **Did nothing on any existing save.** A saved world bypasses `generateMap` entirely
   (`world-serializer.js` assigns the grids directly). Without a load-time hook the road existed only in
   brand-new worlds, and **every current player had no way in**. Re-ensured on load, the same reasoning
   as `mergeShippedCatalog`: shipped world content must reach saves written before it existed.

3. **Even then it did nothing.** The function operated on `this.map` / `this.objects`, and during a world
   load the zone grids are populated *before* the live map is assigned — so it ran, found `this.map`
   empty, and returned 0. Instrumenting the load proved it: `ensureRanDuringApply: true`, crossing still
   absent. It now targets the `dark_woods` slot **by name**. Same shape as the waypoint that was nearly
   placed in the wrong zone: a helper reaching for "whatever is active" instead of the thing it means.

4. **The interaction system registered but was inert.** `installExternalSystems` only stores the returned
   runtime — **it never calls `init()`**. Each system calls its own; `waypoint-system.js` does so at the
   end of its install, and mine did not. The system appeared in `externalSystems`, had a working `init`,
   and `game.zoneLinkSystem` was simply never assigned — **with no error anywhere.** I had written that
   this file "mirrors the waypoint system's shape" and missed the one line that makes it live.
