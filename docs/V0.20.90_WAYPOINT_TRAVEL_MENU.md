# V0.20.90 — Waypoint Travel Menu

Requested: *"the waypoint should open a menu after interacting with it and show all the waypoints you
know. just like diablo 2 does."*

## What was missing

Interacting with a waypoint only **attuned** it. `travelTargets()` and `travelTo()` had existed as API
since V0.20.80, and V0.20.87 added same-zone travel on top — but with **no interface**, so the entire
fast-travel system was reachable only from the developer console. Everything worked and nothing was
usable.

## Behaviour

The interact key **attunes on first touch, then opens the list.** Discovery is still earned by standing
on the stone — spec §5.5 forbids remote unlock, and the menu does not bypass that. An already-attuned
waypoint goes straight to the list, which is the Diablo behaviour. Escape closes it, matching every other
panel.

## The list (spec §5.6)

| | |
|---|---|
| grouped by | **region** — Diablo groups by act, and this is the structure the zone list needs anyway as more regions land |
| discovered | name, region, level range, clickable |
| undiscovered | shown but masked as **???** and disabled |
| current | highlighted, reads **"You are here"** |
| cost | free, stated in the panel |

Masking undiscovered entries as `???` is deliberate: the player learns that *something* exists in a
region without being handed its name or being able to reach it. That matches §5.6's *"may show zone name
if known through lore… not selectable"* while erring toward less disclosure.

**Every disabled button carries its reason as its own label.** In combat the button itself reads *"You
cannot travel while in combat."* That is §5.7's *"blocked travel must provide a specific reason"* placed
where the player is actually looking, rather than in a log line they may not see.

## Implementation

A `panel gameWindow` section filled by `innerHTML`, which is the idiom `mount-system` and the skills
panel already use. No new UI framework, and it inherits the existing window styling for free.

The panel re-renders after a failed travel attempt so the reason on each button stays current, and closes
itself on a successful one.

## Verified by driving the real UI

Not the API beneath it — the actual DOM, with synthesized key events and real `click()` calls:

| check | result |
|---|---|
| interact key on a waypoint | opens the panel **and** attunes, in one action |
| zone headings rendered | Dark Woods, Ashen Valley |
| undiscovered entries | both masked **???**, disabled |
| current waypoint | highlighted, "You are here" |
| **clicking** Ashenfall's button | travels to `ashen_valley`, lands 226,233, panel closes |
| reopening at the destination | Ashenfall now "You are here"; other two re-enabled |
| in combat | all travel buttons disabled, **reason as the label** |
| clicking back to camp | returns to Dark Woods |
| Escape | closes the panel |

This distinction matters here more than usual: the underlying travel API was already verified in
V0.20.80 and V0.20.87, and it was correct the whole time. The thing that was missing could only be caught
by testing what a player can actually press.
