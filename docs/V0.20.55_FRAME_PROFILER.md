# V0.20.55 — Stop Guessing: A Profiler That Runs On The User's Machine

## Why this exists

The frame-rate work across V0.20.49–54 produced **two confident wrong answers**. Both came from the same
source: **every number I quoted was measured in a preview that is permanently `document.hidden`**, where
rAF is throttled to ~0 and canvas2d renders in **software**. That is not the machine the game runs on,
and I kept reporting its numbers as if it were.

## It is worse than merely being slow — it drifts

Driving `render()` in a tight synthetic loop to compare options:

| arm | config | ms/frame |
|---|---|---|
| 1 | current | 127.3 |
| 2 | cache everything | 129.1 |
| 3 | cache nothing | 130.5 |
| 4 | **current again** | **148.2** |

Arm 4 is arm 1 repeated — **21ms slower with identical settings** — and sprite-cache hits/misses read
`0` in every arm. The ablation was noise. I discarded all of it rather than ship a third "fix" derived
from it.

Including a repeat-of-the-first-arm drift check is what caught this. Without it I would have "found" that
cache-everything was fastest and shipped it.

## What was added

```js
await DarkWoodsGame.profileRenderPhases(120)
```

In `systems/performance-verification-system.js`. It:

- wraps the major draw phases **and each external system's `render()` separately** — `renderExternalSystems`
  is usually the single largest bucket, and knowing that without knowing which system is inside it is useless;
- samples the **real `requestAnimationFrame` loop**, not a synthetic one;
- restores every wrapper in a `finally`, so a mid-run throw cannot leave the renderer instrumented;
- returns a report and `console.table`s it;
- includes `version` and `atlasFramesLoaded`, so a **stale-build mix-up is visible immediately** rather
  than costing another round trip.

## The lead it already produced

Measured here, in software rendering — **a lead, not a verdict**:

| phase / system | ms/frame | calls |
|---|---|---|
| **external system `npcRuntime`** | **17.21** | 1 |
| `drawEntity` | 15.82 | 17 |
| `drawTerrainChunks` | 10.29 | 1 |
| `drawObject` | 7.02 | 151 |
| `drawTile` | 3.31 | 58 |
| `drawNameplate` | 2.71 | 17 |

**The camp NPC system costs more per frame than every entity in the scene combined.**

Two things worth knowing about it, established while profiling:

- It culls by a **36-tile world radius**, not the viewport — but with 28 NPCs and 26 genuinely on screen,
  viewport culling would save almost nothing. The cost is per-NPC drawing, not overdraw.
- The camp sits entirely inside `spriteCacheLiveRadius: 18`, so **every camp NPC bypasses the runtime
  sprite cache and draws live every frame**. That was deliberate in V0.18.4 (to fix animation fidelity),
  but the camp is the densest crowd in the game, so it is where that trade costs the most.

Both are hypotheses for the user's profile to confirm or kill. **Neither is being "fixed" blind.**

## Also corrected from V0.20.54

That report claimed **terrain was ~63%** of frame time. That was inferred by subtraction and is **wrong**.
Directly measured, the terrain functions total ~4.75ms of ~62ms — terrain chunks are already cached
(`drawTerrainChunks`, 1 call). The residual I attributed to terrain was mostly `renderExternalSystems`.
