# V0.20.18 — The Settings Remember You Now

## The bug

`game.js` assigned, on every boot:

```js
this.uiPrefs = { showMinimap: true, showMercPanel: true, showPartyHud: true, largeChat: true };
```

…and **nothing ever saved or loaded it.**

**Verified empirically, not by reading.** After toggling a pref:

| probe | result |
|---|---|
| localStorage keys mentioning it | **none** |
| `serializeCharacterState()` carries it | **no** |
| `typeof g.saveUiPrefs` | **`undefined`** |

**Turn off Show Minimap, reload, it's back on.** Three settings the game offers and then forgets.

## The detail that makes it worth telling

That line sits **directly above**:

```js
this.uiPrefs   = { showMinimap: true, … };   // ← forgotten every boot
this.keyBindings = this.loadKeyBindings();   // ← has always persisted correctly
```

Somebody solved this exact problem, correctly, **one line away** — and the settings never got the same
treatment.

## The fix — copy the neighbour, don't invent a scheme

`defaultUiPrefs()` / `loadUiPrefs()` / `saveUiPrefs()`, mirroring `loadKeyBindings`/`saveKeyBindings`
exactly:

| property | why |
|---|---|
| same storage-key shape (`dream-realms.ui-prefs.v1`) | consistent with every other persisted thing here |
| **defaults-then-overlay merge** | a pref added in a *later* version appears for an existing player instead of going missing |
| swallow-and-carry-on errors | corrupt storage must never break boot |
| **per-browser**, not per-character | whether the minimap is visible is a property of the *person playing*, not of their Fighter — same call the keybindings made |

**Saved at the one place a pref mutates** — the `settingKey` branch of the settings-panel click
handler, confirmed by grep to be the **only** assignment to `uiPrefs[...]` in the codebase. One save
site, not a scattering.

## Verification — a real reload, not a simulated one

| check | result |
|---|---|
| toggle → storage written | `{"showMinimap":false,…}` |
| **a genuine page reload** | **`showMinimap: false` survived** |
| did the pref reach the *screen*? | minimap element computed **`display: none`** — not just the object |
| old blob (`{showMinimap:false}` only) | **keeps the old choice AND fills in newer defaults** |
| corrupt JSON (`{not json`) | **falls back to defaults**, boot unbroken |
| suite | **PASS** — 15/15, 396/396, 0 issues, 0 faults |

Test state cleaned up — the user's minimap is back on.

**The distinction mattered.** My first test *simulated* the toggle and passed. Simulating it proves the
functions work; only a real reload proves the bug is dead, because the bug *was* the reload.

## Why this matters beyond three toggles

Found while auditing **Roadmap Item 20 (Accessibility)**, which requires *"accessibility settings must
persist per user."* **This was the hard blocker underneath the entire item.**

A reduced-motion or screen-shake toggle built on `uiPrefs` today would be **forgotten on every
reload** — and an accessibility setting the game forgets is **worse than one it never offered**.
Someone turns off screen shake because it makes them ill; the game shrugs and does it again tomorrow.

### Item 20's audit, recorded

| | |
|---|---|
| **Input support** | **largely done** — `setKeyBinding` / `saveKeyBindings` / `resetKeyBindings` / `loadKeyBindings`, plus Steam Deck support |
| **Accessibility features** | **0 of 12** — no UI/font scale, colourblind, reduced motion, particle density, shake toggle, high contrast, subtitles, dialogue speed, hold-vs-toggle |
| **Screen shake** | **real** — set on hits (`combat-system:1471`), crits (`:1667`) and portals (`portal-system:278`), decayed in `effects-system:276` |
| **…and it has exactly one consumer** | **`game.js:3512`** — `const shake = clamp(Number(this.camera?.shake \|\| 0), 0, 8)`. A toggle is a **one-line gate** covering all four sources |

**The shake toggle is now unblocked**: a `reduceScreenShake` pref in `uiPrefs`, a row in the settings
panel, and that one line. It was the right thing *not* to build before there was somewhere to remember
the answer.

`game.js` (`defaultUiPrefs`/`loadUiPrefs`/`saveUiPrefs` + boot load), `systems/ui-system.js` (save on

## Recommended next

1. **The screen-shake toggle** — genuinely small now: a pref, a row, and the gate at `game.js:3512`.
   The strongest medical case of the twelve (vestibular), and the cheapest.
2. **Reduced motion** — the game has rain, camera shake, and particle effects; the pref would gate
   several systems, so it needs its own audit of what "motion" means here.
3. The remaining nine accessibility features, each needing UI work rather than plumbing.
