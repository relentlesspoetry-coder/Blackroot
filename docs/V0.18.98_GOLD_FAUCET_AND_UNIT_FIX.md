# V0.18.98 — Gold Faucet Fix + Economy-Audit Unit Fix (Roadmap Item 11)

Two bugs fixed, and one significant balance finding surfaced for a design decision.

## The currency model

`COPPER_PER_GOLD = 10000` — 100 copper = 1 silver, 10,000 copper = 1 gold, 1,000,000 = 1 platinum.

**The money paths do not share a unit:**

| path | code | unit |
|---|---|---|
| **Buy** | `event-system.js` `itemPrice = sellValue × 2` → `spendGold()` | **GOLD** (= `sellValue × 20,000` copper) |
| **Vendor sell** | `inventory-system.js` `vendorSellInventoryIndex` → `addCopper(itemSellValue(item))` | **COPPER** |

## Bug 1 (fixed) — gold faucet in the bags-full overflow

`Game.prototype.addItem`'s bags-full branch credited `addGold(item.value)`: the item's **full** value,
in **gold**, bypassing the sell formula entirely.

For `item_head_iron_helm` that paid **760,000 copper**, versus **42 copper** for a deliberate vendor
sale — about **18,000×** (26,000–36,000× across sampled items). Overflowing your bags was therefore
the single most profitable action in the game, and selling gear to a vendor was strictly worse.

Now:

```js
const value = Math.max(1, Math.floor(Number(this.itemSellValue?.(item) ?? item.value ?? 1)));
if (this.addCopper) this.addCopper(value); else this.addGold?.(value);
this.log(`Bags full. ${item.name} sold for ${this.formatCopper(value)}.`);
```

— exactly mirroring `vendorSellInventoryIndex` (same formula, same unit).
**Verified: overflow pays 42 copper == vendor sale 42 copper.**

## Bug 2 (fixed) — the V0.18.97 audit was itself wrong

Last version's `auditEconomy` compared the buy price (**gold**) against the sell-back (**copper**) as
raw numbers, and reported a reassuring *"tightest spread 2.0×"*. That comparison was meaningless — it
was apples to oranges, and it made a badly-scaled economy look healthy.

`auditEconomy` now **normalises both sides to copper** before comparing, and adds a `SANE_SPREAD`
(20×) check with `tightestBuySellRatio` / `widestBuySellRatio` / `outOfScaleCount`.

## The finding (advisory — needs a design decision, deliberately not auto-fixed)

- **395 of 396 items** exceed a sane buy:sell spread — ratios **20,000×** (`item_torn_boot`) to
  **100,000×** (`item_cave_blindfish`).
- Selling a blue helm returns **104 copper**; buying it costs **3,840,000 copper**.
- At ~**232 copper** per level-8 kill, that's ~**16,552 kills** to afford one blue helm.

**Root cause:** item `sellValue`s and mob currency drops are on a **copper** scale, while shop prices
are charged on a **gold** scale — an ~**10,000×** mismatch.

Rescaling the economy is a balance decision with real gameplay consequences, so it is **reported, not
silently changed**. Options:

1. **Charge buy prices in copper** (`addCopper`/copper compare instead of `spendGold`) — prices drop
   10,000×, aligning with copper income. Vendor basics then cost a fraction of a kill, which may be
   *too* cheap.
2. **Rescale item `sellValue`s up** (~100–1000×) so gold-denominated prices make sense against income.
3. **Raise currency drops** to a gold-ish scale — but that undercuts the copper/silver/gold flavour.
4. **A hybrid**: keep copper income, charge buy prices in copper, and re-tune `sellValue`s so vendor
   gear costs a meaningful number of kills.

## Still true and unchanged

- **0 arbitrage** — buying always costs more than selling back, so the economy cannot be farmed.
- **Value ladder perfectly monotonic** (median sell values): junk 1 / common 5 / uncommon 52 /
  rare 150 / epic 260 / legendary 600.
