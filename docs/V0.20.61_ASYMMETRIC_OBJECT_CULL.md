# V0.20.61 — Asymmetric Object Cull, and the Camp Properly Diagnosed

## The fix

Objects were culled with `objectCullPadPx: 920` applied **equally on all four sides**.

Props draw **upward** from their ground anchor. So:

- **anchored above the viewport top** → drawn even further up → **entirely invisible**
- **anchored below the viewport bottom** → a tall tree can still show its crown → **needs the full pad**

Sharing one pad kept a measured **68 of 151 props** alive at screen-y −350 to −411, all invisible.

New `objectCullPadTopPx: 160`, used only for the `sy < 0` test, applied while the **render queue is
built** — so the work is skipped before anything is drawn, not merely at draw time.

## Measured

A/B by mutating the **memoized** `perfSettings` object directly (mutating `DR.CONFIG.PERFORMANCE` does
nothing — V0.17.98 memo), first arm repeated last as a drift check:

| arm | props drawn | total canvas ops |
|---|---|---|
| old symmetric pad (920) | 151 | 41,047 |
| **new top pad (160)** | **120** | **39,261** |
| old again (drift) | 151 | 42,155 |

**~4%.** Small, because the culled props are cheap grass tufts. Nothing visible changed — they were
already off screen. **This is not the camp problem** and is not presented as one.

## The camp, properly diagnosed

The user's report — **8 FPS near camp, 20 FPS away** — is the single most useful measurement of this
investigation, because it isolates the cost to camp content.

Standing in camp, the frame draws **45 characters: 27 NPCs + 18 entities.** `npcRuntime` is the largest
single phase at ~11.7ms.

### Is the NPC cache hurting here, as it did in V0.20.59?

**No.** Interleaved, median of 8 frames:

| NPC path | ms |
|---|---|
| **cached** | **11.7** |
| drawn live | 24.3 |

So NPCs stay cached. The distinction from V0.20.59 is that this is the *existing* cache population — that
version's regression came from *growing* it (485 → 744 entries, 135 → 207MB) and paying more texture
upload, not from using it at all.

### What is left

Simply **the number of procedural characters on screen at once.** 45 characters, each 129–1,212 canvas
ops. No cull, pad, or memo reaches that.

## What would actually fix the camp

1. **A level-of-detail model for background characters.** Fewer drawing instructions per character, and
   crucially **no new cached bitmaps** — which is what makes it different in kind from the three caching
   attempts (V0.20.49, V0.20.53, V0.20.59) that all had to be reverted.
2. **Fewer camp NPCs.** A content decision, not an engineering one.

Away from camp the user already sees 20 FPS — that is the V0.20.56 viewport cull working.
