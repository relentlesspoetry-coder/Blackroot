# V0.20.82 — Through the Treeline

Reported: *"we need a path that exits out of dark woods and into ashenfall, the new zone should be
created outside dark woods dense trees border."*

## Root cause

`ensureDarkWoodsTreeWallBoundary` rings the entire 360×360 map in an impassable wall:

```js
if (edgeDist <= wallDepth) {          // wallDepth = 4
  tile.blocked = true;
  tile.darkWoodsBoundaryWall = true;
  this.objects[y][x] = { type: 'tree', blocked: true, boundaryTreeWall: true, ... };
}
```

The V0.20.81 crossing sat at **x = 344**, and `edgeDist` there is **15** — fifteen tiles *inside* the
wall. So the road ended in the middle of the forest with the treeline still solid beyond it. Functionally
it worked; visually and spatially it was a teleport pad in the woods, not an exit. The request is
correct, and the previous version only solved the reachability half of the problem.

## The gate

The road is now carved from inside the forest **through** the wall to the map edge:

| | |
|---|---|
| span | x **326 → 359** (the map edge) at y **249–251** |
| height | **3 tiles** — a track, not a boulevard |
| clears | blocked tiles **and** the `boundaryTreeWall` **objects** |
| crossing | x = 352, 7 tiles from the edge |

**Clearing the objects matters as much as the tiles.** The boundary trees are `blocked: true` objects in
their own right — unblocking the tile alone would have left a gap the player still could not walk
through, which is the kind of half-fix that tests green on tile data and fails in play.

The wall **above and below** the gate is verified still intact, so this is a track cut through the woods
rather than the map being opened up. Two standing stones frame the mouth so it reads as a deliberate
opening.

Ordering is safe by construction: `ensureAshenValleyRoadLink` runs late in `generateMap` (after the
decoration pass that builds the wall) and again on world load, so it cuts into a wall that already
exists rather than racing it.

## Walking through now travels

A zone border you have to press a key at reads as a signpost, not a way out — and the request was for a
*path that exits*. `zone-link-system` gained an `update()` that triggers on proximity.

**Two guards against the corridor loop** (the spec's validation list names both "no immediate retrigger"
and "no portal loop"):

1. `zoneTransitionCooldown` — already ticked down by `game.js:2889`, set on arrival by
   `travelToOverworldZone`, and used by the cave system for exactly this purpose.
2. **Arrival points moved ~10 tiles short of the opposite crossing.** Previously
   `dark_woods_east_pass.to` was `(34, 300)` — *the exact tile* of `ashen_valley_west_pass`. Under
   key-press travel that was harmless. The moment walking triggered travel it would have bounced the
   player between the two zones forever. This is the failure that would have shipped if walk-through had
   been added without re-reading the coordinates.

The E interaction is kept, since a player may approach and want to choose.

## One bug found in testing

The gate markers **silently placed nothing.** The guard was:

```js
if (objects[my] && (!marker || marker.boundaryTreeWall)) { ... }
```

— replace only empty ground or a boundary tree. A `flower` and a `mushroom` happened to occupy those two
tiles, so both were skipped, with no error. Decor now yields to the markers; anything authored and
meaningful does not.

## Note on test budget

Two verification runs timed out at 30s before completing — 2,800 `update()` + `render()` calls is more
than the harness will sit through. Splitting the walk into shorter legs, and starting nearer the gate,
gave the same coverage inside budget. Worth remembering: a timed-out probe proves nothing, and it is
tempting to read one as a failure of the code rather than of the harness.
