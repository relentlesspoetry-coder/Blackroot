# Phase 2 — Dark Woods 360x360 Expansion: Expand Dark Woods to 360x360

**Phase:** 2 of 28 (Dark Woods Zone Expansion + Resource Identity master plan)
**Previous version:** V0.17.40
**New version:** V0.17.41

---

## 1. Goal

Resize the Dark Woods overworld to 360x360 tiles while every cave/dungeon zone stays 200x200, preserving
the starter camp, spawn behavior, roads, existing landmarks, and cave entrances exactly as authored. No
named regions, landmarks, resources, or enemies were added in this phase (that is Phase 3 onward) — the
new outer band of the map is filled purely by the same procedural forest generation already used
everywhere else in Dark Woods.

## 3. Systems Touched

Map generation (`world-system.js`), movement/collision (`collision-system.js`), the main render loop and
click-to-move targeting (`game.js`), the minimap renderer, ground-item dropping, save/load, and the world
save schema/migration layer. Cave generation (`cave-system.js`) and dungeon generation (`dungeon-system.js`)
were deliberately **not** touched — both still read `CONFIG.MAP_SIZE` directly and remain 200x200, which is
exactly the intended decoupling for this phase.

## 4. Root Cause / Implementation Notes

Phase 1 identified that `DR.CONFIG.MAP_SIZE` was a single frozen global used identically by the overworld
and every cave, with several call sites hardcoding it directly instead of going through the already-proven
`Game.prototype.activeMapSize()` helper (which derives size from the live map array, already used correctly
throughout `collision-system.js`, `pathfinding-worker-system.js`, `bot-player-system.js`, and others).

The implementation approach was to **decouple, not replace**: a new `CONFIG.OVERWORLD_MAP_SIZE = 360`
constant governs only `generateMap()`'s local `size` variable (the overworld/Dark Woods generator, which is
the sole generator that builds `dark_woods` — confirmed via `data/default-zones.js`, which lists only one
`overworld`-type zone). `CONFIG.MAP_SIZE` itself was left at 200 and untouched, so every other consumer of
that constant — `cave-system.js`'s `buildCaveFloorState`/`buildCaveMap`, `dungeon-system.js`'s room/corridor
generation, and the Silk Web Cavern dungeon route — continues to generate 200x200 exactly as before, with
zero changes to those files. This is why the phase could ship without touching cave/dungeon code at all.

For every other hardcoded `CONFIG.MAP_SIZE` reference that operates against the **currently active** map
(player movement, rendering, minimap, item drop, save/load position restore), the fix was to route through
`this.activeMapSize()` (or, in `isRenderableWorldGrid`'s and the minimap's case, the grid's own `.length`
directly), which already correctly resolves to 360 when the active map is the overworld and 200 when it's a
cave — no new flag or "am I in a cave" branch was needed anywhere, since the fix is inherently generic.

**Preserving existing content:** rather than re-centering the camp/spawn into the middle of the new 360x360
canvas, `CONFIG.START_X`/`START_Y` were left at `{100,100}` and every hand-authored absolute-coordinate
feature (camp footprint, four camp approach paths, main road network, water bodies, ruin/stone landmarks,
atmosphere lanterns/glow-mushrooms/dead-trees, and all cave entrance coordinates in `cave-system.js`) was
left completely untouched. Because none of that content depends on the map's total size (only on fixed
absolute coordinates that already existed inside the old 0–199 range), it continues to render and function
identically to before. The practical effect: Dark Woods now has a large new wilderness band from the old
edge (~199) out to the new edge (359) on the south/east side, generated by the same noise-based biome fill,
sparse tree/rock/grass/flower/mushroom dressing, and boundary tree wall that already governed the rest of
the zone — just running across the larger bounds. The boundary tree wall (`ensureDarkWoodsTreeWallBoundary`)
required no changes at all: it already derived its size from `this.map.length` rather than the global, so it
automatically wrapped the new true edge at 359 once the map itself grew.

**Save schema:** `systems/world-serializer.js` previously stamped a single shared `size` onto both
`dark_woods` and `mossfang_cave`'s `width`/`height` fields and validated both grids against that one value.
Since the two zones can now legitimately differ in size, `migratePayload`/`serialize`/`validateNormalizedPayload`
were changed to compute `overworldSize` and `caveSize` independently (preferring each zone's own previously-
serialized `width`/`tiles.length`, falling back to the legacy shared `mapSize` field for old saves, and
finally to each zone's current-build default). A new top-level `zoneMapSizes: { dark_woods, mossfang_cave }`
field was added for forward clarity; the legacy `mapSize` field is retained (now populated with the
overworld's size) for backward compatibility, since nothing else in the codebase reads it directly (verified
by project-wide grep before making the change). The schema string and save-format version were bumped to
signal the change, with the previous `-v2` schema added to the supported/legacy set so existing saves
migrate forward automatically through the same `migrations[]`-logged path already used for v1 → v2.

## 5. Deliberately Deferred (documented, not fixed this phase)

These additional `CONFIG.MAP_SIZE` references were found during Phase 1's audit but are **not** exercised by
anything currently reachable in the new map area (no resources, atmosphere effects, or events are placed
there yet), so fixing them now would be speculative rather than surgical. Flagging them here so whichever
phase first places content in the new outer band checks these first:

- `systems/atmosphere-system.js:159,161,203,205` — fog/particle effect range clamps.
- `systems/crafting-system.js:365,367,781,782` — gathering/fishing node scan range clamps.
- `systems/event-system.js:833,834` — world-event teleport clamp.
- `systems/portal-system.js:95,146,234` — Mage recall-portal placement/return clamp.
- `systems/party-system.js:502` — party member spread/reposition clamp.

None of these can currently be triggered in a way that matters (no resource nodes, events, or portals exist
in the new area yet), but they use the same hardcoded-200 pattern and should be swept when their owning
content phase lands.

`systems/cave-system.js` and `systems/dungeon-system.js` still reference `CONFIG.MAP_SIZE` extensively —
this is intentional and correct, not deferred work; those zones are meant to stay 200x200 in this phase.

## 6. Validation

Commands run:

```bash
find . -name "*.js" -print0 | xargs -0 -n1 node --check
find . -name "*.json" -print0 | xargs -0 -n1 python3 -m json.tool > /dev/null
```

**Results:** both passed with zero errors across every `.js` and `.json` file in the project.

**Limitation:** this environment has no browser, headless or otherwise (`jsdom` and `puppeteer` are not
installed; no `chromium`/`google-chrome` binary is present), so `generateMap()` could not be executed
end-to-end to directly observe a live 360x360 array, and no in-browser boot/console-error check could be
performed. In place of runtime execution, every changed function was traced by hand against its actual
call sequence (e.g., confirmed `buildCaveMap()` — called from inside `generateMap()` — only populates
`this.caveMap`/`this.caveObjects` and never reassigns `this.map`, so the overworld array is what's still
active when `buildStaticMinimap()` runs at the end of `generateMap()`; confirmed `activeMapSize()`'s cache
correctly invalidates whenever `this.map`'s array reference changes between overworld/cave). This is not a
substitute for an actual boot test.

## 7. Manual Tests Needed

- Load `Blackroot V0.17.41.html` in a browser on a fresh profile (no existing save) and confirm Dark Woods
  generates and the game boots with no console errors.
- Walk from spawn toward the new far edges (south/east of the old ~199 boundary) and confirm: terrain
  renders continuously with no visible cutoff, the boundary tree wall appears only at the true new edge, and
  the player cannot walk through it.
- Open the minimap and confirm it shows the full zone (not cropped to the old top-left quadrant).
- Drop an item on the ground while standing in the new outer area and confirm it succeeds.
- Enter Mossfang Cave (or Silk Web Cavern) and confirm the cave still generates and plays at its original
  200x200 scale, with no change in behavior.
- Save the game, reload the page, and load the save — confirm Dark Woods and the cave both restore at their
  correct independent sizes with no console errors or migration warnings surfaced to the player.
- Load an **old** save created before this phase (schema v2, if one is available) and confirm it still loads
  correctly via the migration path.

## 8. Known Limitations

- **Process deviation:** per the project's versioning rules, a ZIP backup of the previous working version
  should have been created immediately before starting this phase's edits. That backup of V0.17.40 was not
  taken before edits began (only the V0.17.39 backup from Phase 1 exists.) This repo has no git history to
  recover the exact pre-Phase-2 state from. A backup of the current V0.17.41 state should be taken now so
  this does not compound, and this step will be done first for every phase going forward.
- The new outer band of Dark Woods (roughly tiles 200–359) is intentionally undecorated beyond the base
  procedural forest fill — no named regions, landmarks, resources, or enemies exist there yet. It is
  functional and walkable but will look and feel sparse until Phases 3–4 (and later content phases) run.
- The mob level-scaling distance constant and the proposed region layout from the Phase 1 report are
  unchanged in this phase (Phase 2 explicitly excludes adding enemies/landmarks) — they remain Phase 3 work.
- The five deferred `CONFIG.MAP_SIZE` sites in Section 5 are unresolved until their owning content phase.
- No in-browser boot test was performed (see Section 6) — the manual tests in Section 7 are required before
  this phase can be considered fully verified.
